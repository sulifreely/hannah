问答 30（进阶）：DNS 解析过程与优化 ​

题目 ​

DNS 解析的完整流程是怎样的？递归查询和迭代查询有什么区别？DNS 缓存有哪些层级？有哪些 DNS 优化手段？DNS 劫持是什么？如何防范？

参考答案 ​

DNS 是什么 ​

DNS（Domain Name System，域名系统）是互联网的「电话簿」，把人类可读的域名（如 www.example.com）翻译成机器可读的 IP 地址（如 93.184.216.34）。

没有 DNS，我们上网就得记住一堆数字 IP 地址。

DNS 解析的完整链路 ​

当你在浏览器输入 www.example.com 时，DNS 解析会经过这样一条链路：

浏览器缓存 → 系统缓存（OS）→ 路由器缓存 → ISP DNS（本地域名服务器）→ 根域名服务器 → 顶级域服务器（.com）→ 权威域名服务器（example.com）
1

逐层往上查，每一层都可能有缓存，命中就直接返回，不用再往上走。

详细过程：

第 1 步：浏览器 DNS 缓存 ​

浏览器首先检查自己的 DNS 缓存里有没有这个域名的记录。

  • Chrome 可以在 chrome://net-internals/#dns 查看
  • 缓存时间比较短，一般几分钟（Chrome 默认 1 分钟左右）
  • 命中就直接返回，不发网络请求

第 2 步：操作系统缓存（hosts 文件 + 系统缓存） ​

浏览器缓存没命中，就调用系统的 DNS 解析函数。

  • 系统先查自己的 DNS 缓存
  • 再查本地的 hosts 文件（/etc/hosts 或 C:\Windows\System32\drivers\etc\hosts）
  • hosts 文件里的域名是手动配置的，优先级很高
  • 命中就直接返回

第 3 步：路由器缓存 ​

系统缓存也没命中，就向路由器发 DNS 请求。

  • 路由器一般也有自己的 DNS 缓存
  • 家里的路由器通常把 DNS 请求转发给 ISP 的 DNS 服务器

第 4 步：ISP DNS（本地域名服务器 / LDNS） ​

ISP（互联网服务提供商，比如联通、电信、移动）的 DNS 服务器。

  • 这是大多数用户的「默认 DNS 服务器」
  • ISP DNS 有大量缓存，大部分常用域名在这里就能命中
  • 如果没命中，ISP DNS 就开始递归查询过程

第 5 步：根域名服务器（Root DNS） ​

ISP DNS 没命中缓存，就向「根域名服务器」查询。

  • 全球共有 13 组根域名服务器（A 到 M 根）
  • 根服务器不存具体域名的 IP，它只知道「顶级域服务器」（如 .com、.org）的地址
  • 根服务器告诉 ISP DNS：「你要查 .com 的域名，去问 .com 的顶级域服务器，地址是 xxx」

注意：13 组根服务器不是 13 台机器，而是 13 个逻辑编号，背后有几千台物理服务器分布在全球（任播技术）。

第 6 步：顶级域服务器（TLD DNS） ​

顶级域（Top-Level Domain）服务器，比如 .com、.org、.cn、.net。

  • ISP DNS 向 .com 顶级域服务器查询 example.com
  • 顶级域服务器也不知道具体 IP，它知道 example.com 的「权威域名服务器」地址
  • 顶级域告诉 ISP DNS：「example.com 的权威服务器是 ns1.example.com，地址是 xxx」

第 7 步：权威域名服务器（Authoritative DNS） ​

权威域名服务器是真正存着这个域名所有 DNS 记录的地方。

  • ISP DNS 向权威服务器查询 www.example.com
  • 权威服务器返回对应的 IP 地址（A 记录或 AAAA 记录）
  • ISP DNS 拿到结果后，缓存起来（缓存时间由 TTL 决定），然后返回给用户

递归查询 vs 迭代查询 ​

这是 DNS 解析中两个核心概念，很多人搞混。

查询类型谁负责怎么做结果
递归查询客户端 → 本地 DNS「你帮我查到底，把最终结果给我」返回最终的 IP 地址或报错
迭代查询本地 DNS → 根 / TLD / 权威「你告诉我下一步该问谁」返回下一个服务器的地址

打个比方：

  • 递归查询就像你问前台：「帮我找一下张三的电话。」前台帮你一层层找，最后把号码给你。
  • 迭代查询就像你问路：「请问去天安门怎么走？」路人说：「你去问前面那个交警。」你去问交警，交警说：「坐 1 号线到西单转 4 号线。」你自己一步步找过去。

实际过程中：

  • 浏览器 → 系统 → ISP DNS：递归查询（用户端只发一次请求，ISP DNS 负责查到底）
  • ISP DNS → 根 → TLD → 权威：迭代查询（ISP DNS 自己一步步往上问）

为什么这么设计？

  • 对用户来说，递归查询简单，只要配一个 DNS 服务器就行
  • 对 DNS 系统来说，迭代查询减轻了根服务器的压力，根服务器只需要指路，不用帮你查到底

DNS 记录类型 ​

常见的 DNS 记录类型：

记录类型全称作用示例

AAAAIPv6 Address域名 → IPv6 地址example.com → 2606:2800:220:1:248:1893:25c8:1946
CNAMECanonical Name域名 → 另一个域名（别名）www.example.com → example.com
MXMail Exchange邮件服务器example.com → mail.example.com
TXTText文本记录（验证、SPF 等）用于域名所有权验证、反垃圾邮件

DNS 优化手段 ​

DNS 解析是网络请求的第一步，DNS 慢了整个请求都慢。常见的优化手段：

1. DNS 预解析（dns-prefetch） ​

让浏览器在空闲时提前解析页面中可能用到的域名，等真正请求时已经解析好了。

html
<!-- 告诉浏览器提前解析这些域名 -->
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="dns-prefetch" href="//api.example.com">
<link rel="dns-prefetch" href="//fonts.googleapis.com">
1
2
3
4

适用场景：

  • 页面上引用了很多第三方域名的资源（CDN、统计、字体、广告）
  • 用户后续可能跳转到的域名

注意：

  • 不要滥用，预解析太多也会消耗资源
  • 只给确实会用到的第三方域名加
  • Chrome 对用户 hover 的链接也会自动做 dns-prefetch

2. 减少 DNS 查询次数 ​

  • 尽量把资源放在同一个域名下（但也要注意 HTTP/1.1 的并发限制）
  • 用 CDN 合并多个域名
  • 不要为了「域名分片」搞太多域名（HTTP/2 时代不需要了）

3. 合理设置 TTL（缓存时间） ​

DNS 记录的 TTL（Time To Live）决定了 DNS 结果能缓存多久。

  • TTL 太长：IP 变更后生效慢，用户可能访问到旧 IP
  • TTL 太短：DNS 查询频繁，解析慢，权威服务器压力大

经验值：

  • 稳定的服务：TTL 设长一点，如 1 小时（3600s）或 1 天（86400s）
  • 经常变更的服务：TTL 设短一点，如 5 分钟（300s）或 1 分钟（60s）
  • 变更前先把 TTL 改短，变更完再改长（平滑切换）

4. HTTPDNS（HTTP DNS） ​

传统 DNS 走 UDP 53 端口，有几个问题：

  • 运营商可能劫持 DNS，把用户引导到广告页面
  • 运营商 DNS 可能有缓存污染
  • 无法根据用户位置智能调度（Local DNS 可能不在用户本地）

HTTPDNS 的原理：

  • 不用传统的 DNS 协议，而是通过 HTTP/HTTPS 请求 DNS 解析结果
  • 客户端直接向 HTTPDNS 服务器发请求，拿到 IP
  • 绕过了运营商的 Local DNS

优点：

  • 防 DNS 劫持（走 HTTP/HTTPS，不容易被篡改）
  • 更精准的调度（直接拿用户真实 IP 做地理位置判断，CDN 调度更准）
  • 解析延迟可控（可以预解析、智能缓存）

缺点：

  • 需要客户端 SDK 支持（App 端常用，网页端需要 JS 配合）
  • 有额外的成本（HTTPDNS 服务一般是付费的）
  • 网页端使用有局限（需要用 JS 动态改请求地址）

常见厂商：

  • 阿里云 HTTPDNS
  • 腾讯云 HTTPDNS
  • Cloudflare 1.1.1.1（也有 DoH / DoT）

5. 使用公共 DNS ​

如果运营商 DNS 慢或不准，可以改用公共 DNS：

  • Google DNS：8.8.8.8 / 8.8.4.4
  • Cloudflare DNS：1.1.1.1 / 1.0.0.1（号称全球最快）
  • OpenDNS：208.67.222.222
  • 国内公共 DNS：114 DNS（114.114.114.114）、阿里 DNS（223.5.5.5）、DNSPod（119.29.29.29）

但公共 DNS 也不一定更快，取决于你和它的网络距离。

6. CDN 智能调度 ​

CDN（内容分发网络）和 DNS 紧密配合：

  • 用户请求 CDN 域名时，DNS 返回离用户最近的 CDN 节点 IP
  • 基于智能 DNS 解析（根据用户地理位置、运营商、节点负载等）
  • 让用户从最近的节点拿数据，速度更快

常见调度方式：

  • 基于地理位置（GeoDNS）：根据用户 IP 所属地区返回最近节点
  • 基于运营商：返回同运营商的节点（避免跨运营商访问慢）
  • 基于负载：返回当前负载较低的节点
  • 基于探测：实时探测各节点质量，返回最优的

7. DNS 预连接（preconnect） ​

比 dns-prefetch 更进一步，不仅解析 DNS，还顺便建立 TCP 连接和 TLS 握手：

html
<link rel="preconnect" href="https://api.example.com">
1

适合你确定马上就要用的域名。但不要用太多，因为 preconnect 会占用连接资源。

DNS 劫持与防范 ​

什么是 DNS 劫持？

  • DNS 解析本来应该返回正确的 IP，但被中间的攻击者篡改了
  • 用户输入的是正确的域名，却被引导到错误的（恶意的）IP 地址
  • 结果就是：你以为你在访问淘宝，实际上访问的是钓鱼网站

常见的 DNS 劫持方式：

1. 本地劫持（篡改 hosts 文件） ​

  • 恶意软件修改用户电脑的 hosts 文件
  • 把银行网站的域名指向钓鱼网站的 IP
  • 因为 hosts 文件优先级最高，直接就被劫持了

防范： 安装杀毒软件，不要乱装软件，定期检查 hosts 文件。

2. 路由器劫持 ​

  • 路由器的 DNS 配置被篡改（比如路由器管理密码太简单，被黑客登录改掉）
  • 所有连这个路由器的设备都被劫持了

防范： 修改路由器默认管理密码，及时升级固件，不要用弱密码。

3. 运营商 DNS 劫持 ​

  • 运营商的 DNS 服务器返回错误的结果
  • 比如你访问一个不存在的域名，运营商给你跳转到它的搜索广告页
  • 更恶劣的情况：把正常网站跳转到广告页

防范： 改用公共 DNS（8.8.8.8、114.114.114.114 等），用 DoH / DoT。

4. 中间人攻击（MITM） ​

  • 攻击者在网络中间拦截 DNS 请求和响应，篡改返回结果
  • 公共 Wi-Fi 环境下更容易发生

防范： 使用 DNS 加密技术（DoH / DoT）。

DNS 安全技术 ​

1. DoH（DNS over HTTPS） ​

  • DNS 查询通过 HTTPS 加密传输
  • 中间人看不到你查了什么域名，也篡改不了结果
  • 常见的公共 DNS 都支持：https://1.1.1.1/dns-query、https://dns.google/dns-query

2. DoT（DNS over TLS） ​

  • DNS 查询通过 TLS 加密传输
  • 和 DoH 类似，只是用的是 TLS 协议而不是 HTTPS
  • 标准端口 853

3. DNSSEC（DNS Security Extensions） ​

  • 给 DNS 记录加上数字签名
  • 递归 DNS 服务器可以验证返回结果的真实性和完整性
  • 防止 DNS 响应被篡改
  • 但需要域名所有者配置 DNSSEC 记录，普及度还不高

4. DNSSEC vs DoH/DoT ​

技术解决什么问题谁来用
DoH / DoT加密 DNS 传输，防窃听和篡改客户端 → 递归 DNS 之间
DNSSEC验证 DNS 数据的真实性，防篡改整个 DNS 解析链路

两者互补，不是替代关系。

其他 DNS 相关概念 ​

1. 任播（Anycast）

  • 多个地方的服务器用同一个 IP 地址
  • 用户请求时，网络路由会把他送到最近的那个节点
  • 根 DNS 服务器、CDN 节点都用了任播技术

2. 智能 DNS / 智能解析

  • 根据用户的 IP 地址、运营商、地理位置等，返回不同的解析结果
  • 让用户访问最近的节点，提升速度
  • CDN 的核心技术之一

3. 泛解析（Wildcard DNS）

  • *.example.com 匹配所有子域名
  • 不用每个子域名都单独配一条记录
  • 但也要小心，泛解析可能被利用做安全攻击

4. DNS 污染 / DNS 缓存投毒

  • 往 DNS 缓存里注入假的记录
  • 让缓存服务器返回错误的结果
  • DNSSEC 可以防范

追问 1：CDN 是如何通过 DNS 实现智能调度的？CNAME 在其中起到什么作用？ ​

答 ​

CDN（内容分发网络）的核心目标就是「让用户从最近的节点拿数据」，而 DNS 解析是实现这个目标的关键一环。

CDN 的典型工作流程：

  1. 用户访问 cdn.example.com/image.jpg
  2. 浏览器进行 DNS 解析
  3. DNS 返回离用户最近的 CDN 节点 IP
  4. 用户向这个 CDN 节点请求资源
  5. 节点有缓存就直接返回，没有就回源站拉取再缓存

关键点：第 2 步 DNS 解析怎么知道用户在哪、该返回哪个节点？

CNAME 的作用 ​

CDN 一般用 CNAME 记录 来接入：

www.example.com.  CNAME  www.example.com.cdnprovider.com.
1

什么意思呢？

  • 你的域名 www.example.com 是一个别名（CNAME）
  • 它真正指向的是 CDN 厂商的域名 www.example.com.cdnprovider.com
  • DNS 解析时，先查到 CNAME，然后继续解析 CNAME 指向的域名
  • 最终由 CDN 厂商的 DNS 服务器返回具体的节点 IP

为什么要用 CNAME？

  1. 解耦：你不用关心 CDN 有哪些 IP，CDN 厂商自己管理节点 IP
  2. 灵活调度：CDN 厂商可以随时调整节点 IP，不用你改 DNS
  3. 智能解析：由 CDN 的 DNS 服务器来做智能调度，返回最优节点

智能 DNS 调度的实现 ​

CDN 厂商的 DNS 服务器（也叫「GSLB」——全局负载均衡）收到解析请求时，会根据这些信息判断该返回哪个节点：

1. 用户地理位置（GeoDNS）

  • 根据请求来源的 IP（实际上是 Local DNS 的 IP）判断用户所在地区
  • 比如北京的用户返回北京节点，上海的用户返回上海节点
  • 有精确的 IP 地理位置库，能精确到城市甚至区

2. 运营商

  • 判断用户是电信、联通、移动还是其他运营商
  • 返回同运营商的节点，避免跨运营商访问（跨运营商通常很慢）

3. 节点健康状态和负载

  • 实时监控所有 CDN 节点的健康状态
  • 如果某个节点挂了，就不再把用户调度过去
  • 如果某个节点负载太高，就少调度一些用户过去

4. 节点距离 / 延迟

  • 有些高级的 CDN 会实时探测各节点到用户的延迟
  • 返回延迟最低的节点

5. 权重 / 比例

  • 按一定比例把用户分到不同节点（灰度发布、流量迁移时用）

一个例子 ​

用户（北京联通）访问 www.example.com：

1. 用户的浏览器查 www.example.com
   ↓
2. 权威 DNS 返回 CNAME：www.example.com → www.example.cdn.com
   ↓
3. 继续解析 www.example.cdn.com
   ↓
4. CDN 的智能 DNS 收到请求：

调度精度的问题 ​

智能 DNS 有一个天生的缺陷：它拿到的不是用户的真实 IP，而是用户 Local DNS 的 IP。

  • 如果用户用了运营商的 DNS，那 Local DNS 一般和用户在同一个地区、同一个运营商，调度还比较准
  • 但如果用户用了公共 DNS（比如 8.8.8.8），那 Local DNS 可能在另一个城市甚至另一个国家，调度就不准了

比如：

  • 一个北京的用户，用了 Google DNS（8.8.8.8）
  • CDN 的 DNS 看到请求来源是 Google DNS 的 IP（可能在上海或国外）
  • 结果给这个用户调度到了上海节点，反而变慢了

怎么解决？——ECS（EDNS Client Subnet）

ECS 是 DNS 协议的一个扩展，允许递归 DNS（Local DNS）在请求里带上用户的 IP 网段（不是完整 IP，保护隐私）。

DNS 请求里包含：用户 IP 段是 203.0.113.0/24（北京联通）
1

这样权威 DNS（CDN 的 DNS）就能拿到用户大致的地理位置，调度更准确。

  • 支持 ECS 的 DNS：Google DNS、Cloudflare DNS、国内大部分公共 DNS
  • 不支持的话，调度精度就差一些

总结 ​

  • CNAME：把你的域名指向 CDN 厂商的域名，让 CDN 接管解析
  • 智能 DNS：CDN 厂商的 DNS 服务器根据用户位置、运营商、节点状态等，返回最优节点 IP
  • ECS：解决公共 DNS 导致的调度不准问题，带上用户 IP 网段

追问 2：浏览器的 DNS 缓存策略是怎样的？Chrome 对 DNS 有哪些优化？ ​

答 ​

浏览器是 DNS 解析的「第一道关卡」，它的缓存策略和优化直接影响用户体验。

Chrome 的 DNS 缓存 ​

缓存位置：

  • 浏览器进程维护一个 DNS 缓存表
  • Chrome 可以在 chrome://net-internals/#dns 查看和清空缓存

缓存时间：

  • 不是固定的，受多个因素影响
  • 一般遵循 DNS 记录的 TTL，但也有自己的上限和下限
  • Chrome 有个默认的缓存上限时间（大约几分钟），即使 TTL 很长也不会存太久
  • 也有最小缓存时间，防止 TTL 太短导致频繁解析

缓存失效：

  • TTL 到期自动失效
  • 浏览器关闭后缓存也会清掉（部分可能保留，但不持久化）
  • 用户主动清除浏览数据

Chrome 的 DNS 优化手段 ​

1. DNS 预解析（dns-prefetch） ​

前面已经讲过，页面可以通过 <link rel="dns-prefetch"> 提示浏览器提前解析。

但 Chrome 自己也会做一些自动预解析：

  • 鼠标悬停预解析：当你把鼠标悬停在链接上时，Chrome 会提前解析那个链接的域名，等你点击时可能已经解析好了
  • 页面加载时预解析：Chrome 会扫描页面中的链接，提前解析可能会点击的域名
  • 地址栏预解析：你在地址栏输入时，Chrome 可能提前解析可能的目标域名

这些都是浏览器自动做的，不需要网站配置。

2. DNS Prefetch 控制 ​

网站可以通过 HTTP 头或 meta 标签控制是否自动预解析：

html
<!-- 关闭自动 DNS 预解析（默认是开启的） -->
<meta http-equiv="x-dns-prefetch-control" content="off">

<!-- 手动指定要预解析的域名 -->
<link rel="dns-prefetch" href="//cdn.example.com">

为什么要关闭？

  • 隐私考虑：你在 A 网站，浏览器却预解析了 B 网站的域名，B 网站能看到请求
  • 但一般来说，开启利大于弊

3. 预连接（preconnect） ​

比 dns-prefetch 更进一步：

  • 不仅解析 DNS
  • 还建立 TCP 连接
  • 还做 TLS 握手（如果是 HTTPS）
html
<link rel="preconnect" href="https://api.example.com">
1

省了 DNS + TCP + TLS 一共 3 个 RTT（甚至更多）。

但注意：preconnect 成本比较高，不要用太多，一般最多 3-4 个。

4. 预加载（preload） ​

比 preconnect 更进一步：

  • 不仅建立连接
  • 还直接把资源下载下来
html
<link rel="preload" href="https://cdn.example.com/key.js" as="script">
1

适合首屏关键资源，但要谨慎使用，避免浪费带宽。

5. Chrome 的「预测服务」 ​

Chrome 有个叫「预测服务」（Prediction Service）的功能：

  • 根据你的浏览历史，预测你接下来可能访问什么网站
  • 在你输入地址时，甚至在你打开浏览器时，就提前解析 DNS、建立连接、甚至预渲染页面
  • 可以在 Chrome 设置里开关：「使用预测服务加快搜索和浏览速度」

6. 提前渲染（NoState Prefetch / Prerender） ​

更极端的优化：

  • 不仅解析 DNS、建立连接
  • 还提前把整个页面渲染好（在后台）
  • 等用户真正访问时，几乎瞬间就能显示

但这个消耗很大（CPU、内存、带宽），所以 Chrome 用得很谨慎，只在它非常确定你会访问时才用。

浏览器 DNS 缓存的一些有趣细节 ​

1. 为什么有时刷新页面 DNS 缓存还在？

  • 浏览器 DNS 缓存是进程级的，刷新页面不会清缓存
  • 要清的话得用 chrome://net-internals/#dns 里的「Clear host cache」
  • 或者重启浏览器

2. 为什么有时 DNS 改了，浏览器还访问旧 IP？

  • 浏览器缓存没过期
  • 操作系统缓存没过期
  • ISP DNS 缓存没过期
  • 各级缓存都要等 TTL 过期才会更新
  • 所以改 DNS 解析时，建议先把 TTL 改短，等旧缓存过期了再改记录

3. 一个页面有多个不同域名的资源，DNS 解析是并行的吗？

  • 是的，浏览器会并行解析多个域名
  • 但解析请求本身很快（缓存命中的话 0ms，不命中的话几十 ms）
  • 真正耗时的是 TCP 连接和资源下载

4. HTTP/2 对 DNS 的影响？

  • HTTP/2 一个域名只需要一条连接（多路复用）
  • 所以不需要像 HTTP/1.1 时代那样搞「域名分片」（sharding）来增加并发
  • 域名少了，DNS 解析次数也少了，首屏更快

追问 3：DNS 查询用的是 UDP 还是 TCP？什么时候会用 TCP？ ​

答 ​

这是一个经典面试题，答案是：主要用 UDP，但某些情况下也会用 TCP。

为什么 DNS 主要用 UDP？ ​

  1. 快：UDP 不需要三次握手，发一个请求包、回一个响应包就完事了
  2. 简单：DNS 查询-响应模式很简单，一个请求对应一个响应
  3. 开销小：UDP 头部只有 8 字节，TCP 头部 20 字节，连接建立还要额外开销
  4. 幂等：DNS 查询是只读的，重复查询也没关系，丢了重发就行

绝大多数 DNS 查询（比如查 A 记录、CNAME 记录）响应都很小，一个 UDP 包就能装下，所以用 UDP 就够了。

什么时候 DNS 会用 TCP？ ​

1. 响应大小超过 512 字节 ​

DNS 协议规定，UDP 模式下 DNS 响应大小不能超过 512 字节（历史限制）。如果响应超过这个大小，服务器会：

  • 只返回前 512 字节
  • 并且设置响应中的 TC（Truncated，截断）位为 1
  • 客户端收到后，知道响应被截断了，就会重新用 TCP 发起完整查询

什么时候响应会超过 512 字节？

  • 域名有很多条 DNS 记录（比如很多 MX 记录、很多 NS 记录）
  • DNSSEC 签名（签名记录会大大增加响应大小）
  • 大型机构的复杂 DNS 配置

注意： EDNS0（扩展 DNS）出现后，UDP 的最大大小可以超过 512 字节（最多可以到 4096 字节甚至更大），但为了兼容和避免 MTU 问题，很多情况下超过一定大小还是会切到 TCP。

2. 区域传输（Zone Transfer） ​

主域名服务器和从域名服务器之间同步数据时用区域传输（AXFR / IXFR）。

  • 区域传输要传整个区域的所有 DNS 记录，数据量大
  • 必须保证可靠传输，不能丢包
  • 所以用 TCP

3. DNS over TLS (DoT) / DNS over HTTPS (DoH) ​

加密 DNS：

  • DoT 直接跑在 TLS 上，TLS 基于 TCP
  • DoH 跑在 HTTPS 上，HTTPS 基于 TCP（HTTP/3 除外，是 QUIC+UDP）

4. 其他特殊情况 ​

  • 有些防火墙或网络环境会拦截 UDP 53 端口，DNS 只能走 TCP
  • 一些安全产品或特殊场景下的 DNS 查询

为什么不是全部用 TCP？ ​

既然 TCP 更可靠，为什么 DNS 不直接全用 TCP？

  1. 性能问题：TCP 三次握手要 1 个 RTT，对于 DNS 这种小包查询来说，开销太大了

    • UDP：请求 → 响应，1 个 RTT
    • TCP：SYN → SYN+ACK → ACK + 请求 → 响应，要 2-3 个 RTT
    • DNS 解析本来就几十 ms，握手就要占一半甚至更多时间

  2. 服务器负载：TCP 连接要维护状态，服务器开销大

    • UDP 是无连接的，服务器处理完就忘
    • 如果所有 DNS 查询都用 TCP，根服务器和 TLD 服务器的连接数会爆炸

  3. 简单性：DNS 查询-响应模式非常适合 UDP，不需要 TCP 的复杂机制（流量控制、拥塞控制、顺序保证等）

DNS 端口 ​

  • UDP 53：标准 DNS 查询端口（最常用）
  • TCP 53：区域传输和大包查询
  • TCP 853：DNS over TLS (DoT)
  • TCP 443：DNS over HTTPS (DoH)
  • UDP 443：DNS over QUIC（DoQ，比较新）

一个常见误区 ​

很多人以为「DNS 就是 UDP」，但实际上现代 DNS 已经很复杂了：

  • 普通查询用 UDP
  • 大包用 TCP（或 EDNS0 + UDP）
  • 加密 DNS 用 TLS / HTTPS（TCP 或 QUIC）
  • 区域传输用 TCP

面试时回答「主要用 UDP，超过 512 字节或区域传输时用 TCP」，再补充一下 DoH/DoT，就比较完整了。
