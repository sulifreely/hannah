问答 29（高级）：前端安全之点击劫持与其他安全问题 ​

题目 ​

什么是点击劫持（Clickjacking）？它的攻击原理是什么？如何防范（X-Frame-Options、CSP frame-ancestors、frame-busting）？它和 XSS、CSRF 有什么区别？还有哪些常见的前端安全问题（SQL 注入、文件上传漏洞、路径遍历、CORS 配置错误）？

参考答案 ​

点击劫持（Clickjacking）是什么 ​

点击劫持是一种视觉欺骗攻击：攻击者在自己的恶意页面里，用一个透明的 iframe 嵌入目标网站，然后在上面覆盖一些诱导用户点击的元素。用户以为自己在点「播放按钮」「领取奖品」，实际上点的是 iframe 里目标网站的「转账」「关注」「删除」等按钮。

攻击原理：

┌─────────────────────────────┐
│  攻击者的页面 evil.com       │
│                             │
│   🎮 点击开始游戏 🎮        │  ← 用户看到的（诱导按钮）
│                             │
│   ┌───────────────────┐     │
│   │  透明 iframe       │     │
│   │  target.com/delete │     │  ← 实际点击的（不可见）
│   │                    │     │
│   └───────────────────┘     │
│                             │
└─────────────────────────────┘

  • iframe 设置为透明（opacity: 0 或 visibility: hidden）
  • 用 CSS 精确定位，让目标按钮正好在诱导按钮的位置下面
  • 用户点击「开始游戏」，实际上点的是 iframe 里的「确认删除」

攻击前提：

  1. 目标网站可以被嵌入在 iframe 中（没有做防嵌入保护）
  2. 用户已经在目标网站登录（Cookie 自动带上）
  3. 有值得攻击的操作按钮（转账、关注、删除、授权等）

典型案例：

  • 诱导用户点击「领奖」，实际在微博/推特上点了「关注」
  • 诱导用户点击「播放」，实际在银行页面点了「转账」
  • 诱导用户点击「确认」，实际开启了摄像头/麦克风权限
  • UI Redressing（界面伪装）：更高级的变种，不是全透明，而是部分可见，让用户以为是正常界面

防范手段 ​

1. X-Frame-Options 响应头 ​

最经典的防范手段，服务端在 HTTP 响应头里加这个字段，告诉浏览器是否允许页面被嵌入 iframe。

三个可选值：

值含义
DENY完全不允许被嵌入任何 iframe，包括同域名
SAMEORIGIN只允许同域名的页面嵌入
ALLOW-FROM uri允许指定来源嵌入（但兼容性不好，很多浏览器不支持）
http
X-Frame-Options: DENY
X-Frame-Options: SAMEORIGIN
X-Frame-Options: ALLOW-FROM https://trusted.com
1
2
3

优点： 简单、有效、兼容性好（IE8+ 都支持） 缺点：

  • ALLOW-FROM 支持不好（Chrome 不支持，Firefox 支持）
  • 只能设置一个域名，不支持多个白名单
  • 是旧标准，逐渐被 CSP 的 frame-ancestors 取代

2. CSP frame-ancestors 指令 ​

Content-Security-Policy 里的 frame-ancestors 是 X-Frame-Options 的现代化替代方案，功能更强。

http
Content-Security-Policy: frame-ancestors 'self' https://trusted.com https://*.example.com;
1

支持的值：

  • 'none'：不允许任何来源嵌入（相当于 DENY）
  • 'self'：只允许同域名（相当于 SAMEORIGIN）
  • 具体域名：https://example.com
  • 通配符：https://*.example.com
  • 协议：https:

优点：

  • 支持多个域名
  • 支持通配符
  • 支持更灵活的表达式
  • 是新标准，推荐使用

注意：

  • 如果同时设置了 X-Frame-Options 和 frame-ancestors，frame-ancestors 优先级更高（浏览器会忽略 X-Frame-Options）
  • 两者可以同时设置，兼容老浏览器

3. frame-busting 脚本（JS 层面的防护） ​

在页面里加一段 JS，检测当前页面是不是在 iframe 里，如果是就「跳出来」。

js
// 最经典的 frame-busting 代码
if (top !== self) {
  top.location.href = self.location.href;
}
1

或者更完善一点：

js
(function() {
  if (window.top !== window.self) {
    // 尝试用各种方式跳出
    try {
      window.top.location.href = window.self.location.href;
    } catch (e) {
      // 跨域不能直接改 top.location，可以用其他方式
      // 比如让页面不可用
      document.body.style.display = 'none';
      alert('本页面不允许在 iframe 中打开');
    }
  }
})();

缺点：

  • 如果用户禁用了 JS，就失效了
  • 攻击者有各种绕过方式（如用 sandbox 属性的 iframe 禁掉 JS、用 X-Frame-Options 的反向利用、用 <iframe security="restricted"> 等）
  • 只能作为辅助手段，不能作为主要防护

为什么还是推荐加？

  • 作为纵深防御的一环
  • 老浏览器可能不支持 X-Frame-Options 和 CSP
  • 增加攻击者的成本

4. SameSite Cookie（辅助） ​

如果会话 Cookie 设置了 SameSite=Lax 或 Strict，那么跨站 iframe 里的请求不会带上 Cookie，即使点击了也没用（因为没登录）。

这不是专门防点击劫持的，但对点击劫持也有防御效果。

UI Redressing（界面伪装/UI 重定向攻击） ​

点击劫持的高级变种，不只是完全透明，而是部分可见、经过精心设计，让用户以为看到的是正常界面，实际上是 iframe 里的内容。

常见手法：

  • 只露出 iframe 里的按钮部分，周围用自己的界面包起来
  • 用 CSS 裁剪（clip 或 overflow: hidden）只显示 iframe 的一部分
  • 多层嵌套，让用户分不清哪个是哪个
  • 利用浏览器的权限弹窗（如通知、摄像头）做伪装

防范：

  • X-Frame-Options / frame-ancestors 仍然有效（根本不让嵌入）
  • 重要操作加二次确认（弹窗确认、短信验证码、密码验证）
  • 敏感操作不要用简单的按钮点击，加手势或验证码

点击劫持 vs XSS vs CSRF ​

三种都是前端常见安全问题，但原理和攻击方式完全不同。

维度点击劫持（Clickjacking）XSSCSRF
核心手段视觉欺骗，骗用户点击注入恶意脚本，在受害者页面执行伪造用户请求，利用 Cookie 自动携带
是否需要脚本不需要（纯 CSS + iframe）需要（注入 JS 执行）不需要（发请求就行）
攻击目标用户的点击操作用户的页面数据和权限用户的登录态和状态变更接口
依赖条件页面能被 iframe 嵌入输入未过滤、输出未转义Cookie 自动携带、服务端无验证
攻击者能读响应吗不能（只能触发点击）能（同源，可读页面内容）一般不能（只发请求）
防范核心X-Frame-Options / frame-ancestors转义 + CSP + HttpOnlySameSite + Token + Origin 校验
属于哪种威胁用户交互劫持代码注入请求伪造

一句话总结区别：

  • XSS：攻击者在你的网站里执行代码
  • CSRF：攻击者借用户的名义发请求
  • Clickjacking：攻击者骗用户自己去点按钮

其他常见前端安全问题 ​

1. SQL 注入（SQL Injection） ​

虽然主要是后端问题，但前端也需要了解和配合防范。

原理： 用户输入的内容被拼接进 SQL 语句，导致执行了恶意 SQL。

js
// ❌ 危险：字符串拼接
const sql = `SELECT * FROM users WHERE username = '${username}' AND password = '${password}'`;
// 如果 username 输入的是 ' OR 1=1 -- 
// SQL 就变成了：SELECT * FROM users WHERE username = '' OR 1=1 --' AND password = '...'
// 结果：直接登录成功
1
2
3
4
5

防范：

  • 核心在后端：用参数化查询 / 预编译语句（Prepared Statement）
  • 输入校验：前端做格式校验，后端做强校验
  • ORM 框架：用成熟的 ORM，避免手写 SQL
  • 最小权限原则：数据库账号不要给太高权限
  • 错误信息：不要把数据库错误详细信息暴露给前端

2. 文件上传漏洞 ​

原理： 用户上传恶意文件（如 .php、.jsp、.exe、带木马的图片），服务器执行或被访问。

常见攻击：

  • 上传 webshell（.php 文件），拿到服务器权限
  • 上传病毒 / 木马，其他用户下载后中招
  • 上传大文件把服务器磁盘占满（DoS）
  • 上传带恶意脚本的 SVG / HTML，用户访问后 XSS

防范：

  • 文件类型校验：检查 MIME 类型 + 文件头（Magic Number），不能只看扩展名
  • 文件重命名：随机文件名，不要用用户传的文件名
  • 存储到独立域名：上传的文件放在独立的静态资源域名（或 OSS），和主站隔离
  • 限制文件大小：防止大文件 DoS
  • 服务端杀毒：上传后扫描病毒
  • 强制下载：Content-Disposition: attachment，不让浏览器直接执行
  • SVG 文件要特别小心（可以内嵌脚本），最好转成 PNG 或做过滤

3. 路径遍历（Path Traversal / Directory Traversal） ​

原理： 用户输入 ../ 之类的路径穿越符，访问到不应该访问的文件。

示例：

// 下载文件接口
/download?file=report.pdf

// 攻击者输入
/download?file=../../../../etc/passwd
/download?file=..\..\windows\system32\config\sam
1
2
3
4
5
6

如果后端直接拼接路径读取文件，就会泄露敏感文件。

防范：

  • 永远不要直接用用户输入拼路径
  • 用白名单校验文件名，只允许特定字符
  • 路径规范化后检查是否在允许的目录内（path.normalize 后判断前缀）
  • 用随机 ID 映射到真实文件名，不暴露真实路径
  • 权限控制：下载前校验用户是否有权限

4. CORS 配置错误 ​

CORS（跨域资源共享）配置不当会导致安全问题。

常见错误配置：

① 反射 Origin（把请求的 Origin 原样返回）+ Allow-Credentials: true

http
// ❌ 危险配置
Access-Control-Allow-Origin: <请求的 Origin>  // 动态反射
Access-Control-Allow-Credentials: true
1
2
3

这样任何网站都可以带 Cookie 请求你的接口，等于完全放开了跨域限制，CSRF 防护形同虚设。

② 通配符 + 凭证

http
// ❌ 浏览器会直接报错（规范不允许），但有些开发者想绕过
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
1
2
3

规范不允许 * 和 Allow-Credentials 同时使用，浏览器会拒绝。

③ 过于宽泛的 Origin 白名单

Access-Control-Allow-Origin: https://*.example.com
1

如果有任何一个子域名被攻破（如 blog.example.com 有 XSS），攻击者就能利用它来访问主站接口。

正确做法：

  • 严格的白名单，只允许确实需要的域名
  • 不要用 * 做凭证请求
  • 不要反射 Origin
  • 结合 CSRF Token 和 SameSite Cookie 做多层防护

5. 其他值得了解的 ​

  • HTTP Host Header 攻击：攻击者伪造 Host 头，导致缓存投毒、密码重置链接泄露
  • 缓存投毒（Web Cache Poisoning）：利用 CDN/代理缓存，把恶意响应缓存下来，影响所有用户
  • CRLF 注入：在 HTTP 头里注入换行，伪造响应头或响应体
  • JSONP 安全：JSONP 回调参数如果未过滤可能导致 XSS，而且容易被 CSRF 利用（现在建议用 CORS 替代）
  • WebSocket 跨域：WebSocket 不受同源策略限制，靠 Origin 头校验，服务端必须校验 Origin
  • 第三方脚本安全：引入的第三方统计、广告脚本如果被劫持，全站遭殃。用 SRI（Subresource Integrity）校验脚本完整性

前端安全防御总原则 ​

  1. 纵深防御（Defense in Depth）：不要只靠一道防线，多层防护

    • 比如防 XSS：转义 + CSP + HttpOnly + 输入校验

  2. 默认安全（Secure by Default）：默认配置应该是最安全的

    • 比如 Cookie 默认加 SameSite=Lax、Secure、HttpOnly

  3. 最小权限原则：只给必要的权限

    • 比如数据库账号只有读写权限，不要给 DBA

  4. 不要信任用户输入：所有用户输入都是不可信的

    • 输入校验、输出转义、参数化查询

  5. 安全左移（Shift Left）：在开发阶段就考虑安全，不要等到上线才补

    • 代码扫描、安全评审、依赖检查

追问 1：frame-busting 脚本有哪些常见的绕过方式？为什么说不能只靠 JS 防护？ ​

答 ​

frame-busting（也叫 frame-breaking）就是用 JS 检测并跳出 iframe 的脚本。它确实能防一些简单的攻击，但有很多绕过方式，所以不能作为主要防护手段。

常见的绕过方式：

1. 禁用 iframe 里的 JS ​

攻击者可以用 HTML5 的 sandbox 属性限制 iframe 里的脚本：

html
<iframe src="target.com" sandbox="allow-forms"></iframe>
1
  • sandbox 属性会给 iframe 加上一堆限制，默认是禁止脚本的
  • 这样 frame-busting 脚本根本不会执行
  • 但同时其他 JS 也不能跑了，攻击场景受限（有些操作可能需要 JS）

2. 多层 iframe 嵌套 ​

攻击者套两层 iframe：

evil.com (最外层)
  └── evil.com/inner (第二层，和最外层同域)
        └── target.com (目标网站，最内层)
1
2
3

目标网站的 frame-busting 检测的是 top !== self，然后尝试把 top.location 改成自己的地址。但因为跨域（target.com 和最外层 evil.com 不同域），top.location 是读不到也改不了的，会抛错。

虽然抛错了页面还是在 iframe 里，攻击者可以继续用。

有些 frame-busting 代码写得不好，只改 top.location 抛错了就没下文了，这就被绕过了。

3. onbeforeunload 欺骗 ​

js
// 攻击者页面
window.onbeforeunload = function() {
  return '确定要离开吗？';
};
1
2
3
4

当 frame-busting 脚本要跳转 top.location 时，浏览器会弹出「确定要离开吗？」的确认框。用户可能点「取消」，跳转就被取消了。

虽然这个技巧比较老，现在浏览器对 onbeforeunload 有很多限制，但在某些场景下仍然有效。

4. 利用 X-Frame-Options 反向 ​

这个比较 trick：如果攻击者的页面也设置了 X-Frame-Options: DENY，然后用某些技术手段...... 这个不太常见，知道有这么回事就行。

5. <iframe security="restricted">（IE 特有） ​

IE 里有个 security="restricted" 属性，可以让 iframe 里的脚本在受限安全区域运行，很多功能被禁用。

6. 用 <object> 或 <embed> 代替 iframe ​

有些 frame-busting 只检测 iframe，没用 <object> 也能嵌入页面。

为什么不能只靠 JS 防护？

  1. JS 可以被禁用：用户（或攻击者）禁了 JS，防护就没了
  2. 跨域限制：跨域 iframe 里的 JS 不能直接操作 top，很多 frame-busting 代码跨域就失效了
  3. 绕过方式多：上面列的这些，而且还有更多奇技淫巧
  4. 不是所有浏览器行为都一致：各种浏览器的安全特性有差异
  5. 攻击者可以不断研究新绕过方式：道高一尺魔高一丈

正确的做法：

  • 主要防线：X-Frame-Options 或 CSP frame-ancestors（服务端响应头，浏览器强制，JS 绕不过）
  • 辅助防线：frame-busting 脚本（增加攻击成本，兼容老浏览器）
  • 纵深防御：重要操作加二次确认、SameSite Cookie

最推荐的配置组合：

http
Content-Security-Policy: frame-ancestors 'self' https://trusted.com;
X-Frame-Options: SAMEORIGIN
1
2

再加上页面里的 frame-busting 脚本作为最后一道防线。

追问 2：SVG 文件上传有哪些安全风险？应该如何防范？ ​

答 ​

SVG（Scalable Vector Graphics）是一种基于 XML 的矢量图格式，它的特殊之处在于：它可以内嵌 JavaScript、HTML、CSS、外部资源引用。这使得 SVG 成为文件上传安全中特别需要注意的一类。

SVG 的安全风险：

1. XSS 脚本注入 ​

SVG 里可以直接写 <script> 标签：

xml
<svg xmlns="http://www.w3.org/2000/svg">
  <script type="text/javascript">
    alert(document.cookie); // 偷 Cookie
    // 或者发起请求、篡改页面
  </script>
</svg>
1

还有各种事件属性也能执行脚本：

xml
<svg xmlns="http://www.w3.org/2000/svg" onload="alert(1)">
  <circle cx="50" cy="50" r="40" fill="red" />
</svg>

如果用户上传这个 SVG，然后直接在浏览器里打开（作为图片显示），脚本就会在你域名的上下文中执行，等同于存储型 XSS。

2. XML 外部实体注入（XXE） ​

SVG 是 XML 格式的，如果服务端解析 SVG 时没有禁用外部实体，可能导致 XXE 攻击：

xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<svg xmlns="http://www.w3.org/2000/svg">
  <text>&xxe;</text>
</svg>

如果服务端解析并返回了 SVG 内容，就可能泄露服务器的敏感文件。

3. 内部网络探测（SSRF） ​

SVG 可以引用外部资源（图片、字体、样式等），这些请求是从服务端发起的（如果服务端做了 SVG 处理），或者从用户浏览器发起的：

xml
<svg xmlns="http://www.w3.org/2000/svg">
  <image href="http://192.168.1.1/admin.png" />
</svg>
1
2
3

如果服务端做了 SVG 转 PNG 之类的处理，那请求是从服务器内网发的，可能探测内网服务（SSRF）。

4. 钓鱼 / 伪装 ​

SVG 可以画任何东西，可以用来做钓鱼页面的一部分，或者伪装成其他类型的文件。

5. 拒绝服务（DoS） ​

SVG 可以写递归引用、超大嵌套、 billion laughs 攻击之类的，把解析器搞挂：

xml
<?xml version="1.0"?>
<!DOCTYPE lolz [
  <!ENTITY lol "lol">
  <!ENTITY lol2 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;">
  <!-- 指数级展开，内存爆炸 -->
]>

2
3
4
5
6

防范措施：

1. 不让浏览器直接解析 SVG（最彻底） ​

  • 上传的 SVG 不要直接以 image/svg+xml 返回
  • 用 Content-Disposition: attachment 强制下载
  • 或者在服务端把 SVG 转成 PNG / JPG 再返回
  • 存储到独立的静态资源域名（和主站隔离，即使有 XSS 也拿不到主站 Cookie）

2. 过滤 / 净化 SVG 内容 ​

如果确实需要显示 SVG（比如产品允许用户上传 SVG 图标），必须经过严格的净化：

  • 白名单标签：只允许 <svg>、<path>、<circle>、<rect> 等纯图形标签
  • 白名单属性：禁止 onload、onclick 等事件属性，禁止 href / xlink:href 指向外部
  • 移除 <script>：所有脚本标签全部删掉
  • 禁用外部实体：XML 解析时禁用 DOCTYPE 和外部实体
  • 禁用外部资源引用：不允许引用外部图片、字体、样式
  • 可以用成熟的库，比如 DOMPurify（支持 SVG 过滤）

3. 服务端解析注意事项 ​

  • XML 解析器要禁用外部实体和 DTD
  • 限制 SVG 文件大小和复杂度（防止 DoS）
  • 解析超时设置，防止阻塞
  • 如果做 SVG 转码（如转 PNG），确保转码工具本身没有漏洞

4. 其他建议 ​

  • 上传文件重命名，不要用原始文件名
  • 放在独立的 CDN / 对象存储域名下，和业务域名隔离
  • 对用户上传的内容设置 CSP，限制脚本执行
  • 如果只是显示用，建议上传时转成 PNG / JPG，既安全又兼容性好

一句话总结：SVG 不是「图片」，是「代码」。处理 SVG 上传要像处理 HTML 上传一样小心。

追问 3：CSP（内容安全策略）有哪些常见的指令和配置错误？如何设计一个合理的 CSP 策略？ ​

答 ​

CSP（Content-Security-Policy）是一个强大的浏览器安全层，用于检测和缓解某些类型的攻击，包括 XSS 和数据注入攻击。

常见 CSP 指令：

指令作用
default-src默认策略，其他没指定的都用这个
script-src脚本来源
style-src样式来源
img-src图片来源
font-src字体来源
connect-srcXHR / WebSocket / fetch 等连接目标
frame-srciframe 来源
frame-ancestors允许被哪些源嵌入（防点击劫持）
object-srcFlash 等插件来源（建议设为 'none'）
media-src音视频来源
worker-srcWeb Worker 来源
base-uri<base> 标签的 href 来源
form-action表单提交的目标地址
upgrade-insecure-requests自动把 HTTP 升级成 HTTPS
report-uri / report-to上报违规报告的地址

常用的来源值：

值含义
'none'不允许任何来源
'self'允许同源
'unsafe-inline'允许内联脚本/样式（危险，降低安全性）
'unsafe-eval'允许 eval() 等动态代码执行（危险）
'nonce-xxx'允许带指定 nonce 的内联脚本
'sha256-xxx'允许内容哈希匹配的脚本
https:允许任何 HTTPS 来源
https://example.com允许指定域名
*.example.com允许子域名

常见配置错误：

1. 过度使用 'unsafe-inline' ​

http
# ❌ 不好：允许所有内联脚本，等于废掉了 CSP 防 XSS 的能力
Content-Security-Policy: script-src 'self' 'unsafe-inline';
1
2

unsafe-inline 允许所有内联 <script> 标签执行，而 XSS 最常见的就是注入内联脚本。加了这个，CSP 对 XSS 的防御能力大大降低。

正确做法：

  • 把内联脚本移到外部 JS 文件
  • 必须内联的脚本用 nonce 或 hash

2. 过度使用 'unsafe-eval' ​

http
# ❌ 不好：允许 eval，很多 XSS  payload 用 eval 执行
Content-Security-Policy: script-src 'self' 'unsafe-eval';
1
2

unsafe-eval 允许 eval()、setTimeout(string)、new Function() 等动态代码执行。很多 XSS 攻击通过这些方式绕过。

正确做法：

  • 尽量不用 eval，改用其他方式实现
  • 如果是第三方库用了 eval，考虑换库或升级版本

3. 过于宽泛的通配符 ​

http
# ❌ 不好：所有 HTTPS 网站都能加载脚本，CDN 被劫持就完蛋
Content-Security-Policy: script-src https:;
1
2
http
# ❌ 不好：data: 协议可以用来执行脚本
Content-Security-Policy: script-src 'self' data:;
1
2

通配符太宽等于没限制，一定要用精确的白名单。

4. 缺少 object-src / base-uri / form-action ​

http
# ❌ 不完整：只设了 script-src，其他都用 default-src
Content-Security-Policy: default-src 'self'; script-src 'self';
1
2

一些容易被忽略但很重要的指令：

  • object-src 'none'：禁用 Flash 等插件（Flash 已经死了，但防御性编程）
  • base-uri 'self'：防止攻击者注入 <base> 标签改所有相对路径的目标
  • form-action 'self'：防止表单被提交到恶意网站

5. report-only 和 enforce 搞混 ​

CSP 有两种模式：

  • Content-Security-Policy：强制模式，违规就拦截
  • Content-Security-Policy-Report-Only：仅报告模式，违规只上报不拦截

很多人一开始就上强制模式，结果把正常功能搞挂了。

正确做法： 先上 report-only 模式，看一段时间的报告，确认没问题了再切换到强制模式。

6. 依赖 'strict-dynamic' 但没理解 ​

'strict-dynamic' 是一个比较新的特性，意思是「受信任的脚本加载的其他脚本也受信任」。用得好可以简化 CSP，用不好反而有漏洞。

  img-src 'self';
  font-src 'self';
  connect-src 'self';

  • 加第三方统计域名

第三步：处理内联脚本 ​

方案 A：nonce（推荐 SPA 和 SSR 场景）

  • SSR 场景天然适用

方案 B：hash（适合固定内容的内联脚本）

http

1

把内联脚本的内容做 SHA-256 哈希，写进 CSP 里。内容变了就要重新算哈希。

第四步：开启报告，逐步收紧 ​

  1. 先用 Content-Security-Policy-Report-Only 观察几周
  2. 看报告里的违规都是什么，哪些是正常业务需要的
  3. 把正常的加进白名单
  4. 确认没问题后，切换到强制模式
  5. 持续监控违规报告，发现新问题及时处理

第五步：定期审计 ​

  • 依赖升级时检查 CSP
  • 新功能上线前检查是否需要调整 CSP
  • 关注新的 CSP 特性和浏览器支持

一个比较完善的 CSP 示例：

http
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-${nonce}' https://cdn.example.com;
  style-src 'self' 'unsafe-inline' https://cdn.example.com;  # 样式的 unsafe-inline 相对风险小
  img-src 'self' data: https://*.example.com blob:;
  font-src 'self' https://cdn.example.com;
  connect-src 'self' https://api.example.com https://analytics.example.com;
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  frame-ancestors 'self';
  upgrade-insecure-requests;
  report-uri /csp-report;

注意事项：

  • 样式的 'unsafe-inline' 相对常见，风险比脚本的小（因为 CSS 里执行脚本能力有限），但也尽量避免
  • 图片的 data: 也常见（小图标用 base64），风险可控
  • upgrade-insecure-requests 适合全站 HTTPS 的站点
  • report-uri 很重要，没有监控的 CSP 等于瞎搞

CSP 的定位：

  • CSP 不是银弹，不能替代输入转义
  • CSP 是「最后一道防线」，即使有 XSS 漏洞，CSP 也能大大降低危害
  • 输入转义 + CSP + HttpOnly = 三层防护，才比较稳
