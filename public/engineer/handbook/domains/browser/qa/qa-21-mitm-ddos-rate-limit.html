问答 21（高级）：中间人攻击、接口防刷与 DDoS ​

题目 ​

中间人攻击（MITM）是怎么发生的？HTTPS 如何防范？接口防刷通常有哪些手段？DoS / DDoS 攻击原理和缓解思路是什么？

参考答案 ​

中间人攻击（MITM） ​

攻击方式

  • ARP 欺骗：局域网内伪造网关 MAC，让流量经过攻击者。
  • DNS 污染 / 劫持：把域名解析到攻击者 IP。
  • 假 WiFi：连上后所有流量都经过攻击者。
  • BGP 劫持：宣告错误路由，把大范围流量吸引到攻击者 AS。
  • 恶意 Root CA：企业 / 恶意软件在系统里植入 CA，让 HTTPS 也能被解密。

HTTPS 如何防范

  • 公钥基础设施 PKI：服务器提供 CA 签发的证书，浏览器用内置 Root CA 校验。
  • 域名匹配 + 有效期 + 吊销状态 + Certificate Transparency 日志 多重校验。
  • 若中间人插入自签证书，浏览器警告「不受信任」。
  • HSTS：Strict-Transport-Security: max-age=63072000; includeSubDomains; preload，浏览器强制走 HTTPS，防「SSL 剥离」（把 HTTPS 变 HTTP）。
  • HSTS Preload List：预置到浏览器源码里，首访就 HTTPS。
  • Certificate Pinning（App 层）：客户端固化公钥指纹，即使系统证书信任被污染也识破。Web 上 HPKP 已废弃，改用 Expect-CT 或客户端策略。

残余风险

  • 用户主动信任恶意 CA（企业电脑 / 木马）。
  • 应用层漏洞（HTTPS 只保护传输，业务逻辑漏洞照样可被利用）。
  • 侧信道（时间攻击、缓存攻击）。

接口防刷 ​

目标：防止爬虫 / 恶意脚本 / 黄牛高频调用接口，保护业务和资源。

常见手段

  1. 限流（Rate Limit）
    • 固定窗口：每分钟最多 60 次；实现简单但边界会有 2 倍流量。
    • 滑动窗口：更平滑，用 Redis ZSET / 双桶实现。
    • 漏桶（Leaky Bucket）：请求以恒定速率处理，突发排队 / 丢弃。
    • 令牌桶（Token Bucket）：允许一定突发，令牌以恒定速率生成，请求消耗令牌。
    • 维度：按 IP、用户、设备指纹、接口路径、参数（如商品 ID）分别限流。
  2. 签名与鉴权
    • 请求带 sign = hmac(secret, timestamp + nonce + body)，服务端校验。
    • timestamp 防重放；nonce 一次性使用。
    • 客户端 JS 里的 secret 只是「提高破解成本」，无法阻止有心人。
  3. 验证码 / 人机识别
    • 图形 / 短信 / 拼图 / 无感（如 hCaptcha、reCAPTCHA、行为验证）。
    • 关键节点触发：登录失败 N 次、异常 IP、批量下单。
  4. 设备指纹 / 风控画像
    • 采集浏览器指纹、Cookie、TLS ClientHello、行为轨迹，识别自动化脚本。
    • 云端风控评分，高风险接口拉起人机验证。
  5. 业务约束
    • 抢购需要预约 / 排队 / 库存分片；一账户一件；地址限制。
    • 关键接口写入去重 key，防止一秒内重复下单。
  6. 网关层配置
    • Nginx limit_req / limit_conn，Kong / Envoy 的 rate limit 插件。
    • CDN / WAF 层限流是最靠外的一道防线。

DoS / DDoS ​

定义

  • DoS：单源攻击，把服务打死。
  • DDoS：分布式，成千上万肉鸡同时打，流量可到几百 Gbps 到 Tbps。

常见类型

  • 网络层洪水（L3/4）：UDP flood、SYN flood、ICMP flood，用带宽 / 连接数打满。
  • 反射放大：向 DNS / NTP / Memcached 发小请求，源 IP 伪造成受害者，服务器把大响应打给受害者，放大几十倍到上千倍。
  • 应用层（L7）：CC 攻击，大量看似正常的 HTTP 请求，特别是慢速请求（Slowloris）耗尽连接。

缓解思路

  1. 上游流量清洗 / DDoS 高防服务：云厂商（Cloudflare、Akamai、阿里云高防）用巨大带宽 + AI 识别过滤。
  2. CDN 分散流量：静态资源天然由 CDN 承载，减少直击源站。
  3. 网络层：
    • SYN cookies 抵抗 SYN flood。
    • RST 洪水 / UDP 洪水由运营商 / IDC 层过滤。
    • BGP 黑洞 / Anycast 分流。
  4. 应用层限流：
    • Nginx limit_req + WAF。
    • 识别异常 UA / 无 referer / 高频重复。
    • JS challenge：先返回一段 JS，只有真正浏览器能执行拿到 token 才放行。
  5. 弹性扩容：K8s HPA / Serverless 自动扩，短期扛住流量。
  6. 降级 / 熔断：非核心接口先熔断，保护核心链路。
  7. 日志与画像：实时分析 5xx / 慢请求，自动加封 IP 段。

追问 1：HSTS 为什么要 preload？攻击者能否绕过？ ​

答：

  • HSTS 是「响应头声明」，只有用户已经访问过一次 HTTPS 站点并拿到 HSTS 头，浏览器才知道下次要强制 HTTPS。
  • 首次访问如果是 HTTP，攻击者仍有机会做 SSL 剥离（把用户重定向到 HTTP，删掉服务器的 HSTS 头）。
  • Preload List 由 Chrome / Firefox / Safari / Edge 共同维护，网站可申请加入。加入后浏览器首次访问就强制 HTTPS，攻击者根本插不进去 HTTP 版本。
  • 申请要求：
    • 主域和所有子域都 HTTPS。
    • HSTS max-age >= 31536000（一年）且带 includeSubDomains、preload。
    • 加入后很难移除（会滞留在浏览器版本中数月甚至数年），要慎重。
  • 绕过难度极高，除非同时控制用户设备（root CA）或找到浏览器 bug。

追问 2：令牌桶和漏桶实现上有什么区别？高并发场景怎么用 Redis 实现分布式限流？ ​

答：

  • 漏桶：请求进桶，按恒定速率漏出。突发流量被桶容量吸收，超过则丢弃。输出速率固定。
  • 令牌桶：令牌以恒定速率填充桶，请求消耗令牌；桶里有令牌就可以突发。允许短时突发，长期速率受限。
  • 实践中令牌桶更常用，因为符合「平均限速，允许小尖峰」的直觉。
  • Redis 分布式实现（简化令牌桶）：
lua
-- KEYS[1] = 桶 key, ARGV = { capacity, rate, now, cost }
local capacity = tonumber(ARGV[1])
local rate     = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local cost     = tonumber(ARGV[4])
local data = redis.call('HMGET', KEYS[1], 'tokens', 'ts')
local tokens = tonumber(data[1]) or capacity
local ts     = tonumber(data[2]) or now
tokens = math.min(capacity, tokens + (now - ts) * rate)
if tokens < cost then return 0 end
tokens = tokens - cost
redis.call('HMSET', KEYS[1], 'tokens', tokens, 'ts', now)
redis.call('EXPIRE', KEYS[1], math.ceil(capacity / rate) + 10)
return 1

追问 3：4 层和 7 层 DDoS 分别怎么打，各自要用什么手段防？ ​

答：

  • 4 层（L4）：不关心业务，靠带宽 / 连接堆死。
    • SYN Flood：大量伪造源 IP 的 SYN 包让服务器建半连接。防：syncookies、SYN queue 调大、上游清洗。
    • UDP Flood / 放大：源伪造成受害者，发大量 DNS / NTP 反射。防：运营商 uRPF、限速、协议感知过滤。
    • ICMP Flood：单纯 ping 大量包。防：网关限速。
    • 关键：在源头 / 骨干层过滤，因为流量已经打进机房网络就来不及。
  • 7 层（L7）：模拟真实用户 HTTP 请求。
    • CC 攻击：高频 GET / POST 打到应用层，可能针对某个昂贵接口。
    • Slowloris：慢速请求，保持连接不发完，耗尽 worker。
    • HTTP/2 Rapid Reset：滥用 stream 快开快关，压垮多路复用。
    • 防：CDN + WAF、JS challenge、行为分析、限流、连接与请求超时收紧、WAF 规则实时更新。
  • 综合：现代方案是「CDN → 高防清洗 → WAF → 源站限流」多级过滤，任何单点都不能保证全防。
