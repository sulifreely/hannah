问答 24（进阶）：IndexedDB 与浏览器存储全对比 ​

题目 ​

浏览器提供了哪些存储方案？它们各自的容量限制、生命周期、同步异步、支持的数据结构和使用场景有什么不同？请重点讲解 IndexedDB 的基本使用（事务、游标、索引）。

参考答案 ​

浏览器存储方案全景对比 ​

现代浏览器提供了多种存储机制，各有侧重，选择哪种取决于数据量、访问频率、是否需要结构化查询、持久性要求等。

存储方案容量限制生命周期同步/异步数据结构访问限制典型场景
Cookie每个域名 ~4KB，总数 ~20 个可设置 expires（会话/持久）同步纯字符串（key=value）同源，自动随请求发送会话管理、用户偏好（小数据）
localStorage~5MB（同源）永久，除非手动清除同步字符串 key-value同源（协议+域名+端口）用户偏好、简单状态缓存
sessionStorage~5MB（同源）标签页关闭即清除同步字符串 key-value同源 + 同标签页单会话临时数据
IndexedDB可用磁盘空间的 ~50%（浏览器策略）永久，除非手动清除或配额不足异步结构化对象（支持索引、事务、游标）同源大量结构化数据、离线数据
WebSQL~5MB-50MB（已废弃）永久异步（回调式）关系型表（SQL）同源已废弃，不要再用

逐个详解 ​

1. Cookie ​

特点：

  • 最小的存储，设计初衷不是给 JS 用的，而是 HTTP 状态管理的一部分
  • 每次 HTTP 请求都会自动带上 Cookie（包括图片、CSS 等所有请求），所以不能存太多

重要属性：

  • expires / max-age：过期时间，不设则为会话 Cookie（关浏览器即失）
  • domain / path：作用域
  • Secure：仅 HTTPS 下发送
  • HttpOnly：JS 无法读取（防 XSS 偷会话）
  • SameSite：跨站是否携带（Strict / Lax / None）

2. localStorage / sessionStorage ​

共同点：

  • 都是 key-value 存储，值只能是字符串
  • API 简单：setItem / getItem / removeItem / clear / key
  • 同步 API，大量数据读写会阻塞主线程

区别：

  • localStorage：同源所有标签页共享，永久保存
  • sessionStorage：仅当前标签页（更准确说是「浏览上下文」）有效，标签页关闭即清除，刷新不清除

为什么是字符串？

  • 设计简单，序列化/反序列化由开发者自己做（JSON.stringify / JSON.parse）
  • 避免复杂对象的深拷贝和循环引用问题
  • 但也带来了性能问题：每次存取都要序列化

3. IndexedDB ​

浏览器端的「NoSQL 数据库」，是目前浏览器端最强大的存储方案。

核心特点：

  • 事务型数据库：所有操作都在事务中进行，保证原子性
  • 对象仓库（Object Store）：类似于表，存的是 JS 对象（可结构化克隆的类型）
  • 索引（Index）：可以对对象的某个属性建索引，支持按索引查询
  • 游标（Cursor）：支持逐条遍历数据，支持范围查询
  • 完全异步：基于事件回调（旧 API）或 Promise（新 API 用 idb 库封装后）
  • 容量大：一般可用磁盘空间的 50%，超出后按 LRU 淘汰

IndexedDB 的基本使用：

① 打开/创建数据库

js
// 原生 API（事件回调风格）
let db;
const request = indexedDB.open('MyDatabase', 1); // 数据库名 + 版本号

request.onupgradeneeded = (event) => {
  // 数据库版本变化时触发（首次创建或版本升级）
  // 在这里创建 object store 和索引
  db = event.target.result;
  
  // 创建一个 object store（表）
  // keyPath 是主键，autoIncrement 表示自增
  const store = db.createObjectStore('books', { 
    keyPath: 'id', 
    autoIncrement: true 
  });
  
  // 建索引：可以按 title、author、price 查询
  store.createIndex('by_title', 'title', { unique: true });
  store.createIndex('by_author', 'author', { unique: false });
  store.createIndex('by_price', 'price', { unique: false });
};

request.onsuccess = (event) => {
  db = event.target.result;
  console.log('数据库打开成功');
};

request.onerror = (event) => {
  console.error('数据库打开失败:', event.target.error);
};

为什么需要版本号？

  • IndexedDB 的 schema（object store 和索引结构）只能在 onupgradeneeded 回调中修改
  • 当你需要新增 store 或索引时，必须提高版本号，浏览器发现版本号变大就会触发 onupgradeneeded
  • 这保证了 schema 变更的可控性，类似于数据库 migration

② 添加数据（事务）

js
function addBook(book) {
  // 开启事务：指定 store 名称和模式（readonly / readwrite）
  const transaction = db.transaction(['books'], 'readwrite');
  const store = transaction.objectStore('books');
  
  // 添加数据
  const request = store.add(book);
  
  request.onsuccess = () => {
    console.log('添加成功，ID:', request.result);
  };
  
  transaction.oncomplete = () => {
    console.log('事务完成');
  };
  
  transaction.onerror = (e) => {
    console.error('事务失败:', e.target.error);
    // 事务中任何一个操作失败，整个事务回滚
  };
}

// 使用
addBook({ title: 'JavaScript高级程序设计', author: 'Zakas', price: 99 });

事务的三个模式：

  • readonly：只读，多个只读事务可以并发
  • readwrite：读写，同一时间同一 store 只能有一个 readwrite 事务
  • versionchange：版本变更，只能在 onupgradeneeded 中使用，用于修改 schema

为什么要有事务？

  • 保证数据一致性：事务中的操作要么全部成功，要么全部失败（回滚）
  • 并发控制：防止多个读写同时进行导致数据错乱

④ 游标遍历

适合遍历大量数据或范围查询：

js
function getAllBooksByAuthor(author) {
  const transaction = db.transaction(['books'], 'readonly');
  const store = transaction.objectStore('books');
  const authorIndex = store.index('by_author');
  
  // 打开游标，参数是键范围
  // IDBKeyRange.only(author) —— 精确匹配
  // IDBKeyRange.lowerBound(price, true) —— 大于（true 表示不包含）
  // IDBKeyRange.bound(min, max, true, false) —— 区间
  const request = authorIndex.openCursor(IDBKeyRange.only(author));
  
  const results = [];
  request.onsuccess = (event) => {
    const cursor = event.target.result;
    if (cursor) {
      results.push(cursor.value);
      cursor.continue(); // 移动到下一条
    } else {
      console.log('遍历完成，共', results.length, '条');
    }
  };
}

⑤ 用 Promise 封装（idb 库风格）

原生 API 是回调风格的，实际开发中常用 idb 库（或自己封装）变成 Promise：

js
// 简易版 Promise 封装
function openDB(name, version, upgrade) {
  return new Promise((resolve, reject) => {
    const request = indexedDB.open(name, version);
    request.onupgradeneeded = (e) => upgrade(e.target.result, e.oldVersion);
    request.onsuccess = () => resolve(request.result);
    request.onerror = () => reject(request.error);
  });
}

// 使用
const db = await openDB('MyDB', 1, (db) => {
  if (!db.objectStoreNames.contains('books')) {
    const store = db.createObjectStore('books', { keyPath: 'id', autoIncrement: true });
    store.createIndex('by_title', 'title', { unique: true });
  }
});

4. WebSQL（已废弃） ​

  • 基于 SQLite 的关系型数据库，用 SQL 查询
  • 2010 年被 W3C 废弃，因为各浏览器实现差异太大，且规范停滞
  • 现在只有部分浏览器还支持，但新项目绝对不要用
  • IndexedDB 是它的替代方案

5. Cache API ​

  • 专门用于缓存 HTTP 请求-响应对，和 Service Worker 配合使用
  • 存储的是完整的 Response 对象，包括 header、body、状态码等
  • 可以按 URL 匹配、批量操作、版本管理
  • 是 PWA 离线能力的核心（详见 qa-23）

6. File System Access API ​

  • 最新的存储能力，可以直接读写用户本地文件系统
  • 需要用户主动授权（通过文件选择器）
  • 支持读、写、创建文件和目录
  • 适合「本地文件编辑器」类应用，如在线 IDE、图片编辑器
  • 兼容性还在逐步完善中

存储持久性与配额 ​

存储配额：

  • Cookie：4KB 硬性限制
  • localStorage / sessionStorage：约 5MB（各浏览器略有不同）
  • IndexedDB + Cache API 等「持久化存储」：共享配额，一般是可用磁盘空间的 50%
  • 超出配额时，浏览器会按 LRU（最近最少使用）原则删除整个源的数据

存储持久性（Storage Persistence）：

  • 默认是「最佳努力（best-effort）」：磁盘空间不足时可能被浏览器清除
  • 可以申请「持久化存储（persistent storage）」：用户授权后，浏览器不会自动清除，只有用户手动清除才行
js
if (navigator.storage && navigator.storage.persist) {
  const isPersisted = await navigator.storage.persist();
  console.log('是否持久化存储:', isPersisted);
}

// 查询剩余配额
if (navigator.storage && navigator.storage.estimate) {
  const estimate = await navigator.storage.estimate();
  console.log('已使用:', estimate.usage, '字节');
  console.log('配额:', estimate.quota, '字节');
}

追问 1：IndexedDB 的事务为什么设计成自动提交的？有什么要注意的坑？ ​

答 ​

IndexedDB 事务的自动提交机制：

  • 当你创建事务后，不需要手动调用 commit()，事务会在所有请求完成且没有新请求加入时自动提交
  • 如果事务中所有请求都成功，事务自动 commit；有任何一个请求出错，事务自动 abort（回滚）

为什么这样设计？

  • IndexedDB 是事件驱动的异步 API，事务的生命周期和事件循环绑定
  • 自动提交简化了 API，开发者不需要手动管理事务的起止
  • 但这也意味着你必须「同步地」把事务中的所有请求都发出去，不能等异步操作回来再继续发请求

常见的坑：

1. 在事务中做异步操作导致事务提前结束

js
// ❌ 错误示例
const transaction = db.transaction(['books'], 'readwrite');
const store = transaction.objectStore('books');

// 先发起一个请求
store.get(1).onsuccess = async (e) => {
  const book = e.target.result;
  // 这里做了一个异步操作（比如 fetch）
  const newPrice = await fetchNewPrice(book.title);
  
  // ⚠️ 等异步回来，事务早就自动提交了！
  // 下面这行会报错：TransactionInactiveError
  store.put({ ...book, price: newPrice });
};

为什么会这样？

  • 你创建事务后，只发了一个 get(1) 请求
  • 这个请求成功后，事务看「没有新请求了」，就自动提交了
  • 等你 await 完 fetch 再去 put，事务已经结束了

正确做法：

  • 先把需要的数据全查出来，再开新事务写
  • 或者把所有操作都放在同步执行路径中
js
// ✅ 正确做法：先查，拿到数据后开新事务写
const transaction1 = db.transaction(['books'], 'readonly');
const store1 = transaction1.objectStore('books');
const book = await new Promise((resolve) => {
  store1.get(1).onsuccess = (e) => resolve(e.target.result);
});

const newPrice = await fetchNewPrice(book.title);

// 开新的读写事务
const transaction2 = db.transaction(['books'], 'readwrite');
const store2 = transaction2.objectStore('books');
store2.put({ ...book, price: newPrice });

2. 事务错误处理

  • 事务中的任何一个请求出错，整个事务都会回滚
  • 如果没在 request 级别处理错误，错误会冒泡到 transaction 级别
  • 可以在 request 上调用 preventDefault() 阻止错误冒泡，让事务继续
js
request.onerror = (e) => {
  e.preventDefault(); // 阻止事务回滚
  console.log('单个请求失败，但事务继续');
};

3. 读写事务的并发限制

  • 同一个 object store 同一时间只能有一个 readwrite 事务
  • 多个 readonly 事务可以并发
  • 所以尽量把操作放在一个事务里完成，避免开多个小事务排队

追问 2：localStorage 是同步的，大量数据读写会阻塞主线程，有什么优化方案？ ​

答 ​

localStorage 同步阻塞的原因：

  • localStorage 操作是磁盘 IO（至少有一层持久化），浏览器在主线程上同步等待 IO 完成
  • 数据量大时（比如存几 MB 数据），序列化 + IO 可能需要几十到上百毫秒
  • 这会阻塞渲染和事件响应，导致页面卡顿

优化方案：

1. 拆包，减少单次存取量

  • 不要把大对象整个存在一个 key 里，拆成多个小 key
  • 按需存取，只读写需要的那部分
js
// ❌ 不好：整个 user 对象序列化
localStorage.setItem('user', JSON.stringify(bigUserObject));

// ✅ 好：拆分存储
localStorage.setItem('user.name', name);
localStorage.setItem('user.preferences', JSON.stringify(prefs));

2. 用 IndexedDB 替代

  • 数据量大（超过几百 KB）时，直接用 IndexedDB
  • IndexedDB 是异步的，不会阻塞主线程
  • 缺点是 API 复杂，可以用 idb 库简化

3. 延迟写入（debounce/throttle）

  • 如果是频繁更新的数据（如滚动位置、输入框内容），不要每次变化都存
  • 用 debounce 或 throttle 合并多次写入
js
// 简单的 debounce 写入
let saveTimeout;
function saveState(key, value) {
  clearTimeout(saveTimeout);
  saveTimeout = setTimeout(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, 500); // 500ms 内的连续写入合并成一次
}

4. 内存缓存 + 后台持久化

  • 在内存中维护一份数据副本，读写都走内存
  • 后台异步地把内存数据同步到 localStorage
js
const cache = new Map();

function getItem(key) {
  if (cache.has(key)) return cache.get(key);
  const value = localStorage.getItem(key);
  cache.set(key, value);
  return value;
}

function setItem(key, value) {
  cache.set(key, value);
  // 用 requestIdleCallback 在浏览器空闲时写入
  if ('requestIdleCallback' in window) {
    requestIdleCallback(() => {
      localStorage.setItem(key, value);
    });
  } else {
    setTimeout(() => {
      localStorage.setItem(key, value);
    }, 0);
  }
}

5. 使用 Web Worker

  • 但注意：Web Worker 中不能访问 localStorage（因为 localStorage 是同步的，设计上不让 worker 用）
  • 不过可以在 worker 中处理数据，通过 postMessage 把序列化好的字符串发给主线程存
  • 这样至少序列化的开销在 worker 中，主线程只负责最终的 setItem

6. 考虑其他存储方案

  • Cache API：如果是 Response/Blob 类数据
  • IndexedDB：结构化数据、大量数据
  • OPFS（Origin Private File System）：File System Access API 的一部分，Web Worker 中也能用，性能好

什么时候可以放心用 localStorage？

  • 数据量小（总存储 < 100KB）
  • 读写不频繁
  • 数据结构简单
  • 需要同步读取（比如初始化时必须拿到配置才能继续）

追问 3：第三方 Cookie 被禁用后，有哪些跨站存储的替代方案？ ​

答 ​

背景：

  • 出于隐私保护，Safari（ITP）、Firefox、Chrome（计划 2024-2025 年）都在逐步禁用第三方 Cookie
  • 第三方 Cookie 指的是：在 A 网站嵌入的 B 网站的 iframe / 脚本 / 图片所设置的 Cookie
  • 这对广告追踪、跨站登录、第三方分析等业务影响很大

替代方案：

1. 第一方 Cookie + 服务端关联

  • 在 A 站设置第一方 Cookie，服务端通过某种方式（如共享用户 ID）关联到 B 站
  • 比如电商网站 A 里嵌入支付网关 B 的 iframe，用户在 A 站登录后，A 站把 token 传给 B 站后端，B 站不用设 Cookie

2. Storage Access API

  • Safari 和 Firefox 提供的 API，让第三方 iframe 可以在用户交互后申请访问第一方存储
  • 需要用户主动点击同意，不是静默的
js
// 在第三方 iframe 中
async function requestStorageAccess() {
  if (document.hasStorageAccess) {
    const hasAccess = await document.hasStorageAccess();
    if (!hasAccess) {
      await document.requestStorageAccess(); // 需要用户手势触发
    }
  }
}

3. CHIPS（Cookies Having Independent Partitioned State）

  • Chrome 提出的方案，给 Cookie 加 Partitioned 属性
  • 第三方 Cookie 按「顶级站点」分区，每个顶级站点有独立的 Cookie jar
  • 这样既保留了第三方功能（如嵌入的购物车记住状态），又不能跨站追踪
http
Set-Cookie: __Host-session=abc123; Secure; HttpOnly; Path=/; SameSite=Lax; Partitioned

4. Shared Storage API

  • Chrome 提出的隐私沙箱方案之一
  • 第三方脚本可以读写跨站共享的、隔离的存储
  • 但读取方式受限，只能通过预定义的函数（如 URL 选择、聚合报告）输出结果，不能直接读原始值
  • 目的是让广告和测量能用，但不能用来追踪个人

5. Fenced Frames + Shared Storage

  • Fenced Frames 是一种隔离的 iframe，不能和父页面通信
  • 配合 Shared Storage，可以在不暴露用户身份的情况下展示个性化广告

6. 服务端方案：用 URL 参数 / postMessage 传递状态

  • 对于嵌入的第三方 iframe，父页面可以通过 postMessage 把必要的状态传过去
  • 或者通过 URL 参数（如 ?user_id=xxx）传递，但要注意安全

7. 第一方上下文（First-party context）

  • 把第三方内容变成第一方：比如用 CNAME 把第三方域名别名到自己的子域名
  • 但 Chrome 的 Privacy Sandbox 也在限制这种做法（CNAME Cloaking）

对前端开发者的影响：

  • 不要再依赖第三方 Cookie 做跨站状态共享
  • 如果你的站点需要嵌入在其他网站里（如 SaaS 产品的嵌入组件），考虑：
    • 用 postMessage 通信
    • 用 URL 参数传递必要状态
    • 引导用户在你的站点直接登录，而不是依赖跨站 Cookie
  • 关注各浏览器的隐私沙箱进展，提前做技术储备
