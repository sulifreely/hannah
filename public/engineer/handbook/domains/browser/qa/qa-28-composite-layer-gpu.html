问答 28（高级）：浏览器渲染原理进阶：合成层 / GPU 加速 / 层爆炸 ​

题目 ​

浏览器渲染流水线的 Layout → Paint → Composite 三个阶段分别做什么？什么是合成层（Composite Layer）？哪些情况会让元素独立成层？GPU 加速的原理是什么？有什么代价？什么是「层爆炸」问题？

渲染流水线回顾：从 DOM 到像素 ​

我们重点讲后三个阶段：Layout → Paint → Composite。

阶段一：Layout（布局 / 回流） ​

做什么： 计算每个元素的几何信息——位置（x, y）和尺寸（width, height）。

  • 输入：Render Tree（带有样式信息的节点树）
  • 输出：每个盒子的精确位置和大小
  • 触发条件：元素尺寸、位置、内容变化，添加/删除 DOM 元素，窗口大小变化等

为什么叫「回流」（Reflow）？

  • 因为布局是一个递归的过程：一个元素的尺寸变化可能影响它的父元素、子元素、兄弟元素
  • 变化会「流」遍整个渲染树，所以叫回流

阶段二：Paint（绘制 / 重绘） ​

做什么： 把元素的视觉内容画出来——颜色、文字、边框、阴影、图片等。

  • 输入：布局后的几何信息 + 样式
  • 输出：多层「位图」（每个合成层一张）
  • 触发条件：颜色变化、背景变化、文字变化等不影响布局的样式变化

Paint 不是一次性把整个页面画完，而是按合成层分层绘制。每个层单独画好，最后合成到一起。

阶段三：Composite（合成） ​

做什么： 把多个 Paint 好的合成层（Composite Layer）按照正确的顺序，合成一张最终的图，交给 GPU 显示到屏幕上。

  • 输入：多个合成层的位图 + 层级关系 + 变换信息
  • 输出：最终屏幕画面
  • 触发条件：transform、opacity 等只影响合成的属性变化

为什么要有合成层？

  • 有些元素的变化可以独立处理，不需要重画整个页面
  • 比如一个元素做 transform 动画，只需要移动它所在的层，其他层都不变
  • 这样动画就能跑在 GPU 上，非常流畅

什么是合成层（Composite Layer） ​

合成层是浏览器为了优化渲染而做的分层机制。不是每个元素都有自己的层，大部分元素都和祖先元素共享同一个层。只有满足特定条件的元素才会被提升为独立的合成层。

类比一下：

  • 把页面想象成一幅画
  • 普通元素就是画在同一张纸上的
  • 合成层就是单独的透明胶片（图层）
  • 最终把这些胶片按顺序叠在一起，就是最终画面

好处：

  • 某一层变化了，只需要重绘这一层，其他层不受影响
  • 合成阶段可以交给 GPU 做，速度快
  • 动画只动某一层的位置/透明度，不需要重绘，性能非常好

哪些情况会形成独立的合成层？ ​

一个元素被提升为合成层，叫做「获得了自己的渲染层」。常见的触发条件：

  transform: translate3d(0,0,0);
  transform: rotateX(45deg);    /* 3D 变换 */
  perspective: 1000px;          /* 透视 */
}

3. video / canvas / iframe 等元素 ​

  • <video> 元素：视频解码和渲染需要单独的层
  • <canvas>：特别是 3D canvas（WebGL）
  • <iframe>：跨域 iframe 会有自己的渲染进程和层

固定定位的元素会独立成层（因为它不随页面滚动，需要单独处理）。

GPU 加速的原理 ​

什么是 GPU 加速？

  • 把本来应该由 CPU 做的渲染工作，交给 GPU 来做
  • GPU 特别擅长做「并行的、简单重复的计算」，比如图形处理

GPU 加速具体做了什么？

  1. 生成合成层：满足条件的元素被提升为独立的合成层
  2. 上传到 GPU：每个合成层的位图数据（纹理）上传到 GPU 显存
  3. GPU 合成：GPU 把多个纹理（层）按照正确的顺序和变换合成最终画面

哪些属性可以「只触发合成，不触发布局和绘制」？

最典型的两个：

  • transform：移动、缩放、旋转、斜切
  • opacity：透明度

为什么这两个属性性能好？

  • 它们不影响布局（不需要 Layout）
  • 它们不影响绘制（不需要重新 Paint 位图）
  • 只需要在合成阶段，GPU 对已有的纹理做变换或透明度调整
  • 整个过程不需要 CPU 参与，主线程完全不阻塞

为什么 transform 动画比 left/top 动画流畅？

css
/* ❌ 慢：每帧都触发 Layout + Paint + Composite */
@keyframes move-bad {
  from { left: 0; top: 0; }
  to { left: 100px; top: 100px; }
}

/* ✅ 快：只触发 Composite */
@keyframes move-good {
  from { transform: translate(0, 0); }
  to { transform: translate(100px, 100px); }
}

  • left/top 是布局属性，变化会触发 Layout → Paint → Composite 全流程
  • transform 在合成层上，变化只需要 GPU 移动纹理，不需要重新布局和绘制

GPU 加速的代价（坑） ​

GPU 加速不是银弹，用多了反而有问题：

1. 内存占用 ​

  • 每个合成层都要占用 GPU 显存
  • 层越多，显存占用越大
  • 移动端显存有限，层太多可能导致浏览器崩溃或页面被 kill
  • 特别是大尺寸的层（如全屏的层），一张 1920×1080 的 32 位位图就要 8MB 显存

2. 层爆炸（Layer Explosion） ​

什么是层爆炸？

  • 本来不需要那么多合成层，但因为各种原因（特别是重叠提升），产生了大量意料之外的合成层
  • 层的数量远远超出预期，导致内存暴涨、性能下降

最常见的原因：

  • 一个元素加了 transform: translateZ(0) 变成合成层
  • 所有叠在它上面的元素（z-index 更高的）也被自动提升为合成层
  • 列表里每一项都加了 translateZ(0)，结果几百个项变成几百个层

怎么发现层爆炸？

  • Chrome DevTools → More Tools → Layers（层面板）
  • 可以看到所有合成层、层级关系、内存占用
  • 或者用 Performance 面板录制，看「Layers」轨道

3. 重绘开销转移 ​

  • 把元素提升为合成层只是省了「每帧重绘」的开销
  • 但如果这个层本身经常需要重绘（比如内容一直变），那层的优势就不大了
  • 反而还要付出「多占一层内存」的代价

4. 闪烁 / 抖动问题 ​

  • 元素在「普通层」和「合成层」之间切换时，可能出现闪烁
  • 比如动画开始时才提升为层，第一帧会有卡顿
  • 所以动画前用 will-change 提前提升，可以避免闪烁
  • 但动画结束后要记得移除 will-change，不然一直占着层

5. 字体渲染差异 ​

  • GPU 渲染文字和 CPU 渲染文字效果可能略有不同
  • 特别是小字体和亚像素渲染（subpixel rendering），GPU 上可能稍模糊
  • 现在这个问题已经越来越不明显了

追问 1：为什么 transform 动画不会触发重排（reflow）？它的位置变化是怎么实现的？ ​

答 ​

这是一个非常核心的问题。要理解为什么 transform 不触发 reflow，我们需要先搞清楚「布局（Layout）」和「合成（Composite）」分别在什么层面。

Layout 是在 CPU 上计算几何信息：

  • Layout 阶段要计算每个元素在文档流中的精确位置和尺寸
  • 它要考虑盒模型、浮动、定位、flex、grid 等复杂的布局规则
  • 一个元素的布局变化可能影响其他元素（比如父元素高度、兄弟元素位置）
  • 所以 left / top / width / height 这些属性变化会触发 Layout，因为它们改变了「元素在布局流中的位置」

transform 是在合成层上做的 GPU 纹理变换：

  • 当一个元素被提升为合成层后，它的位图（纹理）已经上传到 GPU 了
  • transform 改变的不是元素在布局流中的位置，而是「这张纹理在屏幕上怎么显示」
  • 它不影响其他元素的布局，也不影响自己的布局盒子
  • GPU 只需要对这张已经存在的纹理做矩阵变换（平移、旋转、缩放等），就能显示出新的位置
  • 这个过程完全在 GPU 的合成阶段完成，不需要 CPU 参与布局计算

一个关键点：transform 不改变元素的「布局位置」

html
<div class="container">
  <div class="box"></div>
  <span>后面的文字</span>
</div>
1
2
3
4
css
.box { transform: translateX(100px); }
1
  • transform: translateX(100px) 只是视觉上把盒子向右移了 100px
  • 但在布局层面，盒子仍然在原来的位置
  • 后面的文字不会因为盒子移动而改变位置
  • 如果用 margin-left: 100px，后面的文字就会被挤走

这就是为什么 transform 不触发 reflow 的根本原因：它不改变布局树，只改变最终显示时的纹理变换。

具体实现过程：

  1. 初始状态：元素有一个布局位置，Paint 好后变成一张纹理上传到 GPU
  2. 应用 transform：浏览器把 transform 的矩阵信息告诉 GPU
  3. 每帧合成时：GPU 在合成最终画面时，对这个层的纹理应用变换矩阵
  4. 输出到屏幕：变换后的纹理和其他层一起合成显示

为什么这么快？

  • 纹理变换是 GPU 的看家本领，并行计算，极其高效
  • 不需要 CPU 重新计算布局
  • 不需要重新绘制位图（Paint 阶段跳过了）
  • 主线程完全空闲，可以处理 JS 和其他任务

延伸问题：transform 动画真的完全不占主线程吗？

  • 是的，如果只是纯 transform / opacity 动画，主线程几乎不参与
  • 但如果动画有 JS 监听（如 requestAnimationFrame 里做了其他事），那还是会占主线程
  • CSS 动画（@keyframes + animation）的好处就是可以完全跑在合成线程上，主线程阻塞了动画也能继续跑

补充：will-change 的作用就是提前告诉浏览器：「这个元素的 transform 要变了，提前给它提升成合成层，别等到动画开始才提升，那样第一帧会卡。」

追问 2：contain 属性是什么？它和合成层有什么关系？怎么用它做性能优化？ ​

答 ​

contain 是 CSS 的一个属性，用来告诉浏览器：「这个元素内部的变化不会影响外部，外部的变化也不会影响内部。」 浏览器知道这个信息后，就可以做很多优化，比如只重绘这个元素，不检查其他元素。

contain 的值：

值作用
none默认值，没有限制
size元素的尺寸不依赖内部内容，内部变化不影响外部尺寸
layout元素内部的布局不影响外部，外部布局也不影响内部
paint元素内部的绘制不会超出边界，内部重绘不需要重绘外部
style元素内部的样式变化不影响外部（目前支持有限）
strict等价于 contain: size layout paint style
content等价于 contain: layout paint style（除了 size）

最常用的组合是 contain: layout paint;（或 contain: content;）。

contain 怎么帮助性能优化？

1. 减少 Layout 范围

  • 没有 contain 时，一个元素的布局变化，浏览器可能要遍历整个渲染树确认影响范围
  • 有了 contain: layout，浏览器知道内部变化不会影响外部，只需要重新布局这个元素内部
  • 对于复杂的大页面，这个优化效果很明显

2. 减少 Paint 范围

  • 有了 contain: paint，浏览器知道元素内部的绘制不会溢出
  • 内部元素重绘时，只需要重绘这个元素的区域，不需要检查和重绘其他元素
  • 也减少了「层提升」的概率

3. 离屏元素不做绘制

  • 如果一个元素在视口外，且有 contain: paint，浏览器可能会跳过对它的绘制
  • 因为反正看不到，而且绘制内容也不会溢出影响可见区域

和合成层的关系：

  • contain 不等于 合成层，它是一个独立的优化手段
  • contain: paint 可能会让浏览器更容易把元素提升为合成层（因为知道边界明确）
  • 但主要还是在 Layout 和 Paint 阶段做优化，减少不必要的计算
  • 合成层是 Composite 阶段的优化，contain 是 Layout/Paint 阶段的优化，两者是互补的

实际应用场景：

① 虚拟列表（Virtual List）

css
.list-item {
  contain: layout paint; /* 每一项的变化不影响其他项 */
  /* 这样每一项的重绘/重排都局限在自己内部 */
}

② 独立的组件/卡片

css
.card {
  contain: layout paint style;
  /* 卡片内部的样式、布局变化都不会影响卡片外面 */
}

③ 大量独立的小部件

  • 页面有很多独立的 widget，每个 widget 内部可能变化
  • 给每个 widget 加 contain，浏览器可以并行或增量处理

注意事项：

1. contain: size 的副作用

  • contain: size 意味着元素的尺寸完全不依赖内容
  • 如果你没给元素设置固定的宽高，它可能变成 0×0
  • 所以 content（不含 size）比 strict 更常用

2. 不能滥用

  • 不是所有元素加了 contain 都更快
  • 浏览器本身有自己的优化策略，大部分时候已经很好了
  • 只在「确实独立、且频繁变化」的元素上加
  • 加之前最好用 Performance 面板测一下，确认有收益

3. 和 content-visibility 的配合

content-visibility 是一个更新的属性，和 contain 配合使用效果更好：

css
.article-section {
  content-visibility: auto;
  contain-intrinsic-size: 1000px; /* 预估高度，避免滚动条跳动 */
}
1
2
3
4

content-visibility: auto 的意思是：

  • 元素在视口外时，浏览器跳过它的布局和绘制
  • 滚动到视口内时，才开始渲染
  • 可以极大加快首屏渲染和长页面滚动性能
  • 需要配合 contain-intrinsic-size 给一个预估尺寸，不然滚动条会跳

总结一下：

  • 合成层：Composite 阶段的优化，适合动画、变换场景
  • contain：Layout/Paint 阶段的优化，适合独立的、频繁变化的组件
  • content-visibility：更进一步，离屏元素直接跳过渲染，适合长列表/长文章

追问 3：在移动端做渲染性能优化有哪些特殊注意点？移动端的 GPU 和桌面端有什么不同？ ​

答 ​

移动端的渲染性能优化和桌面端思路一致，但有更多限制条件，需要更谨慎。

移动端的特殊性：

1. GPU 显存有限 ​

  • 桌面端 GPU 显存有几 GB，移动端可能只有几百 MB 到 1-2GB（而且是和 CPU 共享内存）
  • 合成层太多 → 显存不够 → 浏览器开始丢层 / 页面卡顿 / 甚至崩溃
  • 一张 1080×1920 的全屏层就要 8MB RGBA 纹理，10 个就是 80MB，100 个就是 800MB

优化：

  • 严格控制合成层数量，尽量少用 translateZ(0) 强制提升
  • 动画结束后及时移除 will-change 和动画层
  • 列表场景避免每一项都独立成层
  • 用 Layers 面板检查层的数量和内存占用

2. CPU 性能弱 ​

  • 移动端 CPU 核心数少、频率低，单线程性能远不如桌面
  • 主线程稍微忙一点就掉帧
  • JS 执行时间长了会明显卡顿

优化：

  • 减少重排重绘：能用 transform 做的动画就不要用布局属性
  • 减少 JS 计算：大数据处理放 Web Worker
  • 拆分长任务：大的计算拆成小任务，用 requestIdleCallback 或 setTimeout 分片
  • 虚拟列表：长列表只渲染可见区域

3. 屏幕刷新率差异大 ​

  • 桌面端一般 60Hz，很多游戏显示器 144Hz 甚至更高
  • 移动端低端机可能还是 60Hz，但旗舰机已经有 120Hz / 144Hz
  • 高刷屏上，如果你的动画只有 60fps，用户会觉得明显不够流畅

优化：

  • 动画不要写死 16.7ms 一帧，用 requestAnimationFrame 自动适配刷新率
  • 用 CSS 动画（跑在合成线程）比 JS 动画更能适配高刷
  • 关键交互（如滚动、动画）要做到 120fps 才够旗舰机的体验

4. 触摸事件与滚动性能 ​

  • 移动端主要交互是触摸和滑动
  • 滚动性能直接影响用户体验
  • touchstart / touchmove 事件处理不好会导致滚动卡顿

优化：

  • 滚动容器加 overflow-y: auto; -webkit-overflow-scrolling: touch;（惯性滚动）
  • 不要在 touchmove 里做重的计算，能放 passive 就加 passive: true
    js
    element.addEventListener('touchmove', handler, { passive: true });
    1
    • passive: true 告诉浏览器：这个监听器不会调用 preventDefault()，浏览器可以不用等 JS 执行完就滚动
    • 大幅提升滚动流畅度
  • 列表项加 contain: layout paint;，减少滚动时的重绘

5. 内存限制严格 ​

  • 移动端浏览器的内存配额比桌面端小很多
  • 页面占内存太多会被后台杀掉
  • JS 堆、DOM 节点数、图片内存都有更大压力

优化：

  • 及时清理不需要的 DOM 和事件监听
  • 图片懒加载，不在视口的不加载
  • 大图片注意解码开销，用 decode() 异步解码
  • WebGL / Canvas 场景注意及时释放纹理

6. 浏览器兼容性和实现差异 ​

  • 移动端主要是 Safari 和 Chrome（以及各种 Chromium 内核的浏览器）
  • Safari 有很多独有的特性和 bug：
    • Safari 的合成层触发条件和 Chrome 不一样
    • Safari 对 position: fixed 的处理有特殊问题
    • Safari 的 backface-visibility 行为不同
    • iOS 上所有浏览器都是 WebKit 内核（因为苹果的政策）

优化：

  • 不要依赖特定浏览器的渲染行为
  • 在真机上测试，特别是 iOS Safari
  • 慎用一些「奇技淫巧」的 GPU 加速 hack

7. 电量消耗 ​

  • 移动端电池容量有限
  • 持续的 GPU 加速、高频动画会很耗电
  • 长时间跑动画的页面，用户手机会发烫

优化：

  • 页面不可见时（visibilitychange）暂停动画和定时器
  • 减少持续动画的数量
  • 不要用 JS 做死循环动画

移动端性能排查工具： ​

  1. Chrome DevTools 远程调试：USB 连手机，在电脑上调试手机里的 Chrome
  2. Safari Web Inspector：Mac 上调试 iOS 设备上的 Safari
  3. Performance 面板：看帧率、主线程占用
  4. Layers 面板：查合成层数量和内存
  5. Memory 面板：查内存占用
  6. 真机测试：模拟器性能和真机差很多，一定要在真机上测

总结移动端优化优先级：

  1. 减少 JS 执行时间（主线程最容易堵）
  2. 控制合成层数量（显存有限，防层爆炸）
  3. 滚动性能优化（用户感知最强）
  4. 减少重排重绘（CPU 弱，尽量走合成）
  5. 内存管理（防止被杀后台）
