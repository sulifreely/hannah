问答 27（进阶）：SSE vs WebSocket vs 长轮询：服务端推送技术对比 ​

题目 ​

服务端向浏览器推送数据有哪些方案？SSE、WebSocket 和长轮询（Long Polling）分别是什么原理？各有什么优缺点？适用场景有什么不同？心跳机制和断线重连策略分别怎么设计？

参考答案 ​

服务端推送（Server Push）是 Web 实时通信的核心问题。HTTP 协议天生是「请求-响应」模式，服务端不能主动给客户端发消息。为了突破这个限制，业界发展出了多种方案。

三种方案原理对比 ​

方案原理连接方式方向协议
长轮询（Long Polling）客户端发请求，服务端 hold 住直到有数据或超时才返回，返回后客户端立即重发短连接（每次请求一个完整 HTTP）服务端→客户端（单向）HTTP
SSE（Server-Sent Events）客户端建立 HTTP 连接，服务端持续往响应里写数据，连接不关闭长连接（一个 HTTP 连接持续写）服务端→客户端（单向）HTTP + text/event-stream
WebSocket通过 HTTP Upgrade 握手，切换成 WebSocket 帧协议，双向通信长连接（一条 TCP 连接）双向（全双工）ws:// 或 wss://

1. 长轮询（Long Polling） ​

原理：

  • 客户端发起一个 HTTP GET 请求
  • 服务端收到请求后，不立即返回，而是 hold 住连接，等有数据了再返回
  • 如果超时了还没数据，返回一个空响应（如 304 或 204）
  • 客户端收到响应后，立刻再发起下一个请求
  • 如此循环往复

示意图：

客户端: GET /events → 服务端 (等待数据...)
客户端: ← 数据A     服务端 (有数据了，返回)
客户端: GET /events → 服务端 (继续等待...)
客户端: ← 数据B     服务端 (有数据了，返回)
...

优点：

  • 兼容性最好：所有浏览器都支持，所有代理服务器都支持
  • 实现简单：服务端和客户端都很容易写
  • 不需要特殊协议：就是普通 HTTP

缺点：

  • 每次消息都要一次完整的 HTTP 请求：Header 开销大（几百字节）
  • 服务端压力大：每个客户端都占一个请求连接，并发高时服务端连接数爆炸
  • 延迟较高：每次返回后要重新发请求，有一次 RTT 的间隔
  • 资源浪费：频繁建连断连，TCP 握手、TLS 握手都要重复做

代码示例（简化版）：

js
// 客户端
async function longPoll() {
  try {
    const response = await fetch('/events?lastId=' + lastEventId);
    const data = await response.json();
    if (data.messages.length > 0) {
      // 处理消息
      data.messages.forEach(handleMessage);
      lastEventId = data.lastId;
    }
  } catch (e) {
    // 网络错误，等一会再重连
    await sleep(1000);
  }
  longPoll(); // 继续轮询
}

2. SSE（Server-Sent Events） ​

原理：

  • 客户端用 EventSource 对象建立 HTTP 连接
  • 服务端返回 Content-Type: text/event-stream 的响应
  • 连接保持打开状态，服务端可以持续往里面写数据
  • 数据是按「事件」格式推送的，每个事件有 id、type、data 字段
  • 浏览器原生支持自动重连（断线后自动重连，并带上 Last-Event-ID）

SSE 数据格式：

: 这是注释行，可以用来做心跳

id: 123
event: message
data: 这是第一条消息

id: 124
event: notification
data: {"type": "like", "from": "alice"}

data: 多行数据
data: 第二行
data: 第三行

  • 每个事件之间用空行分隔
  • id：事件 ID，断线重连时会放在 Last-Event-ID header 里
  • event：事件类型，默认是 message
  • data：事件数据，可以多行
  • retry：告诉浏览器断线后多少毫秒重连
  • : 开头的是注释，常用来做心跳（防止连接超时被中间设备断开）

代码示例：

js
// 客户端
const es = new EventSource('/events');

// 监听默认 message 事件
es.addEventListener('message', (e) => {
  console.log('收到消息:', e.data);
  console.log('事件ID:', e.lastEventId);
});

// 监听自定义事件
es.addEventListener('notification', (e) => {
  const data = JSON.parse(e.data);
  console.log('通知:', data);
});

// 连接打开
es.onopen = () => console.log('SSE 连接已建立');

// 连接错误（断线后会自动重连）
es.onerror = (e) => {
  if (es.readyState === EventSource.CLOSED) {
    console.log('连接被关闭');
  } else {
    console.log('连接出错，正在自动重连...');
  }
};

// 关闭连接
// es.close();

# 服务端（Python/Flask 风格，示意）
@app.route('/events')
def events():
    last_id = request.headers.get('Last-Event-ID', '0')
    
    def generate():
        # 先发送一个重试间隔
        yield 'retry: 3000\n\n'
        
        while True:
            messages = get_new_messages_since(last_id)
            for msg in messages:
                yield f'id: {msg["id"]}\n'
                yield f'event: {msg["type"]}\n'
                yield f'data: {json.dumps(msg["data"])}\n\n'
                last_id = msg["id"]
            
            # 每 15 秒发一个心跳注释，防止连接超时
            if not messages:
                yield ': ping\n\n'
            
            time.sleep(1)
    
    return Response(generate(), mimetype='text/event-stream')

优点：

  • 原生支持：浏览器原生 EventSource API，非常简单
  • 自动重连：浏览器内置断线重连，还会带上 Last-Event-ID，天然支持断线续传
  • 事件语义：内置事件类型，多种消息分类方便
  • 基于 HTTP：走普通 HTTP/HTTPS，代理、CDN、WAF 都支持
  • 单向推送高效：一条连接持续推，没有长轮询的重复建连开销

缺点：

  • 单向通信：只能服务端推给客户端，客户端发消息还要另发 HTTP 请求
  • IE 不支持：老 IE 没有 EventSource（可以 polyfill）
  • HTTP/1.1 下有连接数限制：同一个域名最多 6 个 SSE 连接（HTTP/2 没有这个问题）
  • 只能发文本：原生只支持文本数据，二进制需要编码（如 base64）

3. WebSocket ​

原理：

  • 先通过 HTTP 做一次握手（Upgrade），把连接从 HTTP 切换成 WebSocket 协议
  • 握手成功后，同一条 TCP 连接上可以双向发送数据帧
  • 数据帧很小（头部只有 2-14 字节），开销极低
  • 支持文本帧和二进制帧

const ws = new WebSocket('wss://example.com/chat');

};

  console.log('连接关闭:', event.code, event.reason);
  // 需要自己实现重连

优点：

  • 双向全双工：客户端和服务端随时可以发消息，真正的实时双向通信
  • 开销小：帧头只有几个字节，比 HTTP 高效得多
  • 支持二进制：可以直接传 ArrayBuffer / Blob，适合游戏、音视频等场景
  • 实时性高：消息到达立即处理，没有轮询延迟

缺点：

  • 服务端复杂：长连接管理、扩容、负载均衡都比 HTTP 难
  • 中间设施支持差：一些代理、WAF、CDN 对 WebSocket 支持不好，可能会超时断开
  • 没有内置重连：断线重连、心跳都要自己实现
  • HTTP/2 支持差：WebSocket over HTTP/2（RFC 8441）浏览器支持有限
  • 状态管理复杂：连接状态、会话恢复、消息丢失处理都要自己考虑

适用场景选型 ​

选长轮询：

  • 兼容性要求极高（要支持特别老的浏览器）
  • 消息频率很低（几分钟一条）
  • 实现简单快速，不想引入复杂度
  • 网络环境复杂，不确定中间设备是否支持长连接

选 SSE：

  • 只需要服务端单向推送（通知、消息、实时数据更新）
  • 想要简单的 API，不想自己处理重连逻辑
  • 走 HTTP 就行，不需要双向通信
  • 例如：股票行情推送、新闻通知、日志实时输出、在线人数统计

选 WebSocket：

  • 需要双向实时通信（聊天、协同编辑、游戏）
  • 高频消息（每秒很多条）
  • 需要传二进制数据（音视频、文件传输）
  • 低延迟要求高（实时互动）
  • 例如：即时通讯、在线协作、实时游戏、白板共享

心跳机制与断线重连 ​

为什么需要心跳？

  • 中间设备（Nginx、负载均衡、防火墙）有空闲超时，连接长时间没数据会被断开
  • 有时候连接已经断了，但两端都不知道（「半开连接」）
  • 心跳可以检测连接是否还活着

1. SSE 的心跳

SSE 比较简单，用注释行做心跳：

python
# 服务端每 15 秒发一个注释行
yield ': ping\n\n'
1
2
  • 浏览器收到注释行不会触发 message 事件，但能保持连接活跃
  • 同时可以用 retry 字段告诉浏览器重连间隔

2. WebSocket 的心跳

WebSocket 协议本身有 ping / pong 控制帧：

  • 服务端发 ping，客户端自动回 pong（浏览器自动处理，JS API 看不到）
  • 但浏览器 JS 不能主动发 ping，一般在应用层实现心跳
js
// 客户端心跳实现
let heartbeatTimer = null;
let heartbeatTimeout = null;
const HEARTBEAT_INTERVAL = 30000; // 30秒发一次
const HEARTBEAT_TIMEOUT = 10000;  // 10秒没收到回复算超时

function startHeartbeat() {
  heartbeatTimer = setInterval(() => {
    ws.send(JSON.stringify({ type: 'ping' }));
    
    heartbeatTimeout = setTimeout(() => {
      // 超时没收到 pong，连接可能死了
      console.log('心跳超时，断开重连');
      ws.close();
      reconnect();
    }, HEARTBEAT_TIMEOUT);
  }, HEARTBEAT_INTERVAL);
}

// 收到消息时清除超时
ws.onmessage = (e) => {
  const data = JSON.parse(e.data);
  if (data.type === 'pong') {
    clearTimeout(heartbeatTimeout);
    return;
  }
  // 处理其他消息...
};

3. 断线重连策略

指数退避 + 随机抖动：

js
let reconnectDelay = 1000; // 初始 1 秒
const MAX_RECONNECT_DELAY = 30000; // 最大 30 秒

function reconnect() {
  console.log(`准备在 ${reconnectDelay}ms 后重连...`);
  
  setTimeout(() => {
    console.log('正在重连...');
    connect(); // 重新建立连接
    
    // 指数退避
    reconnectDelay = Math.min(reconnectDelay * 2, MAX_RECONNECT_DELAY);
    // 加随机抖动，避免所有客户端同时重连打爆服务器
    reconnectDelay += Math.random() * 1000;
  }, reconnectDelay);
}

// 连接成功后重置延迟
function onConnect() {
  reconnectDelay = 1000;
  startHeartbeat();
}

追问 1：SSE 和 WebSocket 在 HTTP/2 下分别表现如何？HTTP/2 的 Server Push 和它们是一回事吗？ ​

答 ​

SSE 在 HTTP/2 下：

SSE 在 HTTP/1.1 时有个限制：浏览器对同一个域名的并发连接数有限制（Chrome 是 6 个），而 SSE 会占一个 HTTP 连接，所以你最多只能开 6 个 SSE 连接（而且还占了其他请求的名额）。

HTTP/2 解决了这个问题：

  • HTTP/2 是多路复用的，一个域名只需要一条 TCP 连接
  • SSE 作为 HTTP/2 的一个 stream，不占用额外连接
  • 可以开很多个 SSE 流，不会被连接数限制

但注意：SSE 本身还是单向的，只是连接数限制没了。

WebSocket 在 HTTP/2 下：

WebSocket over HTTP/2（RFC 8441）是一个较新的标准：

  • 允许 WebSocket 握手通过 HTTP/2 的 stream 进行
  • 握手后的数据传输还是 WebSocket 帧格式
  • 好处是可以复用 HTTP/2 的连接，不需要额外建 TCP 连接
  • 但浏览器支持还不普遍（Chrome 支持但需要 flag，Firefox 部分支持）
  • 实际生产中大部分 WebSocket 还是走 HTTP/1.1 Upgrade

HTTP/2 Server Push 和它们完全不是一回事：

HTTP/2 Server Push 是指：

  • 客户端请求 A 资源时，服务端预判客户端还需要 B 资源，主动把 B 也推过去
  • 这是针对「资源预加载」的优化，不是实时消息推送
  • 推送的资源会进入浏览器缓存，等 HTML 解析到需要 B 时直接从缓存读
  • 目的是减少 RTT，加快页面加载

对比一下：

维度HTTP/2 Server PushSSE / WebSocket
目的预加载资源，加速页面加载实时消息推送、通信
时机服务端响应请求时顺便推任意时间持续推
内容静态资源（CSS、JS、图片）业务数据、消息
持续性一次性（请求响应级）持续（长连接级）
触发方服务端主动，但基于当前请求服务端随时主动推
使用场景首屏资源预加载聊天、通知、实时数据

简单说：

  • HTTP/2 Server Push 是「你点了一份饭，我顺便把筷子勺子也给你」——减少你要的次数
  • SSE/WebSocket 是「我给你开个窗口，有消息随时给你递」——持续不断地给

补充：HTTP/2 Server Push 现在的状态

  • 实际上 HTTP/2 Server Push 用得不多，因为很难准确预判客户端需要什么
  • 预判错了反而浪费带宽
  • Chrome 甚至在考虑移除对它的支持
  • 更实用的方案是 103 Early Hints（提前给浏览器提示要预加载哪些资源）

追问 2：WebSocket 和 WebRTC 的 DataChannel 有什么区别？分别适用于什么场景？ ​

答 ​

这是个很深入的问题。WebRTC DataChannel 也是浏览器里的双向数据通道，但和 WebSocket 定位完全不同。

WebRTC DataChannel 是什么？

  • WebRTC 是浏览器之间直接通信（P2P） 的技术
  • DataChannel 是 WebRTC 的一部分，用于在两个浏览器之间直接传数据
  • 不需要经过服务器（或只需要服务器做信令和打洞）
  • 底层是 UDP（可配置为可靠/不可靠、有序/无序）

对比：

维度WebSocketWebRTC DataChannel
连接方式客户端 ↔ 服务器（C/S）浏览器 ↔ 浏览器（P2P）
传输协议TCP（可靠、有序）UDP（可配置）
经过服务器所有数据都经过服务器数据直接走 P2P，不经过服务器
延迟低（经过服务器转发）极低（直连）
NAT 穿透不需要（服务端有公网 IP）需要（STUN / TURN 打洞）
建立复杂度简单（一次 HTTP 握手）复杂（信令 + ICE + DTLS + SCTP）
可靠性可靠、有序可配置：可靠有序 / 不可靠无序
浏览器支持好好（现代浏览器都支持）
服务端成本高（要扛所有流量）低（只做信令，流量不走服务器）

WebRTC DataChannel 的使用场景：

  1. 文件传输：大文件直接 P2P 传，不走服务器，省带宽
  2. 实时游戏：低延迟要求极高，用不可靠模式，丢几帧没关系
  3. 视频会议辅助数据：配合音视频，传白板、聊天消息
  4. 协同编辑：低延迟同步，P2P 架构
  5. P2P 文件共享：类似 BitTorrent 的浏览器版

WebSocket 的使用场景：

  1. 即时通讯：聊天、通知（需要服务端存历史、做推送）
  2. 实时数据推送：股票、行情、新闻
  3. 协同编辑（C/S 架构）：服务端做冲突解决和持久化
  4. 在线游戏（C/S 架构）：需要服务端做权威判定
  5. 需要服务端介入的场景：鉴权、风控、数据持久化

怎么选：

  • 如果你的数据需要经过服务端处理（存数据库、做审核、推送给所有人）→ WebSocket
  • 如果数据就是两个浏览器之间的事，不需要服务端经手 → WebRTC DataChannel
  • 如果是一对多的广播 → 一般 WebSocket 更简单（WebRTC 要用 SFU 或 mesh 架构）
  • 如果追求最低延迟、最大吞吐量、且 P2P 可行 → WebRTC DataChannel

补充：WebTransport（未来趋势）

WebTransport 是一个新的 API，基于 HTTP/3（QUIC）：

  • 支持双向流、单向流、数据报（不可靠）
  • 基于 QUIC，有 0-RTT、连接迁移等特性
  • 是客户端-服务器模式（不是 P2P）
  • 可以理解为「WebSocket + 更底层的能力」，支持不可靠传输
  • 目前还在标准化和实现中，未来可能在很多场景替代 WebSocket

追问 3：在生产环境中使用 WebSocket，如何做负载均衡和横向扩容？ ​

答 ​

WebSocket 的横向扩容比普通 HTTP 服务难很多，核心问题是：WebSocket 是长连接，用户连在某台机器上，给这个用户推消息必须找到他所在的机器。

核心挑战：

  1. 连接的粘性（Sticky）：用户连在机器 A 上，后续消息要发到机器 A
  2. 广播/房间消息：给一个房间的所有人发消息，这些人可能分散在不同机器
  3. 连接数限制：单机连接数有上限（文件描述符、内存）
  4. 平滑扩缩容：新增/下线机器时，连接怎么迁移

常见的架构方案：

方案一：负载均衡 + 消息总线（最常用） ​

工作流程：

  1. 负载均衡层：

    • 四层负载均衡（L4）：按 IP 哈希，同一用户始终落到同一台机器（IP Hash）
    • 七层负载均衡（L7）：按 Cookie 或 URL 参数做粘性会话（Sticky Session）
    • 注意：WebSocket 握手是 HTTP，之后是长连接，LB 需要支持 WebSocket Upgrade

  2. WS 节点：

    • 维护本机的所有 WebSocket 连接
    • 客户端发消息 → 本节点处理 → 要发给其他人的消息 → 发到消息总线
    • 从消息总线订阅消息 → 收到后找到本机上对应的连接 → 推送出去

  3. 消息总线：

    • 所有节点订阅「全局频道」或「房间频道」
    • 某个节点要广播消息时，往消息总线发
    • 所有节点都收到，然后只推给本机上的用户

具体实现（Redis Pub/Sub 风格）：

js
// 每个 WS 节点
const redis = new Redis();
const pubSub = new Redis();

// 订阅全局消息频道
pubSub.subscribe('global-channel', (err) => {
  if (err) console.error('订阅失败', err);
});

// 收到总线消息时，推给本机连接的用户
pubSub.on('message', (channel, message) => {
  const data = JSON.parse(message);
  // 找到目标用户在本机上的连接，推出去
  const ws = connections.get(data.toUserId);
  if (ws) {
    ws.send(data.payload);
  }
});

// 客户端发来消息
ws.on('message', (message) => {
  const data = JSON.parse(message);
  
  if (data.type === 'chat') {
    // 要发给其他人，发到消息总线
    redis.publish('global-channel', JSON.stringify({
      toUserId: data.to,
      payload: data.payload
    }));
  }
});

消息总线选型：

  • Redis Pub/Sub：简单、低延迟，适合中小规模。缺点：消息不持久化，节点断线会丢消息
  • Kafka：高吞吐、持久化、可回溯，适合大规模。缺点：延迟稍高，运维复杂
  • NATS：高性能、轻量，支持多种模式。云原生场景常用
  • RabbitMQ：功能丰富，路由灵活。传统企业级场景

方案二：基于用户路由的网关层 ​

在 LB 和 WS 节点之间加一个「网关层」或「路由层」：

  • 网关知道每个用户在哪台机器上
  • 外部消息先到网关，网关查路由表，转发到对应机器
  • 适合消息量很大、需要做很多接入层逻辑的场景

方案三：P2P / Mesh 架构 ​

节点之间直接互连，形成网状：

  • 每个节点和其他所有节点都有连接
  • 要给别人发消息，直接在节点间转发
  • 适合节点数不多（几十台以内）的场景
  • 缺点：节点数多了之后，节点间连接数是 O(n²)

常见问题处理： ​

1. 连接数限制

  • 单机连接数上限受限于：文件描述符（ulimit -n）、内存、CPU
  • 一台普通机器大概能扛 1-10 万 WebSocket 连接（看业务复杂度）
  • 优化：调内核参数、用高性能 IO 框架（Netty、uWebSockets、gnet）、内存优化

2. 心跳与超时

  • LB / Nginx 有空闲超时，必须用心跳保持连接活跃
  • 一般 30 秒一次心跳，短于中间设备的超时时间

3. 平滑下线（Draining）

  • 要下线一台机器时：
    • 先从 LB 摘流（不再接受新连接）
    • 给所有连接发一个「即将关闭」的通知
    • 等连接自然断开或主动断掉（比如等 30 秒）
    • 再关闭进程
  • 客户端收到关闭通知后，主动重连到其他机器

4. 断线重连与消息可靠

  • 重连时可能切换到不同机器
  • 需要机制确保消息不丢：
    • 消息带序号 / ID
    • 客户端重连时带上 lastId，服务端补发
    • 重要消息走服务端持久化，不依赖 WebSocket 通道本身

5. 多区域 / 多活

  • 跨区域部署时，用户就近接入
  • 消息总线跨区域同步（如 Redis 主从、Kafka MirrorMaker）
  • 或者用全局路由服务，确保同房间的用户在同一个区域

总结：

  • 小流量：单机就够了，不需要考虑扩容
  • 中流量：Nginx + 多台 WS 节点 + Redis Pub/Sub，最常见也最实用
  • 大流量：专业的消息队列 + 网关层 + 精细化运维
  • 超大规模：考虑用专门的 IM 云服务，或自研分布式消息平台
