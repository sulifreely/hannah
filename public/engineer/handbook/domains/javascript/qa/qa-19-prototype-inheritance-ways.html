问答 19（进阶）：原型链与继承的七种实现方式全解析 ​

题目 ​

请系统梳理 JavaScript 中实现继承的七种方式：原型链继承、构造函数继承、组合继承、原型式继承、寄生式继承、寄生组合式继承、ES6 class extends。说明每种的实现思路、核心代码、优缺点，以及它们之间的演进关系。

参考答案 ​

先铺垫一个核心认知：JS 的继承本质上是原型链的构造游戏——让子类型的原型（Child.prototype）能通过 __proto__ 一路查到父类型的原型方法，同时实例属性要能各自独立不共享。

我们从最朴素的方式一步步演进到最优解。

方式一：原型链继承 ​

思路：把子类的 prototype 直接指向父类的一个实例。

js
function Parent(name) {
  this.name = name;
  this.hobbies = ['reading', 'music'];
}
Parent.prototype.sayHi = function () {
  console.log('Hi, I am ' + this.name);
};

function Child() {}
Child.prototype = new Parent('Tom');   // 核心：子类原型 = 父类实例

const c1 = new Child();
const c2 = new Child();
c1.hobbies.push('coding');
console.log(c2.hobbies);   // ['reading', 'music', 'coding']  ← 共享了！

优点：实现简单，父类原型方法能被子类继承。

缺点：

  1. 引用类型属性被所有实例共享（hobbies 改一个全变）。
  2. 创建子类实例时不能给父构造函数传参（name 写死成 'Tom'）。
  3. 父类构造函数在设置原型时就执行了一次，不合理。

方式二：构造函数继承（借用 call/apply） ​

思路：在子类构造函数里调用 Parent.call(this, ...args)，把父类的属性拷贝到子类实例上。

js
function Parent(name) {
  this.name = name;
  this.hobbies = ['reading', 'music'];
}
Parent.prototype.sayHi = function () { /* ... */ };

function Child(name, age) {
  Parent.call(this, name);   // 核心：借父构造函数给 this 挂属性
  this.age = age;
}

const c1 = new Child('Alice', 18);
const c2 = new Child('Bob', 20);
c1.hobbies.push('coding');
console.log(c2.hobbies);   // ['reading', 'music']  ← 独立了
c1.sayHi();                 // TypeError: c1.sayHi is not a function

优点：

  1. 每个实例的属性独立，不会共享引用类型。
  2. 可以给父构造函数传参。

缺点：父类原型上的方法继承不到——sayHi 挂在 Parent.prototype 上，子类跟它没有原型链关系。每个实例都要执行一遍父构造函数，方法也无法复用。

方式三：组合继承（原型链 + 构造函数） ​

思路：用构造函数继承属性，用原型链继承方法，两者组合。

js
function Parent(name) {
  this.name = name;
  this.hobbies = ['reading', 'music'];
}
Parent.prototype.sayHi = function () {
  console.log('Hi, ' + this.name);
};

function Child(name, age) {
  Parent.call(this, name);        // ① 属性继承（第二次调用 Parent）
  this.age = age;
}
Child.prototype = new Parent();   // ② 方法继承（第一次调用 Parent）
Child.prototype.constructor = Child;

const c = new Child('Alice', 18);
c.sayHi();                        // Hi, Alice

缺点：父类构造函数被调用了两次。第一次 new Parent() 往 Child.prototype 上也挂了一份 name 和 hobbies，虽然实例属性会把它"遮住"，但原型上多了一份无用的属性，浪费内存。

方式四：原型式继承 ​

思路：不自定义构造函数，直接以一个已有对象为原型，创建新对象。本质是"浅复制一个对象作为原型"。

js
// 经典实现：道格拉斯·克罗克福德提出的 object() 函数
function object(o) {
  function F() {}
  F.prototype = o;
  return new F();
}

const person = {
  name: 'Tom',
  hobbies: ['reading', 'music'],
};

const p1 = object(person);
const p2 = object(person);
p1.hobbies.push('coding');
console.log(p2.hobbies);   // ['reading', 'music', 'coding']  ← 还是共享

ES5 把这个思路标准化成了 Object.create(o)，就是上面 object() 的规范版，还支持第二个参数定义额外属性。

适用场景：你有一个基础对象，想在它的基础上"再加点东西"，又不想大动干戈写构造函数。

缺点：跟原型链继承一样——引用类型属性共享、不能传参初始化。

方式五：寄生式继承 ​

思路：在原型式继承的基础上，用一个函数"包装"一下，在里面给新对象增强方法，然后返回。

js
function createPerson(original) {
  const clone = Object.create(original);   // 原型式继承
  clone.sayHi = function () {              // 增强
    console.log('Hi, ' + this.name);
  };
  return clone;
}

const person = { name: 'Tom', hobbies: ['reading'] };
const p = createPerson(person);
p.sayHi();   // Hi, Tom

缺点：方法仍然挂在实例上而不是原型上，无法复用；引用类型共享的问题依然存在。

方式六：寄生组合式继承（ES5 最优方案） ​

思路：组合继承的问题是"父构造函数调用了两次"。那我们能不能不调用父构造函数，也能拿到父原型？

用 Object.create(Parent.prototype) 直接以父原型为模板创建一个空对象，赋给 Child.prototype——这样既建立了原型链，又不会执行 Parent。

js
function Parent(name) {
  this.name = name;
  this.hobbies = ['reading', 'music'];
}
Parent.prototype.sayHi = function () { /* ... */ };

function Child(name, age) {
  Parent.call(this, name);          // 属性继承（调用一次 Parent）
  this.age = age;
}

// 核心：用 Object.create 搭桥，不调用 Parent
Child.prototype = Object.create(Parent.prototype);
Child.prototype.constructor = Child;

const c = new Child('Alice', 18);
c.sayHi();

方式七：ES6 class extends ​

语法糖，但不只是语法糖——底层仍然是原型链，但多了一些新特性。

js
class Parent {
  constructor(name) {
    this.name = name;
  }
  sayHi() { console.log('Hi, ' + this.name); }
  static create(name) { return new this(name); }   // 静态方法
}

class Child extends Parent {
  constructor(name, age) {
    super(name);                                    // 必须先调 super
    this.age = age;
  }
  sayHi() {
    super.sayHi();                                  // 调用父类方法
    console.log('I am ' + this.age);
  }
}

extends 做了什么 ​

class Child extends Parent 实际上设置了两条原型链：

1. 实例方法链：Child.prototype.__proto__ === Parent.prototype
2. 静态方法链：Child.__proto__ === Parent
1
2

所以子类既能继承父类的实例方法（通过实例查原型链），也能继承父类的静态方法（通过类本身查原型链）。ES5 手写继承时经常漏掉第二条。

super 的两种身份 ​

  • super(...) 作为函数调用：只能在子类 constructor 里，调用父类构造函数，必须在使用 this 之前。
  • super.method() 作为对象调用：在原型方法中指代父类原型对象，在静态方法中指代父类本身。

class 相比 ES5 继承的额外能力 ​

  1. 静态成员继承（Child.__proto__ === Parent）。
  2. super 关键字，语法层面支持调用父类方法/构造器。
  3. 类声明不可枚举、必须 new 调用，更像传统 OOP 语言。
  4. extends 可以继承内建对象（Array、Error、Date 等），ES5 手写继承 Array 会有问题（length 行为异常）。
  5. 存在 TDZ，不会像函数声明那样整体提升。

追问 1：为什么 ES5 手写继承 Array 会失效？class extends Array 为什么能正常工作？ ​

答 ​

ES5 继承内建对象（比如 Array、Error）有个经典坑：

js
function MyArray() {
  Array.apply(this, arguments);   // 借用构造函数
}
MyArray.prototype = Object.create(Array.prototype);
MyArray.prototype.constructor = MyArray;

const arr = new MyArray(1, 2, 3);
console.log(arr.length);   // 0  ← 不对！应该是 3
arr[0] = 1;
console.log(arr.length);   // 0  ← length 完全不响应

原因：数组的 length 属性、索引行为不是简单地挂在实例属性和原型上的，而是由引擎内部的 [[DefineOwnProperty]] 等内部方法特殊处理的。这些行为跟对象的**内部槽（internal slot）**有关。ES5 用 Array.apply(this, ...) 的方式只是"借"了一下构造函数，但 this 是普通对象，不是真正的数组实例，没有那些内部槽，所以 length 不工作。

class extends Array 为什么行？

ES6 的 class extends 在语义上做了调整：父类构造函数负责创建实例对象（分配内部槽），子类再去增强。也就是说，super() 调用时，父类 Array 创建的是一个真正的数组实例（有 length 的内部实现），然后 this 被返回给子类继续用。所以 length、索引这些数组特性都正常。

这也是为什么 ES6 class 的子类 constructor 里必须先调 super() 才能用 this——因为 this 是父类构造函数创建的。ES5 的思路是"先创建子类 this，再把父属性拷过来"，对内建对象行不通。

追问 2：如何判断一个对象是"自有属性"还是"继承来的属性"？至少说出三种方法并对比。 ​

答 ​

方法一：hasOwnProperty ​

js
obj.hasOwnProperty('key');   // 只查自身，不看原型链
1

返回 true 表示是自有属性。注意：如果对象是 Object.create(null) 创建的，它没有 hasOwnProperty 方法，直接调用会报错。

方法二：Object.hasOwn（ES2022） ​

js
Object.hasOwn(obj, 'key');
1

跟 hasOwnProperty 一样，但更安全——Object.create(null) 的对象也能用，即使对象自己覆盖了 hasOwnProperty 也不受影响。现代代码推荐用这个。

方法三：in 操作符 + hasOwn 组合 ​

js
'key' in obj;                    // 沿原型链查，找到就 true
Object.hasOwn(obj, 'key');       // 只查自身
1
2

如果 in 返回 true 但 hasOwn 返回 false，说明是继承来的属性。

方法四：Object.getOwnPropertyDescriptor ​

js
Object.getOwnPropertyDescriptor(obj, 'key');
1

返回属性描述对象（value、writable、enumerable、configurable），如果是继承属性则返回 undefined。这能顺便拿到属性的元信息。

对比 ​

方法查原型链支持 Object.create(null)额外信息
obj.hasOwnProperty❌❌（自身没这方法）-
Object.hasOwn❌✅-
'x' in obj✅✅-
Object.getOwnPropertyDescriptor❌✅描述符

额外提：for...in 会遍历自身 + 原型链上所有可枚举属性；Object.keys/values/entries 只遍历自身可枚举属性。

追问 3：什么是"原型污染"？怎么防范？ ​

答 ​

什么是原型污染 ​

原型污染（Prototype Pollution）是一种攻击方式：攻击者通过某种手段（比如用户输入的 key）给 Object.prototype 上添加/修改属性，而所有普通对象都继承自 Object.prototype，于是全局都受影响。

典型场景：把用户输入的对象递归 merge 到目标对象上，而没有过滤 __proto__ 这个 key。

js
function merge(target, source) {
  for (const key in source) {
    if (typeof source[key] === 'object' && source[key] !== null) {
      target[key] = target[key] || {};
      merge(target[key], source[key]);
    } else {
      target[key] = source[key];
    }
  }
}

// 攻击者构造这样的输入：
const payload = JSON.parse('{"__proto__": {"isAdmin": true}}');
merge({}, payload);

console.log({}.isAdmin);   // true  ← 所有对象都有了 isAdmin！

当 key === '__proto__' 时，target[key] 拿到的就是 target.__proto__，也就是 Object.prototype，然后往上面写 isAdmin: true，就污染了全局原型。

防范手段 ​

  1. 用 Object.create(null) 创建字典对象：没有原型，污染不了。
  2. 过滤危险 key：__proto__、constructor、prototype，遇到就跳过。
  3. 用 Object.hasOwn 检查 + Object.defineProperty 设置自身属性，不走原型链。
  4. 用 Map 代替对象当字典：Map 的 key 不会走原型链。
  5. JSON 解析时用 reviver 过滤：
js
const safe = JSON.parse(json, (key, value) => {
  if (key === '__proto__' || key === 'constructor') return undefined;
  return value;
});

  1. 库层面：lodash、merge 等工具函数早年都出过原型污染的 CVE，用新版本就好。

理解原型污染的关键：只要你让用户控制了 key，又在普通对象上做"深层赋值"，就有风险。
