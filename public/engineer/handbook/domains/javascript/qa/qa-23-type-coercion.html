问答 23（进阶）：隐式类型转换的规则与坑（== 的陷阱） ​

题目 ​

请说明 JavaScript 的隐式类型转换规则。== 的转换逻辑是怎样的？ToPrimitive、ToString、ToNumber 分别怎么工作？为什么 [] == false 是 true、{} == {} 是 false？为什么推荐用 ===？

参考答案 ​

为什么有隐式类型转换 ​

JS 是弱类型语言：变量没有固定类型，值有类型。运算符、条件判断、字符串拼接等场景下，如果两边类型不一样，JS 引擎会悄悄把一方转成另一方的类型再运算——这就是隐式类型转换。

理解它的价值：不是让你写代码时去用它，而是让你读懂 legacy 代码、排查奇怪 bug、面试能讲清楚。 日常开发一律用 === 和显式转换。

三个核心转换抽象操作 ​

ES 规范里定义了几个抽象的类型转换操作，理解了它们就理解了所有隐式转换：

1. ToNumber(value) —— 转数字 ​

输入结果
undefinedNaN
null+0
true / false1 / +0
string能解析成数字就转数字，否则 NaN；空串 '' 是 +0
symbolTypeError（不能转数字）
bigint直接转（可能丢精度）
object先 ToPrimitive(value, number)，再 ToNumber

特殊：Number('') === 0、Number(' ') === 0、Number('123a') === NaN。

2. ToString(value) —— 转字符串 ​

输入结果
undefined"undefined"
null"null"
true / false"true" / "false"
number数字的字符串形式（0 → "0"，NaN → "NaN"）
symbolTypeError
object先 ToPrimitive(value, string)，再 ToString

特殊：({}).toString() === "[object Object]"、[1,2,3].toString() === "1,2,3"、[].toString() === ""。

3. ToPrimitive(input, PreferredType?) —— 对象转原始值 ​

对象转原始值时，会调用对象的两个方法，优先级取决于 PreferredType：

  • PreferredType 是 number（默认，除非明确指定 string）：
    1. 先调 valueOf()，如果返回原始值，就用它。
    2. 否则调 toString()，如果返回原始值，就用它。
    3. 都没返回原始值 → TypeError。
  • PreferredType 是 string：
    1. 先调 toString()。
    2. 再调 valueOf()。
  • Date 对象比较特殊，默认 PreferredType 是 string（先 toString）。

如果对象有 Symbol.toPrimitive 方法，优先级最高，直接调用它，valueOf 和 toString 都不看了。

js
const obj = {
  [Symbol.toPrimitive](hint) {
    console.log('hint:', hint);   // 'number' / 'string' / 'default'
    return hint === 'number' ? 42 : 'hello';
  },
};

+obj;          // hint: number → 42
String(obj);   // hint: string → "hello"
obj + '';      // hint: default → "hello"

== 的转换规则（抽象相等比较） ​

== 的规则看着复杂，但可以按优先级记住几条：

  1. 类型相同 → 直接比（和 === 一样）。
  2. null == undefined → true，而且它俩跟谁比都是 false（除了自己和对方）。
  3. 数字 vs 字符串 → 字符串转数字，再比数字。
  4. 布尔值 vs 其他 → 布尔值先转数字（true→1, false→0），再继续比。
  5. 对象 vs 原始值 → 对象 ToPrimitive 转原始值，再继续比。
  6. 对象 vs 对象 → 比引用地址（是不是同一个对象）。

流程图大致：

x == y
├─ 类型相同 → x === y
├─ null == undefined → true
├─ 一个 number, 一个 string → ToNumber(string) 再比
├─ 一个 boolean → ToNumber(boolean) 再比
├─ 一个 object, 一个 primitive → ToPrimitive(object) 再比
└─ 其他 → false

几个经典"奇葩"案例解析 ​

1. [] == false → true？为什么 ​

一步步推：

[] == false
  ↓ 规则 4：布尔值转数字
[] == 0
  ↓ 规则 5：对象转原始值（数组也是对象）
  （[].valueOf() 还是数组，不是原始值 → 调 toString）
  [].toString() → ""
"" == 0
  ↓ 规则 3：字符串转数字
  Number("") → 0
0 == 0 → true

2. {} == {} → false ​

两个都是对象，类型相同，走规则 1——比引用地址。两个不同的对象当然地址不同，所以 false。

{} == false

{} == 0

  ({}).toString() → "[object Object]"
"[object Object]" == 0
  ↓ 字符串转数字
  Number("[object Object]") → NaN
NaN == 0 → false

3. null == undefined → true ​

null === undefined    // false

4. 0 == '' → true ​

'' 转数字是 0。

5. ' \t\r\n ' == 0 → true ​

全是空白字符的字符串转数字也是 0。

为什么推荐用 === ​

  1. 行为可预测：类型不同直接返回 false，不用记一堆转换规则。
  2. 减少心智负担：写代码的人不用想"这里会不会隐式转"，读代码的人也一目了然。
  3. 避免意外 bug：'' == 0、' \t ' == false 这些很容易踩坑。
  4. 性能稍好：不用做类型转换（虽然差别微乎其微，但这不是重点）。

ESLint 的 eqeqeq 规则就是强制用 ===，现代项目基本都开着。

例外：判断 null 和 undefined 时，有人喜欢写 x == null 来同时判断两种情况：

js
if (x == null) { /* x 是 null 或 undefined */ }
// 等同：
if (x === null || x === undefined) { /* ... */ }

这是 == 为数不多的"方便用法"。但很多 lint 规则也不允许，看团队约定。

其他隐式转换场景 ​

不只是 ==，这些地方也会发生隐式转换，顺便提一下：

1. 算术运算符 + - * / % ​

  • + 如果有一边是字符串，就转字符串拼接。
  • 其他运算符两边都转数字。
js
'5' + 3      // '53'（字符串拼接）
'5' - 3      // 2（都转数字）
true + 1     // 2（true → 1）
[] + 1       // "1"（[].toString → ""，"" + 1 → "1"）
{} + 1       // "[object Object]1" 或 1（取决于引擎解析，这里 {} 可能被当成空代码块）
+[]          // 0（一元 +，转数字：[].toString→""→Number("")→0）
+{}          // NaN

追问 1：[] + {} 和 {} + [] 结果一样吗？为什么？ ​

答 ​

不一样。在浏览器控制台（非严格模式、顶层代码）里：

js
[] + {}   // "[object Object]"
{} + []   // 0
1
2

为什么 [] + {} 是 "[object Object]" ​

这是正常的 + 运算：

[] + {}
  ↓ 都转原始值
  [].toString() → ""
  ({}).toString() → "[object Object]"
"" + "[object Object]" → "[object Object]"
1
2
3
4
5

为什么 {} + [] 是 0 ​

因为 JS 解析器把开头的 {} 当成了空代码块（空语句块），而不是对象字面量！

所以 {} + [] 被解析成：

js
{}       // 空代码块，什么都不做
+ []     // 一元 + 运算符，把 [] 转成数字
         // Number([].toString()) = Number("") = 0
1
2
3

结果就是 0。

怎么验证 ​

加个括号让 {} 强制作为表达式：

js
({}) + []   // "[object Object]"
1

这就和 [] + {} 结果一样了。

Node.js 里不一样 ​

注意：Node.js 的 REPL 对输入做了特殊处理（会自动包一层括号之类的），{} + [] 在 Node 里也可能得到 "[object Object]"。但在浏览器控制台和标准 JS 语义中，上面的解析是对的。

这个题考的不是"你背不背得下来"，而是看你知不知道 {} 在语句开头会被解析成代码块这个语法歧义点。

追问 2：valueOf 和 toString 的调用顺序到底是怎样的？举几个具体场景。 ​

答 ​

对象转原始值时，ToPrimitive 有个偏好类型（hint），决定先调谁：

  • hint 是 number：先 valueOf，后 toString。
  • hint 是 string：先 toString，后 valueOf。
  • hint 是 default：大部分对象和 number 一样（先 valueOf），Date 对象例外（先 toString）。

哪些场景 hint 是 number ​

  • 一元 +：+obj
  • 算术运算（非 + 的情况）：obj - 1、obj * 2
  • >、< 比较：obj > 10
  • == 一边是数字：obj == 5
js
const obj = {
  valueOf() { console.log('valueOf'); return 100; },
  toString() { console.log('toString'); return 'hello'; },
};

+obj;           // valueOf → 100
obj - 1;        // valueOf → 99
obj == 100;     // valueOf → true
1

哪些场景 hint 是 string ​

  • String(obj)、obj + ''？不，+ 的 hint 是 default
  • alert(obj)、模板字符串 `${obj}`
  • 对象做 key：const o = {}; o[obj] = 1
js
const obj = {
  valueOf() { console.log('valueOf'); return 100; },
  toString() { console.log('toString'); return 'hello'; },
};

String(obj);          // toString → "hello"
`${obj}`;             // toString → "hello"
const o = {}; o[obj] = 1;  // toString，key 是 'hello'

  • + 运算符（两边都是对象 / 对象 + 字符串等不确定的情况）
  • == 一边是字符串一边是对象

追问 3：NaN === NaN 为什么是 false？怎么正确判断一个值是不是 NaN？ ​

答 ​

为什么 NaN 不等于自己 ​

NaN（Not a Number）是"不是数字的数字值"——它本身是 number 类型，但表示一个"无效的数值运算结果"。

js
typeof NaN === 'number'   // true
1

NaN 不等于任何值，包括它自己。这是 IEEE 754 浮点数标准规定的（不是 JS 独有的，所有遵循 IEEE 754 的语言都这样）。

为什么这么规定？因为"非数字"有很多种可能（0/0、Infinity/Infinity、字符串转数字失败……），它们不应该被认为是"同一个东西"。而且如果 NaN == NaN，那用 indexOf 找 NaN 会出问题——它可能找到位置不对的 NaN。

怎么判断 NaN ​

方法一：Number.isNaN(value)（推荐）

js
Number.isNaN(NaN);         // true
Number.isNaN(123);         // false
Number.isNaN('NaN');       // false（不做隐式转换）
Number.isNaN(undefined);   // false
1
2
3
4

Number.isNaN 只有当 value 真的是 NaN（类型是 number 且值是 NaN）才返回 true，不会做隐式类型转换。

方法二：isNaN(value)（不推荐，老 API）

js
isNaN(NaN);         // true
isNaN('NaN');       // true（先转数字，Number('NaN') → NaN）
isNaN(undefined);   // true（Number(undefined) → NaN）
isNaN('abc');       // true
1
2
3
4

全局的 isNaN 会先把参数转成数字再判断，所以很多"不是 NaN 的值"也会返回 true，很容易误用。

方法三：value !== value（经典技巧）

js
function isNaN(x) { return x !== x; }
1

利用 NaN 不等于自己的特性——只有 NaN 自己不等于自己。这个写法在没有 Number.isNaN 的年代很常用，也不会有隐式转换问题。

方法四：Object.is(value, NaN)（ES6）

js
Object.is(NaN, NaN);   // true
1

Object.is 和 === 几乎一样，只有两个区别：

  • Object.is(NaN, NaN) === true（=== 是 false）
  • Object.is(+0, -0) === false（=== 是 true）

更严格的相等判断，特殊场景用。
