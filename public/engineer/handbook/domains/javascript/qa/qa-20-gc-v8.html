问答 20（进阶）：JavaScript 垃圾回收机制（V8 详解） ​

题目 ​

请说明 JavaScript 的垃圾回收机制。V8 引擎的分代回收策略是怎样的？新生代、老生代分别用什么算法？增量标记、三色标记法、写屏障分别是什么、解决什么问题？

参考答案 ​

为什么需要垃圾回收 ​

JS 是自动内存管理的语言：开发者申请内存（创建对象/数组/函数），引擎负责回收不再使用的内存。如果不回收，内存越用越多，最终 OOM（内存溢出）。

垃圾回收（Garbage Collection, GC）的核心问题：哪些内存是"垃圾"（不再被引用）可以回收？

两种基础算法 ​

1. 引用计数（Reference Counting） ​

思路：每个对象记录被引用的次数，引用 +1，解除引用 -1，次数为 0 就回收。

a = { x: 1 }    // 对象引用计数 1
b = a           // 引用计数 2
a = null        // 引用计数 1
b = null        // 引用计数 0 → 回收

致命问题：循环引用

js
function f() {
  const a = {};
  const b = {};
  a.ref = b;
  b.ref = a;   // 互相引用，计数都是 1
}
f();   // 函数结束，a 和 b 都用不上了，但计数永不为 0 → 泄漏！

IE3 ~ IE6 的 COM 对象用的是引用计数，因此有循环引用泄漏问题。现代 JS 引擎（V8、SpiderMonkey）都不用引用计数做主 GC 算法，但理解它有助于理解 WeakMap/WeakSet（它们底层就是弱引用，不参与计数）。

2. 标记清除（Mark-and-Sweep） ​

思路：从一组"根对象（GC Roots）"出发，所有能到达的对象标记为"存活"，没被标记的就是垃圾，回收。

GC Roots（可达的起点）：
├─ 全局对象（window / globalThis）
├─ 执行栈上的变量、参数
├─ 闭包引用的变量
└─ 其他内建对象的内部引用
1
2
3
4
5

流程：

  1. 标记阶段：从根出发，沿引用链走一遍，所有能访问到的对象打标记。
  2. 清除阶段：遍历堆，没标记的对象全部回收。

完美解决循环引用——只要对象不在根的可达链上，即使互相引用也一起被回收。

问题：标记和清除都需要暂停 JS 执行（Stop-The-World, STW），堆越大暂停越长。大应用（比如几 GB 堆）全堆标记一次可能要几百毫秒，用户能感知到卡顿。

V8 的分代回收策略 ​

V8 把堆分成新生代（Young Generation）和老生代（Old Generation），用不同算法，核心依据是代际假说（Generational Hypothesis）：

大部分对象存活时间很短；熬过几次 GC 的对象，通常会活很久。

白话就是："大多数对象用了就扔，少数能留下来的一般会长期存在。"既然生命周期不一样，就用不同策略对付——短命对象用"快而猛"的算法，长命对象用"稳而省"的算法。

堆内存结构（简化）：
┌─────────────────────────────────────┐
│           新生代 (Young)            │
│  ┌──────────┐  ┌──────────┐         │
│  │   From   │  │    To    │  两个半空间 │
│  └──────────┘  └──────────┘         │
├─────────────────────────────────────┤
│           老生代 (Old)              │
│  存活时间长的对象、大对象             │
└─────────────────────────────────────┘

新生代：Scavenge 算法 ​

新生代又分成两个对半空间（semispace）：From 空间和 To 空间。同一时间只有一个在用，另一个空着。

Scavenge 算法流程：

  1. 新对象分配在 From 空间。
  2. 当 From 空间快满时，触发 Minor GC（新生代 GC）。
  3. 从根出发，标记 From 中存活的对象。
  4. 把存活对象复制到 To 空间，同时按顺序紧凑排列（顺便完成了碎片整理）。
  5. From 和 To 角色互换（原来空的 To 变成新的 From，原来的 From 变成空的 To）。

为什么复制？ 因为新生代对象大部分都是垃圾，存活的少，复制成本低。复制过去还顺便解决了内存碎片问题。

晋升（Promotion）：对象在新生代熬过两次 GC 还没死，就认为它是"长命对象"，会被晋升到老生代。另外，如果 To 空间已经用了 25%，为了不影响新对象分配，大一点的存活对象也会直接晋升。

优点：速度极快（只处理存活对象，通常很少），没有碎片。 缺点：有一半空间空着，空间换时间。

老生代：Mark-Sweep & Mark-Compact ​

老生代对象多、存活时间长，再用 Scavenge 就不合适了——复制成本高、还浪费一半空间。V8 用标记清除（Mark-Sweep） + **标记整理（Mark-Compact）**组合。

Mark-Sweep（标记清除） ​

  1. 标记：从 GC Roots 出发，遍历所有可达对象，打上标记。
  2. 清除：把没标记的对象直接回收，把它们的地址加入"空闲列表"。

分配新对象时，从空闲列表里找一块够大的内存分配。

问题：清除之后内存里会留下很多碎片——小的空闲块散落在各处。如果来了一个大对象，可能每个空闲块都不够大，但加起来够，就分配不出去。

Mark-Compact（标记整理） ​

为了解决碎片问题，V8 会定期做 Mark-Compact：

  1. 标记：同上。
  2. 整理：把所有存活对象往一端移动，然后把边界之外的内存全部回收。

相当于"磁盘整理"，存活对象都挤到一边，剩下一整块连续的空闲内存。

代价：整理需要移动对象、更新所有引用这些对象的指针，成本比单纯清除高。所以 V8 不是每次都做 Compact，而是根据碎片情况决定用 Sweep 还是 Compact。

全停顿的痛点与优化 ​

全堆标记清除需要 STW，堆越大停越久。1.5GB 堆 V8 做一次完整标记大概要 1 秒以上，这 1 秒里 JS 完全不执行，页面卡死。

V8 做了好几代优化来解决这个问题。

优化一：增量标记（Incremental Marking） ​

思路：把标记工作拆成一小块一小块，穿插在 JS 执行间隙做，每标记一点就让 JS 跑一会儿，避免一次停太久。

原来： [==== 标记 500ms ====][ JS 执行 ]
现在： [= 5ms =][JS][= 5ms =][JS][= 5ms =][JS]...
1
2

但增量标记有个棘手问题：标记过程中 JS 还在跑，对象之间的引用关系可能变了——比如一个已经被标记为"存活"的对象，新增了一个指向白色（未标记）对象的引用，而这个白色对象如果没被重新扫描到，就会被错误地回收。

这就需要三色标记法 + 写屏障来兜底。

优化二：三色标记法（Tri-color Marking） ​

把对象分成三种颜色：

  • 白色：还没被 GC 访问过。GC 开始时所有对象都是白色。
  • 灰色：已经被访问到，但它的引用对象还没全部处理完（待处理队列里的对象）。
  • 黑色：它自己和它所有的引用对象都处理完了。

标记过程就是"灰色慢慢把白色染成灰色，自己变黑"，直到没有灰色对象。最后剩下的白色对象就是垃圾。

关键不变性（Invariant）：黑色对象不会指向白色对象。因为黑色对象的所有引用都处理过了，如果它引用了某个对象，那个对象一定已经被染色了。

增量标记期间 JS 修改了引用关系，可能会破坏这个不变性——这时候就需要**写屏障（Write Barrier）**来维护。

优化三：写屏障（Write Barrier） ​

每当代码给一个对象的属性赋值时（obj.field = newObj），写屏障会检查：

  • 如果 obj 是黑色的，而 newObj 是白色的，就把 newObj 或者 obj 重新染成灰色，放回待处理队列。

这样就保证了"黑色不指向白色"的不变性永远成立，增量标记就不会漏标对象。

js
// 概念上的写屏障伪代码
function writeBarrier(obj, field, value) {
  if (colorOf(obj) === BLACK && colorOf(value) === WHITE) {
    paintGray(value);   // 把新引用的白色对象变灰，重新入队
  }
  obj[field] = value;
}

更多优化 ​

  • 懒清除（Lazy Sweeping）：标记完之后不急着一次性全清，用到哪清到哪，分摊清除成本。
  • 并行标记（Parallel Marking）：用多个辅助线程一起标记，利用多核 CPU。
  • 并发标记（Concurrent Marking）：标记和 JS 执行完全并行（不是穿插，是同时跑），需要更复杂的写屏障和同步机制。V8 的 Orinoco 垃圾回收器就实现了并发标记。

优化手段解决的问题
增量标记减少单次 STW 时长，拆成多段
三色标记法增量标记期间的正确性保障
写屏障维护三色不变性，防止漏标
懒清除分摊清除阶段成本

追问 1：WeakMap / WeakSet 和垃圾回收有什么关系？为什么它们的 key 只能是对象？ ​

答 ​

普通的 Map 对 key 是强引用——只要 Map 存活，key 对象就不会被回收，即使外部没有任何其他引用。

js
let obj = { data: 'big' };
const map = new Map();
map.set(obj, 'value');

obj = null;   // 外部引用断了
// 但 map 还强引用着 obj，obj 不会被 GC！
1
2
3
4
5

WeakMap 对 key 是弱引用——key 对象的存在不影响 GC。如果除了 WeakMap 之外没有别的地方引用 key 对象了，那这个 key 就会被回收，对应的键值对也自动从 WeakMap 里消失。

js
let obj = { data: 'big' };
const wm = new WeakMap();
wm.set(obj, 'value');

obj = null;   // 外部引用断了
// WeakMap 的弱引用不阻止 GC，obj 会被回收，wm 里这一项也没了
1

为什么 key 只能是对象？ ​

因为弱引用的语义是"如果这个对象没人用了就回收它"——这是针对对象的生命周期而言的。原始类型（number、string、boolean）是值类型，没有"被引用/被回收"的概念，它们不能作为弱引用的目标。

补充：WeakSet 同理，里面存的对象都是弱引用。WeakRef（ES2021）更直接，就是一个对象的弱引用包装。

典型应用 ​

  • 私有数据：给对象存关联数据，但不影响对象本身的回收（旧版私有变量的实现思路）。
  • DOM 节点元数据：给 DOM 元素存额外信息，DOM 被移除后数据自动随节点一起被 GC，不会泄漏。
  • 缓存：缓存 key 是对象，对象没了缓存项自动失效。

追问 2：什么是内存泄漏？前端常见的内存泄漏场景有哪些？如何排查？ ​

答 ​

内存泄漏：程序中已不再使用的内存，由于某种原因没有被 GC 回收，导致内存占用持续升高。注意和"内存占用高但会释放"区分开——泄漏是"只涨不跌"或者"回收得远远不够"。

1. 意外的全局变量

js
function foo() {
  bar = 123;   // 没写 var/let/const，挂到 window 上了！
  this.baz = 456;   // 如果 foo 直接调用，this 是 window
}

Chrome DevTools：

  • Memory 面板 → Take Heap Snapshot：拍堆快照，对比两次快照的差异（Allocation Profiler 也可以看按时间的分配）。

追问 3：V8 为什么不用引用计数做主 GC？引用计数在现代引擎里完全没用了吗？ ​

答 ​

为什么不用引用计数做主 GC ​

主要三个问题：

1. 循环引用无解

最经典的问题。两个对象互相引用，外部都不引用了，计数还是 1，永远回收不了。要解决就得额外做"循环检测"，成本很高，不如直接用标记清除。

2. 每次赋值都要更新计数，开销大

每次 obj.x = y 都要做：旧值引用计数 -1（如果减到 0 还要递归回收它的引用），新值引用计数 +1。频繁操作对象属性的 JS 代码开销不小。

3. 多线程/并发场景下计数原子操作成本高

如果 JS 引擎有多线程（比如辅助 GC 线程），引用计数的增减需要原子操作，更慢。

所以主流 JS 引擎清一色用标记清除做主 GC 算法。

引用计数完全没用了吗 ​

也不是，它在一些"局部场景"里还是好用的：

1. WeakMap / WeakRef 的底层思路

虽然实现上不是简单的计数器，但弱引用的核心思想就是"不增加引用计数/不纳入标记可达性"——本质上和引用计数关注"引用次数"是同一条思路线。

2. 字符串、小整数等的内部引用计数

V8 内部有些资源（比如底层字符串、隐藏类）可能用引用计数管理生命周期，因为它们没有循环引用的问题，用引用计数又快又简单。

3. 其他语言/引擎的启发

Python、Objective-C 的 ARC 都用引用计数做内存管理（Python 也搭配了循环垃圾回收器），但 JS 的对象模型和使用场景（DOM 交互、大量闭包）导致循环引用太普遍，引用计数不合适。

理解引用计数最大的价值在于：理解"可达性"和"引用次数"是两个不同的判定标准，以及 WeakMap/WeakSet 的"弱"到底弱在哪。
