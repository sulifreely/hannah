问答 16（高级）：Node.js 性能监控与诊断 ​

题目 ​

线上 Node.js 服务出现性能问题时，你会从哪些角度排查？有哪些常用的监控指标和诊断工具？Event Loop Lag、内存、CPU 分别怎么监控？

参考答案 ​

线上性能问题排查是一个系统工程，核心思路是：先监控指标定位异常方向，再用诊断工具深入根因。

3. Event Loop Lag 监控

事件循环延迟（Event Loop Lag）是衡量 Node.js 服务健康状况的黄金指标——它直接反映了主线程有没有被阻塞。

js
const { monitorEventLoopDelay } = require('perf_hooks');

const histogram = monitorEventLoopDelay({ resolution: 10 });
histogram.enable();

setInterval(() => {
  console.log({
    min: histogram.min,            // 最小延迟（纳秒）
    max: histogram.max,            // 最大延迟
    mean: Math.round(histogram.mean), // 平均
    p50: histogram.percentile(50),  // 中位数
    p95: histogram.percentile(95),  // 95 分位
    p99: histogram.percentile(99)   // 99 分位
  });
  histogram.reset();
}, 1000);

Async Hooks 与 AsyncLocalStorage

async_hooks 是 Node.js 提供的追踪异步资源生命周期的 API，每个异步操作（Promise、setTimeout、I/O 回调等）都有一个 asyncId，可以追踪它的创建和执行。

直接用 async_hooks 比较底层，更常用的是它的高层封装 AsyncLocalStorage，用来在异步调用链中传递上下文（比如 traceId、用户信息），类似线程本地存储（TLS）。

js
const { AsyncLocalStorage } = require('async_hooks');
const asyncLocalStorage = new AsyncLocalStorage();

function log(msg) {
  const traceId = asyncLocalStorage.getStore();
  console.log(`[traceId=${traceId}] ${msg}`);
}

// 在请求处理中创建上下文
app.use((req, res, next) => {
  const traceId = req.headers['x-trace-id'] || Math.random().toString(36).slice(2);
  asyncLocalStorage.run(traceId, () => {
    next();
  });
});

// 后续任何异步函数里都能拿到 traceId
app.get('/api', async (req, res) => {
  log('处理请求');
  await db.query();
  log('查询完成');
  res.send('ok');
});

追问 1：Event Loop Lag 高一定是 CPU 问题吗？还有哪些可能的原因？ ​

答 ​

不一定。Event Loop Lag 高本质是「事件循环的某个阶段花了太长时间」，原因可能有多种：

  1. CPU 密集计算（最常见）

    • 大循环、复杂正则、JSON.parse 大对象、大量计算。
    • 表现：CPU 使用率高 + lag 高。
    • 排查：火焰图找热点函数。

  2. 同步 I/O

    • 代码里用了 fs.readFileSync、child_process.execSync 等同步 API。
    • 表现：CPU 不一定高，但 lag 高，因为同步 I/O 阻塞了主线程。
    • 排查：grep 代码里的 Sync 调用，看是不是在请求路径上。

  3. GC 停顿

    • 堆内存大、对象分配频繁，导致 Full GC 时间长，Stop-The-World。
    • 表现：lag 有周期性尖峰，同时 --trace-gc 能看到长时间的 GC pause。
    • 排查：调大堆、减少对象分配、查看内存泄漏。

  4. 微任务太多

    • 一瞬间产生大量 Promise microtask（比如 Promise.all 几千个），微任务队列执行时间太长，推迟了下一个宏任务。
    • 表现：lag 高，但 CPU 不一定特别高。
    • 排查：看是不是有批量 Promise 处理没做限流。

  5. 主线程被 debugger 或 inspector 阻塞

    • 开了 --inspect 且连接了调试器，断点或单步会暂停事件循环。
    • 生产环境不要开调试端口。

  6. 进程被操作系统挂起

    • 机器 CPU 被其他进程占满，Node 进程被调度器暂停。
    • 表现：Node 自身 CPU 不高但 lag 高，系统 load 高。
    • 排查：看整机 CPU、load average。

所以排查思路是：先看 lag 高不高 → 再看 CPU 高不高 → 再看 GC 日志 → 再看代码里有没有同步操作。

追问 2：排查内存泄漏的标准流程是什么？如何区分「内存占用高」和「内存泄漏」？ ​

答 ​

先区分：是占用高还是泄漏？

  • 内存占用高：内存上去了但稳定在一个水平，GC 后能回落。可能是缓存大、并发高，不一定是 bug。
  • 内存泄漏：内存持续上涨，GC 后也不回落，最终 OOM 进程崩溃。

简单判断：跑一段时间，看 heapUsed 的趋势。如果像锯齿但整体水平，是正常的（GC 回收）。如果锯齿整体往上走，就是泄漏。

标准排查流程：

  1. 确认是泄漏

    • 用 process.memoryUsage() 定期打点，画内存曲线。
    • 或者用 APM 工具（如 clinic doctor）看内存趋势。

  2. 拍堆快照对比

    • 用 Chrome DevTools Memory 面板或 clinic heapprofiler。
    • 拍三个快照：
      • Snapshot 1：初始状态。
      • Snapshot 2：操作一轮后（内存涨上去）。
      • Snapshot 3：再操作几轮后（确认还在涨）。
    • 选择「Comparison」视图，对比 Snapshot 2 和 Snapshot 1，看什么对象在增长。

  3. 定位泄漏对象

    • 按 Retained Size 排序，找占内存最多的对象。
    • 看 Retaining Path（引用链），找到谁持有了它，为什么没被 GC。
    • 常见的「罪魁祸首」：闭包、全局缓存、事件监听器、模块级变量、定时器。

  4. 定位到代码行

    • 结合对象的构造函数名、属性名，在代码里搜索。
    • 用 --heap-prof 或 allocation sampling 看对象是在哪分配的。

  5. 修复并验证

    • 修复后再跑一遍，确认内存曲线稳定。

常见泄漏场景回顾：

  • 全局缓存只加不删（用 LRU 缓存代替普通对象）。
  • 事件监听只加不移（及时 removeListener，或用 once）。
  • 闭包意外持有大对象。
  • 模块级变量积累数据（模块一直缓存着）。
  • 未关闭的流、socket、定时器。

追问 3：AsyncLocalStorage 的原理是什么？性能开销大吗？和 domain 有什么区别？ ​

答 ​

原理：

AsyncLocalStorage 基于 async_hooks 模块实现。async_hooks 为每个异步资源（Promise、setTimeout、I/O 回调、nextTick 等）分配一个 asyncId，并在异步资源创建、执行前、执行后、销毁时触发钩子。

AsyncLocalStorage.run(store, callback) 做的事情是：

  1. 保存当前 store。
  2. 设置新的 store，关联到当前的 asyncId。
  3. 执行 callback。
  4. 恢复原来的 store。

当异步操作创建新的异步资源时（比如 callback 里调了 setTimeout），async_hooks 的 init 钩子会把父资源的 store 「遗传」给子资源。这样子资源的回调执行时，也能拿到同一个 store。

这就是为什么在 asyncLocalStorage.run 里面发起的任何异步操作（Promise、setTimeout、I/O），在它们的回调里都能通过 getStore() 拿到同一个值。

性能开销：

  • async_hooks 本身有一定开销，因为每个异步操作都要触发钩子。
  • 但 Node.js 对 AsyncLocalStorage 做了优化（用了更轻量的机制），实际开销很小，通常在 1-5% 左右。
  • 对于绝大多数 Web 服务来说，这点开销完全可以接受，换来的是可观测性和调试便利性。
  • 如果是极高 QPS 且对延迟极度敏感的服务，可以评估一下，但大多数情况没问题。
  • 注意：不要在热路径上频繁创建新的 AsyncLocalStorage.run 上下文，复用上下文是没问题的。

和 domain 的区别：

  • domain 是 Node v0.8 引入的老 API，已经废弃（deprecated），不应该再用。
  • domain 主要是为了解决「异步错误捕获」的问题，附带能传点上下文，但设计有缺陷，容易导致内存泄漏，语义也不清晰。
  • AsyncLocalStorage 是专门为「异步上下文传递」设计的，语义清晰，性能更好，是官方推荐的方案。
  • 错误捕获现在推荐用 uncaughtException + unhandledRejection 全局监听，或者框架层面的错误处理中间件。
