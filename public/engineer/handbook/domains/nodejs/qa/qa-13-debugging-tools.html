问答 13（进阶）：Node.js 调试技巧与工具 ​

题目 ​

请介绍 Node.js 常用的调试方法和工具。从最简单的 console 到 Chrome DevTools、VS Code 调试，再到性能分析和火焰图，它们分别适合什么场景？

参考答案 ​

方法一：console 调试（最基础）

最常用也最简单，但效率最低。

js
console.log('变量值:', obj);
console.error('出错了:', err);
console.warn('警告');
console.table([{ a: 1, b: 2 }, { a: 3, b: 4 }]); // 表格输出
console.dir(obj, { depth: null }); // 深层打印对象
console.time('loop');
for (let i = 0; i < 1e6; i++) {}
console.timeEnd('loop'); // loop: 2.345ms
console.trace('这里调用栈是啥'); // 打印调用栈

方法二：debugger 语句 + --inspect

在代码里写 debugger 语句，运行时加 --inspect 参数，就会在断点处停下。

bash
node --inspect app.js          # 启动调试器，默认监听 9229 端口
node --inspect-brk app.js      # 在第一行就断住（等你连上再执行）
node --inspect=0.0.0.0:9229 app.js  # 监听所有地址，远程调试用

然后打开 Chrome，地址栏输入 chrome://inspect，找到你的进程，点击「inspect」，就会打开 DevTools，可以断点、单步、看调用栈、看变量。

也可以用 node inspect app.js 进入命令行调试模式（类似 gdb），但体验不如 DevTools。

方法四：ndb

Google Chrome Labs 出的 Node.js 调试工具，本质就是 Chrome DevTools，但独立成一个 App，体验更好，支持直接 ndb npm test 之类的命令。

方法五：--prof 性能分析（V8 内置采样）

bash
node --prof app.js          # 运行结束生成 isolate-0x...-v8.log
node --prof-process isolate-0x...-v8.log > processed.txt  # 处理日志
1
2

生成的 processed.txt 里有各个函数的执行时间占比，可以找到 CPU 热点。

适用：CPU 性能瓶颈分析，找哪个函数最耗时。

方法六：0x 火焰图

比 --prof 更直观，生成交互式的火焰图 SVG。

bash
npm install -g 0x
0x app.js
# 运行结束后生成一个 .html 文件，浏览器打开看火焰图
1
2
3

火焰图怎么看：

  • 横轴是采样数（占比），越宽表示耗时长。
  • 纵轴是调用栈，从下往上是调用关系。
  • 顶层的「平顶」（宽的横条）就是性能瓶颈。

适用：定位 CPU 性能问题、找热点函数。

方法七：clinic.js

一套 Node.js 诊断工具集，包括：

  • clinic doctor：综合诊断，自动分析瓶颈（CPU/内存/事件循环）并给建议。
  • clinic flame：火焰图。
  • clinic bubbleprof：异步延迟分析，找 I/O 瓶颈。
  • clinic heapprofiler：堆内存分析。
bash
npm install -g clinic
clinic doctor -- node app.js
1
2

适用：遇到性能问题但不知道从哪下手时，先用 doctor 做个全身检查。

追问 1：--inspect 和 --inspect-brk 有什么区别？调试协议是什么原理？ ​

答 ​

区别：

  • --inspect：启动调试器，但代码正常执行，不会自动断住。你需要手动在 DevTools 里加断点，或者代码里有 debugger 语句才会停。
  • --inspect-brk：启动调试器，并且在第一行代码执行前就断住，等调试器连接上之后才继续。适合调试启动阶段的代码、或者脚本一启动就跑完的情况。

调试协议原理：

  • Node.js 的调试基于 Chrome DevTools Protocol (CDP)，和 Chrome 调试页面用的是同一个协议。
  • Node 启动后开一个 WebSocket 端口（默认 9229），调试前端（Chrome DevTools、VS Code、ndb）通过这个端口发命令（如设置断点、单步执行、求值表达式），Node 端发回事件和结果。
  • 底层是 V8 引擎自带的调试能力，Node 只是把它通过 CDP 暴露出来。

安全注意：--inspect 默认只绑定 127.0.0.1，不会被外部访问。如果用 0.0.0.0 要小心，任何人都能连接并执行代码。线上调试要用安全隧道（如 SSH 端口转发），不要直接暴露。

追问 2：调试 TypeScript 项目时，断点怎么对应到源码？source map 是什么原理？ ​

答 ​

TypeScript 最终编译成 JS 执行，调试时你打的断点在 .ts 文件里，但执行的是 .js 文件，这就需要 source map 来做位置映射。

source map 是什么：

  • 一个 .map 文件，记录了编译后代码的每一行每一列，对应源码的哪一行哪一列。
  • 编译时 tsconfig.json 里设置 "sourceMap": true 就会生成。
  • JS 文件末尾通常有一行注释指向 map 文件：//# sourceMappingURL=app.js.map。

调试器如何使用：

  • VS Code 或 DevTools 加载 JS 时，发现有 source map，就会同时加载 map 文件和源文件。
  • 你在 .ts 文件里打的断点，调试器通过 map 换算成 .js 里的位置，在那里真正断住。
  • 断住后展示给你的还是 .ts 源码，变量也是源码里的名字。

VS Code 调试 TS 的配置要点：

json
{
  "type": "node",
  "request": "launch",
  "name": "调试 TS",
  "program": "${workspaceFolder}/src/app.ts",
  "runtimeArgs": ["-r", "ts-node/register"],  // 用 ts-node 直接运行 TS
  "outFiles": ["${workspaceFolder}/dist/**/*.js"],  // 编译后文件位置
  "sourceMaps": true,
  "skipFiles": ["<node_internals>/**"]
}

追问 3：内存泄漏怎么排查？有哪些常见的 Node.js 内存泄漏场景？ ​

答 ​

排查工具：

  1. Chrome DevTools Heap Snapshot：

    • 用 --inspect 连接 DevTools，切到 Memory 面板。
    • 拍堆快照（heap snapshot），对比操作前后的快照，看什么对象在增长。
    • 按「retained size」排序，找占内存多的对象，再看它的引用链（retaining path），找到为什么没被 GC。

  2. clinic heapprofiler：命令行版的堆分析，自动生成报告。

  3. process.memoryUsage()：定时打印内存使用，先确认是不是真的在涨。

常见泄漏场景：

  1. 全局变量意外持有大对象

    js
    const cache = {};
// 只加不删，内存一直涨
function set(key, val) { cache[key] = val; }
    1
    2
    3

    解决：用 LRU 缓存、Map 改 WeakMap、加淘汰策略。

  2. 闭包引用未释放

    js
    function createHandler() {
  const bigData = new Array(1e6).fill('x');
  return function handler() {
    // 虽然没用到 bigData，但闭包会持有它的引用
    console.log('hello');
  };
}
    1
    2
    3
    4
    5
    6
    7

    解决：不需要的变量手动置 null，或者拆分函数。

  3. 事件监听器只加不移除

    js
    // 每次请求都加一个监听，从不移除
app.get('/api', (req, res) => {
  emitter.on('data', () => { ... });
});
    1
    2
    3
    4

    解决：及时 removeListener，用 once 替代 on，或者确保组件销毁时清理。

  4. 未关闭的资源：数据库连接、文件句柄、定时器（setInterval 不清除）等。

  5. 模块缓存：require 进来的模块会一直缓存，如果模块里有不断增长的数组/对象，就是泄漏。

排查思路：先确认是泄漏（内存持续上涨不回落）→ 拍快照对比 → 找增长对象 → 追踪引用链 → 定位代码 → 修复验证。
