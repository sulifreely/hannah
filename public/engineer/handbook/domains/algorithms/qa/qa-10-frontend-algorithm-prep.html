问答 10（入门）：前端面试算法高频考点与准备策略 ​

题目 ​

前端面试中算法考察有什么特点？高频题型有哪些？刷题应该按题型分类刷还是随机刷？面试时拿到一道题应该按什么思路展开？请系统讲一讲前端工程师的算法面试准备策略。

参考答案 ​

一、前端算法面试的特点 ​

和后端 / 算法岗相比，前端面试的算法考察有几个鲜明特点：

  1. 难度中等为主，hard 题少

    • 大部分是 LeetCode easy ~ medium 难度。
    • 很少出现 hard 题，即使出现也是「经典 hard 简化版」或「模板题变形」。
    • 大厂核心岗 / 大厂基础架构岗可能会难一些，但整体还是低于算法岗。

  2. 题型集中，套路化明显

    • 前端常考的就那十来种题型，掌握了覆盖 80% 以上的面试题。
    • 数组、字符串、链表、树、BFS/DFS、滑动窗口、双指针、回溯、DP 入门、位运算。

  3. 偏爱「场景 + 算法」结合

    • 不会只考纯算法，经常结合前端场景：LRU、diff、虚拟列表、URL 解析、树形数据处理、异步并发控制……
    • 具体见 qa-05「前端场景中最常见的算法」。

  4. 更看重「写出来能跑」而不是「最优解」

    • 能写出正确的暴力解 → 大部分面试已经过了一半。
    • 再优化到较优解 → 加分。
    • 思路清晰、代码整洁、能跑通 case → 比单纯最优解但写得乱七八糟强。

  5. JS 语言特性会成为考点

    • 递归深度、闭包、this、原型链、事件循环……这些可能和算法结合考。
    • 比如「用迭代器实现数组扁平化」「手写 Promise.all」本质也是算法思维 + JS 基础。

二、高频题型清单（按出现频率排序） ​

1. 数组双指针 ​

  • 快慢指针：环形链表、删除有序数组重复项、移动零、寻找重复数。
  • 左右指针：两数之和 II（有序数组）、盛最多水的容器、反转字符串、回文串判断。
  • 滑动窗口：长度最小的子数组、无重复字符的最长子串、找到字符串中所有字母异位词。
  • 为什么高频：数组是最基础的数据结构，双指针是数组题最常用的技巧，思路清晰代码短，适合面试短时间内考察。

2. 链表 ​

  • 反转链表（迭代 + 递归）、合并两个有序链表、环形链表（快慢指针判环）、链表中倒数第 k 个节点、相交链表、删除链表的倒数第 N 个节点、排序链表。
  • 为什么高频：链表题考察指针操作和边界处理（空指针、头节点、尾节点），代码量适中，是基础中的基础。

3. 树的遍历 ​

  • 二叉树前 / 中 / 后序遍历（递归 + 迭代）、层序遍历（BFS）、二叉树的最大深度、对称二叉树、翻转二叉树、验证二叉搜索树、二叉树的最近公共祖先。
  • 为什么高频：树是面试常客，既考递归思维，也考迭代写法，题量适中区分度好。

4. BFS / DFS ​

  • BFS：二叉树层序遍历、二叉树的右视图、二叉树的层平均值、N 叉树的层序遍历、岛屿数量（图的 BFS）、打开转盘锁。
  • DFS：岛屿数量、全排列、子集、组合、二叉树的所有路径、被围绕的区域。
  • 为什么高频：BFS/DFS 是图/树/搜索的基础，也是很多算法的基石，考法灵活。

5. 滑动窗口 ​

  • 无重复字符的最长子串、长度最小的子数组、最小覆盖子串、找到字符串中所有字母异位词、最大连续 、水果成篮。
  • 模板：左右指针维护一个窗口，右指针扩大窗口，满足条件时左指针缩小窗口，过程中更新答案。
  • 为什么高频：套路强、模板化明显，学会模板后容易出变种题，考察对「窗口边界 + 条件判断」的理解。

6. 回溯算法 ​

  • 组合、排列、子集、组合总和、电话号码的字母组合、括号生成、N 皇后（低频 hard）。
  • 为什么高频：考察递归 + 剪枝思维，代码量适中，能看出候选人的代码组织能力和边界处理。

7. DP 入门 ​

  • 爬楼梯、斐波那契、最大子数组和、打家劫舍、不同路径、最小路径和、零钱兑换（中等）、最长递增子序列（中等）。
  • 注意：前端很少考 hard DP（如区间 DP、状压 DP），掌握一维和二维基础 DP 就够了。

8. 堆 / 优先队列 ​

  • TopK 问题（第 K 大元素、前 K 个高频元素）、最小的 K 个数、数据流中的中位数。
  • 前端特点：JS 没有内置堆，要么手写堆（面试可能让你手写），要么用数组 + sort 模拟（简单但复杂度高）。
  • 面试时最好能手写一个简单堆。

9. 位运算 ​

  • 汉明距离、比特位计数、只出现一次的数字（I/II/III）、颠倒二进制位、2 的幂。
  • 为什么高频：题目短、思路巧、代码极简，适合面试快问快答。

10. 排序相关 ​

  • 手写快排 / 归并 / 堆排的原理和复杂度对比。
  • 数组中的第 K 大元素（快排思想 / 堆思想）。
  • 归并排序的应用（逆序对、合并 K 个升序链表）。

三、刷题策略：分类刷 vs 随机刷 ​

阶段一：入门期 → 按题型分类刷 ​

推荐按题型分类刷（推荐新手）**

为什么分类刷的好处：

  1. 掌握套路模板：同一类题放一起做，能快速总结出模板和规律。
  2. 建立题型识别能力：看到一道新题能快速判断「这是 XX 类型」→ 用 XX 方法」。
  3. 信心建立快：连续做出同类型的题越做越顺，正反馈强。

怎么分类刷：

  • 按上面的高频顺序，一类一类攻克。
  • 每类先做 3~5 道经典题，掌握模板。
  • 再做 3~5 道变形题，练举一反三。
  • 总共每类 5~10 道，10 类就是 50~100 道，基本覆盖 80% 面试题。

推荐顺序：数组 → 链表 → 树 → BFS/DFS → 滑动窗口 → 回溯 → DP → 堆 → 位运算 → 排序。

阶段二：提高期 → 混合刷 + 周赛 ​

当各类题型都有基础后，开始混合刷（随机刷 / 每日一题 / 周赛）

为什么要混合刷：

  1. 锻炼题型识别能力：面试时题不会告诉你它是哪一类，你得自己判断。
  2. 模拟真实面试：面试就是随机给你一道题，你得自己想思路。
  3. 发现薄弱点：混合刷才能发现「哪类题我总是想不到」。

怎么做：

  • LeetCode 每日一题 + 周末周赛。
  • 做完后总结「我第一时间想不出的题，标记出来，回去补这类。

阶段三：面试前冲刺 → 高频题 + 模拟面试 ​

**面试前 1~2 周：

  1. 刷目标公司的高频题（各平台有企业题库、面经题。
  2. 模拟面试：找人互面，计时，讲思路，说出来。
  3. 复习错题本：把之前做过但做错的 / 不会的再做一遍。

四、面试时的解题五步曲 ​

面试时拿到一道题，不要上来就写代码。按以下步骤来：

第一步：理解题意 + 确认边界 ​

  • 把题目读一遍，用自己的话复述给面试官听，确认理解正确。
  • 问清楚边界条件：
    • 输入规模多大？（决定能不能用暴力）
    • 有没有重复元素？
    • 空输入怎么处理？
    • 返回什么？索引还是值？多个解返回哪个？
  • 举 2~3 个例子，确保理解一致。

为什么重要：理解错了写半天白写，还显得沟通能力差。

第二步：想暴力解 ​

  • 先想一个最朴素的暴力解法是什么。
  • 说出来：「暴力的话就是……时间复杂度是 O(n²)，应该还能优化」。
  • 哪怕你直接想到最优解，也可以先说暴力，显得思路连贯。

为什么重要：

  • 暴力解是保底的，至少说明你有思路。
  • 从暴力出发想优化，思路更自然。

第三步：想优化 ​

从暴力出发，思考怎么优化：

  • 时间超了 → 想怎么减少重复计算：

    • 排序 + 二分？
    • 哈希表存一下？
    • 双指针 / 滑动窗口？
    • DP 记忆化？

  • 空间超了 → 想怎么复用空间：

    • 原地修改？
    • 滚动数组？
    • 位运算压缩状态？

  • 说出你的优化思路和复杂度分析。

第四步：写代码 ​

  • 先搭框架（函数名、参数、返回值）。
  • 再写主体逻辑。
  • 变量名要有意义，不要 a/b/c 满天飞。
  • 边写边说思路，不要沉默。

代码质量要点：

  • 边界处理（空输入、单个元素、首尾端点）。
  • 逻辑清晰，层次分明。
  • 适当的空行和缩进。
  • JS 注意：数组越界、空指针、循环条件、递归终止条件。

第五步：测试 + 走查 ​

写完后不要说「写完了」，而是：

  1. 自己先看一遍：有没有明显的 bug？边界处理了吗？
  2. 拿例子跑一遍：用最简单的 case 手动模拟代码执行。
  3. 想特殊 case：空输入、单个元素、全相同、全逆序、最大最小……
  4. 说复杂度：时间多少、空间多少。
  5. 问面试官：「您看这个思路可以吗？有没有哪里需要优化的？」

五、前端算法面试的额外建议 ​

  1. 不要纠结最优解，先保证正确

    • 能写出正确的暴力解 + 说清思路，已经超过很多人了。
    • 实在想不出最优解，先写暴力，再慢慢想优化。

  2. 一定要说思路

    • 面试考的是「你怎么思考的」，不是「你能不能写出来」。
    • 哪怕写不出来，思路说对了也能拿部分分。

  3. JS 手写能力要过关

    • 手写 Promise、手写 deepClone、手写防抖节流、手写 instanceof……这些也是「算法」。
    • 数组方法的底层实现（map、reduce、flat）也常考。

  4. 积累错题本

    • 做过的题、做错的题、面试遇到的题，都记下来。
    • 面试前翻一遍，比刷新题有用。

  5. 保持手感

    • 每天 1~2 道，保持思考的感觉。
    • 比考前突击刷 100 道效果好。

追问 1：为什么很多前端面试都考算法？前端日常开发不是很少用到吗？算法能力和实际工作是什么关系？ ​

答 ​

这是一个非常好的问题，也是很多前端同学的困惑。分几个层面说：

1. 算法是「思维能力」的 proxy（代理指标）

面试时间很短（通常 45 分钟 ~ 1 小时），面试官要在这么短的时间里判断一个候选人的技术能力，需要一些「快速有效」的考察方式。

算法题是一个很好的筛选器：

  • 能把题做出来 → 说明逻辑思维能力不差
  • 能优化到较好的复杂度 → 说明对性能有意识
  • 代码写得清晰整洁 → 说明代码素养好
  • 能讲清楚思路 → 说明沟通表达能力强
  • 遇到不会的题能一步步推导 → 说明学习能力和抗压能力不错

这些能力（逻辑思维、性能意识、代码素养、学习能力）和前端日常工作高度相关，只是不一定以「写算法题」的形式体现。

2. 前端真的不用算法吗？其实每天都在用，只是你没意识到

  • 你写 Array.prototype.sort → 底层是快排 / 归并 / TimSort。
  • 你用 Map / Set → 底层是哈希表 / 红黑树。
  • 你写 React/Vue → 内部有 diff 算法、Fiber 调度、响应式系统。
  • 你做性能优化 → 时间复杂度、空间复杂度、缓存策略。
  • 你处理树形数据（菜单、部门、评论）→ 树的遍历、递归、回溯。
  • 你做列表渲染优化 → 虚拟列表、二分查找。
  • 你做异步并发控制 → 队列、优先队列、调度算法。

区别只是：**你不用自己手写红黑树，但你需要理解它的原理，才能在出问题 / 做优化 / 选方案时有判断力。

3. 不同层级的工程师，算法的「用武之地」不一样

  • 初级前端：确实很少直接写算法，会用 API 就行。
  • 中级前端：开始做性能优化、复杂组件、工具库，需要算法思维来做方案选型和性能分析。
  • 高级前端 / 架构师：做框架、做基础库、做性能架构、做编译工具，天天和算法打交道。

面试考察算法，本质上是在考察你的上限——你有没有成长为高级工程师的潜力。

4. 为什么前端算法面试「看起来没用」？幸存者偏差

觉得算法没用的，往往是：

  • 做的业务比较简单（展示型页面），确实碰不到复杂算法。
  • 遇到了算法问题但没意识到（用了最朴素的写法，只是性能差一点但能跑）。

而那些做复杂交互、可视化、编辑器、3D、编译器、框架、基建的前端，算法是日常工具。

5. 现实一点的建议

哪怕你觉得工作中用不到，算法也是进大厂的敲门砖。

  • 大厂候选人多，需要筛选成本高，算法是个高效的筛选器。
  • 你可以不认同，但你得适应游戏规则才能进去。
  • 进去之后你可能发现真的用得少，但你至少有了机会。
  • 而且长远看，算法训练出来的思维能力，会潜移默化地提升你解决问题的方式——你自己可能都没意识到，但你的同事和 leader 能感受到。

总结一句话：算法不是前端的全部，但它是一块重要的基石。它衡量的不是「你今天能写多少业务」，而是「你未来能成长到什么高度」。

追问 2：如果只有两周时间准备算法面试，最有效率的刷题路线是什么？ ​

答 ​

两周时间很短，策略 > 努力。不能从第一题开始顺序刷，必须抓高频、抓重点。

第一周：打基础（每天 2~3 小时，约 30~40 道题）

目标：覆盖最高频的 6 大类，每类掌握模板题 + 几道变形。

Day 1-2：数组 + 双指针（约 8 道）

  • 双指针：两数之和 II、盛最多水的容器、移动零、删除有序数组重复项、反转字符串
  • 前缀和：和为 K 的子数组、连续子数组的最大和
  • 重点：快慢指针 vs 左右指针的区别，什么时候用哪种。

Day 3：链表（约 5 道）

  • 反转链表（迭代 + 递归）、合并两个有序链表、环形链表、链表中倒数第 k 个节点、删除链表倒数第 N 个节点
  • 重点：虚拟头节点、快慢指针、递归反转。

Day 4-5：树（约 8 道）

  • 前中后序遍历（递归 + 迭代）、层序遍历、最大深度、对称二叉树、翻转二叉树、验证 BST、最近公共祖先
  • 重点：递归思维，前中后序的区别和应用场景。

Day 6：BFS / DFS（约 6 道）

  • BFS：二叉树层序遍历、二叉树的右视图、岛屿数量
  • DFS：子集、组合、全排列
  • 重点：BFS 队列模板，DFS 回溯模板。

Day 7：滑动窗口 + 位运算（约 5 道）

  • 滑动窗口：无重复字符的最长子串、长度最小的子数组
  • 位运算：只出现一次的数字、汉明距离、2 的幂
  • 重点：滑动窗口模板（什么时候扩右指针、什么时候缩左指针）。

第二周：提高 + 模拟（每天 2~3 小时，约 20~30 道题 + 模拟）

Day 8-9：DP 入门 + 堆（约 6 道）

  • DP：爬楼梯、打家劫舍、最大子数组和、不同路径、最小路径和、最长递增子序列
  • 堆：前 K 个高频元素、第 K 大元素
  • 重点：DP 五步法，TopK 两种解法对比。

Day 10：前端场景算法（约 5 道）

  • LRU、防抖节流、Promise.all、数组扁平化、深拷贝、Promise 限流并发控制、树形数据处理
  • 重点：JS 语言特性和算法的结合。

Day 11-12：混合刷 + 复习

  • 随机挑题做，锻炼题型识别能力。
  • 复习第一周的错题，再做一遍。
  • 每道题控制在 20 分钟内想思路。

Day 13-14：模拟面试 + 查漏补缺

  • 找朋友/平台做模拟面试，计时 + 讲思路。
  • 刷目标公司的面经题、高频题。
  • 整理自己的错题本。
  • 最后一天放松，保持手感。

两周总题量：约 50~70 道。

关键原则：

  1. 质量 > 数量：一道题写出来 + 理解透 + 能复现，> 三道题看答案过一遍。
  2. 模板 > 题量：每类题掌握一个模板，遇到新题往模板上套。
  3. 说出来 > 写出来：面试是要讲思路的，光写不说扣分。
  4. 错题 > 新题：做过的题再做一遍，比刷新题有用。
  5. 中等题为主：easy 熟练了再碰 hard（两周时间 hard 题建议放弃，性价比低）。

如果时间更短（只有一周）怎么办？

砍掉后面的 DP、堆、位运算，只保留前 4 类（数组、链表、树、BFS/DFS），大约 30 道左右，把这些掌握透。这四类覆盖了大部分前端面试 60~70% 的算法题，性价比最高。

核心思路：抓大放小，先保证 80 分的基础分，再冲 20 分的加分题。基础分拿到了，大多数中小厂的算法面都能过。

追问 3：面试时一道题完全没思路怎么办？如何和面试官有效沟通才能拿到尽量多的分数？ ​

答 ​

面试时遇到完全没思路的题很正常，**每个人都会遇到。关键不是「你会不会」，而是「你怎么应对不会的题」。这本身也是面试考察的一部分。

绝对不要做的事：

  1. 不要沉默发呆

    • 沉默 5 分钟，面试官也尴尬，你也紧张。
    • 沉默 = 你不说，面试官不知道你在想什么，也不知道怎么帮你。

  2. 不要直接说「我不会」

    • 一说不会，这道题就 0 分了。
    • 哪怕你真的不会，也要尝试一下。

  3. 不要瞎写一通

    • 思路都没想清楚就写，越写越乱。
    • 不如先说思路，再写。

正确的应对步骤：

第一步：先确认理解，争取时间

「让我先确认一下题意……您是说……对吗？」 「能不能再举个例子吗？我确认一下输入输出。」

  • 复述题目 = 给自己争取思考时间。
  • 也确保你理解对了，避免答非所问。

第二步：从最朴素的思路开始说

「这道题我第一反应是……可能可以用暴力的方式做，就是……时间复杂度可能是 O(n²)……」

  • 哪怕你觉得暴力很蠢，也要说出来。
  • 暴力解至少说明你有思路，不是 0 分。
  • 从暴力出发，你可能自己就想到优化方向了。

第三步：说你的思考过程，哪怕是错的

「我在想……能不能用哈希表优化一下？因为……」 「或者用双指针？因为数组是有序的话……」 「不过这样好像有个问题……就是……」

  • **把你的思考过程说出来，哪怕走了，让面试官知道你在想什么。
  • 面试官可能会给你提示（「你可以想想……」
  • 有提示你就顺着提示走，这很正常。

第四步：实在没思路，主动求提示

「这道题我暂时没想到特别好的方法，您能给一点提示吗？」

  • 主动求提示比沉默强。
  • 面试官一般都会给提示的，他们也希望你做出来。
  • 给了提示能做出来，还是能拿不少分。

第五步：如果还是做不出来，坦诚 + 展现学习能力

如果真的完全不会，坦诚说：

「这道题我确实没什么思路，您能给我讲讲正确的思路是什么吗？我学习一下。

然后认真听，听完可以追问几个问题：

  • 「这个思路的关键 insight 是什么？我怎么才能想到？」
  • 「这类题有什么通用的套路吗？」

为什么这样反而加分？

因为面试官看到你：

  1. 诚实，不瞎编。
  2. 有学习欲，遇到不会的想学。
  3. 会提问，能抓住关键。

这些品质比「会不会某一道题」重要得多。

一些实用技巧：

1. 「我先想想用什么数据结构

  • 找最快查找 → 哈希表？
  • 找第 K 大 → 堆？
  • 有序数组 → 二分？
  • 最短路径 → BFS / Dijkstra？
  • 子数组和 → 前缀和？

从数据结构反推算法，有时候能打开思路。

2. 「这道题和我见过的哪道题像？

  • 是不是两数之和的变形？
  • 是不是爬楼梯的变形？
  • 是不是岛屿数量的变形？

很多新题都是旧题的变形，往你会的题上靠。

3. 「能不能先缩小问题规模

  • n=1 时答案是什么？
  • n=2 呢？
  • n=3 呢？

从小规模入手，找规律，可能就推出递推公式了。

4. 「能不能先写个伪代码？」

哪怕思路不完整，先把框架搭出来。写着写着可能就有思路了。

心态调整：

  • 一道题做不出来天塌不下来。 面试看的是整体表现，不是单题。
  • 你不会的题，别人大概率也不会。 大家水平都差不多，比的是谁心态稳、谁会沟通。
  • 面试是双向选择。 这家不行还有下一家，不用太有心理负担。
  • 把面试当练习。 面多了就有经验了，每次面完都总结，进步很快。

最后一句话：算法面试考的不只是算法，还有沟通能力、学习能力、抗压能力。把这些展现出来，哪怕题没做出来也可能过。
