问答题 18（中级）：useLayoutEffect vs useEffect ​

题目 ​

useLayoutEffect 和 useEffect 有什么区别？它们的执行时机分别是什么？为什么大多数情况下推荐用 useEffect？什么场景下必须用 useLayoutEffect？SSR 中使用 useLayoutEffect 为什么会有警告？

参考答案 ​

执行时机的核心差异 ​

要理解这两个 Hook 的区别，必须先搞清楚 React 更新 DOM 的完整流程：

触发更新 → 组件渲染（render phase） → DOM 变更（commit phase mutation）
     → 【useLayoutEffect 同步执行】 → 浏览器绘制（paint） → 【useEffect 异步执行】
1
2

两者的差异可以用一句话概括：useLayoutEffect 在浏览器绘制之前同步执行，useEffect 在浏览器绘制之后异步执行。

详细时间线 ​

以一次组件更新为例，完整顺序是：

  1. Render 阶段：React 调用组件函数，计算出新的虚拟 DOM，和旧的做 diff。（可中断）
  2. Commit 阶段 - Mutation：React 把 diff 结果应用到真实 DOM 上，DOM 树已经更新了，但用户还看不到。（同步，不可中断）
  3. Commit 阶段 - Layout：同步调用所有 useLayoutEffect 的清理函数，然后同步调用所有 useLayoutEffect 的回调。你在里面读 DOM 布局、改 DOM，都是在绘制之前做的。（同步）
  4. 浏览器绘制：浏览器把最新的 DOM 渲染到屏幕上，用户看到更新后的界面。
  5. Passive Effects：异步调用所有 useEffect 的清理函数和回调。（异步，不阻塞绘制）
jsx
function Example() {
  useEffect(() => {
    console.log('useEffect'); // 绘制后执行
  });

  useLayoutEffect(() => {
    console.log('useLayoutEffect'); // 绘制前执行
  });

  return <div>Example</div>;
}

// 控制台输出顺序：
// useLayoutEffect （先）
// useEffect       （后，在浏览器绘制之后）

为什么大多数情况用 useEffect？ ​

useEffect 是默认选择，原因有两个：

1. 不阻塞浏览器绘制，用户体验更好

useEffect 在绘制之后异步执行，不会阻塞浏览器把内容画到屏幕上。如果副作用里有耗时操作（比如复杂计算、大量 DOM 操作），用户至少能先看到页面，而不是盯着白屏等。

而 useLayoutEffect 是同步执行的，如果里面的代码耗时很长，浏览器就迟迟不能绘制，用户会感觉到页面卡顿、输入无响应。

2. 避免 SSR 警告（后面会详细讲）

服务端渲染时，useLayoutEffect 在服务端根本不会执行，可能导致客户端和服务端渲染出来的内容不一致，产生 hydration mismatch。

React 官方文档明确建议：除非你确定需要在绘制前读取或修改 DOM，否则一律用 useEffect。

什么时候必须用 useLayoutEffect？ ​

useLayoutEffect 的使用场景很明确——需要同步读取 DOM 布局信息，或者需要在用户看到之前修改 DOM，避免闪烁。

场景 1：读取 DOM 布局信息并据此更新状态 ​

假设你需要测量一个元素的高度，然后根据高度调整另一个元素的位置。如果用 useEffect，流程是：

  1. DOM 更新了，浏览器绘制了第一帧（元素在错误的位置）
  2. useEffect 执行，读到高度，更新 state
  3. 组件重新渲染，DOM 再次更新
  4. 浏览器绘制第二帧（元素在正确的位置）

用户会看到元素"跳一下"——先在错误位置出现，然后闪到正确位置。这就是所谓的视觉闪烁（visual flicker）。

用 useLayoutEffect 的话，读取和更新都在绘制前完成，浏览器只绘制一次正确的状态，用户看不到闪烁。

jsx
function Tooltip({ children, targetRef }) {
  const tooltipRef = useRef(null);
  const [position, setPosition] = useState({ top: 0, left: 0 });

  useLayoutEffect(() => {
    // 读取 target 和 tooltip 的尺寸
    const targetRect = targetRef.current.getBoundingClientRect();
    const tooltipRect = tooltipRef.current.getBoundingClientRect();

    // 计算位置，更新 state
    setPosition({
      top: targetRect.bottom + 8,
      left: targetRect.left + targetRect.width / 2 - tooltipRect.width / 2,
    });
  }, []); // 依赖根据实际情况填

  return (
    <div
      ref={tooltipRef}
      style={{ position: 'fixed', top: position.top, left: position.left }}
    >
      {children}
    </div>
  );
}

场景 2：在绘制前修改 DOM 样式/内容 ​

有些 DOM 修改如果在绘制后做，用户会明显看到"先这样再那样"的闪烁。比如：

  • 动态设置 CSS 变量或类名来控制布局
  • 初始化第三方库（比如图表库），需要基于 DOM 尺寸做计算
  • 根据内容自动调整 textarea 高度
jsx
function AutoResizeTextarea({ value }) {
  const textareaRef = useRef(null);

  useLayoutEffect(() => {
    // 把高度重置为 auto，再设为 scrollHeight，实现自适应高度
    // 如果放在 useEffect 里，用户会看到输入框先变矮再变高的闪烁
    textareaRef.current.style.height = 'auto';
    textareaRef.current.style.height = textareaRef.current.scrollHeight + 'px';
  }, [value]);

  return <textarea ref={textareaRef} value={value} />;
}

SSR 中的 useLayoutEffect 警告 ​

如果你在服务端渲染的项目里用了 useLayoutEffect，React 会在控制台打出警告：

Warning: useLayoutEffect does nothing on the server, because its effect cannot be
encoded into the server renderer's output format. This will lead to a mismatch
between the initial, non-hydrated UI and the intended UI.
1
2
3

为什么会有警告？ ​

原因很直接：服务端渲染是在 Node.js 环境里执行的，根本没有 DOM，也没有"浏览器绘制"这回事。 所以 useLayoutEffect 在服务端是不会执行的——它完全被跳过了。

但问题在于：你的组件在客户端 hydrate 之后，useLayoutEffect 会立即执行，可能修改 DOM。这就导致服务端渲染出来的 HTML 和客户端第一次渲染后的 DOM 不一致，产生 hydration mismatch。

而 useEffect 呢？它本来就是在客户端绘制之后才执行的，服务端不执行它完全符合预期，所以不会有 mismatch 问题。

怎么解决？ ​

根据场景不同，有几种方案：

方案 1：换成 useEffect（最推荐）

如果这个副作用不需要在绘制前执行，直接换成 useEffect 就好了，警告自然消失。90% 的场景都可以这么解决。

方案 2：用 suppressHydrationWarning 局部忽略

如果这个 DOM 差异只是初始状态的小问题，后续会被客户端修正，可以给元素加 suppressHydrationWarning 属性告诉 React "这个元素初次渲染不一致没关系"：

jsx
<div suppressHydrationWarning>{/* 内容可能因 useLayoutEffect 而不同 */}</div>
1

方案 3：SSR 时降级为 useEffect

写一个自定义 Hook，在服务端用 useEffect，在客户端用 useLayoutEffect：

jsx
import { useEffect, useLayoutEffect } from 'react';

// 服务端没有 document，用这个判断
const useIsomorphicLayoutEffect =
  typeof window !== 'undefined' ? useLayoutEffect : useEffect;

export default useIsomorphicLayoutEffect;
1

方案 4：只在客户端渲染

如果这个组件完全依赖 DOM 特性，可以用动态导入加 ssr: false（Next.js）或 useEffect + state 的方式让它只在客户端挂载。

一个简单的判断标准：如果你在 effect 里操作了 DOM 且用户能看到视觉变化，就想想会不会有闪烁——会的话就用 useLayoutEffect，否则就用 useEffect。

追问 1 ​

题目 ​

useLayoutEffect 内部如果触发了 setState，会发生什么？会多一次渲染吗？

答 ​

如果在 useLayoutEffect 里调用了 setState 触发更新，React 会在同一个 commit 阶段内同步地重新执行渲染和 layout effects，而不会等到下一次浏览器绘制。

具体流程是：

  1. 第一次 render + commit，DOM 更新
  2. 执行 useLayoutEffect，里面调用了 setState
  3. React 检测到有新的更新，立即同步地再跑一次 render + commit（还是在绘制之前）
  4. 再次执行 useLayoutEffect（这次 setState 依赖没变就不触发了）
  5. 浏览器最终只绘制一次最终状态

这意味着：虽然触发了两次渲染，但用户看不到中间状态，浏览器只绘制一次。 这也正是 useLayoutEffect 能避免闪烁的原因——所有的"调整"都在绘制前完成了。

相比之下，如果在 useEffect 里 setState，用户会看到两帧：第一帧是初始状态，第二帧是更新后的状态。

但要注意：虽然用户看不到中间状态，但两次渲染的性能开销是真实存在的。如果 useLayoutEffect 里的 setState 导致了大量重渲染，还是会阻塞绘制、让页面变慢。所以最佳实践是尽量减少 layout effect 中的状态更新，能在 render 阶段算出来的就别放到 effect 里。

追问 2 ​

题目 ​

类组件中，componentDidMount / componentDidUpdate 和这两个 Hook 的执行时机分别对应吗？

答 ​

componentDidMount 和 componentDidUpdate 的执行时机更接近 useLayoutEffect，而不是 useEffect。

具体来说：

  • componentDidMount / componentDidUpdate：在 DOM 更新之后、浏览器绘制之前同步调用。在里面读取 DOM 布局、setState 都不会产生闪烁（和 useLayoutEffect 一样，setState 会被同步处理，浏览器只绘制一次）。

  • useLayoutEffect：和生命周期方法的时机完全一致——DOM 更新后、绘制前同步执行。

  • useEffect：在绘制之后异步执行，比生命周期方法晚。

所以如果你从类组件迁移到函数组件，原来写在 componentDidMount 里的 DOM 测量/操作逻辑，应该放到 useLayoutEffect 里才能保持完全一致的行为，放 useEffect 可能会出现闪烁。

那为什么 React 不直接把 useEffect 设计成和生命周期方法一样同步执行呢？因为 React 团队认为大多数副作用不需要阻塞绘制，异步执行对用户体验更好。useEffect 是更"安全"的默认选项，useLayoutEffect 是给特殊场景留的逃生舱。这也是函数组件和类组件的一个设计理念差异。

追问 3 ​

题目 ​

useLayoutEffect 和 useEffect 的清理函数执行时机有什么不同？卸载时呢？

答 ​

两者的清理函数执行时机也有差异，分两种情况看：

更新时（依赖变化导致重新执行 effect）：

  • useLayoutEffect 的清理函数：在本次 DOM 更新之后、新的 layout effect 执行之前同步调用。
  • useEffect 的清理函数：在下一次 effect 回调执行之前调用（也是在绘制之后）。

顺序是：

旧 layout effect 清理 → DOM 更新 → 新 layout effect 执行 → 绘制 → 旧 effect 清理 → 新 effect 执行
1

卸载时（组件 unmount）：

  • useLayoutEffect 的清理函数：在 commit 阶段的 mutation 期间同步执行（DOM 被移除之前）。
  • useEffect 的清理函数：也是在 commit 阶段，但比 layout effect 的清理稍晚一点，但仍然是同步的（不是异步的）。

等等，useEffect 的清理在卸载时居然是同步的？没错，这是一个容易混淆的点。useEffect 的回调是异步调度的，但清理函数在卸载时是同步执行的。因为如果清理是异步的，组件都卸载了才执行清理，可能导致内存泄漏或操作已卸载的 DOM。

所以卸载时的顺序是：

layout effect 清理 → effect 清理 → DOM 移除
1

两者都是同步执行，但 layout effect 的清理在前。这也符合直觉：layout effect 管的是布局相关的清理，得在 DOM 还在的时候做；普通 effect 的清理（比如取消订阅、清定时器）不依赖 DOM 是否存在，晚一点也没关系。
