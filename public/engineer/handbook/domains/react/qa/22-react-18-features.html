问答题 22（高级）：React 18 新特性全解 ​

题目 ​

请系统梳理 React 18 的核心新特性：并发渲染（Concurrent Rendering）、自动批处理（Automatic Batching）、useTransition/useDeferredValue、Suspense 服务端支持、React Server Components、useId/useInsertionEffect 等。讲清每个特性解决什么问题、原理是什么、怎么使用。

参考答案 ​

React 18 是 React 历史上最重要的版本之一，它最大的变化是引入了并发渲染（Concurrent Rendering）——这是一个底层架构的根本性变革，很多新特性都是在并发渲染的基础上构建出来的。

一、并发渲染（Concurrent Rendering） ​

这是 React 18 最核心的特性，也是理解其他新特性的基础。

什么是并发渲染？ ​

在 React 18 之前，渲染是同步、不可中断的。一旦开始渲染，就必须一直跑完，中途不能停。如果有一个很大的更新（比如渲染一个一万项的列表），主线程就被占住了，用户的点击、输入都得不到响应，页面感觉"卡死了"。

并发渲染改变了这一点：渲染过程变成可中断、可恢复、可优先级化的。 React 可以：

  • 渲染到一半，停下来去处理更紧急的用户交互
  • 处理完紧急的，再回来继续之前的渲染
  • 甚至发现这次渲染已经过时了，直接丢弃整个结果重来

怎么实现的？ ​

底层是 Fiber 架构早就打好了基础——Fiber 把渲染拆成一个个工作单元，可以随时暂停。React 18 在此之上引入了Lane 模型（车道模型）来管理优先级：

  • 不同的更新有不同的优先级（Lane）
  • 高优先级更新可以打断低优先级更新
  • 低优先级更新被打断后，下次有时间再继续

比如：用户正在输入文字（高优先级），同时后台在过滤列表（低优先级）。过滤可以被输入打断，先保证输入流畅，输入停了再继续过滤。

并发渲染不是什么？ ​

  • 不是让应用变快：该算的还是得算，甚至因为被打断可能更慢。它的目标是让应用感觉更流畅——紧急的操作总能得到响应。
  • 不是默认开启所有组件：并发渲染是opt-in 的，只有用了新特性（useTransition、Suspense 等）的部分才会进入并发模式。普通的 setState 还是同步的，和 React 17 行为一致。
  • 不是多线程：仍然是单线程运行，只是有了优先级调度，不是真的同时跑多个渲染。

二、自动批处理（Automatic Batching） ​

什么是批处理？ ​

React 会把多个 setState 调用合并成一次重渲染，这就是批处理。它能减少重渲染次数，提升性能。

在 React 18 之前，批处理只在 React 事件回调 和 生命周期函数 中生效。如果在 Promise、setTimeout、原生事件回调等"异步场景"里 setState，就不会批处理，每个 setState 都会触发一次渲染。

jsx
// React 17 及之前
function handleClick() {
  setCount(c => c + 1);
  setFlag(f => !f);
  // 只重渲染一次！（React 事件回调中，批处理生效）
}

setTimeout(() => {
  setCount(c => c + 1);
  setFlag(f => !f);
  // 重渲染两次！（setTimeout 中，批处理不生效）
}, 1000);

React 18 的自动批处理 ​

React 18 之后，无论在哪里调用 setState，都会自动批处理——Promise 里、setTimeout 里、原生事件里，全部都会合并成一次渲染。

jsx
// React 18
setTimeout(() => {
  setCount(c => c + 1);
  setFlag(f => !f);
  // 只重渲染一次！（自动批处理，不管在哪里）
}, 1000);

原理 ​

自动批处理的底层是并发渲染的优先级调度。React 18 把所有同步触发的更新都放到同一个优先级批次里，到下一个微任务时再一起处理。所以不管你在多少个地方调用 setState，只要它们是在同一个宏任务里触发的，就会被合并到同一次渲染中。

如果不想批处理怎么办？ ​

极少数场景下，你可能需要强制立即刷新（比如读取最新的 DOM）。可以用 flushSync：

jsx
import { flushSync } from 'react-dom';

function handleClick() {
  flushSync(() => {
    setCount(c => c + 1);
  });
  // 这里 DOM 已经更新了，可以读到最新的
  console.log(divRef.current.offsetHeight);

  setFlag(f => !f); // 这个会单独再渲染一次
}

三、useTransition：过渡更新 ​

解决什么问题？ ​

当你有一个更新会导致大量渲染（比如过滤长列表、切换路由），这个更新会卡主线程，导致用户输入不流畅。

useTransition 让你把这个更新标记为**"过渡更新"（transition）**——它是低优先级的，可以被更高优先级的更新（比如用户输入）打断。

用法 ​

jsx
import { useState, useTransition } from 'react';

function App() {
  const [isPending, startTransition] = useTransition();
  const [filterText, setFilterText] = useState('');
  const [list, setList] = useState(bigList);

  const handleChange = (e) => {
    // 输入框更新是紧急的，立即执行
    setFilterText(e.target.value);

    // 列表过滤是过渡更新，低优先级，可以被打断
    startTransition(() => {
      setList(
        bigList.filter(item => 
          item.name.includes(e.target.value)
        )
      );
    });
  };

  return (
    <div>
      <input value={filterText} onChange={handleChange} />
      {isPending && <div>加载中...</div>}
      <List items={list} />
    </div>
  );
}

  • isPending：布尔值，表示过渡更新是否正在进行中。可以用来显示加载状态。
  • startTransition：函数，接收一个回调，回调里的 setState 都会被标记为过渡更新。

关键特性 ​

  1. 可被打断：过渡更新进行中如果来了更高优先级的更新（比如用户又输入了新字符），React 会中断当前的过渡渲染，先处理紧急更新，然后再重新开始过渡更新。
  2. 不显示中间状态：如果过渡更新被打断了又重新开始，用户不会看到中间的半成品状态，只会看到最终结果。避免了"列表闪一下"的问题。
  3. opt-in：默认所有更新还是紧急的。只有包在 startTransition 里的才是过渡更新。

什么时候用？ ​

  • 过滤、搜索大列表
  • 标签页切换（内容很多的标签页）
  • 路由切换（配合 Suspense）
  • 图表渲染、复杂计算

总之，更新后会有大量渲染、但不需要立即响应用户操作的场景，都可以考虑用 useTransition。

四、useDeferredValue：延迟更新的值 ​

解决什么问题？ ​

和 useTransition 类似，但适用场景不同。useTransition 是你自己控制更新（你调用 setState），useDeferredValue 是接收一个外部传进来的值，生成一个"延迟版本"。

用法 ​

jsx
import { useState, useDeferredValue, useMemo } from 'react';

function SearchResults({ query }) {
  // query 变化时，deferredQuery 会延迟更新
  const deferredQuery = useDeferredValue(query);

  // 用延迟的值计算列表，计算过程不会阻塞紧急更新
  const results = useMemo(() => 
    bigList.filter(item => item.name.includes(deferredQuery)),
    [deferredQuery]
  );

  return <List items={results} />;
}

当 query 变化时：

  • 组件先用旧的 deferredQuery 渲染（保持响应）
  • 后台用新的 query 重新计算
  • 计算完成后，再更新为新结果

和 useTransition 的区别 ​

特性useTransitionuseDeferredValue
控制粒度你主动控制哪些 setState 是过渡的你接收一个值，它自动延迟
使用场景你自己发起更新的地方接收 props / context 传来的值，下游计算很重
是否发起更新是，你调用 setState否，只是延迟值的传播
isPending有，明确知道是否在过渡没有，需要自己判断

简单说：能控制更新的源头就用 useTransition，控制不了（值是从上面传下来的）就用 useDeferredValue。

五、Suspense 服务端支持 & 数据获取 ​

Suspense 早在 React 16 就有了，但当时只能配合 React.lazy 做代码分割的加载状态。React 18 大大扩展了 Suspense 的能力：

1. 服务端 Suspense（Streaming SSR） ​

React 18 之前，SSR 是"全有或全无"的——服务端必须把整个页面都渲染完才能返回 HTML，客户端必须等所有 JS 下载完才能 hydrate。

React 18 的流式 SSR 配合 Suspense，改变了这一点：

  • 服务端：用 Suspense 包裹的组件如果数据还没好，就先返回其他部分的 HTML，等这个组件的数据好了，再通过流把它的 HTML 推送到浏览器。
  • 客户端：不用等所有 HTML 都到齐、所有 JS 都下载完再 hydrate。Suspense 包裹的部分可以等它自己的 JS 和数据都好了再 hydrate。

这叫 Streaming HTML + Selective Hydration（流式 HTML + 选择性水合）。

2. Suspense for Data Fetching ​

Suspense 还可以用来声明式地处理数据加载状态。组件在数据加载过程中"挂起"（suspend），React 自动显示最近的 Suspense fallback，数据好了再渲染组件。

jsx
function ProfilePage() {
  return (
    <Suspense fallback={<Loading />}>
      <ProfileHeader />  {/* 数据加载中时显示 Loading */}
      <ProfilePosts />
    </Suspense>
  );
}

  • 可组合：多个 Suspense 可以嵌套，每层有自己的 fallback。

目前 Suspense for Data Fetching 需要配合支持 Suspense 的框架（如 Relay、Next.js App Router）或库（SWR v2 也支持了 Suspense 模式）使用。

六、React Server Components（RSC） ​

React Server Components 是一个全新的组件模型，它不是 React 18 才发布的（目前还在持续演进中），但它是和 React 18 并发特性紧密相关的下一代架构。

核心思想 ​

把组件分成两类：

1. 服务端组件（Server Components）

  • 只在服务端渲染，代码不会发送到客户端
  • 可以直接访问数据库、文件系统、后端 API
  • 没有 state（因为不在客户端运行），不能用浏览器 API
  • 零 JS bundle 开销，用再多依赖也不增加客户端体积

2. 客户端组件（Client Components）

  • 在客户端渲染和 hydrate
  • 可以用 state、effect、浏览器 API
  • 代码需要发送到客户端，有 bundle 开销
  • 用 "use client" 指令标记

解决什么问题？ ​

  1. 减小 JS bundle 体积：大量只用于渲染的组件（比如 Markdown 渲染、数据格式化）都放在服务端，客户端不用下载它们的代码。
  2. 消除数据瀑布：服务端组件直接连数据库，不用"客户端请求 → 服务端查询 → 返回数据 → 客户端再请求子数据"这种瀑布。
  3. 简化架构：不用专门写 BFF 层，服务端组件直接访问后端。

RSC 目前最成熟的实现是 Next.js App Router，它深度集成了 RSC，是体验 RSC 最好的方式。

七、新的 Hook ​

1. useId ​

CSS-in-JS 库专用的 Hook，普通业务代码基本用不到。

它的执行时机在 DOM 变更之前、useLayoutEffect 之前，专门用来注入样式。CSS-in-JS 库可以在这个阶段插入 <style> 标签，避免后面读取布局时样式还没到位导致的闪烁。

}, [rules]);

八、其他重要变化 ​

1. 根 API 变化 ​

React 18 废弃了旧的 ReactDOM.render，改用 createRoot：

jsx
// React 17（仍然兼容，但不推荐）
import ReactDOM from 'react-dom';
ReactDOM.render(<App />, document.getElementById('root'));

// React 18
import { createRoot } from 'react-dom/client';
const root = createRoot(document.getElementById('root'));
root.render(<App />);

2. Strict Mode 更严格 ​

开发环境下的 StrictMode 现在会有意地"挂载 → 卸载 → 再挂载"组件，用来检测：

  • effect 的清理逻辑是否完善
  • 组件是否能承受多次渲染
  • 是否存在只执行一次的副作用假设

这有助于提前发现并发渲染下可能出问题的代码。

总结 ​

React 18 的新特性可以分成几个层次：

  1. 底层基石：并发渲染（Concurrent Rendering）——让渲染可中断、可优先级化。
  2. 并发特性：useTransition、useDeferredValue、Suspense 增强——基于并发渲染的用户级 API。
  3. 体验改进：自动批处理、useId——日常开发直接受益。
  4. 未来方向：React Server Components——改变 React 应用的架构范式。

升级到 React 18 基本是无痛的——不使用新特性的话，行为和 React 17 几乎一致。但一旦开始使用并发特性，你的应用就能在保持响应性的同时，处理更复杂的渲染任务。

追问 1 ​

题目 ​

React 18 中，"并发渲染"和"时间切片（Time Slicing）"是一回事吗？有什么区别？

答 ​

它们相关但不完全是一回事。时间切片是并发渲染的一种表现形式，并发渲染是更广义的概念。

时间切片（Time Slicing）：

  • 指的是把长时间的渲染任务切成多个小片段（每段 5ms 左右），每段执行完后把主线程还给浏览器，让浏览器响应用户输入、处理动画等。
  • 解决的是"长任务阻塞主线程"的问题。
  • 是并发渲染的一个基础能力。

并发渲染（Concurrent Rendering）：

  • 是一个更广泛的概念，指 React 有能力"同时"处理多个更新，按优先级调度、可中断、可恢复。
  • 它包含了时间切片，但不止于此——还包括优先级调度、可中断的更新、过渡更新、Suspense 挂起、批量处理等。
  • 时间切片是"怎么切任务"，并发渲染是"怎么管理、调度这些任务"。

打个比方：

  • 时间切片就像把一个大蛋糕切成小块，一口吃一块。
  • 并发渲染是一整套用餐礼仪——先吃哪块（优先级）、吃一半被打断了怎么办（可中断）、哪块可以慢点吃（过渡更新）、哪块掉了就不要了（丢弃过时的更新）。

所以时间切片是并发渲染的底层机制之一，并发渲染是建立在时间切片之上的一整套调度体系。

追问 2 ​

题目 ​

在并发渲染模式下，为什么说"渲染函数必须是纯函数"更加重要了？违反了会有什么后果？

答 ​

在并发渲染模式下，render 阶段（组件函数执行的阶段）可能被多次执行——被打断后重来、高优先级插入后重新开始、React 丢弃过期的重新渲染等。如果你的组件函数有副作用（不纯），这些副作用可能会被执行多次，导致意想不到的 bug。

具体来说，渲染阶段可能重复执行的场景：

  1. 更新被打断后重新执行：低优先级渲染到一半被高优先级打断，等处理完高优的，低优的要从头再来。组件函数会再执行一次。
  2. StrictMode 下的双调用：开发环境 StrictMode 会故意让组件渲染两次，帮你发现问题。
  3. Suspense 挂起后恢复：组件在渲染中挂起（suspend）了，等数据回来后会重新渲染。
  4. 并发模式下的重试机制：React 可能因为各种原因放弃当前渲染、重新开始。

追问 3 ​

题目 ​

useTransition 和普通的防抖（debounce）有什么区别？各有什么适用场景？

答 ​

useTransition 和防抖看起来都能"让慢操作不阻塞快操作"，但它们的原理和适用场景完全不同。

防抖（Debounce） ​

  • 原理：在事件停止触发一段时间后才执行操作。比如输入停止 300ms 后再过滤列表。
  • 控制方式：基于时间——延迟 N 毫秒执行。
  • 是否可中断：新的触发会取消旧的定时器，从新开始计时。
  • 感知：用户输入时完全不更新，等输入停了才突然跳变。

useTransition ​

  • 原理：把更新标记为低优先级，高优先级更新可以打断它。低优先级更新一有空闲就做，做完了就更新。
  • 控制方式：基于优先级调度——React 自己决定什么时候做、做多久。
  • 是否可中断：可以被高优先级更新打断，打断后重来。
  • 感知：如果计算很快，用户几乎感觉不到延迟；如果计算慢，也不会卡住输入，只是结果晚一点出来。配合 isPending 可以显示加载状态。

核心区别 ​

维度防抖useTransition
控制维度时间（延迟多久）优先级（谁先谁后）
延迟原因你主动设置延迟被高优先级任务打断才延迟
计算时机停止触发后才开始计算边触发边计算（被打断就重来）
结果延迟固定至少延迟 N 毫秒不固定，看主线程忙不闲
复杂度简单直观需要理解 React 优先级机制
适用范围任何 JS 场景仅 React 渲染相关的更新

适用场景 ​

用防抖的场景：

  • 接口请求（搜索联想、输入校验）——避免每次输入都发请求，减轻服务端压力。
  • 窗口 resize、scroll 事件处理——降低事件触发频率。
  • 任何"等用户操作停了再做"的场景。

用 useTransition 的场景：

  • 大量 DOM 渲染（长列表过滤、图表更新）——计算是在本地做的，但渲染量大，会卡主线程。
  • 标签页切换、路由切换——内容很多，但用户可以等一下。
  • 任何"更新后会有大量 React 渲染"的场景。

两者可以结合使用：

比如搜索功能：

  • 输入框 → 防抖 300ms → 发请求（用防抖减少请求次数）
  • 请求回来后 → 更新列表 → 用 useTransition 包裹（避免列表渲染卡主线程）

一个管"发不发请求"（I/O 层面），一个管"渲染卡不卡"（CPU 层面），各司其职。
