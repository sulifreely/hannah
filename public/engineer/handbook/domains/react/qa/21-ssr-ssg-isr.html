问答题 21（高级）：React 服务端渲染原理 ​

题目 ​

请阐述 React SSR 的完整流程（从服务端 renderToString 到客户端 hydrate），解释水合（hydration）是什么、为什么会有 hydration mismatch。再对比 SSG 静态生成、ISR 增量再生成的区别和适用场景，并结合 Next.js 说明它们的实现方式。

参考答案 ​

为什么需要服务端渲染？ ​

传统的客户端渲染（CSR）流程是：浏览器下载一个几乎空的 HTML 文件 → 下载 JS bundle → JS 执行 → 请求数据 → 渲染页面。用户在 JS 加载和执行完成之前，看到的是白屏。

服务端渲染（SSR）要解决的就是两个核心问题：

  1. 首屏性能：用户打开页面就能看到内容，不用等 JS 下载执行完。
  2. SEO（搜索引擎优化）：搜索引擎爬虫能直接爬到完整的 HTML 内容，不用等 JS 执行。

SSR 完整流程 ​

一次 SSR 请求的完整生命周期分为服务端和客户端两部分：

服务端阶段 ​

用户请求 → 服务端接收请求
    → 调用 renderToString 把 React 组件渲染成 HTML 字符串
    → 服务端把 HTML 塞进响应，返回给浏览器

服务端渲染时的特点：

  • 没有 DOM：renderToString 在 Node.js 环境运行，没有 window、document，所以组件中不能直接访问浏览器 API。
  • 没有生命周期：useEffect、useLayoutEffect 在服务端都不会执行。只有组件函数本身会被调用一次，生成初始 HTML。
  • 没有交互：生成的 HTML 是静态的，按钮点了没反应，输入框输不了字——因为 JS 还没加载。

客户端阶段 ​

浏览器拿到 HTML 后：

浏览器接收 HTML → 立即显示内容（用户能看到页面了）
    → 下载 JS bundle（这时候用户已经能看内容了）
    → JS 执行，进行 hydrate（水合）
    → 页面变成可交互的

水合（Hydration）是什么？ ​

Hydration 直译是"水合作用"，可以理解为：给干的（静态的）HTML 浇上水（JS 事件和状态），让它活起来。

更准确地说，hydration 是 React 在客户端将服务端渲染的 HTML 与 React 组件树进行匹配和绑定的过程：

  1. 构建 Fiber 树：React 在客户端重新渲染一遍组件树，生成虚拟 DOM（Fiber 树）。
  2. 匹配 DOM 节点：React 拿着虚拟 DOM，去和服务端渲染出来的真实 DOM 一一对比。
  3. 绑定事件监听器：给对应的 DOM 元素绑定 onClick、onChange 等事件处理函数。
  4. 建立状态关联：把组件的 state、effect 等和对应的 DOM 节点关联起来。

匹配成功后，React 就接管了这个 DOM 树，后续的更新就和普通客户端渲染一样了。

为什么不直接重新渲染整个 DOM？ 因为服务端已经渲染出 HTML 了，用户正在看呢。如果客户端再重新渲染一遍，用户会看到页面闪一下，性能也更差。hydration 的目标是复用已有 DOM，只绑定事件和状态，不重新创建 DOM 节点。

Hydration Mismatch（水合不匹配） ​

什么是 mismatch？ ​

hydration 过程中，React 期望客户端第一次渲染出来的虚拟 DOM 和服务端渲染的 HTML 结构完全一致。如果不一致，就会出现 hydration mismatch。

常见的不一致原因：

  1. 组件里用了浏览器特有的 API（window、document、localStorage），服务端没有这些东西，渲染结果不一样。
  2. 使用了随机数或当前时间：服务端渲染时生成了一个随机数，客户端渲染时又生成了另一个，结果对不上。
  3. 条件渲染依赖了只有客户端才有的数据：比如 if (typeof window !== 'undefined') 这种判断，服务端和客户端走了不同分支。
  4. HTML 结构被外部修改：比如 CMS 在 HTML 里插了额外的标签，或者浏览器自动修正了不合法的 HTML 结构（比如 <p> 里包了 <div>，浏览器会自动拆成两个 <p>）。

mismatch 会怎么样？ ​

React 18 之前，mismatch 时 React 会在控制台打个警告，然后从第一个不匹配的地方开始，把后面的 DOM 全部重新渲染。这意味着 hydrate 等于白做了——本来想复用 DOM，结果大部分都重建了，性能反而更差。

React 18 改进了策略：遇到 mismatch 时，会尝试更精细地修复，而不是整块重建。但仍然会有警告，且修复过程有性能开销，所以还是要尽量避免。

怎么避免 mismatch？ ​

  1. 服务端和客户端用同一份数据：把服务端获取的数据通过 HTML 注入（比如 window.__INITIAL_STATE__）传给客户端，客户端 hydrate 时用同一份数据渲染。
  2. 不要在渲染中访问浏览器 API：把用到 window 的逻辑放到 useEffect 里（effect 只在客户端执行）。
  3. 用 suppressHydrationWarning：如果某些地方确实允许不一致（比如时间戳），给元素加 suppressHydrationWarning 属性告诉 React 忽略这个差异。
  4. 动态导入 + ssr: false：完全依赖浏览器特性的组件，干脆不在服务端渲染，等客户端再挂载。

SSG：静态站点生成（Static Site Generation） ​

SSG 和 SSR 很像，区别在于渲染时机不同：

  • SSR：每次请求到来时，服务端实时渲染 HTML。
  • SSG：在构建时就把所有页面渲染成静态 HTML 文件，用户请求时直接返回现成的 HTML。
构建时运行 build 命令
    → 遍历所有路由
    → 每个路由调用 renderToString 生成 HTML
    → 把 HTML 文件存到 CDN / 静态服务器
    ↓
用户请求时
    → CDN 直接返回静态 HTML
    → 客户端 hydrate

优点：

  • 极快：不用每次请求都渲染，CDN 直接返回，响应速度是毫秒级。
  • 成本低：不需要 Node.js 服务，放静态服务器就行。
  • 稳定：没有服务端运行时，不会因为请求量变大而崩。

缺点：

  • 内容是固定的：构建后内容就不变了，要更新就得重新构建。
  • 页面多了构建慢：几千几万页的话，构建时间会很长。

适用场景：

  • 内容不经常变的页面：博客、文档、营销页、官网。
  • 数据更新频率低，可以接受"构建后才更新"的场景。

在 Next.js 中，默认就是 SSG——只要页面没有 getServerSideProps，就是构建时静态生成。

ISR：增量静态再生成（Incremental Static Regeneration） ​

ISR 是 Next.js 提出的一个概念，可以理解为**"SSG + 后台增量更新"**——大部分时候返回静态页面，但每隔一段时间或者在特定条件下，后台重新生成一遍这个页面，下次请求就拿到新内容了。

核心思路：

  1. 首次构建时生成静态页面（和 SSG 一样）。
  2. 用户请求时，先返回缓存的旧页面（保证快）。
  3. 如果发现页面"过期"了（超过了 revalidate 时间），就在后台重新生成新版本。
  4. 下一次请求时就能拿到新版本了。
用户请求 /post/123
    → 缓存里有这个页面，直接返回（旧版）
    → 后台检查：距离上次生成超过 revalidate 时间了吗？
    → 是的话，后台偷偷重新生成新版本
    ↓
下一个用户请求 /post/123
    → 返回新版本

Next.js 中的实现：

在 getStaticProps 中返回 revalidate 字段：

js
export async function getStaticProps({ params }) {
  const res = await fetch(`https://api.example.com/post/${params.id}`);
  const post = await res.json();

  return {
    props: { post },
    revalidate: 60, // 60 秒后重新生成（以秒为单位）
  };
}

优点：

  • 兼顾了 SSG 的速度和 SSR 的动态性——平时是静态页面的速度，内容又能定期更新。
  • 不用全量重新构建，只更新有变化的页面。

缺点：

  • 不是实时的——用户可能看到旧内容，要等下一次刷新才看到新的。
  • 是 Next.js 的特性，原生 React 不直接支持，要自己实现。

适用场景：

  • 内容会更新，但不需要实时更新的页面：新闻列表、商品详情、博客文章。
  • 接受"最多延迟 N 秒"的数据新鲜度。

Next.js 中的实现方式 ​

Next.js 把这几种渲染模式封装得非常优雅，通过不同的数据获取函数来区分：

SSG（静态生成）：

  • getStaticProps：构建时获取数据，生成静态页面。
  • getStaticPaths：动态路由场景下，指定哪些路径需要预生成。
  • 特点：构建时生成，CDN 缓存，速度极快。

SSR（服务端渲染）：

  • getServerSideProps：每次请求时获取数据，服务端实时渲染。
  • 特点：每次请求都渲染，数据始终最新，但响应慢一些。

ISR（增量再生成）：

  • getStaticProps + revalidate 选项。
  • 特点：静态生成 + 后台增量更新，兼顾速度和新鲜度。

客户端渲染（CSR）：

  • 不用任何数据获取函数，或者用 SWR / React Query 在客户端 fetch。
  • 特点：首屏是 loading，但后续交互流畅。

Next.js 的强大之处在于，同一个应用里可以混合使用多种渲染方式——首页用 SSG，用户中心用 SSR，商品详情用 ISR，搜索结果用 CSR。每个页面自己选最合适的模式。

React 18 对 SSR 的增强：Streaming SSR + Selective Hydration ​

React 18 之前的 SSR 是"全有或全无"的：服务端得把整个页面都渲染完才能返回 HTML，客户端得等整个页面的 JS 都下载完才能 hydrate。如果某个部分数据很慢，整个页面都卡住。

React 18 引入了两个重要改进：

1. Streaming SSR（流式 SSR）

不用等整个页面渲染完再返回，而是一边渲染一边通过流（stream）发送给浏览器。浏览器收到一部分就显示一部分，用户不用等全页。

用 renderToPipeableStream 代替 renderToString：

js
import { renderToPipeableStream } from 'react-dom/server';

const stream = renderToPipeableStream(<App />, {
  onShellReady() {
    // 外壳准备好了就开始流式发送
    stream.pipe(res);
  },
});

2. Selective Hydration（选择性水合）

客户端也不用等所有 JS 下载完再 hydrate。哪个部分的 JS 先下载完，就先 hydrate 哪个部分，用户可以提前和已经 hydrate 的部分交互。

这两个特性加起来，彻底改变了 SSR 的体验——不再是"等一个大的全页面"，而是"一点点出来，出来的就能用"。

追问 1 ​

题目 ​

SSR 中，服务端获取的数据怎么同步到客户端？为什么不能让客户端自己再 fetch 一遍？

答 ​

服务端获取的数据需要通过某种方式传递给客户端，这样客户端 hydrate 时才能用同样的数据渲染，避免 hydration mismatch。

标准做法：脱水（dehydration）与注水（rehydration）

  1. 服务端脱水：服务端渲染时获取了数据，把数据序列化成 JSON，塞进 HTML 的一个 <script> 标签里（通常是 window.__INITIAL_STATE__ 之类的全局变量）。
  2. 客户端注水：客户端 JS 执行时，从 window.__INITIAL_STATE__ 里读出数据，作为初始 state 传给组件，进行 hydrate。
jsx
// 服务端
const initialState = { user: data };
const html = renderToString(<App initialState={initialState} />);
const fullHtml = `
  <html>
    <body>
      <div id="root">${html}</div>
      <script>
        window.__INITIAL_STATE__ = ${JSON.stringify(initialState)};
      </script>
      <script src="/bundle.js"></script>
    </body>
  </html>
`;

// 客户端
const initialState = window.__INITIAL_STATE__;
hydrateRoot(document.getElementById('root'), <App initialState={initialState} />);

为什么不能让客户端自己再 fetch 一遍？

主要有三个原因：

  1. 避免重复请求：服务端已经 fetch 过一次了，客户端再 fetch 一遍浪费资源，也增加了服务端压力。对于数据量大的页面，浪费尤其明显。

  2. 避免闪屏：如果客户端自己重新 fetch，那 hydrate 时用的是空数据/默认数据，页面先显示空状态，等数据回来再变内容。用户会看到"从空到有"的闪烁，体验很差。而且这也会导致 hydration mismatch（服务端 HTML 有数据，客户端首次渲染没数据）。

  3. 保持一致性：服务端和客户端可能因为时间差、地域差等原因拿到不一样的数据（比如数据在这期间更新了），导致 mismatch。用同一份初始数据是最稳妥的。

当然，如果是后续的交互（比如用户点了分页），那客户端自己 fetch 是正常的。只有首屏的初始数据需要做水合同步。

追问 2 ​

题目 ​

SSR 中有哪些常见的性能陷阱？如何优化？

答 ​

SSR 不是银弹，用不好反而比 CSR 还慢。常见的性能坑和优化手段：

1. 服务端渲染慢，首字节时间（TTFB）长

服务端每次请求都要执行 React 渲染、查询数据库、调用 API，响应时间自然比直接返回静态文件慢。如果接口慢，用户就盯着白屏等。

优化：

  • 缓存：相同的请求不要每次都渲染。页面级缓存（CDN、Nginx 缓存）、组件级缓存（组件渲染结果缓存）、数据缓存（Redis 缓存接口数据）。
  • SSG / ISR：能静态生成就静态生成，不用每次都 SSR。
  • Streaming SSR：React 18 的流式渲染，不用等全页渲染完再返回，先发一部分出去，用户至少能看到骨架屏。
  • 优化接口：并行请求、减少不必要的数据、数据库加索引。

2. 客户端水合慢，页面"看起来有了但点不动"

用户已经能看到页面内容了，但 JS 还没下载完、hydrate 还没完成，这时候点击按钮没反应，输入框输不进去，体验很糟糕。这叫**"可交互时间（TTI）长"**的问题。

优化：

  • 代码分割：减小首屏 JS bundle 体积，让 hydrate 更快完成。
  • Selective Hydration：React 18 的选择性水合，优先 hydrate 用户正在交互的部分。
  • 渐进式水合：一部分一部分地 hydrate，先让关键区域可交互。
  • islands architecture（岛屿架构）：Astro、Marko 等框架的思路——页面大部分是静态 HTML，只有少量"交互岛屿"需要 hydrate，大大减少 JS 体积。

3. 服务端内存压力大

每次请求都要创建新的 React 组件树、执行渲染，高并发时 Node.js 服务内存会暴涨。而且如果不小心在服务端用了全局变量，还可能出现请求之间的数据串扰。

优化：

  • 不要在模块作用域缓存请求相关的数据，每个请求的上下文要隔离。
  • 合理设置实例数和限流：Node.js 单实例能扛的并发有限，该扩容就扩容。
  • 多用缓存：缓存命中就不用进渲染流程了。

4. 第三方库不兼容 SSR

很多库（尤其是 UI 组件库）默认假设有浏览器环境，直接在模块顶层访问 window，在服务端运行就报错。

优化：

  • 选用支持 SSR 的库。
  • 用动态导入（next/dynamic with ssr: false）让某些组件只在客户端渲染。
  • 用 typeof window !== 'undefined' 做判断，但要注意避免 hydration mismatch。

题目 ​

React Server Components（RSC）和传统 SSR 有什么区别？它解决了什么问题？

答 ​

React Server Components 和传统 SSR 看起来都是"在服务端渲染 React"，但设计理念和解决的问题完全不同。

传统 SSR 的本质：

  • 是一种首屏优化手段——把首屏 HTML 提前渲染出来，让用户更快看到内容。
  • 组件还是那些组件，只是在服务端多跑了一次 renderToString。
  • hydrate 之后，所有组件都还是客户端组件，该下载的 JS 一点没少。
  • 解决的是"首屏慢"的问题。

React Server Components 的本质：

  • 是一种架构范式——组件分为"服务端组件"和"客户端组件"两类。
  • 服务端组件只在服务端渲染，永远不会发送到客户端，也不会被 hydrate。
  • 只有标记了 "use client" 的客户端组件，才会被发送到客户端并 hydrate。
  • 解决的是**"JS 太多、数据请求瀑布"**的问题。

具体区别：

维度传统 SSRReact Server Components
组件模型所有组件都是客户端组件，服务端只是预渲染一遍明确区分服务端组件和客户端组件
JS 体积所有组件的 JS 都要发送到客户端只有客户端组件的 JS 才发送，大大减少 JS 体积
数据获取通常在路由层面统一获取（如 getServerSideProps）每个服务端组件可以自己获取数据，没有 props 层层传递
数据瀑布容易有瀑布（服务端拿数据 → 渲染 → 客户端 hydrate → 再拿数据）服务端直接连数据库/后端，减少网络往返
交互能力所有组件都能交互（hydrate 后）只有客户端组件能交互，服务端组件只输出 UI
状态管理所有组件都能有 state服务端组件不能有 state（因为它们不在客户端运行）
重新渲染导航时客户端路由切换，重新渲染导航时服务端重新渲染对应组件，返回新的 UI 片段

RSC 解决的核心问题：

  1. Zero-Bundle-Size 组件：服务端组件的代码不会发送到客户端，你可以放心地引入各种工具库、大依赖，不会增加 JS bundle 体积。

  2. 消除数据瀑布：服务端组件可以直接访问数据库、文件系统、后端服务，不用先把数据传到客户端再往下传。也不用"父组件请求数据 → 传给子组件 → 子组件再请求自己的数据"这种瀑布。

  3. 简化架构：以前要考虑 BFF 层、GraphQL、REST API，现在服务端组件直接连数据库，省了一层。

SSR 和 RSC 不是互斥的——RSC 可以和 SSR 配合使用。服务端组件在服务端渲染成初始 HTML（类似 SSR），客户端组件在客户端 hydrate。它们是互补的关系。

Next.js App Router 就是基于 RSC 构建的，是目前 RSC 最成熟的实现。
