问答题 24（中级）：React Router 原理与实现 ​

题目 ​

请阐述 React Router 的核心原理：两种路由模式（HashRouter vs BrowserRouter）的区别与实现、history API 的作用、路由匹配的机制、嵌套路由与动态路由的实现、<Link> vs <NavLink> 的区别，以及 withRouter 与 useHistory / useNavigate 的演进。

参考答案 ​

React Router 是 React 生态中最主流的路由库，它的核心思想是：路由和 UI 是同步的——URL 是什么样，页面就渲染成什么样。 它把 URL 的变化和 React 组件的渲染关联起来，让单页应用（SPA）也能有"页面切换"的体验。

一、两种路由模式：HashRouter vs BrowserRouter ​

React Router 提供了两种最常用的路由模式，它们的底层实现完全不同。

1. HashRouter（哈希模式） ​

原理：利用 URL 的 hash（# 后面的部分） 来模拟路由。hash 变化不会触发页面刷新，但会触发 hashchange 事件，React Router 监听这个事件来更新组件。

http://example.com/#/home        →  Home 组件
http://example.com/#/about       →  About 组件
http://example.com/#/user/123    →  User 组件
1
2
3

实现原理：

  • 监听 window.addEventListener('hashchange', callback)。
  • 读取 window.location.hash 来获取当前路由。
  • 导航时修改 window.location.hash 来触发 hash 变化。
jsx
// 简化版 HashRouter 原理
function HashRouter({ children }) {
  const [path, setPath] = useState(window.location.hash.slice(1) || '/');

  useEffect(() => {
    const handleHashChange = () => {
      setPath(window.location.hash.slice(1) || '/');
    };
    window.addEventListener('hashchange', handleHashChange);
    return () => window.removeEventListener('hashchange', handleHashChange);
  }, []);

  // 通过 context 把 path 传给下游的 Route 组件
  return <RouterContext.Provider value={path}>{children}</RouterContext.Provider>;
}

优点：

  • 兼容性好：支持所有浏览器，包括很老的 IE。
  • 不需要服务端配置：刷新页面不会 404，因为服务端只看 # 前面的部分，永远返回 index.html。
  • 部署简单：放静态服务器上就能跑。

缺点：

  • URL 丑：有个 # 号，不优雅。
  • SEO 不友好：搜索引擎通常不处理 hash 部分的内容（不过现在 Google 已经支持了，但其他搜索引擎不一定）。
  • 不能用锚点（#anchor）：因为 hash 已经被路由占用了。

适用场景：

  • 简单的内部系统、后台管理系统。
  • 部署环境不能改服务端配置的场景。
  • 需要兼容老浏览器的场景。

2. BrowserRouter（History 模式 / HTML5 模式） ​

原理：利用 HTML5 提供的 History API（pushState、replaceState、popstate 事件）来实现前端路由。URL 看起来和正常的多页应用一样，没有 # 号。

http://example.com/home        →  Home 组件
http://example.com/about       →  About 组件
http://example.com/user/123    →  User 组件
1
2
3

实现原理：

  • 用 history.pushState() 和 history.replaceState() 来修改 URL，不会触发页面刷新。
  • 监听 window.popstate 事件（浏览器前进/后退按钮触发）来响应 URL 变化。
  • 读取 window.location.pathname 获取当前路径。
jsx
// 简化版 BrowserRouter 原理
function BrowserRouter({ children }) {
  const [path, setPath] = useState(window.location.pathname);

  useEffect(() => {
    const handlePopState = () => {
      setPath(window.location.pathname);
    };
    window.addEventListener('popstate', handlePopState);
    return () => window.removeEventListener('popstate', handlePopState);
  }, []);

  // navigate 函数：调用 pushState 并手动更新 state
  const navigate = (to) => {
    window.history.pushState({}, '', to);
    setPath(to);
  };

  return (
    <RouterContext.Provider value={{ path, navigate }}>
      {children}
    </RouterContext.Provider>
  );
}

优点：

  • URL 美观：没有 # 号，和传统多页网站一模一样。
  • SEO 友好：搜索引擎可以正常抓取每个路由的内容。
  • 功能更强：可以用 state 在路由间传递数据（pushState 的第一个参数）。

缺点：

  • 需要服务端配合：用户刷新 /about 页面时，浏览器会向服务器请求 /about 这个路径，如果服务端没有配置 fallback 到 index.html，就会返回 404。
    • Nginx 配置：try_files $uri /index.html;
    • Apache 配置：FallbackResource /index.html
    • Express：用 history 中间件
  • 兼容性稍差：需要支持 HTML5 History API 的浏览器（IE10+，基本不是问题了）。

适用场景：

  • 面向用户的官网、营销页、内容站（SEO 重要的场景）。
  • 对 URL 美观度有要求的项目。
  • 大多数现代 Web 应用。

二、history 库：React Router 的底层基石 ​

React Router v4+ 自己不直接操作浏览器 history，而是依赖一个叫 history 的库。它封装了三种 history 模式的差异，提供统一的 API：

history 类型对应的 Router 组件适用场景
createBrowserHistory<BrowserRouter>现代浏览器，HTML5 History API
createHashHistory<HashRouter>兼容老浏览器或静态部署
createMemoryHistory<MemoryRouter>测试、SSR、React Native

history 库提供的核心 API：

  • history.location：当前位置对象（{ pathname, search, hash, state }）
  • history.push(path, state)：跳转到新路径（往历史栈里加一条记录）
  • history.replace(path, state)：替换当前路径（不新增历史记录）
  • history.go(n)：前进/后退 n 步
  • history.listen(callback)：监听地址变化

React Router 的所有路由组件（Router、Route、Link 等）都是基于这个 history 对象工作的。它把"URL 怎么变"和"组件怎么渲染"分开了——history 管 URL，React Router 管组件。

三、路由匹配原理 ​

React Router 的核心是"根据当前 URL，决定渲染哪些组件"。这个过程叫路由匹配。

1. 基本匹配：路径比较 ​

最简单的匹配：当前 pathname 和 Route 的 path 完全相等（或前缀匹配，取决于 exact 属性）。

React Router 内部用 path-to-regexp 这个库把路径字符串转换成正则表达式，然后用正则去匹配当前 URL。

path="/user/:id"
    ↓ 转换成正则
/^\/user\/([^\/]+)$/
    ↓ 匹配
/user/123  →  ✅ 匹配，params = { id: '123' }
/user      →  ❌ 不匹配
/user/123/posts → ❌ 不匹配（默认 exact 时）

3. 嵌套路由（Nested Routes） ​

嵌套路由是 React Router v6 的核心特性之一。路由可以像组件一样嵌套，父路由匹配了，才会尝试匹配子路由。父路由的组件通过 <Outlet /> 来渲染子路由的内容。

jsx
// 路由配置
<Routes>
  <Route path="/" element={<Layout />}>
    <Route index element={<Home />} />
    <Route path="about" element={<About />} />
    <Route path="users" element={<Users />}>
      <Route path=":id" element={<UserDetail />} />
    </Route>
  </Route>
</Routes>

// Layout 组件
function Layout() {
  return (
    <div>
      <Header />
      <Outlet /> {/* 子路由渲染到这里 */}
      <Footer />
    </div>
  );
}

匹配流程：

  1. URL 是 /users/123
  2. 先匹配父路由 / → 匹配成功，渲染 <Layout />
  3. 在 Layout 的 <Outlet /> 位置，继续匹配子路由 users → 匹配成功，渲染 <Users />
  4. 在 Users 的 <Outlet /> 位置，继续匹配子路由 :id → 匹配成功，渲染 <UserDetail />

嵌套路由让"布局嵌套"和"路由嵌套"对应起来，结构非常清晰。

4. 匹配优先级 ​

当有多个 Route 都能匹配当前 URL 时，React Router v6 会自动按**"具体程度"**排序，更具体的路由优先匹配。你不用再担心路由顺序了——比如 /user/new 和 /user/:id，无论顺序怎么写，/user/new 都会优先匹配。

这是 v6 相比 v5 的一个重大改进——v5 是按顺序匹配，写在前面的优先，所以必须把具体的路由放在动态路由前面，否则会被动态路由"吞掉"。v6 自动处理了这个问题。

四、导航方式 ​

1. Link 组件 ​

<Link> 是最常用的导航组件，它渲染成一个 <a> 标签，但点击时不会真的跳转页面，而是调用 history.push 来切换路由。

jsx
<Link to="/about">关于我们</Link>
// 渲染成：<a href="/about">关于我们</a>
// 点击时：调用 history.push('/about')，不刷新页面

为什么不直接用 <a> 标签？

  • <a> 标签点击会刷新页面，用户体验差。
  • <Link> 会阻止默认行为，用 history API 切换，页面不刷新，体验更流畅。
  • 但它仍然渲染成 <a> 标签，所以右键"在新标签页打开"仍然有效，SEO 也没问题。

2. NavLink 组件 ​

<NavLink> 是 <Link> 的特殊版本——当它指向的路由和当前 URL 匹配时，会自动加上 active 类名，方便做高亮效果。

jsx
<NavLink to="/home" className={({ isActive }) => isActive ? 'active' : ''}>
  首页
</NavLink>

五、React Router v6 的核心变化 ​

如果你熟悉 v5，v6 有几个重大变化需要知道：

  1. 用 <Routes> 替代 <Switch>：

    • v5: <Switch> 按顺序匹配第一个命中的
    • v6: <Routes> 自动按优先级匹配，不用关心顺序

  2. component/render → element：

    • v5: <Route path="/" component={Home} />
    • v6: <Route path="/" element={<Home />} />（直接传 React 元素，更灵活）

  3. useHistory → useNavigate：

    • 更简洁的 API，返回一个函数而不是对象。

  4. 嵌套路由内置支持：

    • v5 嵌套路由很麻烦，要自己写 match.path
    • v6 内置了 <Outlet /> 和相对路径，嵌套路由非常自然

  5. 移除了 withRouter：

    • 全面拥抱 Hook 时代。

总结 ​

React Router 的原理其实很简单：

  1. 监听 URL 变化（hashchange 或 popstate）
  2. 根据新 URL 匹配路由（path-to-regexp 把路径转正则，匹配对应组件）
  3. 重新渲染匹配的组件（通过 Context 通知下游更新）

理解了这个核心循环，就理解了 React Router 的 80%。剩下的就是嵌套路由、动态参数、导航方式等 API 层面的东西了。

追问 1 ​

题目 ​

浏览器的前进后退按钮是如何触发 React Router 更新的？为什么点击 Link 不会触发 popstate 事件？

答 ​

这是一个很好的问题，涉及到 History API 的细节。分两种情况看：

1. 前进/后退按钮 → 触发 popstate 事件 ​

当用户点击浏览器的前进、后退按钮，或者调用 history.back()、history.forward()、history.go(n) 时，浏览器会触发 popstate 事件。

js
window.addEventListener('popstate', (event) => {
  console.log('路由变了', window.location.pathname);
  console.log('state:', event.state); // pushState 时传入的 state 对象
});
1
2
3
4

React Router 监听这个事件，当事件触发时，读取最新的 location.pathname，更新内部状态，从而触发组件重新渲染、匹配新的路由。

2. 点击 Link → 不触发 popstate 事件 ​

但是！调用 history.pushState() 或 history.replaceState() 不会触发 popstate 事件。这是 History API 的设计——只有用户主动操作浏览器（前进后退）才触发，代码主动调用 pushState 不触发。

那 React Router 是怎么知道 URL 变了呢？

答案很直接：React Router 自己知道 URL 变了，因为它就是调用 pushState 的那一方。

当你点击 <Link> 时：

  1. Link 组件拦截点击事件，调用 e.preventDefault() 阻止默认跳转。
  2. 调用 history.push(to)（也就是调用 history.pushState）。
  3. history 库在 pushState 之后，手动触发自己的监听器。
  4. React Router 收到通知，更新内部的 location state。
  5. 组件重新渲染，匹配新路由。

简化一下：

js
// history 库内部的简化实现
function createBrowserHistory() {
  const listeners = [];

  const push = (to, state) => {
    window.history.pushState(state, '', to); // 修改 URL
    // 手动通知所有监听器（pushState 不会触发 popstate）
    listeners.forEach(listener => listener({ pathname: to, state }));
  };

  const listen = (listener) => {
    listeners.push(listener);
    return () => { /* 移除监听 */ };
  };

  // 只有浏览器前进后退才会触发 popstate
  window.addEventListener('popstate', () => {
    listeners.forEach(listener => listener(window.location));
  });

  return { push, listen, /* ... */ };
}

所以 React Router 的更新来源有两个：

  • 浏览器操作（前进后退）：通过 popstate 事件 → history 库的 listener → React Router 更新。
  • 代码主动导航（Link、navigate）：直接调用 history.push/replace → history 库手动触发 listener → React Router 更新。

两者殊途同归，最终都是通过 history 库的监听器机制来通知 React Router 更新的。这样就统一了两种场景的处理逻辑。

补充：为什么 pushState 不触发 popstate？ ​

这是 HTML5 规范有意设计的。原因是：如果 pushState 也触发 popstate，那你在 popstate 监听器里又 pushState，就会无限递归。而且，代码主动调用 pushState，代码自己本来就知道 URL 变了，不需要通过事件来通知。事件是给"浏览器导致的、代码控制不了的变化"用的。

追问 3 ​

题目 ​

MemoryRouter 是做什么的？它的原理是什么？什么场景下会用到？

答 ​

MemoryRouter 是 React Router 提供的第三种 Router，它不依赖浏览器的 URL，而是把路由历史存在内存里。

原理 ​

和 HashRouter、BrowserRouter 不同，MemoryRouter 完全不操作 window.location，也不监听浏览器事件。它自己在内存里维护一个"历史栈"，所有路由操作都在这个内存栈上进行。

jsx
// 简化版 MemoryRouter 原理
function MemoryRouter({ initialEntries = ['/'], initialIndex = 0, children }) {
  const [entries, setEntries] = useState(initialEntries);
  const [index, setIndex] = useState(initialIndex);

  const navigate = (to, { replace = false } = {}) => {
    if (replace) {
      const newEntries = [...entries];
      newEntries[index] = to;
      setEntries(newEntries);
    } else {
      // push：截断当前位置后面的记录，加新的
      const newEntries = entries.slice(0, index + 1);
      newEntries.push(to);
      setEntries(newEntries);
      setIndex(newEntries.length - 1);
    }
  };

  const go = (n) => {
    setIndex(Math.max(0, Math.min(entries.length - 1, index + n)));
  };

  const currentPath = entries[index];

  return (
    <RouterContext.Provider value={{ path: currentPath, navigate, go }}>
      {children}
    </RouterContext.Provider>
  );
}

适用场景 ​

1. 单元测试 / 集成测试

这是最常见的场景。测试组件时，你不需要真的改 URL，也不需要浏览器环境。用 MemoryRouter 可以在 Node.js 环境（jsdom）里测试路由逻辑，而且可以通过 initialEntries 控制初始路由。

3. React Native / 嵌入式环境

React Native 里没有浏览器的 URL 概念，路由也用 MemoryRouter 的思路（不过 React Native 通常用 react-navigation 而不是 React Router，但原理类似）。

4. 嵌套应用 / 微前端

主应用有自己的路由，子应用也有自己的路由，但子应用的路由不想影响主应用的 URL，就可以用 MemoryRouter 把路由隔离在内部。

5. 弹窗 / 抽屉的"内部路由"

比如一个多步骤的弹窗，你想在弹窗内部实现"上一步/下一步"的路由式导航，但又不想改变浏览器 URL，就可以用 MemoryRouter。

总结 ​

MemoryRouter 是一个"不跟浏览器挂钩"的纯内存路由实现。它的 API 和其他 Router 完全一致，只是底层操作的是内存数组而不是浏览器地址栏。需要路由逻辑但不需要真实 URL 变化的场景，都是 MemoryRouter 的用武之地。
