问答题 20（高级）：React 性能优化全景图 ​

题目 ​

请从组件层、状态层、列表层、渲染层、构建层等多个角度，系统梳理 React 应用的性能优化手段。讲清每个手段解决什么问题、原理是什么、什么时候该用，以及常见的误用。

参考答案 ​

React 的性能问题归根结底就两类：渲染得太多（不必要的重渲染）和渲染得太慢（单次渲染开销大）。性能优化的全景图，就是围绕这两个问题，在不同层面给出的解决方案。

一、组件层：减少不必要的重渲染 ​

父组件重渲染时，默认所有子组件都会跟着重渲染一遍，哪怕子组件的 props 完全没变。组件层优化的目标就是：props 没变的子组件，跳过重渲染。

1. React.memo：组件级缓存 ​

React.memo 是一个高阶组件，它包裹你的组件后，会对新旧 props 做浅比较，如果 props 没变，就跳过重渲染，直接复用上次的渲染结果。

jsx
const Child = memo(({ name, age }) => {
  console.log('Child 渲染了');
  return <div>{name} - {age}</div>;
});

4

原理：memo 在 Fiber 节点上标记了 MemoComponent 类型，协调过程中如果发现 props 浅相等，就跳过该子树的 render 阶段。

注意：

  • 默认是浅比较（Object.is 比较每个 prop），如果 props 里有对象/数组/函数，引用变了就会失效。
  • 可以传第二个参数自定义比较函数：memo(Comp, areEqual)，返回 true 表示相等（跳过渲染）。
  • 不要滥用：memo 本身有比较的开销，简单组件比较的时间可能比重新渲染还长。

2. useMemo：缓存计算结果 ​

当组件里有昂贵的计算（比如大数组过滤、复杂数据转换），每次渲染都重算会浪费性能。useMemo 把计算结果缓存起来，依赖不变就复用上次的值。

jsx
const filteredList = useMemo(() => {
  return bigList.filter(item => item.active && item.score > 60);
}, [bigList, threshold]);
1
2
3

原理：把计算函数和依赖数组存起来，每次渲染时对比依赖，没变就直接返回缓存的值。

注意：

  • React 文档明确说 useMemo 是性能提示，不是语义保证——React 有权在内存紧张时丢弃缓存重新计算。所以计算函数必须是纯函数，不能依赖它"只执行一次"。
  • 别什么都 memo。简单计算（比如数组 map 几个元素）直接算就行，缓存的开销可能比计算还大。

3. useCallback：缓存函数引用 ​

当你把函数作为 props 传给 memo 子组件时，如果父组件每次渲染都创建新的函数引用，memo 就失效了。useCallback 缓存函数引用，依赖不变就返回同一个函数。

jsx
const handleClick = useCallback((id) => {
  setSelected(id);
}, []);
1
2
3

原理：useCallback(fn, deps) 等价于 useMemo(() => fn, deps)，一个专门给函数用的语法糖。

什么时候不需要：

  • 函数没有传给 memo 子组件，也没出现在其他 Hook 的依赖数组里。
  • 子组件本身没被 memo 包裹，传稳定引用也不会减少渲染。

三者的关系 ​

三者经常配合使用，形成一条优化链：

父组件用 useCallback 稳定函数引用
     ↓
父组件用 useMemo 稳定对象/数组引用
     ↓
子组件用 React.memo 浅比较 props，没变就跳过渲染

二、状态层：从根源减少渲染次数 ​

组件层优化是"治标"——状态变了但子组件不关心就不渲染。状态层优化是"治本"——让状态变化的影响面尽可能小。

1. 状态下沉（State Colocation） ​

把状态放到尽可能低的组件里，也就是"谁需要这个状态，就把状态放在谁那里"。状态的影响范围越小，重渲染的组件就越少。

jsx
// ❌ 不好：状态在父组件，父组件重渲染时所有子组件都跟着重渲染
function Parent() {
  const [input, setInput] = useState(''); // 只有 SearchBar 用到
  const [list, setList] = useState([]);
  return (
    <div>
      <SearchBar value={input} onChange={setInput} />
      <ItemList items={list} />
      <Footer /> {/* Footer 根本不需要 input，但每次输入都会重渲染 */}
    </div>
  );
}

// ✅ 好：状态下沉到真正需要它的组件
function SearchBar() {
  const [input, setInput] = useState(''); // 状态移到子组件里
  return <input value={input} onChange={e => setInput(e.target.value)} />;
}

状态下沉是成本最低、收益最高的优化手段，优先级远高于 memo/useMemo。很多人一上来就加 memo，其实只要把状态搬个家，问题就解决了。

2. 状态拆分 ​

不要把不相关的状态塞到一个对象里。一个 state 对象里只要有一个字段变了，所有依赖这个 state 的组件都会重渲染。拆成多个独立的 state，变化的影响面更小。

jsx
// ❌ 不好：改 user 时，只关心 theme 的组件也会重渲染
const [appState, setAppState] = useState({ user: null, theme: 'light', lang: 'zh' });

// ✅ 好：拆成多个 state，各变各的，互不影响
const [user, setUser] = useState(null);
const [theme, setTheme] = useState('light');
const [lang, setLang] = useState('zh');

3. 原子化状态管理 ​

当状态需要跨组件共享时，Context 是最方便的方案，但 Context 有个问题：Provider 的 value 一变，所有消费者都重渲染，哪怕消费者只关心 value 里的一个字段。

解决方案是原子化状态库（Zustand、Jotai、Recoil）——状态被拆成一个个独立的原子（atom 或 selector），组件订阅哪个原子，只有那个原子变化时才重渲染。

jsx
// Jotai：每个 atom 独立，订阅粒度细
const countAtom = atom(0);
const nameAtom = atom('');

function Counter() {
  const [count, setCount] = useAtom(countAtom); // 只订阅 count
  // name 变了，这个组件不会重渲染
}

4. 服务端状态与客户端状态分离 ​

别把服务端数据（用户列表、文章详情）存在全局状态库里。这类数据的本质是缓存，应该用专门的服务端状态库（TanStack Query、SWR、Apollo Client）来管理。

这些库内置了：

  • 缓存与重取策略
  • 自动刷新
  • 分页 / 无限滚动
  • 乐观更新
  • 请求去重

把服务端状态从客户端状态里剥离出来，全局状态库里只剩下真正的 UI 状态，既减少了重渲染，又省了自己写一堆 loading/error 逻辑的功夫。

三、列表层：大数据列表的性能优化 ​

列表是性能问题的重灾区。一个长列表，哪怕每个列表项很简单，成百上千个一起渲染也会卡。

1. 正确使用 key ​

这是列表优化的第一步，也是最基础的一步。key 帮助 React 识别哪些元素改变了、哪些是新增的、哪些被删除了。

不要用 index 作为 key（除非列表是静态的、不会重新排序）。当列表项被插入、删除或重新排序时，index 作为 key 会导致 React 错误地复用 DOM，产生 bug 和不必要的重渲染。

jsx
// ❌ 不好：用 index 当 key
list.map((item, index) => <Item key={index} {...item} />);

// ✅ 好：用唯一 ID
list.map(item => <Item key={item.id} {...item} />);

2. 虚拟列表（Virtual List） ​

如果列表有成百上千项，全部渲染到 DOM 里既占内存又慢。虚拟列表的思路是：只渲染可视区域内的列表项，上下各留一点缓冲，滚动时动态替换内容。

│   Item 103    │
│   Item 104    │

3. 列表项 memo 化 ​

列表项通常是纯展示组件，props 变化不频繁。给列表项包一层 React.memo，可以避免父组件重渲染时所有列表项跟着重渲染。

jsx

  <div onClick={() => onClick(item.id)}>{item.name}</div>
));

四、渲染层：降低首屏和交互时的渲染压力 ​

1. 懒加载与代码分割（Code Splitting） ​

把代码拆成多个 chunk，用户访问到哪个页面/功能才加载对应的代码，减少首屏需要下载和解析的代码量。

React.lazy + Suspense：

jsx
const Dashboard = lazy(() => import('./Dashboard'));

function App() {
  return (
    <Suspense fallback={<Loading />}>
      <Dashboard /> {/* 渲染时才动态加载 Dashboard 的代码 */}
    </Suspense>
  );
}

常见分割粒度：

  • 路由级分割：每个路由页面一个 chunk，最常见也最推荐。
  • 组件级分割：大组件（比如富文本编辑器、图表库）单独分割。
  • 功能级分割：某些不常用的功能（比如导出、设置面板）按需加载。

2. 组件懒挂载 ​

有些组件虽然在页面上，但初始状态不可见（比如弹窗、抽屉、折叠面板）。没必要一开始就渲染它们，可以等用户打开时再渲染。

jsx
function Modal({ open, children }) {
  if (!open) return null; // 不渲染，自然也就不会执行里面的 effect 和计算
  return <div className="modal">{children}</div>;
}

更极端一点，第一次打开后可以保留渲染结果（避免关闭再打开时重新初始化），用 CSS 控制显隐。看场景取舍：初始化慢就保留 DOM，内存紧张就销毁重建。

3. 防抖与节流 ​

高频事件（scroll、resize、mousemove、input）触发的状态更新，如果不加处理，一秒内可能触发几十次重渲染。

  • 防抖（debounce）：事件停止触发一段时间后才执行一次。适合搜索输入框——用户停止输入 300ms 后再发请求。
  • 节流（throttle）：每隔一段时间最多执行一次。适合滚动事件——16ms 内最多执行一次，保证 60fps。
jsx
// 用 useDebounce Hook
const [searchTerm, setSearchTerm] = useState('');
const debouncedTerm = useDebounce(searchTerm, 300);

useEffect(() => {
  fetchResults(debouncedTerm);
}, [debouncedTerm]);

4. 长任务拆分（并发渲染） ​

React 18 的并发渲染特性（useTransition、useDeferredValue）可以把更新分成紧急和非紧急两类，紧急更新优先处理，非紧急更新可以被打断，让页面保持响应。

const [isPending, startTransition] = useTransition();
const [filterText, setFilterText] = useState('');

const handleChange = (e) => {
  setFilterText(e.target.value); // 紧急更新：输入框立即响应
  startTransition(() => {
    setList(bigList.filter(/* 慢计算 */)); // 非紧急：列表过滤可以延后
  });

五、构建层：减小包体积，加快加载速度 ​

1. Tree Shaking ​

Tree Shaking 是打包工具（Rollup、Webpack、Vite）的优化手段，把没用到的代码从最终 bundle 里剔除掉。

前提条件：

  • 使用 ES Module 语法（import / export），不要混用 CommonJS。
  • 确保第三方库支持 tree shaking（看 package.json 里有没有 sideEffects 字段、有没有 module 字段）。
  • 避免副作用导入（比如 import './polyfill'）。

2. 代码压缩与优化 ​

生产构建必须开启：

  • JS 压缩（Terser / esbuild / swc）：去掉空格、重命名变量、死代码消除。
  • CSS 压缩：去掉空格、合并选择器。
  • 图片优化：压缩图片、用 WebP/AVIF 格式、雪碧图。
  • Gzip / Brotli 压缩：服务端开启，传输体积能小 60%~80%。

3. 第三方库优化 ​

  • 用轻量替代方案：moment.js → dayjs / date-fns；lodash → 按需引入或用原生 API；antd → 按需加载或用更小的组件库。
  • 按需引入：import { debounce } from 'lodash-es' 而不是 import _ from 'lodash'。
  • 外部化（externals）：大库用 CDN，不打进 bundle 里。
  • 代码分割：大的第三方库单独拆成 vendor chunk，利用浏览器缓存。

4. 预加载与预获取 ​

  • preload：提前加载当前页面需要的关键资源（字体、关键 JS）。
  • prefetch：预加载下一个页面可能需要的资源，用户跳转到下一页时就已经缓存好了。

六、性能优化的正确姿势 ​

先测量，再优化 ​

不要凭感觉优化！ 先用工具找到真正的瓶颈：

  • React DevTools Profiler：看哪些组件重渲染了、为什么重渲染、渲染花了多久。
  • Chrome DevTools Performance：看主线程哪里卡、长任务在哪里。
  • Lighthouse：整体性能评分和建议。

找到瓶颈后再针对性优化，否则很可能花了大力气优化了一个根本不慢的地方。

优化的优先级 ​

  1. 业务逻辑优化：算法优化、减少不必要的计算，收益最大。
  2. 状态层优化：状态下沉、状态拆分、原子化管理，从根源减少渲染。
  3. 组件层优化：memo、useMemo、useCallback，减少不必要的重渲染。
  4. 列表层优化：虚拟列表，处理大数据量。
  5. 渲染层优化：懒加载、并发渲染，提升感知性能。
  6. 构建层优化：减小包体积，加快加载速度。

避免过度优化 ​

优化是有成本的：

  • 代码复杂度上升（依赖数组、memo 嵌套）
  • 维护难度增加（改一个 prop 可能要改一堆依赖）
  • 缓存本身也有开销（比较 props、存缓存值）

默认不用优化，测出问题再优化。 优化完也要再测一遍，确认真的变快了。

追问 1 ​

题目 ​

为什么 React 默认不帮我们做"自动 memo"，而是要开发者手动加 memo/useMemo/useCallback？

答 ​

这是一个很好的问题，React 团队当然知道自动 memo 更方便，但他们选择不做，有几个深层原因：

1. 浅比较也有成本

如果每个组件默认都 memo，那每次父组件渲染都要对所有子组件的 props 做一次浅比较。对于简单组件（比如就渲染一个 div），比较 props 的时间可能比重新渲染还长。React 选择"默认全量重渲染 + 开发者手动优化"，是相信大多数组件重渲染的成本很低，不值得默认加比较开销。

2. 引用相等的判断标准不明确

什么时候该认为"props 没变"？是浅比较还是深比较？深比较性能太差，浅比较又容易失效。React 不替你做这个决定，而是把选择权交给你——你可以用 memo 默认浅比较，也可以传自定义比较函数，甚至用 useMemo 自己控制粒度。

3. 保持简单性和可预测性

默认全量重渲染的心智模型非常简单：父组件渲染，子组件就跟着渲染。开发者很容易理解和预测。如果默认 memo 了，你得时刻想"这个组件会不会不更新？是不是被 memo 拦住了？"反而增加了心智负担。

4. 编译器方向的探索

不过，React 团队也在探索"编译器自动优化"的方向。React Forget（React 编译器）就是这样一个项目——它可以在编译时自动分析哪些组件需要 memo、哪些值需要缓存，自动插入 memo/useMemo/useCallback，开发者不用手动写了。这可能是未来的方向。

目前阶段，React 的哲学是：默认简单，需要时手动优化，未来可能编译器自动处理。

追问 2 ​

题目 ​

"状态提升"和"状态下沉"不矛盾吗？什么时候该提升，什么时候该下沉？

答 ​

不矛盾，它们是两个不同维度的优化，适用场景不同：

状态下沉：为了性能——状态放得越低，影响范围越小，重渲染的组件越少。

状态提升：为了共享——当多个组件需要同一个状态时，得把状态提到它们的公共祖先里，大家才能都读到。

正确的做法是 LIFT-UP-Then-PUSH-DOWN：

  1. 先把状态提升到需要共享它的最低公共祖先（LCA）。
  2. 然后看看这个祖先下面的子组件，哪些不需要这个状态——能不能把不需要的那部分抽出去，或者用 memo 优化。

举个例子：

       App
      /   \
  Header  Main
          /  \
      List   Detail
1
2
3
4
5

如果 List 和 Detail 需要共享 selectedId，那把 selectedId 提升到 Main 就够了，不用提到 App。这样 Header 不会因为 selectedId 变化而重渲染。

再配合状态拆分：如果 Main 里既有共享状态又有自己的本地状态，把本地状态下沉到对应的子组件，Main 只留共享的部分。

一个经验法则：状态放在"它能工作的最低层级"。既保证所有需要它的组件都能拿到，又尽量减少影响范围。

追问 3 ​

题目 ​

React 18 的并发渲染（Concurrent Rendering）和传统的性能优化有什么本质区别？它解决了什么以前解决不了的问题？

答 ​

传统的性能优化（memo、虚拟列表、防抖等）解决的是**"少渲染"的问题——通过各种手段减少渲染的数量或范围。但只要渲染发生了，它就是同步、不可中断**的。一旦有一个大的更新（比如过滤一个一万条的列表），主线程就被占住了，用户的点击、输入都得不到响应，页面看起来"卡死了"。

并发渲染解决的是**"渲染不阻塞"**的问题——它把渲染过程变成可中断、可优先级调度的。紧急的更新（比如用户输入、点击）可以打断不那么紧急的更新（比如列表过滤、图表渲染），先处理完紧急的，再回来处理慢的。这样页面始终保持响应，用户不会感觉到"卡"。

本质区别：

  • 传统优化：减少渲染量（让渲染变快）
  • 并发渲染：让渲染可中断（让渲染不阻塞）

以前解决不了的问题：

  1. 大列表的即时过滤：以前过滤一万条数据会卡 200ms，输入框都动不了。现在用 useDeferredValue，输入框立即响应，列表慢慢更，用户觉得很流畅。

  2. 路由切换不卡：切换到一个复杂页面时，以前整个页面会卡一下（等新页面渲染完）。现在用 Suspense + 过渡更新，旧页面保持可交互，新页面在后台慢慢渲染，准备好了再平滑切过去。

  3. 状态更新有优先级：以前所有更新都是"同步且平等"的，谁先来谁先执行。现在有了 Lane 模型，用户交互是高优先级，数据加载是低优先级，高优先级可以抢占低优先级，保证用户体验。

但注意：并发渲染不是银弹，它不能让你的应用变快——它只能让应用感觉上更流畅。该有的计算量一点没少，只是调度方式变了。如果你的渲染本身就很慢，并发渲染只是让它不卡，但完成时间可能更长（因为被打断了好几次）。所以传统优化还是要做，并发渲染是锦上添花，不是雪中送炭。
