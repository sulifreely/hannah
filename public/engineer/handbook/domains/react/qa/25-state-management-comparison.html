问答题 25（高级）：状态管理方案深度对比 ​

题目 ​

请对比 Redux（含 Redux Toolkit）、Zustand、Jotai、Recoil、MobX 这五种主流状态管理方案的设计理念（Flux / 原子化 / 响应式）、核心概念、适用场景、性能特点，并分析为什么 Zustand 和 Jotai 近年来越来越流行，以及 Redux Toolkit 解决了原生 Redux 的哪些痛点。

参考答案 ​

React 状态管理的江湖从来不是一家独大。从早期的 Flux、Redux 一统天下，到 MobX 另辟蹊径，再到近年原子化状态库（Recoil、Jotai、Zustand）的崛起，每个方案背后都有不同的设计哲学。理解它们的差异，才能在项目中做出正确的选型。

一、设计理念对比 ​

五种方案分别代表了三种不同的设计哲学：

方案设计哲学核心思想状态形态
Redux / RTKFlux 架构单一 store、单向数据流、纯函数 reducer大的树形对象
Zustand极简 FluxHook 化的 store、轻量 Flux多个独立 store（也可一个大的）
Jotai原子化（Atomic）状态拆成最小粒度的 atom，像分布式 useState大量独立 atom
Recoil原子化（Atomic）Meta 提出的原子状态 + selector 图大量 atom + selector
MobX响应式（Reactive）可变状态 + 观察者模式，自动追踪依赖可观察的对象 / 类

二、各方案详解 ​

1. Redux & Redux Toolkit（RTK） ​

Redux 是 Flux 架构最著名的实现，曾经几乎是 React 状态管理的"标准答案"。

核心概念：

  • 单一 Store：整个应用的状态存在一个大的对象树里。
  • Action：描述"发生了什么"的普通对象（{ type: 'ADD_TODO', payload: ... }）。
  • Reducer：纯函数，接收旧 state 和 action，返回新 state。
  • Dispatch：唯一的更新方式——dispatch 一个 action。
  • 中间件（Middleware）：扩展 dispatch 的能力（异步、日志、埋点等）。

原生 Redux 的痛点：

  • 模板代码太多：每个 feature 都要写 action types、action creators、reducer switch-case，文件一堆。
  • 不可变更新写起来麻烦：深层对象更新要层层展开，{ ...state, a: { ...state.a, b: ... } }。
  • 异步方案混乱：thunk、saga、observable，选哪个？新人一脸懵。
  • 配置复杂：各种中间件、devtools、enhancer 配起来一堆代码。
  • TypeScript 类型体操：写类型定义很繁琐。

Redux Toolkit（RTK）的救场：

RTK 是 Redux 官方的"最佳实践封装"，解决了原生 Redux 几乎所有痛点：

jsx
import { createSlice, configureStore } from '@reduxjs/toolkit';

// createSlice 一次性生成 action + reducer + action types
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => {
      state.value += 1; // 可以直接"改"，底层用 Immer 保证不可变
    },
    decrement: (state) => {
      state.value -= 1;
    },
  },
});

const store = configureStore({
  reducer: {
    counter: counterSlice.reducer,
  }, // 自动加了 devtools、thunk、immutability check 等
});

RTK 解决了什么问题？

原生 Redux 痛点RTK 的解决方案
模板代码多createSlice 一个函数搞定 action + reducer + types
不可变更新难写内置 Immer，直接"修改" draft state，自动生成新 state
配置复杂configureStore 一键配置 devtools、thunk、中间件
异步方案混乱createAsyncThunk + createEntityAdapter 标准化异步逻辑
TypeScript 麻烦类型自动推导，减少类型体操
数据重复请求RTK Query（RTK 的一部分）专门处理服务端缓存

优点：

  • 规范统一，团队协作成本低。
  • DevTools 极强（时间旅行、action 日志、state diff）。
  • 生态成熟，中间件丰富（thunk、saga、observable、persist 等）。
  • 企业级项目有大量最佳实践。

缺点：

  • 仍然有一定的心智负担（Flux 概念、action/reducer 拆分）。
  • 小项目用起来还是重（"杀鸡用牛刀"）。
  • 学习曲线比轻量方案陡。

2. Zustand ​

Zustand 是目前社区最火的轻量状态库，主打"简单到离谱"。

核心思想：把 store 定义成一个 Hook，调用 create 函数就完了。没有 Provider、没有 reducer、没有 action，爱怎么写怎么写。

jsx
import { create } from 'zustand';

const useStore = create((set, get) => ({
  count: 0,
  user: null,

  increment: () => set(state => ({ count: state.count + 1 })),
  setUser: (user) => set({ user }),

  fetchUser: async (id) => {
    const res = await fetch(`/api/user/${id}`);
    set({ user: await res.json() });
  },
}));

// 组件里用
function Counter() {
  const count = useStore(state => state.count);
  const increment = useStore(state => state.increment);
  return <button onClick={increment}>{count}</button>;
}

特点：

  • 极简 API：一个 create 函数搞定所有事，学习成本 5 分钟。
  • 没有 Provider：直接导出 Hook，在哪里都能用。
  • 灵活无约束：想怎么组织状态都行——一个大 store 或多个小 store，同步异步随便写。
  • TypeScript 友好：类型推导基本全自动。
  • 生态丰富：devtools、persist、immer、redux-devtools 等 middleware 一应俱全。
  • 性能不错：selector 浅比较，按需订阅。

为什么流行？

  1. API 太简单了：看完 README 就能用，几乎没有学习成本。
  2. 足够灵活：不像 Redux 有那么多规矩，想怎么写怎么写。
  3. 小而美：体积小（不到 1KB gzipped），功能却很全。
  4. 对新人友好：不用学 Flux、不用懂 reducer，会写 Hook 就会用 Zustand。

缺点：

  • 太灵活 = 缺规范：大团队用容易写得乱七八糟，每个人风格不一样。需要自己定规范。
  • 调试能力不如 Redux：虽然支持 redux devtools，但体验还是不如原生 Redux。
  • 没有强约束：容易写出"状态到处改、逻辑分散"的代码。

3. Jotai ​

Jotai 是原子化状态管理的代表，号称"React 精神的状态库"。

核心思想：状态拆成一个个最小粒度的 atom（原子），每个 atom 就像一个"共享的 useState"。atom 之间可以组合派生，形成图状依赖。

jsx
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

// 定义原子
const countAtom = atom(0);
const doubledAtom = atom(get => get(countAtom) * 2); // 派生 atom

// 组件里用
function Counter() {
  const [count, setCount] = useAtom(countAtom); // 像 useState 一样
  const doubled = useAtomValue(doubledAtom);

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <p>双倍：{doubled}</p>
    </div>
  );
}

特点：

  • 心智最贴近 React：atom 就是"共享版的 useState"，会用 useState 就会用 Jotai。
  • 细粒度订阅：改一个 atom，只有订阅了这个 atom 的组件重渲染。不需要 selector，不需要 memo。
  • 没有全局 key：atom 就是普通对象，引用即身份，不像 Recoil 需要全局唯一的 key 字符串。
  • 派生 atom 强大：同步、异步、有写逻辑的派生都支持，天然支持 Suspense。
  • TypeScript 体验极佳：类型全自动推导。

为什么流行？

  1. 简单：API 极少，核心概念只有 atom，学会了就全会了。
  2. 灵活：从简单的全局变量到复杂的图状依赖，都能 handle。
  3. 性能好：天生细粒度，不需要做太多优化。
  4. React 原生感：就像 React 自己的功能，用起来毫无违和感。

缺点：

  • 没有全局 store 概念：调试起来不如 Redux 方便（没有统一的状态树、没有时间旅行）。
  • atom 组织依赖规范：atom 放哪里、怎么命名、怎么分类，需要团队自己定规矩。
  • 状态分散：大量 atom 散落在各处，新人上手可能不知道状态在哪。

4. Recoil ​

Recoil 是 Meta（Facebook）官方推出的原子化状态库，曾经被寄予厚望。

核心思想：和 Jotai 类似，也是 atom + selector 的原子化模型，但 Recoil 有更完整的"状态图"概念。

jsx
import { atom, selector, useRecoilState, useRecoilValue } from 'recoil';

const textState = atom({
  key: 'textState', // 需要全局唯一的 key
  default: '',
});

const charCountState = selector({
  key: 'charCountState',
  get: ({ get }) => {
    const text = get(textState);
    return text.length;
  },
});

function TextInput() {
  const [text, setText] = useRecoilState(textState);
  return <input value={text} onChange={e => setText(e.target.value)} />;
}

特点：

  • Meta 官方背景：背靠大树，一度被认为是"未来方向"。
  • atom + selector 模型完整：异步 selector、依赖追踪都很成熟。
  • 支持并发模式：和 React 并发特性集成好。
  • DevTools 体验不错。

现状：

Recoil 这几年社区活跃度明显下降——版本发布缓慢、1.0 长期处于实验状态、GitHub issues 堆积。Meta 内部对 Recoil 的投入似乎也不如以前。相比之下，Jotai 社区更活跃、迭代更快、API 更简洁。

缺点：

  • key 全局唯一：大项目里 key 冲突是个头疼的问题，不像 Jotai 用引用当身份。
  • 需要 <RecoilRoot> Provider：比 Jotai 多了一层。
  • 社区活跃度下降：有被 Jotai 赶超的趋势。
  • 文档和 API 还在变动：不够稳定。

5. MobX ​

MobX 走的是和 Redux 完全不同的路——响应式（reactive）路线。

核心思想：状态是可变的，你直接修改状态，MobX 自动追踪哪些组件依赖了哪些状态，状态变了自动通知对应组件更新。像 Vue 的响应式系统。

jsx
import { makeAutoObservable } from 'mobx';
import { observer } from 'mobx-react-lite';

// 定义可观察的 store
class Store {
  count = 0;
  user = null;

  constructor() {
    makeAutoObservable(this); // 自动变成可观察的
  }

  increment() {
    this.count++; // 直接改！不用返回新对象
  }

  async fetchUser(id) {
    const res = await fetch(`/api/user/${id}`);
    this.user = await res.json(); // 直接赋值
  }
}

const store = new Store();

// observer 包裹的组件会自动追踪依赖
const Counter = observer(() => (
  <button onClick={() => store.increment()}>
    {store.count}
  </button>
));

特点：

  • 写起来最自由：直接修改状态，不用 dispatch、不用返回新对象、没有 immutable 的限制。
  • 自动追踪依赖：你不用手动写 selector，MobX 自动知道组件用了哪些状态，精准更新。
  • 面向对象风格：class + 方法，对后端转前端的工程师很友好。
  • 性能很好：细粒度更新，用到什么订阅什么。

为什么没有 Redux 流行？

  1. 太灵活：状态可以被任何地方修改，出了 bug 不知道是谁改的。大团队协作时容易失控。
  2. 和 React 哲学不符：React 推崇不可变、纯函数、单向数据流；MobX 是可变、响应式、双向数据流。
  3. 调试困难：状态变化没有统一的 action 日志，追踪问题不如 Redux 方便。
  4. 学习曲线：响应式的心智模型和 React 不一样，需要理解"什么会被追踪、什么不会"。

优点：

  • 开发效率极高，写代码非常爽。
  • 性能好，自动优化。
  • 适合中小型项目、快速原型、个人项目。

三、性能对比 ​

方案性能特点优化手段适合的状态规模
Redux / RTK全量通知 + selector 比较，大 state 时需要优化reselect、shallowEqual、拆分 slice大状态树也可以，但需要认真优化
Zustand和 Redux 类似，selector 比较shallow、选择器拆分中小规模，太大了也要优化
Jotai / Recoil天生细粒度，atom 级别订阅几乎不需要，自动细粒度大量小状态、细粒度更新的场景最优
MobX自动细粒度追踪，精准更新几乎不需要优化各种规模都表现不错

简单总结：

  • 状态少：都差不多，不用纠结。
  • 状态多但组件依赖集中：Redux / Zustand 没问题。
  • 状态多且组件依赖分散（每个组件只依赖一小部分）：Jotai / MobX 性能最优。
  • 深层嵌套的大对象：MobX 最方便（直接改），Redux/Zustand 配合 Immer 也还行。

四、选型指南 ​

先想清楚两个问题 ​

1. 这是客户端状态还是服务端状态？

  • 服务端状态（从接口拿的数据、缓存）：用 TanStack Query (React Query) 或 SWR，不要用上面任何一个。它们是专门干这个的，缓存、重取、重试、乐观更新、分页全给你做好了。
  • 客户端状态（UI 状态、用户偏好、临时状态）：再从上面的方案里选。

很多项目根本不需要全局状态库——useState + useContext + TanStack Query 就够了。

选型推荐 ​

场景推荐方案理由
大型企业级项目、多人协作Redux Toolkit规范强、调试强、生态成熟、团队协作成本低
中小型项目、快速迭代Zustand简单、灵活、学习成本低，什么都能做
大量细粒度状态、表单复杂Jotai原子化天生适合，不用操心优化
喜欢响应式、个人项目MobX写起来最快、最爽
状态非常简单（几个全局变量）useState + useContext不用引入额外库，原生就够
服务端数据缓存TanStack Query / SWR专业的事交给专业的库
Meta 技术栈 + 原子化Recoil（谨慎）社区活跃度存疑，新项目更推荐 Jotai

我的实际推荐 ​

  • 90% 的项目：useState + useContext + TanStack Query。真的够用。
  • 需要客户端全局状态：Zustand。目前社区最推荐的"万金油"方案。
  • 大公司、强规范需求：Redux Toolkit。稳。
  • 状态特别细碎复杂：Jotai。原子化的最佳选择。

五、为什么 Zustand / Jotai 越来越流行？ ​

总结一下轻量状态库崛起的原因：

1. Redux 太重了，很多场景用不上

Redux 设计的初衷是解决"大型应用的状态管理"问题，但现实中大部分项目没有那么大。为了一个小全局状态引入一整套 Flux 架构，写一堆 action/reducer，实在太折腾。开发者需要更轻量的方案。

2. Hooks 改变了游戏规则

Hooks 出现之前，全局状态基本只能靠 HOC（connect）或 Render Props，Redux 那套模式虽然繁琐但也没有更好的选择。Hooks 出现后，状态可以用函数的方式封装和使用，Zustand/Jotai 这种"Hook 原生"的库应运而生，使用体验比 connect 好太多。

3. 原子化模型更符合 React 心智

Jotai 的 atom 就像"共享的 useState"，会用 useState 就会用 Jotai，几乎没有学习成本。相比 Redux 的 action/reducer/middleware 一整套概念，原子化的心智负担低多了。

4. TypeScript 体验好

Zustand 和 Jotai 的类型推导都非常自然，基本不需要手动写类型。Redux 虽然 RTK 已经优化了，但还是有一定的类型体操量。

5. 社区氛围的转变

从"大而全"转向"小而美"。开发者越来越偏好体积小、API 简单、按需组合的工具。Zustand 不到 1KB，Jotai 也只有几 KB，和 Redux + 中间件的体积比起来轻量太多。

但 Redux 也没输——在大型企业级项目里，Redux 的规范和调试能力仍然是不可替代的。它只是从"默认选择"变成了"特定场景的选择"。

总结 ​

方案一句话定位适合人群
Redux Toolkit工业级状态管理，稳大型团队、企业项目、规范至上
Zustand最简洁的 store，什么都能做中小型项目、快速迭代、想少写点代码
Jotai原子化状态的最佳选择细粒度状态多、喜欢 React 风格
RecoilMeta 出品的原子库，但前景不明已经在用的项目维护，新项目谨慎选择
MobX响应式路线，写起来最爽个人项目、快速原型、喜欢 OOP 风格

没有最好的，只有最合适的。理解每个方案的设计哲学，根据项目规模、团队特点、业务场景来选，才是高级工程师该有的判断力。

追问 1 ​

题目 ​

Redux 的三大原则（单一数据源、状态只读、纯函数修改）到底有什么价值？为什么要搞这么多"规矩"？

答 ​

Redux 的三大原则不是凭空制定的条条框框，每一条都是为了解决实际问题而存在的。理解了它们解决什么问题，就理解了 Redux 的精髓。

原则一：单一数据源（Single Source of Truth） ​

内容：整个应用的状态存在唯一的一个 store 对象树里。

解决什么问题？

  • 状态共享和同步：如果状态分散在各个组件、各个模块里，同一份数据可能有多份副本，很容易不同步。比如用户信息在 A 组件里改了，B 组件里的那份还是旧的。单一 store 保证了"所有组件读的是同一份数据"。
  • 调试方便：出问题了，只要看这一个 store 里的状态，就能知道应用当前处于什么状态。不需要去各个组件找状态。
  • 服务端渲染容易：服务端把数据塞进一个 store，客户端 hydrate 直接用，非常方便。
  • 全局功能容易实现：撤销/重做、持久化、状态快照、时间旅行，有了单一 store 都很好做。

代价：

  • 所有状态在一起，一个地方变了，所有订阅者都要检查是否变化（需要 selector 优化）。
  • 状态树可能变得很大很复杂，需要合理组织（slices）。

原则二：状态是只读的（State is Read-Only） ​

内容：你不能直接修改 state，唯一能改变 state 的方式是 dispatch 一个 action。

解决什么问题？

  • 可追踪：所有状态变化都是由 action 触发的，你可以记录每一个 action，知道"什么时候、因为什么、改了什么"。这就是时间旅行调试的基础。
  • 可预测：你想知道状态为什么变了，只要看 action 日志就行，不用去排查"哪里偷偷改了状态"。
  • 避免竞态：如果状态可以被任意修改，多个异步操作同时改状态，结果不可控。所有修改都走 action，至少变更的来源是清晰的。
  • 集中管理：所有修改逻辑都在 reducer 里，你可以在一个地方看到所有状态变更的可能性。

打个比方：状态是银行账户，action 是交易记录。你不能直接改账户余额（只读），必须通过一笔交易（action）来改，每一笔都有记录，出了问题可以对账。

代价：

  • 写起来麻烦。改一个值要定义 action、写 reducer、dispatch，比直接 state.count++ 复杂多了。
  • 模板代码多（RTK 已经大大缓解了）。

原则三：使用纯函数修改（Changes are Made with Pure Functions） ​

内容：reducer 必须是纯函数——同样的输入，永远输出同样的结果，没有副作用。

解决什么问题？

  • 可测试：纯函数的测试太简单了——给定输入，断言输出。不用 mock、不用搭建环境、不用考虑时序。
  • 可预测：reducer 没有副作用，你不用担心改个状态会触发什么奇怪的副作用。它就是输入到输出的映射。
  • 可组合：纯函数可以任意组合、拆分、复用。大 reducer 拆成小 reducer，每个小 reducer 管自己的部分。
  • 时间旅行：纯函数 + action 日志 = 可以回放。给我初始 state 和一串 action，我就能重放整个过程，得到任意时刻的 state。这就是 Redux DevTools 时间旅行的原理。
  • 不可变性带来的安全：每次返回新的 state 对象，旧的 state 不会被修改。不用担心"谁把我用的 state 改了"的问题。

代价：

  • 不可变更新写起来麻烦（深层嵌套展开很痛苦）。
  • 每次创建新对象有一定性能开销（不过 Immer 已经解决了写法问题，性能开销对大多数场景也可以忽略）。

三大原则的本质 ​

Redux 的三大原则，最终指向同一个目标——可预测性（Predictability）。

  • 状态在哪？→ 只有一个 store，好找。
  • 状态怎么变的？→ 都是 action 触发的，有日志。
  • 这次更新会产生什么结果？→ reducer 是纯函数，同样的输入永远同样的输出。

当你的应用很复杂、很多人一起维护、出 bug 很难复现时，这种"可预测性"的价值就体现出来了。你知道状态不会莫名其妙地变，你知道每一步变化都是可追溯的，你对代码有掌控感。

这就是 Redux 的核心价值：不是功能多强大，而是通过一系列约束，让复杂应用的状态变得可预测、可调试、可维护。

小项目当然感觉不到——代码少的时候，怎么写都不会乱。但当项目大到一定程度、团队人多到一定程度，你会开始怀念这些"规矩"带来的安全感。

追问 2 ​

题目 ​

原子化状态管理（Jotai / Recoil）和 Context 的本质区别是什么？为什么说 Context 性能差，原子化就性能好？

答 ​

很多人觉得 Jotai 就是"更好用的 Context"，但它们的底层机制和性能模型有本质区别。理解了这个区别，才能明白为什么原子化库性能更好。

Context 的工作原理 ​

Context 的核心是**"发布-订阅"模型，但订阅的粒度是整个 Context value**。

Provider value = { a, b, c, d }
     ↓  （value 一变，通知所有消费者）
┌───────┬───────┬───────┐
Consumer1  Consumer2  Consumer3
（只用 a） （只用 b）  （只用 c）
   ↓         ↓         ↓
  全部重渲染  全部重渲染  全部重渲染
1
2
3
4
5
6
7

当 Provider 的 value 发生任何变化（哪怕只是其中一个字段变了），所有订阅了这个 Context 的组件都会重渲染。Consumer 用没用到那个变化的字段不重要——反正 value 是一个新对象，所有消费者都要跑一遍。

为什么会这样？ 因为 React 的 Context 机制是"全有或全无"的——它只能判断 value 变没变（Object.is 比较），没法判断"value 里的哪个字段变了、哪些消费者用到了这个字段"。它没有细粒度的依赖追踪能力。

怎么优化？

  • 拆 Context：把不同变化频率的状态分到不同 Context，各自的消费者不互相影响。但拆多了 Provider 嵌套很丑。
  • memo + 拆分组件：在消费者那一层用 memo 优化，但治标不治本——组件还是会收到新 props，只是浅比较后跳过渲染。
  • use-context-selector：第三方库，让 Context 支持 selector，实现细粒度订阅。但这不是 React 原生的。

原子化库（Jotai / Recoil）的工作原理 ​

原子化库的核心是**"每个 atom 是独立的订阅单元"**。

atomA  atomB  atomC  atomD
  ↓      ↓      ↓      ↓
CompA  CompB  CompC  CompD
（只订阅A）（只订阅B）（只订阅C）（只订阅D）
1
2
3
4

每个 atom 是独立的，每个组件订阅了哪些 atom，库是知道的。当某个 atom 变化时，只有订阅了这个 atom 的组件会重渲染，其他组件完全不受影响。

它是怎么做到的？

原子化库内部对每个 atom 维护了自己的订阅者列表。当组件调用 useAtom(xxxAtom) 时，就把这个组件加到 xxxAtom 的订阅列表里。atom 的值变化时，遍历订阅列表通知对应组件更新。

本质上，它是把"一个大的 Context 通知"拆成了"N 个小的 atom 通知"，粒度细到每个 atom，自然性能就好了。

本质区别总结 ​

维度Context原子化库（Jotai/Recoil）
订阅粒度整个 Context value，变一点通知所有消费者每个 atom 独立通知，变哪个通知谁
依赖追踪不追踪消费者具体用了 value 的哪些部分知道每个组件订阅了哪些 atom
重渲染范围value 变 → 所有消费者重渲染只有订阅了变化 atom 的组件重渲染
Provider 数量状态多了要拆很多 Context，Provider 嵌套地狱不需要 Provider（Jotai）或只需要一个（Recoil）
状态组织一个大对象树大量独立的小 atom
性能优化成本高，需要手动拆 Context / memo / selector低，天生细粒度

原子化就一定比 Context 好吗？ ​

不一定。看场景：

Context 仍然很好用的场景：

  • 状态变化不频繁（比如主题、语言、当前用户信息）。
  • 状态本身就很少（一两个全局变量）。
  • 注入不变的服务（比如 API 客户端实例、配置对象）。
  • 组件层级很深，props 传递太麻烦，但状态变化频率很低。

原子化更适合的场景：

  • 状态多且分散，每个组件依赖不同的子集。
  • 状态变化频繁，需要精准控制重渲染范围。
  • 状态之间有复杂的派生关系（A 变了 B 跟着变，B 变了 C 跟着变）。
  • 大量细粒度的 UI 状态（表单每个字段、表格每一列配置等）。

一句话总结：Context 是"大喇叭广播"——不管你听不听，我喊一嗓子所有人都听到；原子化是"点对点通知"——你订了哪个就给你发哪个。广播适合消息少、大家都关心的事；通知适合消息多、每个人关心不同内容的场景。

追问 3 ​

题目 ​

服务端状态（Server State）和客户端状态（Client State）为什么要分开管理？只用一个全局状态库不行吗？

答 ​

这是一个非常重要的问题——很多人状态管理用不好，根本原因是没搞清楚"服务端状态"和"客户端状态"是两种完全不同的东西，混在一起管理只会越来越乱。

什么是服务端状态？什么是客户端状态？ ​

服务端状态（Server State / 远端状态 / 缓存状态）：

  • 存在服务器上，你本地只是存了一份副本（缓存）。
  • 数据的"真相来源（source of truth）"在服务端，不是你客户端。
  • 随时可能过期——别人可能修改了服务端的数据，你本地的缓存就旧了。
  • 需要考虑缓存、重取、过期、重试、乐观更新、分页、去重等等。
  • 例子：用户列表、文章内容、商品详情、评论数据。

客户端状态（Client State / UI 状态 / 本地状态）：

  • 完全存在客户端，服务端不知道也不关心。
  • 数据的真相来源在客户端，你完全控制它。
  • 不会"过期"——你不改它就不变。
  • 相对简单，就是 useState 那一套。
  • 例子：弹窗开关、当前选中的 tab、输入框内容、主题色、菜单展开收起。

为什么混在一起管会出问题？ ​

很多人用 Redux / Zustand 存所有东西，包括从接口拿的数据。这会带来一堆问题：

1. 你得自己写一堆缓存逻辑

每次都要写：

  • 请求开始 → 设置 loading = true
  • 请求成功 → 保存数据、loading = false
  • 请求失败 → 保存 error、loading = false
  • 组件卸载 → 要不要取消请求？
  • 重复访问同一个数据 → 要不要复用之前的结果？
  • 数据多久算过期？什么时候重新拉取？
  • 窗口重新聚焦 → 要不要刷新数据？
  • 分页、无限滚动、预加载……

这些逻辑每个接口写一遍，代码量巨大，而且容易写得不一样（有的地方处理了错误，有的地方没处理）。

2. 状态越来越臃肿

store 里一半是 UI 状态，一半是接口数据缓存，混在一起，越来越乱。哪些是本地的、哪些是从服务端拿的，时间长了分不清。

3. 重复造轮子

你辛辛苦苦写的那套 loading / error / 缓存 / 重试逻辑，人家专门的库已经写得完美了，而且经过了大量项目验证。你自己写的大概率有 bug（比如竞态问题、内存泄漏、边界情况没考虑）。

专门的服务端状态库解决了什么问题？ ​

以 TanStack Query（React Query）为例，它内置了：

功能你手写要多久React Query
自动缓存每个接口写一遍✅ 内置
重复请求去重自己搞全局 Map✅ 同一时间相同查询只发一次
后台刷新自己监听焦点/联网✅ 窗口聚焦、重连自动刷新
过期策略自己加时间戳判断✅ 配置 staleTime 就行
重试机制自己写递归重试✅ 内置，可配置次数和间隔
分页 / 无限滚动自己维护游标✅ useInfiniteQuery 搞定
乐观更新自己写回滚逻辑✅ 内置 optimistic update API
预加载自己提前调接口✅ 可预取数据，用户点的时候已经有了
请求取消自己用 AbortController✅ 自动处理
竞态处理自己搞版本号✅ 自动处理，旧请求回来不会覆盖新的
DevTools没有✅ 有专门的 devtools

这还只是一部分功能。你花一周时间手写的缓存逻辑，可能还不如人家一个库的零头强大。

分开管理的最佳实践 ​

服务端状态 → TanStack Query / SWR / Apollo Client

所有从接口拿的数据都走这些库，不要存到全局 store 里。它们就是你的"数据缓存层"。

客户端状态 → useState / useContext / Zustand / Jotai / Redux

真正的 UI 状态、交互状态，才用状态管理库。你会发现，把服务端状态剥离后，需要全局管理的客户端状态其实少得可怜——可能就一个当前用户、一个主题色、几个全局弹窗。

那全局状态库还有用吗？ ​

当然有。但它的职责变了：

以前：全局状态库 = 所有状态的容器（什么都往里塞）。 现在：全局状态库 = 只管客户端 UI 状态（服务端状态交给专门的库）。

这是状态管理领域的一个重要趋势——关注点分离。不同性质的状态用不同的工具，各司其职，代码更清晰，维护成本更低。

一个判断标准 ​

不确定某个状态该放哪里？问自己一个问题：

刷新页面后，这个状态应该从哪里恢复？

  • 从服务端重新获取 → 服务端状态 → TanStack Query
  • 应该保留用户刚才的操作（比如弹窗关了还是开着）→ 客户端状态 → useState / 全局状态库
  • 应该重置为默认值 → 组件本地状态 → useState

想清楚这个问题，你的状态管理就不会乱。
