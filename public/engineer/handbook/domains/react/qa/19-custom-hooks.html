问答题 19（中级）：自定义 Hook 设计模式与最佳实践 ​

题目 ​

请阐述自定义 Hook 的命名规范、抽离逻辑的判断标准、常见的组合模式，并举例说明几个经典自定义 Hook（useToggle、usePrevious、useLocalStorage、useDebounce）的实现。同时谈谈自定义 Hook 有哪些常见的坑（依赖数组、条件调用等）。

参考答案 ​

什么是自定义 Hook？ ​

自定义 Hook 是一种在函数组件之间复用有状态逻辑的机制。它不是 React 的 API，而是一种约定——只要一个函数的名字以 use 开头，并且内部调用了其他 Hook，它就是一个自定义 Hook。

本质上，自定义 Hook 就是把组件里的一段逻辑（包括 state、effect、ref 等）抽出来变成一个可复用的函数。它和普通函数的区别在于：自定义 Hook 可以调用其他 Hook，可以"借用"组件的 React 状态和生命周期能力。

命名规范 ​

1. 必须以 use 开头，小驼峰命名

这不是建议，是强制约定。React 的 ESLint 插件（eslint-plugin-react-hooks）靠这个命名来判断一个函数是不是自定义 Hook，从而检查 Hook 规则是否被遵守。如果你不以 use 开头，Hook 规则检查就会失效，容易出 bug。

jsx
// ✅ 好的命名
useToggle
useLocalStorage
useDebounce
useFetch

// ❌ 不好的命名
toggleHook        // 不以 use 开头
useToggleHook     // 冗余，Hook 不体现在命名里

2. 动词开头，体现"做什么"

和 React 内置 Hook 风格一致，用动词开头，后面跟名词或描述。比如 useState（用状态）、useEffect（用副作用）、useToggle（用开关）、usePrevious（用上一个值）。

3. 返回值清晰

返回单个值直接返回，返回多个值用数组（模仿 useState）或对象。如果返回值多于 2 个，推荐用对象，方便调用方按需解构：

jsx
// 两个值用数组（和 useState 一致，调用方可以重命名）
const [on, toggle] = useToggle(false);

// 多个值用对象（调用方按需取，顺序不重要）
const { data, loading, error, refetch } = useFetch(url);

抽离逻辑的判断标准：什么时候该抽自定义 Hook？ ​

不是所有重复代码都要抽成 Hook。判断标准可以参考以下几条：

1. 逻辑中用到了 React Hook（state/effect/ref 等） ​

如果一段逻辑只用到了普通 JavaScript，没有调用任何 React Hook，那它就是一个普通工具函数，不需要叫 Hook。比如 formatDate(date)、debounce(fn, delay)，这些就是普通函数，别画蛇添足叫 useFormatDate。

2. 逻辑在多个组件中重复出现 ​

如果同一段有状态的逻辑（state + effect + handler）在两个以上组件里出现，就可以考虑抽成自定义 Hook。DRY（Don't Repeat Yourself）原则同样适用。

3. 逻辑本身是一个内聚的单元 ​

抽出来的 Hook 应该只做一件事，并且做好。如果一个 Hook 里同时管了数据请求、表单验证、弹窗状态，那它就是一个"上帝 Hook"，还不如不抽。好的 Hook 职责单一，可以像积木一样组合使用。

4. 组件太复杂，需要按关注点拆分 ​

即使逻辑只在一个组件里用，如果组件太大、Hook 太多，也可以把相关的逻辑抽到一个自定义 Hook 里，让组件代码更清晰。这时候 Hook 的作用是"按关注点组织代码"，而不是"复用"。

jsx
// 之前：所有逻辑堆在组件里，乱糟糟
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  // ... 一堆 effect 和 handler
  // ... 还有一堆和编辑相关的 state
  // ... 还有一堆和关注相关的 state
}

// 之后：按关注点拆成多个自定义 Hook
function UserProfile({ userId }) {
  const { user, loading, error, refetch } = useUser(userId);
  const { isEditing, startEdit, saveEdit, cancelEdit } = useEditUser(user);
  const { isFollowing, toggleFollow } = useFollowUser(userId);
  // 组件里只负责渲染和组合，逻辑都在 Hook 里
}

常见自定义 Hook 实现 ​

下面手写几个最经典的自定义 Hook，理解了它们的实现，写其他 Hook 就触类旁通了。

1. useToggle：布尔值切换 ​

最简单也最常用的自定义 Hook，封装布尔值的切换逻辑。

jsx
function useToggle(initialValue = false) {
  const [value, setValue] = useState(initialValue);

  const toggle = useCallback(() => {
    setValue(v => !v);
  }, []);

  // 返回 [当前值, 切换函数, 直接设置函数]
  return [value, toggle, setValue];
}

// 使用
const [isModalOpen, toggleModal, setModalOpen] = useToggle(false);
// isModalOpen: 当前值
// toggleModal: () => setIsModalOpen(v => !v)
// setModalOpen: 直接设置，和 useState 的 setter 一样

2. usePrevious：保存上一次渲染的值 ​

利用 useRef 跨渲染周期保存数据的特性，记录上一次渲染的 state/props 值。

jsx
function usePrevious(value) {
  const ref = useRef();

  // 每次渲染后更新 ref 的值
  useEffect(() => {
    ref.current = value;
  }, [value]);

  // 返回的是"上一次"的值（因为 effect 在渲染后才执行）
  return ref.current;
}

// 使用
const [count, setCount] = useState(0);
const prevCount = usePrevious(count);

// 首次渲染: count=0, prevCount=undefined
// 点击一次后: count=1, prevCount=0
// 点击两次后: count=2, prevCount=1

原理：每次渲染时，usePrevious 返回的是 ref.current（上一次渲染结束后 effect 存进去的值）。然后当前渲染结束后，effect 执行，把 ref.current 更新为本次的值。下一次渲染时，读到的就是这次的值了。所以永远慢一拍——这正是我们想要的"上一次的值"。

3. useLocalStorage：状态持久化到 localStorage ​

把 useState 和 localStorage 结合起来，状态变化时自动同步到 localStorage，刷新页面也不丢。

jsx
function useLocalStorage(key, initialValue) {
  // 惰性初始化：从 localStorage 读，读不到就用初始值
  const [value, setValue] = useState(() => {
    try {
      const item = window.localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch (error) {
      console.warn(`Error reading localStorage key "${key}":`, error);
      return initialValue;
    }
  });

  // 自定义 setter：更新 state 的同时同步到 localStorage
  const setStoredValue = useCallback((newValue) => {
    try {
      // 支持函数式更新，和 useState 一致
      const valueToStore =
        newValue instanceof Function ? newValue(value) : newValue;
      setValue(valueToStore);
      window.localStorage.setItem(key, JSON.stringify(valueToStore));
    } catch (error) {
      console.warn(`Error setting localStorage key "${key}":`, error);
    }
  }, [key, value]);

  return [value, setStoredValue];
}

// 使用
const [theme, setTheme] = useLocalStorage('theme', 'light');
// 刷新页面后 theme 还是上次设置的值

4. useDebounce：防抖值 ​

输入框搜索、窗口 resize 等场景常用。输入停止一段时间后，值才"真正"更新。

jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    // 值变化时，设置一个定时器，delay 后更新 debouncedValue
    const timer = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    // 下一次值变化时，先清掉上一个定时器
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debouncedValue;
}

// 使用：搜索输入框
const [searchTerm, setSearchTerm] = useState('');
const debouncedSearchTerm = useDebounce(searchTerm, 300);

// 用 debounced 值发请求，用户停止输入 300ms 后才真正发请求
useEffect(() => {
  if (!debouncedSearchTerm) return;
  fetch(`/api/search?q=${debouncedSearchTerm}`).then(/* ... */);
}, [debouncedSearchTerm]);

Hook 组合模式 ​

自定义 Hook 的强大之处在于组合——一个自定义 Hook 可以调用其他自定义 Hook，像搭积木一样构建复杂逻辑。

jsx
// useUser 内部用了 useFetch 和 useLocalStorage
function useUser(userId) {
  const cachedUser = useLocalStorage(`user-${userId}`, null);
  const { data, loading, error, refetch } = useFetch(`/api/user/${userId}`);

  // 优先用接口数据，接口还没返回时用缓存
  return {
    user: data || cachedUser,
    loading,
    error,
    refetch,
  };
}

自定义 Hook 常见的坑 ​

坑 1：依赖数组遗漏或错误 ​

这是最常见的坑。自定义 Hook 内部用了 useEffect、useCallback、useMemo，但依赖数组没写全，导致闭包陷阱。

jsx
// ❌ 错误：遗漏了 delay 依赖
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(timer);
  }, [value]); // 没写 delay！delay 变了定时器不会重建
  return debouncedValue;
}

怎么避免：

  • 始终开启 ESLint 的 exhaustive-deps 规则。
  • 不要为了消除警告而盲目加依赖——先想清楚这个依赖该不该加。
  • 如果某个函数依赖不稳定，用 useCallback 包一下，或者用 useRef 存起来。

坑 2：在条件或循环中调用 Hook ​

Hook 规则说：只能在组件顶层调用 Hook，不能在 if、for、嵌套函数里调用。自定义 Hook 也一样，如果你的自定义 Hook 内部有条件调用 Hook 的情况，就违反了规则。

jsx
// ❌ 错误：Hook 在条件里调用
function useUser(userId) {
  if (!userId) return null; // 提前返回了，下面的 Hook 可能不会执行
  const [user, setUser] = useState(null); // 违反规则！
  // ...
}

坑 3：自定义 Hook 返回的函数引用不稳定 ​

如果自定义 Hook 返回的函数每次渲染都是新引用，下游组件用 memo 或在 useEffect 依赖中使用时，会出问题。

jsx
// ❌ 错误：increment 每次渲染都是新函数
function useCounter(initialValue = 0) {
  const [count, setCount] = useState(initialValue);

  const increment = () => setCount(c => c + 1); // 每次都新建
  const decrement = () => setCount(c => c - 1);

  return { count, increment, decrement };
}

坑 4：过度抽象，Hook 之间耦合太深 ​

一开始抽 Hook 很爽，抽多了发现 Hook 之间互相依赖、参数传来传去，比放在组件里还难理解。

怎么判断是不是过度抽象了：

  • 一个 Hook 有 5 个以上参数，调用时参数传一大坨。
  • 改一个 Hook，得同时改 3 个其他 Hook。
  • 读代码时，要在 5 个 Hook 文件之间跳来跳去才能搞懂逻辑。

如果出现以上情况，说明抽得太细了，适当合并回去反而更好。代码清晰 > 强行复用。

最佳实践总结 ​

  1. 命名以 use 开头，遵循小驼峰，让 ESLint 能正确检查。
  2. 职责单一：一个 Hook 只做一件事，复杂逻辑通过组合多个 Hook 实现。
  3. 返回值约定：两个值用数组，多个值用对象。
  4. 稳定引用：返回的函数用 useCallback 包裹，方便下游优化。
  5. 遵守 Hook 规则：不在条件/循环中调用 Hook。
  6. TypeScript 友好：给自定义 Hook 加上类型定义，使用方体验更好。
  7. 不要过度抽象：先写在组件里，重复了再抽；抽了之后更复杂就合并回去。

追问 1 ​

题目 ​

自定义 Hook 能访问调用它的组件的 props 吗？它是怎么拿到组件上下文的？

答 ​

自定义 Hook 不需要主动访问组件的 props——它本身就是在组件的渲染过程中被调用的，所以组件的作用域对它天然可见。你可以通过参数把 props 传进去，也可以在 Hook 内部直接闭包捕获外部变量（但不推荐，耦合太强）。

更准确地说，自定义 Hook 能访问组件上下文，靠的是两个机制：

1. JavaScript 闭包

自定义 Hook 是在组件函数内部调用的，它可以访问组件函数作用域里的所有变量，包括 props、state、其他 Hook 的返回值。这和普通的函数闭包没有任何区别。

2. React 的 Hook 链表机制

React 靠调用顺序来匹配 Hook 和对应的状态。不管是内置 Hook 还是自定义 Hook，最终都是内置 Hook 在工作。自定义 Hook 只是把"调用哪些内置 Hook"这件事封装了一下，实际的状态还是存在调用它的那个组件的 Fiber 节点上。

举个例子：

jsx
function useCounter() {
  const [count, setCount] = useState(0); // 这个 state 存在调用 useCounter 的组件上
  // ...
}

function MyComponent() {
  useCounter(); // MyComponent 的 Fiber 上存了 count 这个 state
  return null;
}

所以自定义 Hook 不是"把状态存在 Hook 里"，而是"借用组件的状态存储空间"。这也是为什么 Hook 规则要求调用顺序固定——React 得靠顺序来找到对应的状态。

理解了这一点，就明白了：自定义 Hook 的状态是属于调用它的那个组件实例的。两个组件调用同一个自定义 Hook，它们的状态是完全独立的，互不影响。

追问 2 ​

题目 ​

如何设计一个"可以取消"的自定义 Hook（比如 useFetch 支持取消请求）？有哪些需要注意的地方？

答 ​

设计可取消的 Hook，核心要考虑两件事：什么时候取消和怎么取消。以 useFetch 为例：

jsx
function useFetch(url, options = {}) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  useEffect(() => {
    if (!url) return;

    // 用 AbortController 来控制取消
    const controller = new AbortController();
    const signal = controller.signal;

    const fetchData = async () => {
      setLoading(true);
      try {
        const res = await fetch(url, { ...options, signal });
        const json = await res.json();
        setData(json);
        setError(null);
      } catch (e) {
        if (e.name !== 'AbortError') { // 取消导致的错误不算真错误
          setError(e);
        }
      } finally {
        setLoading(false);
      }
    };

    fetchData();

    // 清理函数：url 变化或组件卸载时，取消上一次请求
    return () => controller.abort();
  }, [url]); // 简化，实际要处理 options 的依赖

需要注意的点：

  1. 用 AbortController 取消 fetch：这是浏览器原生 API，是取消 fetch 请求的标准方式。axios 也支持取消令牌（CancelToken / AbortController）。

  2. 清理函数中取消：当依赖变化（url 变了）或组件卸载时，effect 的清理函数会执行，这时候调用 abort() 取消上一次请求，防止"请求返回后组件已卸载"的警告，也避免旧请求覆盖新请求的数据。

  3. 忽略取消导致的错误：abort() 会让 fetch 的 promise reject，错误 name 是 AbortError。这种"我们主动取消"的情况不应该当作错误处理，也不该设置 error 状态。

  4. 组件卸载后不要 setState：除了取消请求，还要注意——如果请求没走 fetch（比如用其他库），要确保卸载后不调用 setState。可以用一个 ref 标记挂载状态：

  return () => { isMounted.current = false; };

追问 3 ​

题目 ​

为什么自定义 Hook 不能在普通函数里调用？只能在组件或其他 Hook 里调用？

答 ​

这涉及到 Hook 的底层实现原理。React 维护 Hook 状态靠的是调用顺序和当前正在渲染的组件。

具体来说：

1. Hook 状态存在组件 Fiber 节点上

每个函数组件对应一个 Fiber 节点，Fiber 节点上有一个 memoizedState 属性，它是一个链表，链表上的每个节点对应一个 Hook 的状态。React 是按调用顺序把 Hook 和链表节点一一对应的——第一个 useState 对应链表第一个节点，第二个 useState 对应第二个节点，以此类推。

2. React 靠"当前渲染组件"指针找到对应的链表

渲染组件时，React 会设置一个"当前正在工作的 Fiber"指针（workInProgress）。调用 Hook 时，React 就从这个指针指向的 Fiber 上取 Hook 链表，按顺序匹配。

3. 普通函数里调用的问题

如果在普通函数（不是组件、不是自定义 Hook）里调用 Hook，有两种情况：

  • 普通函数在组件渲染过程中被调用：其实没问题。比如你在组件里调用了一个工具函数，工具函数里又调用了 Hook——只要调用顺序稳定，是能工作的。但 React 不推荐这么做，因为工具函数不该依赖 Hook，应该通过参数传值。而且这样违反了命名约定（不以 use 开头的函数里有 Hook），ESLint 会报错。

  • 普通函数在组件外调用（比如事件回调里、定时器里、promise 回调里）：这时候根本不在渲染过程中，"当前工作 Fiber"指针不存在或指向错误的组件，Hook 不知道该把状态存在哪里，行为完全不可预测。

所以规则的本质是：Hook 必须在组件的渲染阶段调用，这样 React 才能知道把状态和哪个组件关联、按什么顺序匹配。 自定义 Hook 之所以可以，是因为它本身就在组件渲染过程中被调用，相当于把 Hook 调用"转发"了一下。

这也是 Hook 规则（只在顶层调用、只在组件或 Hook 中调用）的根本原因——React 需要稳定的调用顺序来正确维护状态。
