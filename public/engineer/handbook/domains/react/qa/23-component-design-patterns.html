问答题 23（中级）：React 组件设计模式 ​

题目 ​

请系统梳理 React 中常见的组件设计模式：高阶组件（HOC）、Render Props、children 作为函数、组合 vs 继承、受控组件 vs 非受控组件、容器组件与展示组件、Compound Components。讲清每种模式的用途、实现方式、优缺点和适用场景。

参考答案 ​

组件设计模式是 React 开发的精髓之一。好的模式能让代码更易复用、更易维护、更易测试。下面逐一梳理常见的模式。

一、高阶组件（Higher-Order Component, HOC） ​

是什么？ ​

高阶组件是一个函数，它接收一个组件作为参数，返回一个新的组件。它是一种"组件增强"的模式——给原组件添加上额外的 props 或能力。

jsx
// 高阶组件：给组件添加一个 "user" prop
function withUser(WrappedComponent) {
  return function EnhancedComponent(props) {
    const user = useCurrentUser();
    return <WrappedComponent {...props} user={user} />;
  };
}

// 使用
const ProfileWithUser = withUser(Profile);

典型应用场景 ​

  • 注入 props：把 context、store、路由参数等注入到组件的 props 里。

    • Redux 的 connect
    • React Router 的 withRouter（v5 及之前）
    • 各种 withXXX 工具函数

  • 条件渲染：权限控制、加载状态、错误边界等。

    jsx
    function withAuth(WrappedComponent) {

    const user = useUser();

  • 劫持生命周期/副作用：在原组件挂载/卸载时做一些额外操作（比如埋点、订阅）。

优缺点 ​

优点：

  • 逻辑复用性强，可以跨多个组件共享。
  • 对原组件无侵入——原组件不用改，外面包一层就增强了。
  • 可以组合使用：withRouter(withUser(withTheme(Comp)))。

缺点：

  • 嵌套地狱：多个 HOC 组合起来，组件树会很深，调试困难。

  • props 来源不透明：打开组件，你不知道某个 prop 是哪里来的——是父组件传的？还是哪个 HOC 注的？
  • 命名冲突：两个 HOC 注入了同名的 prop，后者会覆盖前者，而且很难排查。
  • ref 丢失：HOC 默认不会把 ref 转发到内层组件，需要用 forwardRef 手动处理。
  • 静态方法丢失：原组件上的静态方法（如 Comp.preload）在 HOC 包装后会丢失，需要手动 hoist。

现状 ​

HOC 在类组件时代非常流行（Redux connect、withRouter 都是典型），但在 Hook 时代，大多数场景都被自定义 Hook 取代了。Hook 没有嵌套问题、props 来源清晰、不会有命名冲突。现在写新代码优先用自定义 Hook，HOC 更多是历史遗留或框架级 API 在用。

二、Render Props（渲染属性） ​

是什么？ ​

Render Props 是指：一个组件接收一个函数类型的 prop（通常叫 render 或 children），在自己的渲染逻辑中调用这个函数，把内部状态传给它。

jsx
// 提供鼠标位置的组件
class MouseTracker extends React.Component {
  state = { x: 0, y: 0 };

  handleMouseMove = (e) => {
    this.setState({ x: e.clientX, y: e.clientY });
  };

  render() {
    return (
      <div onMouseMove={this.handleMouseMove}>
        {/* 调用传入的 render 函数，把 state 传出去 */}
        {this.props.render(this.state)}
      </div>
    );
  }
}

// 使用
<MouseTracker render={({ x, y }) => (
  <div>鼠标位置: {x}, {y}</div>
)} />

优缺点 ​

优点：

  • 数据来源清晰：你清楚地知道 x, y 是从 MouseTracker 来的。
  • 没有命名冲突：变量名你自己取，想叫什么叫什么。
  • 组合灵活：想用就用，不想用就不写，不像 HOC 那样是全量注入。

缺点：

  • 嵌套地狱（和 HOC 一样）：多个 Render Props 组合时嵌套很深。

  • 代码结构不直观：层层回调，读起来费劲。
  • 性能优化麻烦：每次渲染都传一个新的函数引用，内层组件如果用 memo 会失效。

现状 ​

Render Props 在 Hook 出现之前是"共享有状态逻辑"的主流方案之一。Hook 出现后，绝大多数 Render Props 的场景都可以用自定义 Hook 替代，代码更简洁、没有嵌套问题。

但 Render Props 也不是完全没用了——当你需要在"渲染位置"和"状态来源"之间解耦时，Render Props 仍然很有用。比如 React Router 的 <Routes>、Formik 的 <Field>，这种"组件内部决定什么时候渲染、渲染什么由你定"的场景，Render Props 还是很合适的。

三、组合（Composition）vs 继承 ​

React 为什么推崇组合？ ​

React 官方明确推荐组合而不是继承来复用代码。为什么？

继承的问题：

  • 强耦合：子类依赖父类的实现细节，父类一改，子类全崩。
  • 不灵活：你只能继承一个父类，想复用多个来源的逻辑就不行了。
  • 层级爆炸：为了复用不断加父类，继承树越来越深，越来越难理解。
  • props 和 state 不透明：你不知道哪些属性是自己的、哪些是继承来的。

组合的优势：

  • 松耦合：组件之间通过 props 通信，不依赖对方的内部实现。
  • 灵活：想组合多少个就组合多少个，随用随拼。
  • 清晰：数据流向明确，props 从哪里来一目了然。
  • 易测试：每个组件独立，单独测试即可。

组合的常见形式 ​

1. children 组合（最基本）

父组件把 children 当插槽用，子组件只管渲染自己的内容。

jsx
function Card({ title, children }) {
  return (
    <div className="card">
      <h3>{title}</h3>
      <div className="card-body">{children}</div>
    </div>
  );
}

// 使用
<Card title="用户信息">
  <p>姓名：张三</p>
  <p>年龄：28</p>
</Card>

什么时候可能用到继承？ ​

几乎不会。React 官方文档说："在 Facebook，我们用 React 开发了成千上万个组件，还没有发现任何需要用组件继承来解决的场景。"

四、受控组件 vs 非受控组件 ​

这是表单组件中最经典的模式。

受控组件（Controlled Component） ​

表单元素的值由 React 的 state 控制，状态的"单一数据源"在 React 这边。

jsx
function ControlledInput() {
  const [value, setValue] = useState('');

  const handleChange = (e) => {
    setValue(e.target.value); // 每次输入都更新 state
  };

  return <input value={value} onChange={handleChange} />;
}

特点：

  • 表单元素的值和 React state 始终同步。
  • 你可以实时读取、修改、校验输入的值。
  • 每次输入都会触发重渲染。

适用场景：

  • 需要实时校验（输入时显示错误提示）
  • 需要联动控制（一个输入框的值影响另一个）
  • 需要格式化输入（比如手机号自动加空格）
  • 提交前需要动态修改值

绝大多数场景下，推荐用受控组件——逻辑更清晰、更可控。

非受控组件（Uncontrolled Component） ​

表单元素的值存在 DOM 本身 里，React 不控制它，需要用时再通过 ref 去 DOM 里取。

jsx
function UncontrolledInput() {
  const inputRef = useRef(null);

  const handleSubmit = () => {
    // 提交时才去 DOM 里取值
    const value = inputRef.current.value;
    console.log(value);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input ref={inputRef} defaultValue="hello" />
      <button type="submit">提交</button>
    </form>
  );
}

特点：

  • 状态在 DOM 里，React 不管。
  • 输入时不会触发重渲染。
  • 用 defaultValue 设置初始值，而不是 value。

适用场景：

  • 非常简单的表单，不需要实时校验和联动。
  • 集成第三方非 React 的 DOM 库。
  • 文件输入（<input type="file">）——它是只读的，只能用非受控方式。
  • 性能极端敏感的场景：输入非常频繁且不想每次都重渲染。

如何选择？ ​

  • 默认用受控组件：逻辑清晰、功能强，绝大多数场景都合适。
  • 简单表单用非受控也行：如果只是个登录框，用户名密码，没必要搞受控，提交时取值就行了。
  • 文件上传只能非受控：input type="file" 的值是只读的，只能通过 ref 操作。

一个经验法则：字段少、逻辑简单 → 非受控也 OK；字段多、逻辑复杂 → 必须受控。

五、容器组件与展示组件（Container / Presentational） ​

这是一种经典的组件职责分离模式，由 Dan Abramov 提出，也叫"聪明组件 vs 笨组件"。

核心思想 ​

把组件分成两类：

展示组件（Presentational / Dumb）：

  • 只管长什么样，不管数据从哪来、怎么改。
  • 数据通过 props 传入，回调通过 props 传出。
  • 没有自己的 state（除非是 UI 状态，比如展开收起）。
  • 不依赖全局状态、不直接发请求。
  • 可复用性高。
jsx
// 展示组件：只负责渲染
function UserList({ users, onSelect }) {
  return (
    <ul>
      {users.map(user => (
        <li key={user.id} onClick={() => onSelect(user.id)}>
          {user.name}
        </li>
      ))}
    </ul>
  );
}

容器组件（Container / Smart）：

  • 只管数据和逻辑，不管长什么样。
  • 订阅全局状态、发请求、处理业务逻辑。
  • 渲染展示组件，把数据和回调通过 props 传下去。
  • 通常没有自己的样式。
jsx
// 容器组件：管数据，不管样式
function UserListContainer() {
  const [users, setUsers] = useState([]);
  const { addNotification } = useNotifications();

  useEffect(() => {
    fetchUsers().then(setUsers);
  }, []);

  const handleSelect = (id) => {
    addNotification(`选中了用户 ${id}`);
  };

  return <UserList users={users} onSelect={handleSelect} />;
}

优缺点 ​

优点：

  • 关注点分离：样式和逻辑分开，各改各的，互不影响。
  • 展示组件可复用：同样的列表样式，可以用在不同的数据来源上。
  • 易于测试：展示组件只看 props，纯渲染，测试很简单。
  • 业务逻辑集中：容器组件里都是逻辑，便于梳理业务流程。

缺点：

  • 多了一层抽象：简单组件也分的话，反而增加文件数量和跳转成本。
  • Hook 出现后边界模糊了：以前类组件时代，容器组件用 connect 包裹，展示组件是函数组件，界限很清。现在一个 Hook 就能搞定数据获取，容器组件被大大简化了。

现状 ​

这个模式在 Hook 时代仍然有价值，但不再是"一个组件拆成两个文件"的硬规则了。更多是一种思想——把"长什么样"和"数据逻辑"分开。可以在同一个文件里，上面写逻辑 Hook，下面写展示组件，或者干脆就一个组件，但心里要清楚哪部分是展示、哪部分是逻辑。

六、Compound Components（复合组件） ​

复合组件是一种"多个组件协同工作、共同完成一个功能"的模式。用户使用时，通过组合子组件来自定义内容和顺序，但内部状态和逻辑由父组件统一管理。

最典型的例子是 HTML 的 <select> 和 <option>：

html
<select>
  <option value="1">选项一</option>
  <option value="2">选项二</option>
</select>

它们俩是配合工作的——select 管状态，option 管内容，但 option 又能影响 select 的行为。React 中模拟这种模式，就是 Compound Components。

实现示例（Tabs 组件） ​

jsx
import { createContext, useContext, useState } from 'react';

const TabsContext = createContext();

// 父组件：管理状态
function Tabs({ defaultValue, children }) {
  const [activeKey, setActiveKey] = useState(defaultValue);
  return (
    <TabsContext.Provider value={{ activeKey, setActiveKey }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

// 子组件：Tab 列表
function TabList({ children }) {
  return <div className="tab-list">{children}</div>;
}

// 子组件：单个 Tab
function Tab({ eventKey, children }) {
  const { activeKey, setActiveKey } = useContext(TabsContext);
  const isActive = activeKey === eventKey;
  return (
    <button
      className={isActive ? 'tab active' : 'tab'}
      onClick={() => setActiveKey(eventKey)}
    >
      {children}
    </button>
  );
}

// 子组件：Tab 面板
function TabPanel({ eventKey, children }) {
  const { activeKey } = useContext(TabsContext);
  if (activeKey !== eventKey) return null;
  return <div className="tab-panel">{children}</div>;
}

// 把所有子组件挂到父组件上，方便导入
Tabs.TabList = TabList;
Tabs.Tab = Tab;
Tabs.TabPanel = TabPanel;

export default Tabs;

核心实现机制 ​

通常用 Context 来实现：父组件把状态和方法放到 Context 里，子组件通过 useContext 读取，从而实现"父子协同工作，但不通过 props 显式传递"。

优缺点 ​

优点：

  • 灵活的布局：用户可以自由排列子组件的顺序、增减子组件，甚至在中间插入其他内容。
  • API 优雅：用起来像原生 HTML 元素，语义清晰。
  • 状态共享隐式：子组件不需要显式传一堆 props，通过 Context 自动拿到状态。

缺点：

  • 隐式依赖：子组件必须放在父组件里面，否则拿不到 Context。单独使用子组件会报错。
  • 调试麻烦：状态通过 Context 传递，追踪起来不如 props 直观。
  • 类型定义复杂：TypeScript 场景下 Context 的类型定义会比较绕。

适用场景 ​

  • 布局组件：Tabs、Accordion、Dropdown、Modal 等由多个部分组成的组件。
  • 表单组件：Form、Field 等需要协同工作的组件组。
  • 任何"整体 + 多个部分"结构的组件。

总结：模式怎么选？ ​

模式解决的问题推荐程度（Hook 时代）
自定义 Hook复用有状态的逻辑⭐⭐⭐⭐⭐ 首选
组合（children/props）UI 结构的复用与灵活布局⭐⭐⭐⭐⭐ 基础
受控组件表单状态管理⭐⭐⭐⭐ 默认选择
Compound Components多组件协同的 UI 组件⭐⭐⭐⭐ 适合组件库
容器/展示组件关注点分离⭐⭐⭐ 作为思想，不用硬拆
Render Props共享渲染逻辑⭐⭐ 多数场景被 Hook 取代
HOC组件增强 / 横切关注点⭐⭐ 历史遗留，新代码少用
继承——⭐ 几乎不用

没有最好的模式，只有最合适的模式。理解每种模式的优缺点，根据场景灵活选用，才是高级工程师的素养。

追问 1 ​

题目 ​

自定义 Hook 和 HOC / Render Props 相比，为什么说 Hook 是更好的代码复用方案？它解决了前两者的哪些痛点？

答 ​

自定义 Hook 在"复用有状态逻辑"这个场景下，几乎全面优于 HOC 和 Render Props。它解决了前两者几乎所有的痛点：

1. 解决了嵌套地狱（Wrapper Hell） ​

HOC 和 Render Props 最大的问题就是嵌套——复用 3 个逻辑，组件树就多了 3 层。调试时打开 React DevTools，一堆 <WithXxx> 或 <MouseTracker>，很难找到真正的组件。

jsx
// HOC 的嵌套
export default withRouter(withUser(withTheme(Profile)));

// Render Props 的嵌套
<UserProvider>
  {user => (
    <ThemeProvider>
      {theme => (
        <Router>
          {route => <Profile user={user} theme={theme} route={route} />}
        </Router>
      )}
    </ThemeProvider>
  )}
</UserProvider>

jsx
function Profile() {
  const user = useUser();
  const theme = useTheme();
  const route = useRouter();
  // 直接用，没有嵌套
}

2. 解决了 props 来源不透明的问题 ​

HOC 注入的 props，你打开组件文件根本不知道从哪来的——是父组件传的？还是哪个 HOC 注的？叫什么名字？都不直观。

Hook 的数据来源非常清晰：const user = useUser()——你一眼就知道 user 是从 useUser 这个 Hook 来的，想追代码直接点进去就行。

3. 解决了命名冲突问题 ​

两个 HOC 注入了同名的 prop，后注入的会覆盖先注入的，而且很难排查。Render Props 稍微好点（变量名自己取），但还是要注意不要和外层变量重名。

Hook 完全没有这个问题——返回值的名字是你自己定义的，想叫什么叫什么，重名了直接改就行，不会有"覆盖"的问题。

4. 解决了静态方法丢失 / ref 转发问题 ​

HOC 包装后，原组件的静态方法会丢失，需要用 hoist-non-react-statics 手动搬运。ref 也不会自动转发，要手动 forwardRef。这些都是 boilerplate。

Hook 是在组件内部调用的，不改变组件本身的结构，所以静态方法、ref 都和普通组件一样，没有这些麻烦。

5. 更轻量，心智负担小 ​

HOC 你得理解"组件进、组件出"的函数式思想，Render Props 你得理解"把渲染逻辑当 prop 传"的思路。而 Hook 就是普通的函数调用，学起来更简单，用起来更自然。

Hook 不是万能的 ​

当然，Hook 也不是所有场景都能替代：

  • UI 结构的复用：还是得用组合（children / props 传 ReactNode）。Hook 只管逻辑，不管 UI。
  • 横切关注点：比如"所有路由切换时都埋个点"，这种用 HOC 包一层路由也很方便。但多数场景 Hook 更直接。
  • 组件库的 Compound Components 模式：需要父子组件协同的，Context + 子组件的模式还是最佳实践。

但在"复用有状态的逻辑"这个维度上，自定义 Hook 毫无疑问是目前最好的方案。

追问 2 ​

题目 ​

受控组件每次输入都重渲染，不会有性能问题吗？什么时候需要考虑非受控组件来优化性能？

答 ​

受控组件每次输入都会触发重渲染，这是事实。但绝大多数情况下，这点性能开销可以忽略不计。原因如下：

1. React 的重渲染很快 ​

React 的虚拟 DOM diff 已经优化得非常好了。一个简单的输入框重渲染一下，diff 到真实 DOM 更新，耗时不到 1ms，用户完全感知不到。

不要过早优化——在你测出性能问题之前，默认认为受控组件的性能是够的。

2. 重渲染的是哪些组件？ ​

受控组件的重渲染通常只影响当前表单组件和它的子组件。如果表单组件不大，重渲染的开销微乎其微。真正的性能问题往往出在"父组件重渲染导致一大堆子组件跟着重渲染"——但这不是受控组件的锅，是你状态放的位置不对（可以用状态下沉、memo 等优化）。

3. 什么时候真的有性能问题？ ​

以下场景下，受控组件可能真的会卡：

a. 非常复杂的富文本编辑器 / 大型表单 如果一个页面有几十个字段，每次输入都要整个大表单重渲染一遍，确实可能卡。但这种场景很少见，而且可以用其他方式优化（比如把表单拆成多个子组件、用 memo 包裹、用 Formik 或 React Hook Form 的订阅优化）。

b. 输入时需要做昂贵的计算 比如输入时实时格式化、实时语法高亮、实时计算复杂结果，如果计算量很大，每次输入都算一遍可能卡。

  • 优化方式：用 useMemo 缓存计算结果，或者用 useDeferredValue 延迟计算。

c. Canvas / WebGL 等高性能场景 比如用 input 控制 3D 场景的参数，每次输入都要重绘整个场景，可能掉帧。

  • 优化方式：节流（throttle），16ms 内最多更新一次，保证 60fps。

4. 非受控组件是性能优化的好选择吗？ ​

是，但优先级不高。因为：

  1. 非受控组件的缺点很明显：数据不在 React 的控制下，做校验、联动、动态修改值都很麻烦。代码复杂度上升了很多。
  2. 有很多其他优化手段：memo、useMemo、状态拆分、防抖节流、React Hook Form 等，这些手段在不放弃受控组件便利性的前提下就能解决大部分性能问题。
  3. 非受控组件本身也有限制：比如你没法在输入时实时修改值（比如自动格式化手机号），因为值不在你手里。

建议的优化路径 ​

如果你的表单真的有性能问题，按以下顺序优化：

  1. 检查状态放置位置：把状态下沉到真正需要的组件，不要放在顶层导致全局重渲染。
  2. 用 memo 优化重渲染的子组件：配合 useCallback 稳定回调引用。
  3. 用 React Hook Form 等库：它们用非受控 + 订阅的方式，只有真正变化的字段会触发对应组件重渲染，性能很好，同时又保留了类似受控组件的开发体验。
  4. 最后再考虑手写非受控组件：只有以上都试过了还不够，再考虑。

大多数项目，第 1、2 步就够了。

追问 3 ​

题目 ​

Compound Components 模式中，除了用 Context 共享状态，还有其他实现方式吗？各有什么优缺点？

答 ​

Compound Components 的核心是"父组件管理状态，多个子组件协同工作，子组件可以灵活排列"。状态共享的方式主要有三种，各有优劣：

方式 1：React.cloneElement + 遍历 children ​

最早的实现方式（React Context 出现之前的老办法）。父组件遍历 children，用 React.cloneElement 把状态和方法作为 props 注入到每个子组件里。

jsx
function Tabs({ defaultValue, children }) {
  const [activeKey, setActiveKey] = useState(defaultValue);

  // 遍历 children，给每个子组件注入 props
  const childrenWithProps = React.Children.map(children, child => {
    if (React.isValidElement(child)) {
      return React.cloneElement(child, {
        activeKey,
        setActiveKey,
      });
    }
    return child;
  });

  return <div className="tabs">{childrenWithProps}</div>;
}

优点：

  • 不依赖 Context，兼容性好（远古 React 版本也能用）。
  • props 传递显式，调试起来简单一些。

缺点：

  • 只能传给直接子组件：孙子组件拿不到，得层层往下传。如果用户在 Tab 和 TabList 之间包了一层 <div>，props 就传不到 Tab 了。
  • 不够灵活：子组件的层级必须固定，不能随意嵌套。
  • 类型推断麻烦：TypeScript 场景下，cloneElement 的类型不好写。
  • 性能：每次渲染都要遍历和 clone 子元素。

方式 2：Context（最常用） ​

就是前面例子里讲的方式——父组件把状态放到 Context Provider 里，子组件用 useContext 读取。

优点：

  • 任意层级都能拿到：子组件、孙子组件、深层嵌套都能用，用户可以随意包 div、加中间层。
  • API 简洁：子组件直接 useContext 就行，不用层层传 props。
  • 性能优化空间大：可以拆成多个 Context、用 memo 等。

缺点：

  • 隐式依赖：子组件必须在 Provider 里面才能用，单独用会报错，用户可能不知道。
  • 调试困难：状态通过 Context 流，追踪起来不如 props 直观。
  • 消费者重渲染问题：Context value 变化时，所有消费者都会重渲染（可以通过拆分 Context、用 use-context-selector 等优化）。

方式 3：状态管理库（Zustand / Jotai 等） ​

对于特别复杂的复合组件（比如一个功能完整的表格组件，有筛选、排序、分页、选择等很多子部件），Context 可能不够用（Context 太多也乱），可以直接用一个轻量状态库来管理内部状态。

jsx
// 用 Zustand 创建一个组件内部的 store
const useTableStore = create((set, get) => ({
  data: [],
  sortKey: null,
  sortOrder: 'asc',
  setSort: (key) => set({ sortKey: key }),
  // ... 很多状态和方法
}));

优点：

  • 状态管理能力强：复杂逻辑、中间件、DevTools 都有。
  • 订阅粒度细：用 Zustand 的 selector，子组件只订阅自己需要的字段，避免不必要的重渲染。
  • 适合大型复合组件：状态多、逻辑复杂的场景比 Context 更顺手。

缺点：

  • 引入了额外依赖。
  • 多个实例之间状态隔离要处理好（比如同一个页面有两个 Table，store 不能串）。通常用 Context + store 实例的方式解决。
  • 有点"杀鸡用牛刀"，简单的复合组件犯不上。

怎么选？ ​

  • 简单组件（状态少，2-3 个）：Context 就行，轻量直接。
  • 中等复杂度：Context + 拆分成多个 Context，控制好重渲染。
  • 大型组件库级别的复杂复合组件：可以考虑用轻量状态库，开发体验和可维护性更好。
  • 兼容老版本 React：cloneElement 方式（但现在基本不用考虑了）。

目前行业主流是 Context 方式，大多数组件库（Ant Design、Radix UI、React Aria 等）的 Compound Components 都是基于 Context 实现的。
