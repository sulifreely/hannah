问答 18（高级）：前端错误监控与告警系统设计 ​

题目 ​

请说明前端需要监控哪些类型的错误、各类错误的采集原理、错误堆栈如何通过 SourceMap 反解、用户行为回溯怎么做、如何做去重与合并，以及告警分级与降噪策略。

参考答案 ​

一、为什么需要前端错误监控 ​

"线上报错了，用户投诉了我们才知道"——这是很多团队的痛点。

前端运行在用户的浏览器里，环境千差万别（不同浏览器、不同设备、不同网络、不同操作路径），你不可能在发布前覆盖所有场景。错误监控就是你的"眼睛"，让你知道线上到底发生了什么。

没有错误监控的痛点：

  • 用户遇到 bug 了，开发不知道，等用户投诉才去查。
  • 没有堆栈信息，用户只会说"点了没反应"，根本没法定位。
  • 不知道错误影响了多少用户，优先级不好排。
  • 发版后有没有引入新 bug，全靠猜。

有了错误监控之后：

  • 线上一报错，立即告警，主动发现问题。
  • 有完整的堆栈信息，直接定位到源码位置。
  • 知道错误影响范围（多少用户、多少次、哪些页面）。
  • 发版后监控错误率，有问题及时回滚。

二、需要监控哪些错误 ​

前端的错误类型很多，不同类型的采集方式不一样。

1. JS 运行时错误 ​

最常见的错误：

  • 语法错误（SyntaxError）
  • 类型错误（TypeError，最常见，比如 Cannot read property 'xxx' of undefined）
  • 引用错误（ReferenceError）
  • 范围错误（RangeError）
  • 自定义错误（throw new Error(...)）

采集方式：window.onerror

javascript
window.onerror = function(message, source, lineno, colno, error) {
  // message: 错误信息
  // source: 发生错误的脚本 URL
  // lineno: 行号
  // colno: 列号
  // error: Error 对象，有 stack 属性
  
  reportError({
    message,
    source,
    lineno,
    colno,
    stack: error?.stack,
  });
  
  return true; // 阻止默认行为（不在控制台打印）
};

2. Promise 未捕获错误 ​

Promise 的 reject 如果没有被 catch，window.onerror 捕获不到，要用 unhandledrejection 事件。

javascript
window.addEventListener('unhandledrejection', function(event) {
  // event.reason 是 Promise reject 的原因，可能是 Error 对象
  const reason = event.reason;
  
  reportError({
    type: 'unhandledrejection',
    message: reason?.message || String(reason),
    stack: reason?.stack,
  });
});

3. 静态资源加载错误 ​

图片、JS、CSS、字体等资源加载失败（404、500、网络错误）。

采集方式：error 事件 + 捕获阶段

javascript
// 资源加载错误不会冒泡，必须在捕获阶段监听
window.addEventListener('error', function(event) {
  // JS 运行时错误的 target 是 window
  // 资源加载错误的 target 是具体的元素（img/script/link）
  if (event.target !== window) {
    const target = event.target;
    const url = target.src || target.href;
    
    reportResourceError({
      type: 'resource-error',
      tagName: target.tagName,
      url: url,
      pageUrl: location.href,
    });
  }
}, true); // 注意：第三个参数是 true，捕获阶段

4. 接口请求错误 ​

AJAX / Fetch 请求失败（4xx、5xx、网络错误）。

采集方式：劫持 XMLHttpRequest 和 fetch

javascript
// 劫持 XHR
const originalXHROpen = XMLHttpRequest.prototype.open;
XMLHttpRequest.prototype.open = function(method, url) {
  this._url = url;
  this._method = method;
  this._startTime = Date.now();
  return originalXHROpen.apply(this, arguments);
};

const originalXHRSend = XMLHttpRequest.prototype.send;
XMLHttpRequest.prototype.send = function() {
  const handleError = () => {
    reportApiError({
      type: 'xhr-error',
      method: this._method,
      url: this._url,
      status: this.status,
      duration: Date.now() - this._startTime,
      response: this.responseText?.slice(0, 500), // 只截前 500 字符
    });
  };
  
  this.addEventListener('error', handleError);
  this.addEventListener('abort', handleError);
  this.addEventListener('load', () => {
    if (this.status >= 400) {
      handleError();
    }
  });
  
  return originalXHRSend.apply(this, arguments);
};

const originalFetch = window.fetch;
window.fetch = function(input, init) {
  const url = typeof input === 'string' ? input : input.url;
  const method = init?.method || 'GET';
  const startTime = Date.now();

      return response;
    })
    .catch(error => {
      reportApiError({
        type: 'fetch-network-error',
        method,
        url,
        message: error.message,
        duration: Date.now() - startTime,
      });
      throw error;
    });
};

5. 白屏错误 ​

页面加载了但内容是空的（JS 报错导致首屏没渲染出来，或者框架崩了）。

采集方式：检测页面关键位置有没有内容

javascript
// 简单版：检测 #app 容器里有没有内容
setTimeout(() => {
  const app = document.getElementById('app');
  if (app && app.children.length === 0) {
    reportWhiteScreen({
      type: 'white-screen',
      url: location.href,
      html: app.innerHTML.slice(0, 1000),
    });
  }
}, 5000); // 5 秒后检查

6. 性能异常（也算广义的"错误"） ​

性能太差导致用户体验糟糕，虽然不是代码报错，但也是问题。

  • LCP > 4s
  • INP > 500ms
  • 内存泄漏（JS Heap 持续增长）
  • 长时间卡顿（long task > 200ms）

7. 框架级错误 ​

React/Vue 等框架有自己的错误捕获机制：

// React 错误边界
class ErrorBoundary extends React.Component {

      message: error.message,
      stack: error.stack,
      componentStack: errorInfo.componentStack,
    });
  }
  render() {
    return this.props.children;
  }
}

三、错误堆栈解析与 SourceMap 反解 ​

线上的 JS 都是压缩混淆后的，堆栈里的行号列号都是压缩后的，根本看不懂。需要用 SourceMap 反解到源代码位置。

反解流程：

1. 前端上报错误（压缩后的行号、列号、文件名）
     ↓
2. 后端/消费端拿到错误信息
     ↓
3. 找到对应版本的 SourceMap 文件
     ↓
4. 用 source-map 库反解
     ↓
5. 得到源代码的文件名、行号、列号、函数名
     ↓
6. 存入数据库 / 展示

反解实现（Node.js）：

javascript
const { SourceMapConsumer } = require('source-map');
const fs = require('fs');

async function parseStackTrace(stack, release) {
  // 从堆栈字符串中提取每个栈帧
  const frames = parseStackFrames(stack);
  
  const parsedFrames = [];
  
  for (const frame of frames) {
    // frame: { filename, line, column, function }
    
    // 找到对应的 .map 文件（根据 release 版本 + 文件名）
    const mapFile = findSourceMapFile(frame.filename, release);
    
    if (mapFile) {
      const rawSourceMap = fs.readFileSync(mapFile, 'utf8');
      const consumer = await new SourceMapConsumer(rawSourceMap);
      
      const original = consumer.originalPositionFor({
        line: frame.line,
        column: frame.column,
      });
      
      parsedFrames.push({
        ...frame,
        originalFilename: original.source,
        originalLine: original.line,
        originalColumn: original.column,
        originalFunction: original.name,
        sourceCode: getSourceCode(original.source, original.line, consumer),
      });
      
      consumer.destroy();
    } else {
      parsedFrames.push(frame); // 没有 map 就保留原始的
    }
  }
  
  return parsedFrames;
}

SourceMap 管理：

  • 每个版本的 SourceMap 都要存好，和 release 版本对应。
  • 可以存在对象存储（OSS/S3）里，数据库里存索引。

四、用户行为回溯（Breadcrumbs） ​

只有错误堆栈还不够——"为什么会走到这个代码路径？用户做了什么操作？"

用户行为回溯（也叫面包屑 Breadcrumb）就是记录错误发生前的用户操作轨迹，帮助复现问题。

记录哪些行为：

  1. 用户交互：点击、输入、滚动、按键
  2. 路由变化：页面跳转
  3. 接口请求：XHR/Fetch 请求的 URL、状态、耗时
  4. 控制台日志：console.log / warn / error
  5. 性能事件：LCP、FCP 等关键节点
  6. 自定义事件：业务关键操作（比如"加入购物车"、"提交订单"）

实现示例：

javascript
const breadcrumbs = [];
const MAX_BREADCRUMBS = 20; // 只保留最近 20 条

function addBreadcrumb(type, message, data = {}) {
  breadcrumbs.push({
    type,
    message,
    data,
    timestamp: Date.now(),
  });
  
  // 超过最大数量，删掉最老的
  if (breadcrumbs.length > MAX_BREADCRUMBS) {
    breadcrumbs.shift();
  }
}

// 记录点击事件
document.addEventListener('click', (e) => {
  const target = e.target;
  addBreadcrumb('click', `${target.tagName} ${target.id || target.className}`, {
    tagName: target.tagName,
    id: target.id,
    class: target.className,
  });
}, true);

// 记录路由变化
window.addEventListener('popstate', () => {
  addBreadcrumb('route', location.pathname);
});

// 记录接口请求
//（在前面的 XHR/Fetch 劫持里加）
addBreadcrumb('xhr', `${method} ${url}`, { status, duration });

// 上报错误时，把面包屑一起带上
function reportError(error) {
  send({
    ...error,
    breadcrumbs: [...breadcrumbs], // 复制一份
  });
}

五、错误去重与合并 ​

如果不做去重，同一个错误发生一万次，你就会收到一万条告警，直接被消息淹没。

怎么定义"同一个错误"

核心是生成一个错误指纹（fingerprint），相同指纹的错误被认为是同一个。

指纹生成策略：

默认指纹 = 错误信息 + 堆栈中第一个有意义的栈帧（文件名 + 行号 + 函数名）
1

但光靠这个不行，因为：

  • 同一个 bug，错误信息里可能带动态参数（比如用户 ID），导致每次 message 不一样。
  • 行号列号每次发版可能变（因为代码行数变了）。

更合理的策略：

  1. 规范化错误信息：把 message 里的动态内容（数字、ID、URL 路径参数）替换成通配符。

    • User 12345 not found → User * not found
    • GET /api/user/12345 failed → GET /api/user/* failed

  2. 规范化堆栈：

    • 去掉 node_modules 里的栈帧（第三方库的帧不算），取第一个业务代码的栈帧。
    • 用函数名 + 文件名的相对路径做标识，不用行号（行号经常变）。

  3. 自定义指纹：允许业务代码手动指定错误的 fingerprint。

javascript
function generateFingerprint(error) {
  // 1. 规范化 message
  const normalizedMessage = normalizeMessage(error.message);
  
  // 2. 取第一个业务代码栈帧
  const firstFrame = getFirstAppFrame(error.stack);
  
  // 3. 组合指纹
  if (firstFrame) {
    return `${normalizedMessage}::${firstFrame.function}::${firstFrame.filename}`;
  }
  return normalizedMessage;
}

function getFirstAppFrame(stack) {
  const frames = parseStackFrames(stack);
  // 跳过 node_modules 里的，找第一个业务代码的
  return frames.find(f => !f.filename.includes('node_modules'));
}

合并策略：

  • 相同指纹的错误，合并成一条"错误 Issue"。
  • 每条 Issue 记录：首次发生时间、最近发生时间、发生次数、影响用户数。
  • 新发生的错误 → 创建新 Issue → 触发告警。
  • 已经存在的错误 → 只更新计数，不重复告警。

六、告警分级与降噪 ​

告警太多等于没有告警。必须分级 + 降噪，确保收到的每条告警都是值得关注的。

告警分级 ​

级别严重程度触发条件通知方式响应时间
P0 / Critical极严重大面积白屏、核心功能不可用、错误率 > 10%电话 + 短信 + IM立即
P1 / High严重核心功能报错、错误率 > 1%、影响用户多IM @所有人30 分钟内
P2 / Warning警告非核心功能报错、错误率上升IM 普通消息当天内
P3 / Info提示零星错误、低频邮件 / 看板每周汇总

怎么判断"严重程度"：

  • 影响范围：影响用户比例越高越严重。
  • 功能重要性：支付/登录/下单等核心链路最严重。
  • 是否有 workaround：用户能不能通过刷新、重进解决。
  • 发生频率：是偶发还是必现。

告警降噪 ​

降噪手段 1：阈值 + 持续时间

不要一报错就告警，要有阈值和持续时间：

  • ❌ 1 分钟内发生 1 次就告警 → 误报多
  • ✅ 连续 5 分钟错误率超过 1% 才告警 → 稳定可靠

降噪手段 2：新错误才告警

  • 已经存在的老错误，不要每次发生都告警。
  • 只有新出现的错误（之前没见过的 fingerprint）才告警。
  • 老错误如果数量突增（比如突然涨了 10 倍），再告警。

降噪手段 3：告警合并与抑制

  • 同一个错误 1 小时内只告警一次。
  • 同一类问题导致的多条告警，合并成一条（比如后端接口挂了导致前端 10 个接口报错，合并成一条"接口异常"告警）。

降噪手段 4：告警静默 / 抑制

  • 已知问题：正在处理的错误，先静默，不用反复告警。
  • 计划内变更：发版期间的告警，可以暂时抑制（发版后 10 分钟内的错误额外关注，但不批量告警）。
  • 低峰期：凌晨的低级别告警，可以等白天再说。

降噪手段 5：采样 / 升级

  • 低频错误（一天几次）：只记录，不告警，定期看板上看。
  • 中频错误（一小时几次）：发 IM 消息。
  • 高频错误（一分钟几十次）：升级到电话告警。

追问 1：Sentry 的错误捕获原理是什么？它比我们自己手写 onerror 好在哪里？Sentry 的 SDK 大致包含哪些核心模块？ ​

答 ​

Sentry 是目前最流行的开源错误监控平台，它的 SDK 比我们自己手写 window.onerror 强大得多。理解它的原理和设计，对自己做监控系统很有参考价值。

Sentry 的核心设计

Sentry 的 SDK 不是简单地包一层 window.onerror，而是有一套完整的错误捕获、上下文增强、数据上报的体系。

核心模块拆解：

1. 错误捕获层（Integrations）

Sentry 用"集成（Integrations）"的方式模块化地捕获各类错误，而不是写死在 SDK 里。每个 Integration 负责一类错误的采集。

核心内置 Integrations：

  • GlobalHandlers：全局错误处理器，封装了 window.onerror 和 unhandledrejection。
  • TryCatch：劫持 setTimeout、setInterval、requestAnimationFrame 等异步 API，用 try/catch 包裹回调，确保异步错误也能捕获到堆栈。
  • Breadcrumbs：面包屑，记录用户行为（点击、XHR、console、路由等）。
  • BrowserTracing：性能追踪（Web Vitals、Transaction 等）。
  • Dedupe：错误去重。
  • InboundFilters：过滤掉不需要上报的错误（比如某些第三方库的错误）。
  • HttpContext：劫持 XHR/Fetch，记录请求信息。

为什么 TryCatch 很重要：

很多异步错误，window.onerror 也能捕获，但有时候堆栈不完整。TryCatch 集成的做法是把常见的异步 API 都包裹一层：

javascript
// 简化示意
const originalSetTimeout = window.setTimeout;
window.setTimeout = function(callback, delay, ...args) {
  const wrappedCallback = function() {
    try {
      return callback.apply(this, arguments);
    } catch (e) {
      Sentry.captureException(e);
      throw e;
    }
  };
  return originalSetTimeout.call(this, wrappedCallback, delay, ...args);
};

4. 堆栈处理（Stack Trace）

Sentry 对错误堆栈的处理非常精细：

  • 堆栈解析：支持各种浏览器的堆栈格式（Chrome、Firefox、Safari、IE 各不一样），统一解析成结构化的栈帧。
  • SourceMap 反解：服务端用 SourceMap 把压缩后的栈帧还原到源码位置。
  • 栈帧增强：可以上传源码后，在 Sentry 界面直接看到报错行的前后代码。
  • In-app 标记：标记哪些栈帧是你的业务代码（in-app），哪些是第三方库的，默认只展示 in-app 的，减少干扰。
  • 忽略库文件：node_modules 里的错误可以折叠起来，重点看自己的代码。

5. 指纹与分组（Fingerprinting）

Sentry 自动给错误生成指纹，相同指纹的合并到同一个 Issue。

默认分组规则：

  • 基于错误信息 + 栈帧（有一套复杂的算法）。
  • 会自动规范化 message（去掉动态内容）。
  • 会考虑 in-app 栈帧（优先用业务代码的栈帧分组）。

也支持自定义：

  • 页面加载性能

  • 用户交互性能
  • 分布式追踪（前端 → 后端全链路）

7. 传输与缓存

  • 批量上报：错误攒一批再发，减少请求。
  • sendBeacon 兜底：页面关闭时用 sendBeacon。
  • 离线缓存：网络不好时存 localStorage，下次上线再发。
  • 重试机制：上报失败了重试几次。
  • 采样率：可以设置 tracesSampleRate，性能数据采样上报。

什么时候用 Sentry，什么时候自建

  • 小团队 / 创业公司：直接用 Sentry（免费版足够），把精力放在业务上。
  • 中型团队：Sentry 付费版，或者自建 Sentry。
  • 大型团队 / 数据敏感：自建，或者基于 Sentry 二次开发。
  • 特殊定制需求：自建，但可以参考 Sentry 的架构设计。

Sentry 的架构对自建监控的启发

即使你要自建，也可以借鉴 Sentry 的设计：

  1. 用 Integration / 插件的方式扩展采集能力，不要写死。
  2. 错误事件要有处理管道（beforeSend / 过滤 / 去重 / 采样）。
  3. 要有 Scope / 上下文机制，方便附加用户信息、业务信息。
  4. 堆栈解析要全面，兼容各种浏览器格式。
  5. 错误要做指纹分组，不然告警会爆炸。
  6. 上报要做批量 + 缓存 + 离线兜底。

追问 2：线上 JS 报错 "Script error." 是什么原因？怎么解决？ ​

答 ​

这是前端错误监控里一个非常经典的坑——你会看到大量 Script error.，没有行号、没有列号、没有堆栈，完全无法定位。

什么是 "Script error."

当页面加载了一个跨域的 JS 文件（比如 CDN 上的 JS），这个 JS 里发生了错误，window.onerror 捕获到的错误信息只有 "Script error."，没有详细的错误消息、行号、列号、堆栈。

这不是 bug，而是浏览器的安全策略——CORS（跨域资源共享）的限制。

为什么浏览器要这么做

想象一下：

  • 你访问 evil.com，它的页面里引入了 https://mail.google.com/mail.js（假设 Gmail 有这个 JS 文件）。
  • 如果 mail.js 里有错误，错误信息可能包含敏感信息（比如用户的邮箱、未读邮件数等）。
  • 如果 evil.com 能通过 window.onerror 拿到完整的错误信息，它就可以窃取用户在 Gmail 里的信息。

为了防止这种跨域信息泄露，浏览器规定：跨域脚本的错误详情不能被页面读取，只返回一个模糊的 "Script error."。

这就是"Script error." 的来源——不是没有错误详情，是浏览器不让你看。

怎么解决

分两步：服务器端配置 CORS + 前端加 crossorigin 属性。

第一步：CDN / 静态资源服务器配置 CORS 头

在返回 JS 文件的响应头里加上：

Access-Control-Allow-Origin: *
1

或者更严格一点，指定你的域名：

Access-Control-Allow-Origin: https://www.yourdomain.com
1

意思是："这个 JS 文件允许指定的源跨域读取，包括错误详情。"

不同的 CDN / 服务器配置方式不一样：

nginx
# Nginx 配置
location ~* \.(js|css)$ {
    add_header Access-Control-Allow-Origin *;
    # ...
}

yaml
# 阿里云 OSS / 腾讯云 COS 的 CORS 配置
# 在控制台的 "跨域访问设置" 里添加规则
AllowedOrigin: *
AllowedMethod: GET, HEAD
AllowedHeader: *

5

第二步：script 标签加 crossorigin 属性

html
<!-- 加了 crossorigin="anonymous" -->
<script src="https://cdn.example.com/app.js" crossorigin="anonymous"></script>
1
2

crossorigin="anonymous" 的意思是：请求这个资源时，不带身份凭证（cookie、HTTP 认证信息等），但允许读取响应内容（包括错误详情）。

为什么两步都要有：

  • 只配服务器 CORS，script 不加 crossorigin → 浏览器还是按普通请求发，不会读取 CORS 头，错误还是被屏蔽。
  • 只加 script crossorigin，服务器没配 CORS → 服务器不返回 Access-Control-Allow-Origin，浏览器直接拦截整个 JS 加载，连 JS 都执行不了。

必须同时配置才生效。

特殊情况：内联脚本（inline script）

内联的 <script>...</script> 没有跨域问题，因为它和页面同源。但内联脚本通常没有文件名，堆栈里不好定位。解决方式是给内联脚本加 //# sourceURL=xxx.js 或者用 nonce + SourceMap。

特殊情况：第三方脚本的错误

如果是你引入的第三方 JS（广告、统计、第三方 SDK）报的错：

  • 你没法让第三方的服务器配 CORS（你管不着）。
  • 这些错误大部分时候也跟你没关系（不是你的代码 bug）。
  • 建议：直接过滤掉，不纳入告警。
javascript
// Sentry 里过滤第三方错误
Sentry.init({
  denyUrls: [
    // 谷歌广告
    /googleadservices\.com/i,
    // 百度统计
    /hm\.baidu\.com/i,
    // 其他第三方...
  ],
});

  2. 多收集上下文：虽然没有堆栈，但至少知道"在哪个页面、用户做了什么操作、什么时候"，靠这些线索推测。
  3. 用 try/catch 包裹：如果你知道哪段第三方代码可能报错，可以自己用 try/catch 包一层，在 catch 里上报。try/catch 里拿到的 Error 对象是有完整堆栈的（因为代码已经在你页面的上下文里执行了）。

  Sentry.captureException(e);
}

追问 3：错误监控系统如何做告警降噪，才能既不漏掉重要问题，又不被告警淹没？ ​

答 ​

这是所有监控系统的核心难题——告警太少会漏，告警太多会麻木。好的告警策略是"该响的时候响，不该响的时候安静"。

下面是一套从实践中总结出来的降噪方法论，按层次递进：

第一层：错误分级——不是所有错误都一样重要

先给错误分等级，不同等级不同对待。

分级维度：

  1. 错误类型：

    • 白屏 / 框架崩溃 → 最高级
    • JS 运行时错误 → 中高级
    • 接口错误 → 中级
    • 资源加载错误（图片、CSS）→ 低级
    • 第三方脚本错误 → 忽略

  2. 影响范围：

    • 影响用户数（UV）
    • 发生次数（PV）
    • 错误率（错误 PV / 总 PV）

  3. 功能重要性：

    • 核心链路（支付、登录、下单）→ 高
    • 主要功能（列表、详情）→ 中
    • 边缘功能（帮助中心、设置）→ 低

  4. 是否可恢复：

    • 用户刷新就好 → 低
    • 完全用不了 → 高

一个实用的分级公式：

严重程度 = f(错误类型权重 × 影响范围 × 功能重要性 ÷ 可恢复性)
1

实际操作中不用真的写公式，靠规则配置就行：

javascript
const levelRules = [
  // P0：白屏率超过 0.5%，且用户数 > 100
  {
    level: 'P0',
    match: (event) => 
      event.type === 'white-screen' && 
      event.stats.userCount > 100 && 
      event.stats.rate > 0.005,
  },
  // P1：核心页面 JS 错误率 > 1%
  {
    level: 'P1',
    match: (event) =>
      event.type === 'js-error' &&
      event.pageType === 'core' &&
      event.stats.rate > 0.01,
  },
  // P2：普通 JS 错误，新出现的
  {
    level: 'P2',
    match: (event) =>
      event.type === 'js-error' &&
      event.isNew &&
      event.stats.count > 10,
  },
];

第二层：抑制策略——不要同一件事反复告警

同一个错误，或者同一类问题，不要反复发告警。

抑制手段 1：同错误冷却

同一个错误（同一个 fingerprint），在一段时间内只告警一次。

  • 新错误第一次出现 → 立即告警。
  • 之后 1 小时内再发生 → 不告警，只累计数量。
  • 1 小时后如果还在发生 → 再告警一次（提醒还没解决）。
  • 错误数量突然激增（比如 ×10）→ 升级告警。

抑制手段 2：告警风暴抑制

短时间内突然出现大量不同的错误（比如后端挂了导致前端 100 个接口报错），不要发 100 条告警。

做法：

  • 5 分钟内告警超过 20 条 → 触发风暴模式。
  • 风暴模式下，只发一条"告警风暴，共 N 条错误"的汇总告警。
  • 风暴结束后补发详细汇总。

抑制手段 3：已知问题静默

  • 已经在处理的 bug → 手动标记为"处理中"，暂时静默。
  • 计划内的操作 → 提前设置维护窗口，期间告警静默。
    • 比如发版后的 10 分钟，预期会有一些波动，不要大惊小怪。
    • 但发版期间的错误要重点关注，可以单独出一个"发版监控"的视图。

抑制手段 4：工作时间分级

  • 工作时间（9:00 ~ 21:00）：P1 以上 @所有人，P2 正常发。
  • 非工作时间（21:00 ~ 9:00）：只有 P0 打电话，P1 发消息不 @人，P2 以下等第二天再说。
  • 周末：类似夜间，更宽松一点。

第三层：基线与异常检测——不是超过固定阈值才叫问题

很多人用固定阈值（比如"错误率 > 1% 告警"），但这很粗糙：

  • 平时错误率是 0.01%，突然涨到 0.1%——虽然没到 1%，但已经涨了 10 倍，可能有问题。
  • 平时错误率就是 2%（历史包袱重），1% 的阈值根本没意义。

更好的方式是基于历史数据的异常检测：

方法 1：同比 / 环比

  • 和昨天同一时间比（同比）：涨了 50% → 告警。
  • 和上一小时比（环比）：涨了 30% → 告警。
告警条件 = 当前值 > 过去 7 天同时间段平均值 × 1.5
1

方法 2：移动平均 + 标准差

更科学一点，用统计学的方法：

基线 = 过去 N 小时的移动平均值
波动 = 标准差

告警条件 = 当前值 > 基线 + 3 × 标准差（3σ 原则，99.7% 置信度）
1
2
3
4

也就是"明显偏离正常范围"就告警，不管绝对值是多少。

方法 3：趋势检测

  • 错误率在持续上升（连续 5 分钟都在涨）→ 告警。
  • 不一定已经很高，但趋势不对，提前预警。

第四层：告警关联——把相关的告警合并成一个

很多告警其实是同一个根因导致的，不要发多条。

关联策略：

1. 前后端关联

场景：后端接口挂了，前端同时出现 10 个接口错误 + 20 个 JS 错误（因为接口数据不对导致 JS 报错）。

做法：

  • 检测到大量接口错误时，自动把相关的 JS 错误标记为"可能由接口异常导致"。
  • 只发一条核心告警（"接口异常，可能导致前端错误增多"），其他的收敛在详情里。

2. 同一页面的多个错误

场景：一个页面崩了，上面所有组件都报错，产生几十个错误。

做法：

  • 按页面聚合，同一个页面 1 分钟内的多个错误合并成一条。
  • 只告警"XX 页面错误率飙升"，具体错误详情点进去看。

3. 发布关联

场景：发版后出现了一批新错误。

做法：

  • 自动和最近的发布关联，告警消息里带上"可能是 v1.2.3 发布引入的"。
  • 甚至可以自动对比：发布前错误率 X，发布后错误率 Y，增加了 Z%。

第五层：用户反馈闭环——告警有没有用，用户说了算

最后，告警系统本身也要迭代优化。

机制 1：告警反馈

每条告警下面加两个按钮：

  • 👍 有用（确实是问题）
  • 👎 误报（不是问题 / 不用管）

收集反馈，不断调整告警规则和阈值。误报多的规则，要么调松，要么关掉。

机制 2：On-Call 复盘

每次 On-Call 后复盘：

  • 这周收到了多少告警？
  • 哪些是真的有问题？哪些是误报？
  • 误报的为什么会误报？怎么改进规则？
  • 有没有漏掉的问题？为什么没告警？

持续迭代，告警质量会越来越高。

机制 3：告警数是核心指标

把"每周告警数"、"误报率"作为监控系统本身的核心指标。

  • 告警越来越多 → 系统在变差，要清理。
  • 误报率 > 50% → 大家已经不信告警了，必须整改。

一个反直觉的经验

好的告警系统，应该是安静的。

  • 一天到晚响个不停的告警系统，等于没有告警系统——大家都麻木了。
  • 真正好的系统，平时很安静，一响就真的有问题，大家会立即响应。
  • 宁可少告警，也不要乱告警。少了可以加，多了大家麻木了就麻烦了。

告警的黄金法则：

  1. 每条告警都应该是可行动的（收到就知道该干啥）。
  2. 告警不应该需要人去判断是不是问题（是的话才告警，不是的话别发）。
  3. 告警太多会被忽略（宁少勿滥）。

总结降噪层次：

  • 第一层：分级（不同错误不同对待）
  • 第二层：抑制（同一件事别反复说）
  • 第三层：基线（用异常检测替代固定阈值）
  • 第四层：关联（相关的合并成一条）
  • 第五层：闭环（不断迭代优化）

做到这五层，你的告警系统就既不会漏报，也不会被淹没。
