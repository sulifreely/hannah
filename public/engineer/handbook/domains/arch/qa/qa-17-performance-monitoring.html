问答 17（高级）：前端性能监控系统设计 ​

题目 ​

请说明为什么需要自建前端性能监控、需要采集哪些性能指标（Web Vitals、Navigation Timing、Resource Timing 等）、数据上报策略（采样、批量、sendBeacon），以及数据可视化与告警设计。对比一下 Sentry 等开源方案。

参考答案 ​

一、为什么需要自建前端性能监控 ​

很多人会问：Chrome DevTools 里的 Performance/Lighthouse 不能看性能吗？为什么还要专门搞个监控系统？

DevTools 的局限：

  • 只能看你自己的电脑、你自己的网络下的性能，代表不了真实用户。
  • 只能看你主动测的时候的性能，用户那边卡成翔了你不知道。
  • 没有历史数据，不知道性能是变好还是变坏了。
  • 没有维度下钻（不同地区、设备、网络下的性能差异）。

真实用户监控（RUM - Real User Monitoring）的价值：

  1. 了解真实用户的体验：不是"我这里不卡"，而是"95% 的用户首屏在 2 秒以内"。
  2. 发现性能退化：某次发版后 LCP 涨了 500ms，告警出来，及时回滚。
  3. 定位长尾问题：P95/P99 的用户为什么慢？是网络差？还是设备烂？还是某个地区 CDN 有问题？
  4. 指导优化优先级：哪个页面最慢？哪个资源最大？先优化什么收益最高？
  5. 业务影响量化：性能和转化率的关系——首屏慢 1 秒，转化率降多少？
  6. SLA/SLO 度量：对外承诺的性能指标，用数据说话。

一句话：DevTools 是"实验室数据"，监控是"真实世界数据"。优化不能靠感觉，要靠数据。

二、需要采集哪些指标 ​

1. Web Vitals（核心 Web 指标） ​

Google 提出的衡量用户体验的核心指标，是目前业界的标准。

指标全称衡量什么良好阈值
LCPLargest Contentful Paint最大内容绘制——首屏最大内容元素出现的时间< 2.5s
FIDFirst Input Delay首次输入延迟——用户第一次交互到浏览器响应的时间< 100ms
CLSCumulative Layout Shift累计布局偏移——页面元素意外移动的总和< 0.1
INPInteraction to Next Paint交互到下一帧绘制的时间（替代 FID 的新指标）< 200ms

为什么是这几个：

  • LCP：加载性能（用户多快看到内容）
  • FID/INP：交互性能（用户点了有没有反应）
  • CLS：视觉稳定性（页面会不会乱跳）

分别对应加载、交互、视觉稳定三个维度，比较全面。

怎么采集：

Web Vitals 可以用 Google 官方的 web-vitals 库：

javascript
import { getLCP, getFID, getCLS, getINP } from 'web-vitals';

getLCP(metric => {
  console.log('LCP:', metric.value);
  // 上报
  report('lcp', metric.value);
});

getFID(metric => report('fid', metric.value));
getCLS(metric => report('cls', metric.value));
getINP(metric => report('inp', metric.value));

2. Navigation Timing（导航时间） ​

performance.timing 或 PerformanceNavigationTiming 提供了页面加载各个阶段的精确时间。

javascript
const nav = performance.getEntriesByType('navigation')[0];

// 关键时间点
const timing = {
  // DNS 查询
  dnsLookup: nav.domainLookupEnd - nav.domainLookupStart,
  // TCP 连接
  tcpConnect: nav.connectEnd - nav.connectStart,
  // TLS 握手
  tlsHandshake: nav.connectEnd - nav.secureConnectionStart,
  // TTFB（首字节时间）
  ttfb: nav.responseStart - nav.requestStart,
  // 响应下载
  responseDownload: nav.responseEnd - nav.responseStart,
  // DOM 解析
  domParse: nav.domInteractive - nav.responseEnd,
  // DOMContentLoaded
  dcl: nav.domContentLoadedEventEnd - nav.fetchStart,
  // 页面完全加载
  load: nav.loadEventEnd - nav.fetchStart,
  // 白屏时间（FP）
  firstPaint: performance.getEntriesByName('first-paint')[0]?.startTime,
  // 首次内容绘制（FCP）
  fcp: performance.getEntriesByName('first-contentful-paint')[0]?.startTime,
};

3. Resource Timing（资源加载时间） ​

每个静态资源（JS、CSS、图片、字体）的加载时间。

javascript
const resources = performance.getEntriesByType('resource');
resources.forEach(resource => {
  console.log(resource.name, resource.duration, resource.transferSize);
  // 上报大资源、慢资源
  if (resource.duration > 3000 || resource.transferSize > 500 * 1024) {
    report('slow-resource', {
      name: resource.name,
      duration: resource.duration,
      size: resource.transferSize,
    });
  }
});

三、数据上报策略 ​

上报策略很重要——上报太频繁会影响用户性能（甚至把自己打挂），上报太少数据又不准。

1. 采样率（Sampling） ​

不是每个用户的每条数据都要上报。对于大流量网站，全量上报成本太高（存储、带宽、计算），而且也没必要——采样 1% 已经足够反映整体趋势。

javascript
const SAMPLE_RATE = 0.01; // 1% 采样

function shouldSample() {
  return Math.random() < SAMPLE_RATE;
}

function report(type, data) {
  if (!shouldSample()) return;
  // 真正上报
  send(type, data);
}

采样策略：

  • 基础指标（Web Vitals、Navigation Timing）：1%~10% 采样，足够看趋势。
  • 错误/异常：100% 上报（错误本来就少，而且很重要）。
  • 自定义业务指标：根据业务重要性和数据量决定。
  • 长尾数据：可以只上报 P95/P99 的数据，或者慢于阈值的才上报。

注意：采样要均匀随机，不能有偏差（比如只在每天的某个时间段采样），否则数据不准。

2. 批量上报（Batching） ​

不要每条数据发一个请求，攒一批再发，减少请求数。

javascript
const buffer = [];
const BATCH_SIZE = 20;
const BATCH_INTERVAL = 5000; // 5 秒

function report(type, data) {
  buffer.push({ type, data, time: Date.now() });
  
  if (buffer.length >= BATCH_SIZE) {
    flush();
  }
}

let timer = null;
function startTimer() {
  timer = setInterval(() => {
    if (buffer.length > 0) {
      flush();
    }
  }, BATCH_INTERVAL);
}

function flush() {
  if (buffer.length === 0) return;
  
  const batch = buffer.splice(0, buffer.length);
  sendBatch(batch);
}

3. sendBeacon（信标上报） ​

用户离开页面时（关闭标签页、跳转），普通的 XMLHttpRequest 或 fetch 可能发不出去——页面都关了，请求被取消了。

navigator.sendBeacon() 就是为了解决这个问题的：

javascript
window.addEventListener('beforeunload', () => {
  if (buffer.length > 0) {
    const data = new Blob(
      [JSON.stringify(buffer)],
      { type: 'application/json' }
    );
    navigator.sendBeacon('/api/report', data);
  }
});

// visibilitychange 更可靠
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    flush(); // 页面隐藏时也上报
  }
});

sendBeacon 的特点：

  • 异步执行，不阻塞页面跳转。
  • 浏览器保证"尽量发送"，就算页面关了也会在后台发。
  • 请求是 POST，体积有限制（通常 64KB 左右）。
  • 不能设置自定义 header（可以用 Blob + type 模拟 content-type）。

为什么用 visibilitychange 不用 beforeunload/unload：

  • 移动端很多情况下 beforeunload 不会触发（比如直接杀进程）。
  • visibilitychange 在页面切后台时就触发，更可靠。
  • 现代浏览器推荐用 visibilitychange + sendBeacon 的组合。

4. 上报方式对比 ​

方式优点缺点适用场景
new Image()兼容性好，跨域简单只能 GET，数据量有限，不能确认是否成功简单统计，老浏览器
XMLHttpRequest灵活，能知道结果页面关闭时可能发不出去页面还在时的上报
fetch现代 API，Promise 风格同上，keepalive 有限制同上
sendBeacon页面关闭也能发，不阻塞只能 POST，数据量有限，无回调页面离开时的上报

最佳实践：

  • 页面还在时，用 fetch / XHR 批量上报。
  • 页面离开时，用 sendBeacon 兜底上报剩余数据。
  • 数据量大的话，优先批量 + 采样，减少请求数。

四、数据存储与可视化 ​

数据存储 ​

上报的数据存哪里？取决于数据量和查询方式：

方案优点缺点适用
Elasticsearch全文检索，聚合分析强存储成本高，运维复杂中型以上团队
ClickHouse列式存储，海量数据查询快运维有一定门槛大数据量，分析多
Prometheus + Grafana时序数据，监控告警一体存储原始明细不太适合指标聚合，趋势监控
云服务（ARMS、Datadog）开箱即用，不用自己搭按用量付费，数据在第三方小团队，不想运维
MySQL / PG简单，熟悉数据量大了查询慢小流量，简单场景

一般推荐 ClickHouse 或 Elasticsearch 存明细数据，Prometheus 存聚合后的指标数据。

数据可视化 ​

有了数据，要怎么展示？常见的看板：

1. 大盘概览

  • 核心指标的趋势图（LCP/FID/CLS 的 P50/P75/P95/P99）
  • 按天/周/月的变化趋势
  • 好/中/差的用户比例（绿色/黄色/红色占比）

2. 页面维度

  • 哪个页面最慢？排序一下，优先优化 Top N。
  • 每个页面的各项指标对比。

3. 下钻分析

  • 按地区：哪个省/国家的用户体验最差？
  • 按设备：iOS vs Android？高端机 vs 低端机？
  • 按网络：Wi-Fi vs 4G vs 3G？
  • 按浏览器：Chrome vs Safari？
  • 按版本：新版本性能有没有提升？

4. 资源分析

  • 最慢的 Top 20 资源
  • 最大的 Top 20 资源
  • 4xx/5xx 的资源

5. 告警配置

  • LCP P75 超过 3s → 告警
  • 错误率超过 1% → 告警
  • 某个地区性能突降 50% → 告警

五、告警设计 ​

监控没有告警等于没有监控。但告警也不能乱发——告警太多等于没有告警（狼来了效应）。

告警设计原则：

  1. 只告警真正重要的：不要每个指标波动都告警，只告警"用户体验明显受损"的程度。
  2. 用百分位不用平均值：P95 涨了 50% 比平均值涨了 20% 更值得关注。
  3. 设置合理的阈值：基于历史数据设基线，比基线高 X% 就告警。
  4. 持续时间：连续 N 分钟超过阈值才告警，防止毛刺。
  5. 分级告警：P1（严重）→ 打电话；P2（警告）→ 飞书/钉钉；P3（提示）→ 邮件/看板。
  6. 告警聚合：同一个问题导致的多条告警，合并成一条。
  7. 自动恢复通知：问题恢复了也要通知，让人安心。

常见告警规则示例：

  • LCP P95 > 4s，持续 5 分钟 → P2 告警
  • JS 错误率 > 1%，持续 3 分钟 → P2 告警
  • 页面白屏率 > 0.5%，持续 10 分钟 → P1 告警
  • 接口错误率 > 5%，持续 2 分钟 → P1 告警

六、开源方案对比：Sentry 等 ​

方案类型定位优点缺点
Sentry错误 + 性能监控全栈 APM生态好，功能全，支持 SourceMap性能监控是后来加的，深度一般；自建成本高

怎么选：

  • 小团队 / 创业公司 → Sentry 免费版 + 基础监控
  • 中型团队，预算充足 → 商业 SaaS（Datadog / New Relic / ARMS）
  • 大型团队，数据敏感，有资源 → 自建，或者商业版 + 补充自建
  • 只看 Web Vitals → Google Search Console + GA 免费的就够了

什么时候值得自建：

  • 数据量大，商业版太贵。
  • 数据敏感，不能出公司。
  • 有特殊的定制化需求，商业版满足不了。
  • 有专门的团队来做这件事。

追问 1：Web Vitals 的 LCP、FID、CLS 三个指标具体是怎么计算的？各自有哪些影响因素？如何优化？ ​

答 ​

理解指标的计算原理和影响因素，才能有针对性地优化。

1. LCP（Largest Contentful Paint）

衡量什么：视口内最大的内容元素渲染完成的时间。反映用户多快能看到页面的主要内容。

哪些元素算"内容元素"：

  • <img> 元素
  • <picture> 里的 <img>
  • <video> 的封面图（poster）
  • 通过 url() 加载的背景图
  • 包含文本的块级元素（<p>、<h1> 等）

怎么算"最大"：

  • 图片：按显示面积算（宽 × 高）。
  • 文本：按文本块的面积算。
  • 注意：是按显示在视口内的面积，不是原始大小。被裁剪的部分不算。

计算时机：

  • 页面加载过程中，不断更新"当前最大元素"。
  • 用户交互（点击、滚动、按键）后停止计算（因为用户交互后内容变化可能是用户主动触发的）。
  • 最终上报的是"交互前最大的那个元素的渲染时间"。

影响 LCP 的因素（按优先级）：

  1. TTFB（首字节时间）慢 → 服务器响应慢，HTML 回来得晚，什么都白搭。
    • 优化：CDN、缓存、服务端性能优化、边缘计算。
  2. 渲染阻塞的 JS 和 CSS → JS 和 CSS 会阻塞渲染，推迟所有内容的绘制。
    • 优化：JS 加 defer/async，CSS 拆分（首屏关键 CSS 内联）。
  3. 资源加载慢 → 最大元素是图片的话，图片加载慢直接拉低 LCP。
    • 优化：图片格式（WebP/AVIF）、压缩、响应式图片（srcset）、预加载（preload）、CDN。
  4. 客户端渲染慢 → 首屏靠 JS 渲染内容，JS 下载 + 执行完了才有内容。
    • 优化：SSR/SSG、减少首屏 JS、代码分割、预渲染。

优化 LCP 的四大手段（官方推荐）：

  1. 优化关键路径，尽快让 HTML 到达。
  2. 优化 CSS 和 JS，减少阻塞。
  3. 优化图片资源（格式、压缩、预加载）。
  4. 优化客户端渲染（SSR、减少首屏 JS）。

2. FID（First Input Delay）

衡量什么：用户第一次和页面交互（点击、按键、触摸），到浏览器真正开始处理事件的时间差。反映页面的"响应能力"。

注意：

  • FID 只测"延迟"，不测事件处理本身的时间，也不测渲染更新的时间。
  • 只测第一次交互。
  • 为什么只测第一次？因为第一次交互的体验最关键，用户对"卡不卡"的第一印象就在这里。

为什么会有延迟：

  • 浏览器主线程在忙（在执行一大段 JS），用户点了，事件只能在队列里等着，等主线程空了才处理。
  • 所以 FID 高的本质原因是：主线程阻塞。

影响 FID 的因素：

  1. 长任务（Long Task） → JS 执行时间超过 50ms 的任务。
    • 50ms 是怎么来的？人眼能感知到 100ms 以上的延迟。浏览器一帧 16.6ms（60fps），留 3 帧（~50ms）给浏览器处理输入，剩下的时间用来执行 JS。
    • 优化：拆分长任务，用 requestIdleCallback 做非紧急任务，Web Worker 跑大计算。
  2. 第三方脚本 → 广告、统计、A/B 测试等第三方脚本，经常执行很久。
    • 优化：延迟加载、异步加载、用 Web Worker。
  3. JS 体积太大 → 下载 + 解析 + 编译 + 执行都占时间。
    • 优化：代码分割、Tree Shaking、移除无用代码。

INP（Interaction to Next Paint）——FID 的替代者：

FID 有局限：只测第一次交互，只测延迟，不测后续。 INP 是 Google 新推出的指标，2024 年 3 月正式替代 FID 成为核心指标。

INP 的特点：

  • 测所有交互（点击、按键、触摸），不是只测第一次。
  • 测的是"从交互到下一帧绘制完成的总时间"（包含事件处理 + 渲染更新）。
  • 最终值取 P98（所有交互中最慢的 2% 的那个值），而不是第一次。
  • 更全面地反映页面的交互性能。

INP 的良好阈值是 < 200ms。

3. CLS（Cumulative Layout Shift）

衡量什么：整个页面生命周期内，所有意外的布局偏移的总和。反映页面的"视觉稳定性"。

什么是"布局偏移"：

  • 一个已经渲染的元素，突然改变了位置（比如上面插入了一张图，把下面的内容挤下去了）。
  • 用户正要点一个按钮，按钮突然跑了，点到了别的地方——这就是 CLS 高导致的糟糕体验。

怎么计算一次布局偏移的分数：

布局偏移分数 = 影响分数 × 距离分数

影响分数：受影响的元素占视口面积的比例
距离分数：元素移动的距离占视口尺寸的比例（取宽和高中较大的那个）
1
2
3
4

举个例子：

  • 视口 1000 × 800 = 800,000 px²
  • 一个 1000 × 200 的元素（占视口 25%）向下移动了 100px（视口高度的 12.5%）
  • 影响分数 = 0.25
  • 距离分数 = 0.125
  • 布局偏移分数 = 0.25 × 0.125 = 0.03125

累积（Cumulative）：

  • 把所有布局偏移的分数加起来。
  • 但不是简单地把所有时间的都加起来，而是分"会话窗口"（session window）——连续的布局偏移之间间隔不超过 1 秒，且总时长不超过 5 秒，算一个窗口。取所有窗口中的最大值。
  • 简单理解：页面加载时的一堆偏移算一波，滚动时的另一堆偏移算另一波，取最大的那波。

什么不算"意外偏移"：

  • 用户主动触发的（比如点击展开一个菜单，下面的内容被挤下去——这是用户预期的）。
  • 在已有空间内的变化（比如一个元素大小变了，但不影响其他元素的位置）。
  • CSS transform 变化（不影响布局，只影响视觉位置）。

影响 CLS 的常见原因：

  1. 图片没有预尺寸 → 图片加载前占 0 高度，加载完撑起来，下面的内容被挤下去。
    • 优化：给 <img> 加 width 和 height 属性，或者用 aspect-ratio 预留位置。
  2. 广告、嵌入内容 → 广告异步加载，不知道尺寸，加载完推挤内容。
    • 优化：给广告位预留固定大小的容器。
  3. 动态插入的内容 → 比如"相关推荐"模块，数据回来后插入到页面中间，把下面的挤下去。
    • 优化：预留占位空间，或者放在页面底部（不会影响已经看过的内容）。
  4. Web 字体 → 字体加载前后，文字宽度变化，导致换行变化，布局偏移。
    • 优化：font-display: optional/swap，或者预加载字体，或者用 size-adjust 调整。
  5. 动画 → 某些动画改变了元素的布局属性（width/height/margin/padding）。
    • 优化：用 transform 和 opacity 做动画（不影响布局，只影响合成）。

CLS 优化口诀：图片留尺寸，广告留位置，字体提前备，动态插底部。

三个指标的关系：

  • LCP：加载快不快（看到内容要多久）
  • FID/INP：交互顺不顺（点了有没有反应）
  • CLS：视觉稳不稳（页面会不会乱跳）

分别对应性能体验的三个维度，一个都不能少。优化时要全面看，不能只盯着一个指标。

追问 2：前端性能监控的数据量很大，怎么设计存储和查询方案才能既省钱又高效？ClickHouse 和 Elasticsearch 在这个场景下各有什么优劣？ ​

答 ​

这是一个非常实际的工程问题——监控系统最头疼的就是存储成本和查询性能的平衡。

数据量估算

先算一下数据量有多大，心里有个数：

假设：

  • 日活 100 万用户
  • 采样率 5% → 5 万用户上报
  • 每个用户平均访问 10 个页面
  • 每次页面浏览上报 20 条指标（Web Vitals + Navigation Timing + 资源等）
  • 每条数据 1KB

那么：

  • 每天上报条数 = 5万 × 10 × 20 = 1000 万条/天
  • 每天数据量 = 1000万 × 1KB = 10 GB/天
  • 每月 = 300 GB/月
  • 保留 3 个月 = 近 1TB

这还只是性能指标，如果加上错误日志、用户行为，数据量还要翻几倍。

所以存储和查询方案的选择非常关键。

常见存储方案对比

方案存储成本查询速度运维复杂度适合场景
Elasticsearch高（索引占空间，存 3 副本）快高全文检索、明细查询、维度多
ClickHouse低（列存压缩比高）极快中海量数据、聚合分析、时序数据
Prometheus中（只存指标，不存明细）快低时序指标、告警、趋势图
MySQL / PG中数据量大了慢低小数据量、简单查询
数仓（Hive / BigQuery）中低慢（分钟级）高离线分析、T+1 报表

不要所有数据都存同一种存储，按使用频率和保留时间分层：

ClickHouse vs Elasticsearch 详细对比

这两个是最常用的明细存储，详细对比一下：

1. 存储模型

  • Elasticsearch：倒排索引 + 行存（实际是列式分片，但索引很占空间）。

    • 每个字段都有倒排索引，查询快但索引占空间大。
    • 原始数据存在 _source 里，也是一份。
    • 通常配 1 个主分片 + 2 个副本（3 副本），存储空间 ×3。
    • 压缩比一般：1TB 原始数据存进去可能要 3~5TB。

  • ClickHouse：列式存储 + LSM Tree。

    • 每列单独存储，相同类型的数据在一起，压缩比极高（通常 5~10 倍压缩比）。
    • 没有倒排索引（但有稀疏主键索引、跳数索引等），靠列存 + 并行扫描取胜。
    • 副本可选，不需要三副本也能行（取决于数据重要性）。
    • 压缩比优秀：1TB 原始数据可能只占 100~200GB。

存储成本结论：ClickHouse 只有 ES 的 1/5 ~ 1/10。监控数据量大，成本差异很可观。

2. 查询性能

  • Elasticsearch：

    • 全文检索、模糊查询：极强（倒排索引的强项）。
    • 简单聚合：快。
    • 复杂聚合（海量数据、多维度下钻）：中等，数据量大了会慢。
    • 明细查询：快。

  • ClickHouse：

    • 全文检索：弱（虽然有全文索引，但不如 ES 强）。
    • 简单聚合：极快。
    • 复杂聚合、多维度下钻：极强，专门为 OLAP 设计，速度比 ES 快 5~10 倍。
    • 明细查询：也快，但不如按列查那么夸张。

查询性能结论：聚合分析 ClickHouse 强，全文检索 ES 强。

3. 运维复杂度

  • Elasticsearch：

    • 集群运维复杂：分片、副本、扩容、reindex、segment merge……坑很多。
    • 需要专门的 ES 运维经验。
    • JVM 调优、磁盘水位、分片策略……事情不少。

  • ClickHouse：

    • 相对简单，集群配置比 ES 少。
    • 但也有自己的坑：表引擎选择、分区策略、物化视图、数据去重……
    • 中文资料相对少一些（但也足够用了）。

运维复杂度结论：两者都有门槛，ES 可能更复杂一点。

4. 生态与工具

  • Elasticsearch：生态极其丰富，Kibana 可视化很成熟，Logstash/Beats 等采集工具齐全。
  • ClickHouse：生态也在快速发展，Grafana 有官方插件，可视化可以用 Grafana / Metabase / Superset 等。

怎么选

场景推荐原因
主要做聚合分析、性能指标统计ClickHouse快 + 省
需要全文检索、错误日志搜索Elasticsearch倒排索引强
预算有限、数据量大ClickHouse存储成本低很多
已经有 ES 基础设施、团队熟悉Elasticsearch复用现有能力
错误监控 + 性能监控都要做两者混用：错误存 ES，性能存 ClickHouse各取所长

很多公司的实际架构是两者混用：

  • 性能指标数据 → ClickHouse（量大，主要做聚合分析）
  • 错误日志、异常堆栈 → Elasticsearch（量少一些，需要全文检索）
  • 聚合后的指标 → Prometheus（用于告警和大盘）
  • 可视化 → Grafana 统一展示

其他省钱技巧

  1. 采样上报：最有效的省钱手段。性能指标采样 1% 已经够了，数据量直接减 99%。
  2. 只存必要的字段：不要什么都往上报，只存分析需要的字段。
  3. 降精度存储：时间戳存秒级不用毫秒级，数值存整数不用浮点（比如存毫秒整数，不用秒的小数）。
  4. 冷热分离：前面讲的，老数据移到便宜存储。
  5. 预聚合 + 物化视图：ClickHouse 的物化视图，可以预计算好常用的聚合结果，查询直接查预聚合表，又快又省。
  6. 分区表：按天/周分区，老数据直接 drop 分区，比 delete 快多了。
  7. 字符串转枚举：很多维度（浏览器、操作系统、地区）取值有限，存枚举（数字）比存字符串省空间多了。

总结：前端性能监控数据量大、查询以聚合分析为主，ClickHouse 通常是更优选择——存储便宜、查询快。如果需要全文检索，可以和 ES 混用。再加上冷热分层、采样、预聚合等优化，成本可以控制得很好。

关键模块详解

1. SDK（数据采集层）

最前端的部分，嵌在用户页面里，负责采集和上报。

核心功能：

  • Web Vitals 采集（基于 web-vitals 库）
  • Navigation Timing / Resource Timing 采集
  • JS 错误采集（window.onerror、unhandledrejection）
  • 自定义事件上报（业务埋点）
  • 上下文信息收集（URL、UA、网络、设备等）
  • 采样控制
  • 批量上报 + 缓存
  • sendBeacon 兜底上报
  • 去重 / 节流（防止刷爆）

开发要点：

  • 体积要小：SDK 自己不能太大，否则影响性能（建议控制在 10KB 以内 gzip）。
  • 不能影响主业务：SDK 报错不能影响页面，所有代码 try/catch 包裹。
  • 兼容性：老浏览器要做降级（没有 PerformanceAPI 就不上报）。
  • 可配置：采样率、上报地址、开关等可以动态配置。

2. 接入层（Gateway）

接收上报数据的入口。

核心功能：

  • CORS 处理（前端跨域上报）
  • 限流 / 熔断（防止被刷爆）
  • 鉴权（如果需要）
  • 数据落盘 / 转发（收到数据后扔到消息队列，不要同步处理）
  • 日志记录

技术选型：Nginx / OpenResty / API Gateway 都行。

为什么要有接入层，不直接写数据库：

  • 解耦：上报请求和数据处理解耦，数据库挂了不影响上报。
  • 削峰：突发流量时，消息队列扛住，后端慢慢消费。
  • 多目的地：一份数据可以同时发往多个存储/处理系统。

3. 消息队列（缓冲层）

削峰填谷，解耦。

  • Kafka 最常用，吞吐量高，生态成熟。
  • 数据量小的话，Redis 也可以。

4. 数据处理层

消费队列里的数据，做清洗、转换、分流、聚合。

核心工作：

  • 数据校验：格式不对、字段缺失的丢掉。
  • 数据清洗：UA 解析（解析出浏览器、操作系统、设备）、IP 解析（解析出地理位置）、URL 归一化（去掉 query 里的敏感参数）。
  • 维度扩充：补充应用信息、版本信息、环境信息。
  • 分流：性能数据 → ClickHouse；错误数据 → ES；聚合指标 → Prometheus。
  • 实时聚合：用 Flink / 自研流式计算，实时计算分钟级 / 5 分钟级的聚合指标。
  • 告警检测：实时检测异常，触发告警。

技术选型：

  • 简单场景：自己写消费程序（Node.js / Go / Python）。
  • 复杂场景：Flink / Spark Streaming。

5. 存储层

前面追问 2 详细讲过，推荐：

  • 性能明细 → ClickHouse
  • 错误日志 → Elasticsearch
  • 聚合指标 → Prometheus

6. 可视化层

把数据展示出来。

  • Grafana：做通用大盘、趋势图，最快能用。
  • 自研大盘：业务定制化的、需要特殊交互的，自己写前端。
  • 告警通知：Alertmanager 或自研，对接飞书/钉钉/短信/邮件。

7. 后台管理

  • 项目管理（接入新的业务线）
  • 配置管理（采样率、告警规则等）
  • 权限管理
  • SourceMap 上传管理

开发优先级（MVP → 迭代）

不要想一步到位，从最小可用版本开始，逐步迭代。

第一阶段（MVP，1~2 周）——能用就行

目标：先有数据，能看趋势。

  1. ✅ SDK 核心功能：Web Vitals + Navigation Timing 采集 + 简单上报。
  2. ✅ 简单的接收接口：一个 HTTP 接口，接了就写数据库。
  3. ✅ 基础存储：先存 MySQL 或 Elasticsearch（哪个熟用哪个）。
  4. ✅ 简单的可视化：Grafana 连数据库，搞几个核心指标的趋势图。

先跑起来，验证方案可行，数据是有用的。

第二阶段（完善，2~4 周）——好用

目标：功能完善，能支撑日常分析。

  1. ✅ SDK 增强：错误采集、Resource Timing、自定义事件、采样、批量、sendBeacon。
  2. ✅ 接入层 + 消息队列：加 Kafka，解耦 + 削峰。
  3. ✅ 数据处理管道：清洗、UA 解析、IP 解析、分流。
  4. ✅ 存储升级：性能数据迁到 ClickHouse，错误存 ES。
  5. ✅ 告警系统：基础的阈值告警，飞书/钉钉通知。
  6. ✅ 多维分析：按页面、地区、设备等维度下钻。

第三阶段（高级，持续迭代）——精通

目标：智能化，驱动业务。

  1. 🔄 SourceMap 反解：错误堆栈自动还原到源码。
  2. 🔄 用户行为回溯：错误发生前的用户操作路径，类似 Sentry 的 breadcrumb。
  3. 🔄 智能告警：基于历史数据的异常检测（不是固定阈值，而是同比环比自动判断）。
  4. 🔄 性能预算：设置每个页面的性能预算，超过了 CI 阶段就卡。
  5. 🔄 业务关联分析：性能和转化率、留存率的关系分析。
  6. 🔄 慢页面 / 慢资源自动定位：自动找出 Top N 慢的页面和资源，给出优化建议。
  7. 🔄 白屏检测：检测页面白屏率。
  8. 🔄 录屏回放：出问题时，能看用户当时的操作录屏（类似 FullStory）。

常见坑与经验

  1. 不要一开始就追求完美：先有再优，先跑起来再优化。很多团队死在"设计了一个完美的架构，还没做完就放弃了"。
  2. SDK 体积和性能是生命线：SDK 不能大，不能慢，不能影响业务。自己引入性能问题的监控，是最讽刺的。
  3. 数据质量很重要：上报的数据要可靠、可解析、维度一致，不然分析出来的结论是错的，还不如没有。
  4. 告警不能乱发：告警太多大家就麻木了。宁可少告警，也不要发没用的告警。
  5. 采样策略要灵活：默认低采样，但可以针对特定页面、特定用户开高采样。出问题时可以临时上调采样率。
  6. 数据安全和隐私：不要上报敏感信息（用户输入的内容、密码、token 等），URL 里的敏感参数要脱敏。用户 ID 要匿名化。

总结：从 0 到 1 做前端性能监控，核心是**"先有数据，再做分析，最后智能"**。SDK 是基础，存储是关键，可视化是价值出口。不要追求一步到位，小步快跑，持续迭代。
