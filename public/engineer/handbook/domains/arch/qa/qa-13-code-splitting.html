问答 13（进阶）：Code Splitting 策略与实践 ​

题目 ​

请说明 Code Splitting（代码分割）的意义、三种常见的分割方式，并深入讲解 Webpack splitChunks 的核心配置项。再谈谈路由级分割 vs 组件级分割的选择，以及 preload / prefetch 的使用场景。

参考答案 ​

一、Code Splitting 解决什么问题 ​

没有代码分割时：

  • 整个应用打成一个 bundle，首屏要下载全部代码，哪怕用户只看首页。
  • 改一行代码，整个 bundle 的 content hash 变化，缓存全部失效，用户得重新下载。

代码分割的核心收益：

收益说明
首屏加载更快只下载首屏需要的代码，减少首屏 JS 体积，提升 FCP / LCP
缓存利用率更高不常变的代码（第三方库、公共组件）单独打包，长期缓存；业务代码变了，用户只下业务代码
按需加载用户走到哪个页面/功能，才加载对应代码，节省流量和时间
并行下载多个小 chunk 可以并行下载，比单个大文件更快（HTTP/2 下更明显）

二、三种代码分割方式 ​

  • 粒度较粗，是"应用级"的分割。

三、深入 splitChunks 配置 ​

Webpack 的 splitChunks 是代码分割的核心，配置项很多，理解每个参数的含义才能调优到位。

全局配置项 ​

javascript
splitChunks: {
  chunks: 'async',          // 哪些 chunk 参与分割：'initial' | 'async' | 'all'
  minSize: 20000,           // 提取出来的 chunk 最小体积（字节），小于这个不单独拆
  minRemainingSize: 0,      // 分割后剩余 chunk 的最小体积（v5 新增）
  minChunks: 1,             // 至少被几个 chunk 共享才提取
  maxAsyncRequests: 30,     // 按需加载时，最大并行请求数
  maxInitialRequests: 30,   // 入口页，最大并行请求数
  enforceSizeThreshold: 50000, // 超过这个体积，强制分割（忽略上面的限制）
  cacheGroups: { /* ... */ } // 缓存组，核心中的核心
}

关键参数详解：

1. chunks

  • 'async'（默认）：只对异步 chunk（动态 import 出来的）做分割。
  • 'initial'：只对初始 chunk（入口 chunk）做分割。
  • 'all'：两者都做，最常用。

为什么默认是 async：因为异步 chunk 的分割最安全，不会影响首屏；而初始 chunk 分割不当可能导致首屏请求数过多、反而变慢。但实际项目里一般都改成 'all'，配合 cacheGroups 精细化控制。

2. minSize

  • 默认 20KB（v5）。
  • 作用：防止拆出太小的 chunk，得不偿失（一个 HTTP 请求的开销可能比多下载几 KB 还大）。
  • 经验值：20KB~50KB 比较合理，太小的模块不如不拆。

3. minChunks

  • 默认 1。
  • 作用：一个模块被多少个 chunk 共享了，才值得提取出来。
  • 注意：默认值 1 看起来奇怪（一个模块只在一个 chunk 里用，为什么要提取？），这是配合 cacheGroups 用的，比如 vendors 组的 minChunks 是 1，意思是"只要是 node_modules 里的，哪怕只用了一次也提出来"——因为第三方库不常变，单独打包利于缓存。

4. maxAsyncRequests / maxInitialRequests

  • 限制并行请求数，防止拆得太碎导致请求过多。
  • maxInitialRequests 是入口的首屏请求数，更重要，不能太大（一般 10~30 合理）。
  • 为什么要有这个限制？HTTP/1.1 下同域名并发请求数有限制（Chrome 是 6 个），请求太多会排队。HTTP/2 下可以放宽。

cacheGroups（缓存组） ​

cacheGroups 是 splitChunks 的核心，它定义了"按照什么规则把哪些模块分到哪些 chunk 里"。可以定义多个缓存组，每个组有自己的规则。

javascript
cacheGroups: {
  // 第三方库
  vendors: {
    test: /[\\/]node_modules[\\/]/,  // 匹配模块路径
    name: 'vendors',                 // 输出的 chunk 名
    priority: -10,                   // 优先级，数字越大越优先
    chunks: 'all',
  },
  // 公共组件
  common: {
    name: 'common',
    minChunks: 2,                    // 至少被 2 个 chunk 引用
    priority: -20,
    reuseExistingChunk: true,        // 如果一个模块已经被其他组提取了，直接复用
  },
  // 大体积第三方库单独拆
  echarts: {
    test: /[\\/]node_modules[\\/]echarts[\\/]/,
    name: 'echarts',
    priority: 0,                     // 优先级比 vendors 高，先匹配
    chunks: 'all',
  },
}

核心参数：

  • test：正则/函数，匹配模块路径。匹配上了就进这个组。
  • priority：优先级。一个模块可能同时匹配多个组（比如 echarts 既匹配 vendors 又匹配 echarts），谁 priority 高听谁的。
  • name：输出的 chunk 文件名。可以是字符串，也可以是函数。
  • reuseExistingChunk：如果一个模块已经被其他 chunk 包含了，直接复用那个 chunk，不重复打包。推荐开 true。
  • enforce：强制分割，忽略 minSize、minChunks、maxAsyncRequests、maxInitialRequests 的限制。

四、路由级分割 vs 组件级分割 ​

路由级分割（Route-level） ​

javascript
// React Router 示例
import { lazy, Suspense } from 'react';

const Home = lazy(() => import('./pages/Home'));
const About = lazy(() => import('./pages/About'));
const Dashboard = lazy(() => import('./pages/Dashboard'));

function App() {
  return (
    <Suspense fallback={<Loading />}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />

优点：

  • 天然的分割点，每个页面对应一个 chunk，逻辑清晰。
  • 实现简单，框架层都有支持（React.lazy、Vue 的异步组件）。
  • 首屏只加载当前路由的代码，收益明显。

缺点：

  • 粒度较粗，如果一个页面很大（比如后台管理页面，左边菜单 + 右边内容），首屏还是要加载整个页面的代码。
  • 页面内的大组件（富文本、图表）还是会跟着页面一起加载。

组件级分割（Component-level） ​

  • 粒度更细，可以精确到"用户用到了才加载"。

缺点：

  • 过度分割会导致请求数过多，体验反而下降。
  • 每个懒加载组件都要处理 loading 状态，代码更繁琐。
  • 用户交互时才加载，可能有"点击后等半天"的卡顿感。

怎么选 ​

最佳实践：

  1. 默认路由级分割：这是基准线，所有路由都做懒加载，投入产出比最高。
  2. 大组件做组件级分割：体积超过 100KB 的组件（富文本编辑器、图表库、地图 SDK、视频播放器等）做懒加载。
  3. 条件性功能做组件级分割：只有 VIP 用户能用的功能、低频使用的高级功能。
  4. 避免过度分割：不要每个小组件都懒加载，loading 闪烁 + 请求开销的代价可能超过收益。
  5. 配合 preload/prefetch：预判用户下一步行为，提前加载。

五、preload / prefetch ​

现代浏览器支持 <link rel="preload"> 和 <link rel="prefetch">，可以用来预加载资源。Webpack 通过 magic comment 支持。

preload（预加载） ​

javascript
import(/* webpackPreload: true */ './ChartComponent');
1

生成的 HTML 里会插入：

html
<link rel="preload" as="script" href="chart.chunk.js">
1

作用：告诉浏览器"这个资源马上就要用，提前下载好"。浏览器会在空闲时提前下载，但不执行，等真正 import 时直接从缓存里拿。

适用场景：

  • 高概率会用到的资源，比如当前页面下一步大概率会点击的功能。
  • 首屏渲染后立即需要的资源（字体、首屏图片、关键 JS）。

prefetch（预获取） ​

javascript
import(/* webpackPrefetch: true */ './AboutPage');
1

生成的 HTML 里会插入：

html
<link rel="prefetch" as="script" href="about.chunk.js">
1

作用：告诉浏览器"这个资源未来可能会用到，闲了就下载"。优先级很低，不会抢占当前页面的带宽。

适用场景：

  • 用户可能访问的下一个页面（比如首页的"关于我们"链接）。
  • 低频但重要的功能（比如设置页面、帮助中心）。

preload vs prefetch 对比 ​

特性preloadprefetch
优先级高（和当前页面资源同优先级）低（浏览器闲了才下载）
时机马上就要用未来可能用
缓存存在内存缓存，立即可用存在 HTTP 缓存，下次用的时候可能命中
滥用风险高（抢占当前页面带宽）低（闲了才下，不影响当前）
典型场景首屏关键资源、当前页面下一步操作下一个页面、低频功能

注意事项：

  • 不要滥用 preload，会影响当前页面的加载性能。
  • prefetch 可以放心用，因为是低优先级，浏览器自己控制时机。
  • 移动端要慎重，考虑用户流量。

六、代码分割的最佳实践 ​

  1. 路由懒加载是标配：所有非首屏路由都做懒加载。
  2. 拆出第三方库：vendors chunk 单独打，长期缓存。
  3. 大库单独拆：echarts、lodash、antd 等大体积库单独拆 chunk，避免和其他库混在一起。
  4. 公共组件提取：被多个页面共用的组件/工具函数提到 common chunk。
  5. 控制 chunk 数量：不要拆得太碎，maxInitialRequests 控制在合理范围。
  6. 合理设置 minSize：太小的 chunk 不如不拆，20~50KB 起步。
  7. 用 content hash 做缓存：output.filename: '[name].[contenthash].js'，内容不变文件名不变，充分利用缓存。
  8. 配合 prefetch：预判用户行为，提前加载下一页代码。
  9. 监控产物体积：用 Bundle Analyzer 定期分析，防止体积膨胀。

追问 1：代码分割是不是拆得越细越好？有什么副作用？怎么衡量"拆得合不合理"？ ​

答 ​

代码分割不是越细越好，过度分割会带来一系列副作用。这是一个典型的"权衡"问题。

过度分割的副作用：

1. 请求数过多，首屏反而变慢

  • 把 100KB 的代码拆成 20 个 5KB 的 chunk，首屏要发 20 个请求。
  • HTTP/1.1 下同域名并发数有限（Chrome 6 个），请求排队。
  • 就算 HTTP/2 多路复用，每个请求也有 TLS 握手、DNS 解析、HTTP 头部的开销。
  • 结果：20 个 5KB 的文件，总下载时间可能比 1 个 100KB 的文件还长。

2. 运行时开销

  • 每个 chunk 都有 webpack runtime 的包装代码，拆得越细，重复的包装代码越多。
  • 模块之间的依赖关系变复杂，运行时要做更多的模块解析。
  • 虽然单个开销不大，但积少成多。

3. 缓存失效的连锁反应

  • 拆得太细，稍微改一点代码可能影响多个 chunk 的依赖关系。
  • 比如 common chunk 里加了一个工具函数，所有依赖它的 chunk 的 content hash 可能都会变（因为它们的依赖列表变了）。
  • 当然 Webpack 5 的 deterministic module IDs 已经缓解了这个问题，但还是存在。

4. 用户体验问题

  • 组件级懒加载用多了，用户点击每个功能都要等加载，体验碎片化。
  • Loading 状态频繁出现，用户会觉得"这个应用怎么老是在加载"。
  • 网络慢时尤其明显。

5. 调试和排查问题变难

  • 几十个 chunk，线上报错了要先定位是哪个 chunk 的问题。
  • SourceMap 配置也更复杂。

怎么衡量"拆得合不合理"

可以从这几个维度评估：

维度 1：首屏请求数

  • 入口页的初始 JS 请求数（maxInitialRequests）建议控制在 10 个以内。
  • 超过 15 个就要警惕了，考虑合并一些小的 chunk。

维度 2：单个 chunk 的最小体积

  • 最小的 chunk 不要小于 20~50KB（gzip 前）。
  • 小于这个体积的，合并起来可能更划算。
  • 可以通过 minSize 配置控制。

维度 3：首屏 JS 体积

  • 首屏（入口 + 同步 chunk）的 JS 总大小（gzip 后）建议控制在 100~300KB。
  • 超过 500KB 就要重点优化了。
  • 这是最核心的指标，代码分割的最终目的就是减小首屏体积。

维度 4：缓存命中率

  • 看 CDN 日志或者性能监控，缓存命中率有多少。
  • 如果 vendors chunk 的缓存命中率很低（经常变），说明拆分策略有问题——可能把经常变的业务代码混进去了。

维度 5：用户体验指标

  • FCP（First Contentful Paint）：首屏内容绘制时间。
  • LCP（Largest Contentful Paint）：最大内容绘制时间。
  • TTI（Time to Interactive）：可交互时间。
  • 这些指标有没有因为代码分割而改善。

一个实用的判断方法

用 Webpack Bundle Analyzer 看产物体积分布：

  1. 有没有特别大的 chunk（比如超过 500KB）？ → 可以再拆
  2. 有没有特别多的小 chunk（比如几十个小于 10KB 的）？ → 可以合并
  3. vendors chunk 是不是包含了所有 node_modules？ → 大的库可以单独拆
  4. 业务代码和第三方库是不是混在一起？ → 分开更利于缓存

经验法则

  • 路由级分割：必做，收益最大，副作用最小。
  • vendors 提取：必做，缓存收益大。
  • 大组件懒加载：选择性做，只对体积大（>100KB）、使用频率低的组件做。
  • 公共代码提取：谨慎做，只有当确实被多个地方共用且体积不小的时候才提。不要为了"复用"而复用，有时候重复打包一点反而缓存更好。
  • 宁少勿多：拆少了还可以再拆，拆多了合并回去更麻烦。从粗到细，逐步优化。

  • 文件名可预测，知道哪个文件是什么。
  • 排查问题方便，看到 vendors.js 就知道是第三方库。
  • 可以做一些特殊处理（比如单独上传到某个 CDN、特殊的缓存策略）。
  • 适合固定的、边界清晰的 chunk（比如 runtime、vendors）。

缺点：

  • 如果这个 chunk 变得很大，没法继续自动拆分（因为名字固定了，再拆就冲突了）。
  • maxAsyncRequests / maxInitialRequests 的限制可能被绕过。
  • Webpack 官方说 name: true（自动生成）是推荐方式，因为能更好地利用缓存。

javascript
cacheGroups: {
  vendors: {
    test: /[\\/]node_modules[\\/]/,
    // 不配置 name，Webpack 自动生成
    chunks: 'all',
  },

优点：

  • Webpack 可以灵活地把模块分到多个 chunk，不会因为名字冲突而合并。
  • 缓存更精细：如果只有 react 变了，只有 react 那个 chunk 的 hash 变，lodash 的 chunk 不受影响。

  • 文件名很长，不好看。
  • 不可预测，想针对某个 chunk 做特殊处理比较麻烦。
  • 线上排查问题时，看到一堆长名字的文件有点晕。

方式 3：函数式命名（name(module, chunks, cacheGroupKey)）

javascript
cacheGroups: {
  vendors: {
    test: /[\\/]node_modules[\\/]/,
    name(module, chunks, cacheGroupKey) {
      // 获取包名
      const moduleFileName = module
        .identifier()
        .split('/')
        .reduceRight(item => item);
      const allChunksNames = chunks.map(item => item.name).join('~');
      return `${cacheGroupKey}-${allChunksNames}-${moduleFileName}`;
    },
    chunks: 'all',
  },
}

分情况处理：

1. 对于 vendors（第三方库）：推荐自动命名，但做"包级"拆分

    test: /[\\/]node_modules[\\/]/,

    chunks: 'all',
    priority: 0,

总结原则：

  • 稳定且独立的 chunk 固定命名：runtime、react、antd 这种边界清晰、更新频率可预期的。
  • 变化频繁的 chunk 自动命名：业务代码的异步 chunk，用 Webpack 自动命名或 magic comment 语义化命名。
  • 不要为了"好看"而强行固定命名：缓存命中率比文件名好看重要得多。
  • 配合 content hash：无论命名方式如何，文件名里都要加 [contenthash]，这是长期缓存的基础。

追问 3：代码分割后，首屏加载需要等待多个 JS 文件下载，如何优化首屏加载速度？有哪些手段可以让多个 chunk 并行加载、尽快执行？ ​

答 ​

代码分割后首屏需要加载多个 JS 文件（runtime + vendors + common + main），如果加载顺序不对或串行加载，会拖慢首屏。以下是优化手段：

手段 1：合理的 chunk 加载顺序

JS 执行有依赖关系：runtime → vendors → common → main。必须按顺序执行，但可以并行下载。

HTML 里的 <script> 标签默认是"遇到就下载并执行，阻塞后续解析"。但现代浏览器有预解析扫描器（preload scanner），会提前看后面的 script 标签，并行下载。

但如果 JS 是动态注入的（比如某些脚手架的做法），浏览器的预解析扫描器可能看不到，导致下载延迟。

优化方法：

  • 确保首屏的 JS 用 <script> 标签直接写在 HTML 里，不要通过 JS 动态注入。
  • 使用 defer 属性（不阻塞 HTML 解析，解析完后按顺序执行）。
html
<!-- 推荐：defer + 按依赖顺序排列 -->
<script defer src="runtime.abc.js"></script>
<script defer src="react.abc.js"></script>
<script defer src="antd.abc.js"></script>
<script defer src="common.abc.js"></script>
<script defer src="main.abc.js"></script>
1
2
3
4
5
6

defer 的特点：

  • 并行下载，不阻塞 HTML 解析。
  • 按标签出现顺序执行（保证依赖顺序）。
  • 在 DOMContentLoaded 前执行完毕。

手段 2：preload 关键资源

对首屏必须的 JS 文件用 <link rel="preload">，让浏览器提前发现并下载。

html
<link rel="preload" as="script" href="runtime.abc.js">
<link rel="preload" as="script" href="react.abc.js">
<link rel="preload" as="script" href="main.abc.js">
1
2
3

什么时候用 preload？

  • 如果 JS 是动态加载的（通过 JS 注入 script 标签），浏览器的预解析扫描器看不到，这时 preload 特别有用。
  • 如果 JS 放在页面底部，preload 可以让浏览器更早开始下载。
  • 但不要滥用，preload 的资源优先级很高，太多了会抢占其他资源的带宽。

手段 3：HTTP/2 多路复用

HTTP/1.1 下同域名并发请求数有限制（Chrome 6 个），多个 JS 文件要排队下载。HTTP/2 支持多路复用，同一个连接上可以并行传输多个请求，没有并发数限制。

优化方法：

  • 确保 CDN/服务器支持 HTTP/2。
  • 把静态资源都放在同一个域名下（HTTP/2 下多个域名反而不好，因为每个域名都要建连接）。
  • HTTP/1.1 时代的"域名分片"（把资源分散到多个域名提高并发）在 HTTP/2 下是反模式。

手段 4：内联关键代码（Inline Critical JS）

把很小但关键的代码（比如 webpack runtime、首屏骨架屏代码）直接内联到 HTML 里，减少一个 HTTP 请求。

javascript
// webpack.config.js
const HtmlWebpackPlugin = require('html-webpack-plugin');
const InlineChunkHtmlPlugin = require('inline-chunk-html-plugin');

module.exports = {
  plugins: [
    new HtmlWebpackPlugin(),
    new InlineChunkHtmlPlugin(HtmlWebpackPlugin, [/runtime/]),
  ],
};

  • runtime chunk（很小，几 KB），内联更划算。
  • 骨架屏的 CSS/JS。
  • 页面初始化必须的极小代码。

手段 5：优化 chunk 大小，减少请求数

不是拆得越细越好（参见追问 1）。合理的 chunk 大小和数量是首屏性能的基础。

  • minSize 设大一点（30~50KB），避免小 chunk。
  • maxInitialRequests 控制在 10 以内。
  • 把零散的小库合并到 vendors 里。

手段 6：CDN 缓存 + 强缓存

代码分割的一大目的就是提高缓存利用率，所以缓存策略必须跟上。

  • 带 [contenthash] 的静态资源设置强缓存：Cache-Control: max-age=31536000, immutable。
  • HTML 文件设置协商缓存：Cache-Control: no-cache。
  • 用 CDN 加速静态资源下载，缩短物理距离。

手段 7：代码压缩 + Tree Shaking

每个 chunk 本身的体积也要优化：

  • 生产模式开启压缩（Terser）。
  • Tree Shaking 去掉无用代码。
  • CSS 也做代码分割和压缩。
  • 图片和字体优化，不要混在 JS 里（用 asset module 或 file-loader 单独输出）。

手段 8：按需加载 + prefetch 配合

首屏只加载必须的代码，非首屏的用懒加载 + prefetch：

javascript
// 首页加载完后，闲的时候预加载其他页面
import(/* webpackPrefetch: true */ './pages/About');
import(/* webpackPrefetch: true */ './pages/Dashboard');
1
2
3

这样用户点击其他页面时，代码可能已经下载好了，秒开。

手段 9：模块联邦 / 独立部署

如果是微前端或大型应用，可以用 Module Federation 把公共库做成共享的：

  • 多个应用共享同一份 React、antd。
  • 用户从 A 应用跳到 B 应用，公共库已经在缓存里了。
  • 但架构复杂度高，小项目没必要。
