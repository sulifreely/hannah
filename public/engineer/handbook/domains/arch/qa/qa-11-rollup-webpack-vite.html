问答 11（进阶）：Rollup vs Webpack vs Vite 深度对比 ​

题目 ​

请从定位差异、构建原理、产物格式、Tree Shaking、生态与适用场景等维度，深度对比 Rollup、Webpack、Vite 三者，并解释为什么 Vite 在开发模式下更快。

参考答案 ​

一、定位差异 ​

工具核心定位诞生背景设计哲学
Rollup库打包器（Library Bundler）2015 年，为 ES Module 而生专注于把源码打包成可发布的库，产物干净、体积小
Webpack应用打包器（Application Bundler）2012 年，为复杂 SPA 而生一切皆模块，管尽所有资源（JS/CSS/图片/字体...），生态万能
Vite下一代前端构建工具2020 年，浏览器原生 ESM 普及后开发时不打包、按需编译；生产时用 Rollup 打包，追求极致开发体验

一句话区分：写库用 Rollup，写复杂应用用 Webpack，想开发快用 Vite。

二、构建原理差异 ​

Webpack：Bundle-based，一次构建全量打包 ​

entry → 模块依赖图 → chunk 分组 → bundle 产物
1
  • 从入口开始，递归解析所有依赖，形成完整的模块依赖图（Module Graph）。
  • 根据 splitChunks、入口、动态 import 把模块分到不同 chunk。
  • 每个 chunk 经过模板拼装（加上 webpack runtime），输出为 bundle 文件。
  • 开发模式也是打包，只不过不压缩、保留模块结构，但依然要先把整张依赖图跑一遍才能启动。

Rollup：ESM-native，专注打包 ​

entry → 解析 ESM 依赖 → tree-shake → 输出多种格式
1
  • 原生支持 ES Module，天然静态分析能力强。
  • 核心卖点：顶级 Tree Shaking（基于 ESM 静态结构，能精准消除未使用代码）。
  • 开发模式下 Rollup 也有 watch，但本质还是重新打包，只是增量更快。
  • 设计上更"纯粹"：默认只处理 JS，CSS/图片等需要插件介入。

Vite：No Bundle（开发）+ Rollup（生产） ​

开发模式：

浏览器请求 index.html → 逐个 import 模块 → Dev Server 按需编译返回
1
  • 启动一个 Koa/Connect HTTP 服务器，不做打包。
  • 浏览器请求哪个模块，服务器才编译哪个模块（按需编译）。
  • 依赖预构建：用 esbuild 把 node_modules 里的 CJS 转成 ESM 并合并，缓存在 node_modules/.vite/deps。
  • 源码转换：.ts / .jsx / .vue 由 esbuild 或对应编译器现场编译。

生产模式：直接用 Rollup 打包，Vite 只是一层配置封装和插件适配。

三、产物格式对比 ​

格式RollupWebpackVite（生产）
ESM (es)✅ 原生支持⚠️ output.library.type: 'module'（v5+）✅ 继承 Rollup
CJS (cjs)✅✅✅
UMD (umd)✅✅✅
IIFE (iife)✅⚠️ 需要配置✅
SystemJS (system)✅ 插件⚠️✅
多格式同时输出✅ 一行配置⚠️ 多份配置或插件✅ 继承 Rollup

为什么库打包偏爱 Rollup：

  • Rollup 输出的 ESM 产物没有额外 runtime 包裹，几乎就是你写的代码拼起来的，利于消费方再次 tree-shake。
  • Webpack 产物自带 webpack runtime（模块加载器、chunk 加载逻辑），体积较大，不适合被别人当作库引用。
javascript
// Rollup 产物（ESM，干净）
function add(a, b) { return a + b; }
export { add };

// Webpack 产物（自带 runtime，几十行起步）
(function(modules) {
  var installedModules = {};
  function __webpack_require__(moduleId) { /* ... */ }
  return __webpack_require__(__webpack_require__.s = 0);
})([
  function(module, __webpack_exports__, __webpack_require__) {
    // 你的代码...
  }
]);

四、Tree Shaking 效果 ​

维度RollupWebpack 5Vite（生产）
基础能力✅ 原生顶级✅ usedExports✅ 继承 Rollup
副作用识别✅ sideEffects✅ sideEffects✅
嵌套 export 消除✅ 更彻底✅ v5 改进✅
跨模块内联✅ Scope Hoisting✅ ModuleConcatenationPlugin✅
CSS Tree Shaking⚠️ 插件⚠️ 插件✅ 内置支持（部分）

为什么 Rollup 的 Tree Shaking 被认为更好：

  • Rollup 从设计之初就以 ESM 为一等公民，静态分析做得更深。
  • Rollup 的 Scope Hoisting（作用域提升）更自然，能把多个模块合并到一个作用域里，消除中间变量，进一步减少体积。
  • Webpack 因为历史包袱（CJS 兼容、复杂的 chunk 机制），某些边界场景下消除不够彻底。

五、生态对比 ​

维度RollupWebpackVite
插件数量中等（~1000+）海量（~10000+）快速增长（兼容 Rollup 插件）
Loader 体系❌ 插件一体化✅ Loader/Plugin 分离❌ 插件一体化（继承 Rollup）
框架支持需配置成熟（Vue CLI / CRA 都基于它）官方一等（Vue / React / Svelte 都有官方插件）
社区活跃度稳定成熟稳定爆发式增长
学习曲线平缓陡峭平缓

六、适用场景 ​

Rollup 适合：

  • 打包 JavaScript 库（React、Vue、Lodash 都用 Rollup 构建）
  • 组件库、工具函数库
  • 对产物体积和干净程度要求高的场景

Webpack 适合：

  • 复杂的单页应用（SPA）
  • 需要丰富的构建定制能力
  • 模块联邦（Module Federation）需求
  • 大型团队、成熟项目、历史包袱重的项目

Vite 适合：

  • 新项目、追求开发体验
  • 中后台应用、营销页、快速原型
  • Vue / React / Svelte 现代框架项目
  • 愿意接受少量开发/生产不一致风险的团队

七、为什么 Vite 开发模式快 ​

核心原因：不打包，按需编译

  1. 冷启动快：

    • Webpack 冷启动要先构建完整依赖图（模块数 × 编译时间）。
    • Vite 冷启动只要启动 Dev Server + 预构建依赖（依赖数量少，且用 esbuild），源码完全不处理。
    • 项目越大，这个差距越明显：1000 个模块的项目，Webpack 可能要 30s，Vite 可能 1s。

  2. 热更新快（HMR）：

    • Webpack HMR 需要重新编译受影响的 chunk（哪怕是增量，也得跑一遍 compilation pipeline）。
    • Vite HMR 只需让浏览器重新请求变化的那个模块，服务器端只转换这一个文件，通常 10~50ms。

  3. 依赖预构建用 esbuild：

    • esbuild 用 Go 编写，比 Node.js 写的 Babel/Webpack 快 10~100 倍。
    • 预构建只做两件事：CJS → ESM、合并小文件减少请求数。

  4. 浏览器缓存利用充分：

    • 预构建依赖打 hash，强缓存（Cache-Control: max-age=31536000），二次启动几乎不花时间。
    • 源码模块用协商缓存（304 Not Modified），没改动的模块不重新编译。

代价：

  • 开发环境和生产环境不一致（开发是 ESM 按需加载，生产是 Rollup 打包），少数 bug 只在生产复现。
  • 首次页面加载会发很多 HTTP 请求（一个模块一个请求），HTTP/2 下没问题，但弱网下体验略差。
  • 某些 CJS 依赖动态预构建会导致短暂卡顿或页面 reload。

追问 1：Vite 生产构建为什么用 Rollup 而不是 esbuild？ ​

答 ​

虽然 esbuild 速度极快，但 Vite 生产构建选择 Rollup 主要有以下原因：

1. 功能成熟度差异

  • Tree Shaking 精细度：esbuild 的 Tree Shaking 对副作用的识别不如 Rollup 精细。比如某些模块有顶层副作用（修改全局变量、注册副作用），Rollup 能通过 sideEffects 更精准地处理；esbuild 偏保守，可能保留一些本可消除的代码。
  • 代码分割（Code Splitting）：Rollup 的 chunk 分割策略更成熟，支持 manualChunks、动态 import 自动拆包、公共依赖提取等；esbuild 的代码分割能力相对基础。
  • CSS 处理：Rollup 生态里 CSS Modules、CSS-in-JS、PostCSS 插件非常成熟；esbuild 的 CSS 功能相对简单。

2. 生态与插件

  • Rollup 有庞大的插件生态，Vite 可以直接复用大量 Rollup 插件（@rollup/plugin-node-resolve、@rollup/plugin-commonjs、rollup-plugin-terser 等）。
  • 框架级插件（Vue SFC、React Fast Refresh、Svelte）在 Rollup 生态里已经验证过。
  • esbuild 插件生态还在发展中，功能不如 Rollup 丰富。

3. 产物质量

  • Rollup 产物更"干净"，冗余代码少，适合生产环境。
  • esbuild 追求速度，某些优化（比如作用域提升的深度）不如 Rollup。

4. 未来方向：Rolldown

  • Vite 团队正在开发 Rolldown（用 Rust 编写的 Rollup 兼容打包器），目标是兼顾 esbuild 的速度和 Rollup 的功能质量。
  • 未来 Vite 生产构建可能会从 Rollup 迁移到 Rolldown，实现开发和生产双快。

总结：不是 esbuild 不好，而是生产构建对产物质量、功能完整性、生态成熟度的要求更高，Rollup 在这些方面更有优势。开发构建追求速度，所以用 esbuild；生产构建追求质量，所以用 Rollup。这是一个典型的工程权衡。

追问 2：Webpack 就不能做到 Vite 那样的开发体验吗？它的瓶颈在哪？ ​

答 ​

Webpack 理论上也能优化开发体验，但它的架构设计决定了上限，核心瓶颈在以下几点：

1. 设计哲学：Bundle-based

  • Webpack 的核心是"打包器"，它的整个架构围绕"构建模块依赖图 → 生成 bundle"展开。
  • 开发模式下的 devtool: 'eval'、cache: true 都是在"打包"的基础上做加速，本质上还是要先打包再提供给浏览器。
  • 而 Vite 的思路是"不打包，让浏览器自己加载模块"，从根上绕过了打包开销。

2. 模块粒度与 HMR

  • Webpack 的 HMR 是基于 chunk 的：一个模块变了，要找到它所在的 chunk，重新生成这个 chunk 的热更新版本，再通过 WebSocket 推给浏览器。
  • 这个过程需要走完整的 compilation 流程（哪怕是增量）：解析依赖、loader 转换、模块图更新、chunk 生成。
  • Vite 的 HMR 是基于模块的：哪个模块变了，直接让浏览器重新请求这个模块，服务器只转换这一个文件。

3. Loader 链的开销

  • Webpack 的 Loader 链（从右到左、从下到上）虽然灵活，但每次构建都要跑一遍完整的 loader pipeline。
  • 即使有持久化缓存（Webpack 5 的 cache: { type: 'filesystem' }），首次构建、缓存失效时还是很慢。
  • Vite 用 esbuild 做转换，Go 语言的执行效率远超 Node.js 写的 Babel/TS Loader。

4. Webpack 有哪些优化手段？

  • 持久化缓存：cache: { type: 'filesystem' }，二次构建提速明显，但冷启动还是慢。
  • SWC / esbuild-loader：把 Babel 换成 SWC 或 esbuild-loader，JS 转换速度提升 10 倍以上。
  • thread-loader：多进程 loader 执行，利用多核 CPU。
  • exclude / include：缩小 loader 处理范围。
  • DLLPlugin（过时）：提前打包第三方依赖。

5. 为什么 Webpack 不做 ESM dev server？

  • 不是不能做，而是包袱太重。Webpack 有大量历史功能（CJS 兼容、AMD、UMD、复杂的 chunk 机制、Module Federation 等），如果要改成 ESM-first，几乎是推倒重来。
  • 而且 Webpack 的核心价值是"生态万能"，不是"开发最快"，用户选 Webpack 也是因为生态和功能，而不是开发速度。
  • Vite 是从零开始设计的，没有历史包袱，天然能选择最优解。

结论：Webpack 通过各种优化能接近 Vite 的开发速度，但受限于架构设计，很难做到同一量级。这也是 Vite 能迅速流行的根本原因——它选择了一条不同的路。

追问 3：如果让你为一个新项目选择构建工具，你的决策框架是什么？ ​

答 ​

我会从以下几个维度建立决策框架，而不是简单地说"新项目就用 Vite"：

维度 1：项目类型与规模

项目类型推荐原因
库 / 组件库Rollup产物干净、Tree Shaking 好、支持多格式输出
小型应用 / 原型 / 营销页Vite开发快、配置简单、启动快
中大型 SPA（100+ 页面）Vite 或 WebpackVite 开发体验好；Webpack 生态更稳、坑更少
微前端 / Module FederationWebpackMF 是 Webpack 5 原生功能，最成熟
SSR / Next.js / Nuxt框架自带Next.js 用 Turbopack/Webpack，Nuxt 用 Vite

维度 2：团队技术栈与熟悉度

  • 如果团队一直用 Webpack，且项目有复杂的 Webpack 配置积累，贸然切 Vite 有迁移成本。
  • 如果是新团队、新项目，Vite 学习曲线更平缓，上手更快。
  • 如果项目依赖大量冷门的 Webpack 插件，Vite 可能找不到替代品。

维度 3：对开发/生产一致性的要求

  • Vite 开发是 ESM 按需加载，生产是 Rollup 打包，存在环境差异。
  • 大多数场景下没问题，但有少数坑：比如某些依赖在开发时正常、生产打包后出问题（常见于 CJS 包的默认导出、循环依赖等）。
  • 如果项目对稳定性要求极高（比如金融级应用），Webpack 的"开发生产一致"可能更让人放心。

维度 4：特殊功能需求

  • 需要 Module Federation → Webpack（Vite 的 @originjs/vite-plugin-federation 也能用，但成熟度稍差）
  • 需要复杂的代码分割策略 → 两者都可以，Webpack 的 splitChunks 配置更灵活
  • 需要 Web Worker、Service Worker 等高级特性 → 两者都支持
  • 需要 SSR/SSG → Vite 配合 Vite SSR 或 Nuxt/SvelteKit 等框架更现代

维度 5：长期维护成本

  • Vite 是新工具，API 还在演进（虽然已经很稳定了），升级时可能有 breaking change。
  • Webpack 非常成熟，API 稳定，社区问题多，遇到问题容易搜到答案。
  • Vite 的配置更简单，新人上手快；Webpack 配置复杂，需要专人维护。

我的默认推荐：

  • 新项目、团队没有历史包袱 → Vite
  • 库/组件库 → Rollup
  • 需要 Module Federation 或有复杂 Webpack 插件依赖 → Webpack
  • 不确定，但想用现代工具 → Vite（Rollup 生产构建质量有保障，开发体验又好，风险其实很低）

一句话原则：默认 Vite，有特殊需求再选其他。Vite 已经是事实标准的下一代构建工具，生态成熟度足够覆盖绝大多数场景。
