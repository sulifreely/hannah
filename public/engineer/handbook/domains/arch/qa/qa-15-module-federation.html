问答 15（高级）：Module Federation 模块联邦原理 ​

题目 ​

请说明 Module Federation（模块联邦）解决了什么问题、核心概念（host / remote / shared）、运行时加载机制，以及 shared 依赖的版本管理与 singleton 机制。它和微前端框架是什么关系？适用场景有哪些？

参考答案 ​

一、Module Federation 解决什么问题 ​

Module Federation（简称 MF）是 Webpack 5 推出的一个革命性特性，它允许一个应用在运行时动态加载另一个应用的代码，并共享依赖。

在 MF 出现之前，跨应用共享代码有这些痛点：

  1. npm 包共享的痛点：

    • 公共组件抽成 npm 包，每次改了要发版、所有消费方升级，效率低。
    • 多个应用各自打包 npm 包，代码重复，体积大。
    • 发版链路长，紧急修复要等所有应用重新发版。

  2. 微前端的痛点：

    • qiankun 等框架是"应用级"的，粒度太粗，想共享一个组件都很麻烦。
    • 每个子应用各自打自己的 React、antd，重复加载，浪费性能。
    • 技术栈不同的应用无法共享组件。

  3. 多团队协作的痛点：

    • A 团队做的组件，B 团队想用，要么走 npm（慢），要么复制粘贴（难维护）。
    • 没有一个"运行时共享"的机制。

Module Federation 的核心价值：

让多个独立构建的应用，可以在运行时像调用本地模块一样调用对方的模块，并且共享依赖。

关键词：运行时、模块级共享、依赖共享。

二、核心概念 ​

Host（宿主）与 Remote（远程） ​

角色含义类比
Host消费方，加载别人模块的应用主应用 / 宿主应用
Remote提供方，暴露模块给别人用的应用子应用 / 远程应用

这个关系是相对的：一个应用可以同时是 host 和 remote——既消费别人的模块，也暴露自己的模块给别人。

三要素：exposes / remotes / shared ​

一个完整的 MF 配置有三个核心部分：

javascript
// webpack.config.js
const ModuleFederationPlugin = require('webpack/lib/container/ModuleFederationPlugin');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'app1',           // 应用名，全局唯一
      filename: 'remoteEntry.js', // 生成的入口文件名
      
      // 1. 暴露给别人的模块（Remote 角色用）
      exposes: {
        './Button': './src/components/Button',
        './utils': './src/utils',
      },
      
      // 2. 要消费的远程模块（Host 角色用）
      remotes: {
        app2: 'app2@http://localhost:3002/remoteEntry.js',
      },
      
      // 3. 共享依赖
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true },
        antd: { singleton: true },
      },
    }),
  ],
};

1. exposes（暴露模块）

  • Remote 应用声明"我有哪些模块可以给别人用"。
  • key 是模块名（别人 import 时用的名字），value 是本地路径。
  • 构建时 Webpack 会为每个暴露的模块单独打包。

2. remotes（引用远程模块）

  • Host 应用声明"我要用到哪个远程应用的模块"。
  • 格式：名字: "应用名@URL/remoteEntry.js"
  • 代码里就可以像本地模块一样 import：import Button from 'app2/Button'

3. shared（共享依赖）

  • 双方约定"这些依赖我们共用同一份"，不要重复打包、重复加载。
  • 比如 react: { singleton: true } 表示 React 用单例模式，整个页面只加载一份。
  • 这是 MF 最有价值的特性之一——解决了微前端里"多份 React 导致 hooks 失效"的经典问题。

三、运行时加载机制 ​

MF 的加载过程发生在运行时，不是构建时。这是它和 npm 包最大的区别。

整体流程 ​

Host 应用启动

  ↓
检查 remoteEntry.js 是否已加载

  │          remoteEntry 里包含模块清单和 get 方法

  └─ 已加载 → 调用 remote 的 get(moduleName) 方法

关键文件：remoteEntry.js ​

每个 Remote 应用会生成一个 remoteEntry.js 文件，它是 MF 的"入口清单"，体积很小（几 KB），包含：

  • 这个应用暴露了哪些模块
  • 每个模块对应的 chunk 文件 URL
  • 一个全局变量（应用名），上面挂着 get 和 init 方法

remoteEntry.js 的作用类似"目录"——先拿到目录，知道有什么东西、在哪里，再按需加载具体模块。

Host 端的使用方式 ​

javascript
// Host 应用里的代码
import('app2/Button').then(module => {
  const Button = module.default;
  // 使用 Button 组件
});

// 或者配合 React.lazy
const Button = React.lazy(() => import('app2/Button'));

  4. 如果 Button 模块还依赖其他 chunk（比如 Button 用到了 antd 的某个组件），会自动加载那些 chunk。

四、shared 依赖的版本管理与 singleton ​

shared 是 MF 里最复杂也最容易踩坑的部分，理解它的机制非常重要。

为什么需要 shared ​

没有 shared 时：

  • Remote 应用打包了自己的 React
  • Host 应用也打包了自己的 React
  • 页面上有两份 React，体积大不说，还会导致 context 不共享、hooks 报错（因为 React 用了内部的"当前渲染器"全局变量，多份的话会乱）

有了 shared 后：

  • 双方都声明 React 是共享依赖
  • 运行时只加载一份 React，大家都用同一份
  • 体积小，context 也通了

shared 的核心配置 ​

javascript
shared: {
  react: {
    singleton: true,        // 是否单例模式（全局唯一）
    requiredVersion: '^18', // 要求的版本范围
    version: '18.2.0',      // 当前版本（自动从 package.json 读）
    eager: false,           // 是否预先加载（不动态 import）
    import: 'react',        // 回退时用哪个模块
    shareScope: 'default',  // 共享作用域名
  },
}

单例模式（singleton） ​

singleton: true 表示"整个页面只允许有一份这个依赖"。

版本协商机制：

  • 多个应用共享同一个依赖，各自声明自己的版本。
  • 运行时 MF 会选择一个"最高版本"作为共享版本。
  • 如果某个应用的 requiredVersion 和最高版本不兼容怎么办？
    • 如果 singleton: true：强制用最高版本，哪怕不兼容（可能出问题）。
    • 如果 singleton: false：加载多份，各自用各自的版本。

典型场景：

  • React、ReactDOM 必须 singleton，因为它们内部有全局状态，多份会崩。
  • Lodash 可以不 singleton，各用各的版本也没问题。
  • antd 建议 singleton，否则样式重复、context 不共享。

版本冲突怎么办 ​

假设：

  • Host 用 React 18.2.0
  • Remote 用 React 17.0.2
  • 都配置了 singleton: true

结果：

  • 页面上只会有一份 React，版本是 18.2.0（Host 的版本，因为 Host 先加载）。
  • Remote 用的是 React 18，但它是按 React 17 开发的，可能出兼容问题。

如何避免：

  1. 尽量统一依赖版本（最好的方式）。
  2. 如果无法统一，不要设 singleton: true，允许多版本共存（但 React 不能多版本）。
  3. 用 requiredVersion 声明版本范围，不兼容时降级或报错。

eager 模式 ​

eager: true 表示"不要把这个共享依赖做成异步 chunk，直接打进主 bundle 里，预加载"。

  • 正常的 shared 依赖是按需加载的：用到了才加载。
  • eager: true 会让它一开始就有，不会有异步加载的延迟。
  • 代价是首屏体积变大。

适合场景：

  • 首屏必须的依赖（React、antd 这种）。
  • 不想有"组件第一次渲染时闪烁一下"的体验。

五、MF 和微前端框架的关系 ​

很多人把 MF 和微前端等同起来，其实不对——MF 是微前端的一种实现方式，但不是唯一方式，也不等价于微前端。

对比 ​

维度Module Federationqiankun / single-spa
粒度模块级（组件、函数、工具）应用级（整个子应用）
加载方式运行时动态 import 模块加载整个应用的 HTML + JS + CSS
依赖共享原生支持 shared需要手动 externals 或约定
样式隔离不提供（需要自己处理）提供（Shadow DOM / scoped CSS）
JS 沙箱不提供提供（Proxy 沙箱 / 快照沙箱）
路由管理不提供，需要自己搞内置路由匹配和生命周期
技术栈强绑定 Webpack技术栈无关

它们是互补关系，不是竞争关系：

  • 用 qiankun 做应用级的微前端框架（路由、沙箱、生命周期）。
  • 用 MF 做模块级的共享（共享组件、共享工具库）。
  • 完全可以混用：主应用是 qiankun 壳，子应用之间通过 MF 共享组件。

六、适用场景 ​

场景 1：微前端的组件共享

多个子应用之间需要共享组件、工具函数、状态管理等，用 MF 比 npm 包高效得多。

场景 2：大型应用的渐进式升级

老应用是 React 16 + antd 3，新模块想用 React 18 + antd 5。可以：

  • 老应用作为 Host
  • 新模块作为 Remote，用新技术栈开发
  • 老应用里按需加载新模块的页面
  • 逐步把老页面替换成新的

场景 3：跨团队协作

A 团队做用户中心，B 团队做订单中心，各自独立开发部署。通过 MF：

  • 订单中心可以直接用用户中心的头像组件、地址选择器
  • 用户中心改了，订单中心不用发版，线上直接生效

场景 4：插件化架构

主应用提供插件机制，第三方开发者开发插件（Remote），用户安装后动态加载进来。比如低代码平台的组件市场。

场景 5：多应用共享一套基础组件库

传统方式：组件库发 npm 包，各应用升级。 MF 方式：组件库作为 Remote，各应用作为 Host 直接用。组件库发版后，所有应用线上立即生效。

七、MF 的缺点与坑 ​

  1. 强依赖 Webpack：所有应用都必须用 Webpack 5（Vite 可以用插件兼容，但成熟度稍差）。
  2. 版本管理复杂：shared 依赖的版本冲突是高频坑点。
  3. 没有沙箱和样式隔离：直接加载远程模块，JS 和 CSS 都是全局的，可能互相污染。
  4. 部署依赖关系：Host 依赖 Remote 的 remoteEntry.js，如果 Remote 挂了，Host 也会受影响。
  5. 类型共享麻烦：跨应用的 TypeScript 类型定义需要额外处理。
  6. 调试复杂：多个应用混在一起，出问题了不好排查。
  7. 安全风险：加载远程代码执行，Remote 被篡改的话 Host 也会受影响。

追问 1：Module Federation 的 remoteEntry.js 里到底有什么？为什么要先加载它再加载具体模块？不能直接加载模块吗？ ​

答 ​

这是理解 MF 运行机制的关键问题。remoteEntry.js 是 MF 架构里非常核心的设计。

remoteEntry.js 的内容

remoteEntry.js 是一个很小的 JS 文件（通常几 KB），它的作用是"模块目录 + 加载器"。

简化后的结构大概是这样：

javascript
// remoteEntry.js（简化示意）
window.app2 = {
  // 初始化方法，Host 加载 remoteEntry 后会调用
  init(sharedScope) {
    // 把自己的 shared 依赖注册到共享作用域里
    // 协商版本
  },
  
  // 获取模块的方法，返回 Promise
  get(moduleName) {
    const modules = {
      './Button': () => import('http://cdn/app2/button.chunk.js'),
      './utils': () => import('http://cdn/app2/utils.chunk.js'),
      // ... 其他模块
    };
    return modules[moduleName]();
  },
  
  // 模块清单
  __scope: 'default',
  __modules: ['./Button', './utils'],
};

实际上更复杂，还包含了 shared 依赖的版本信息、chunk 映射表等，但核心就是这两个方法：init 和 get。

为什么要有 remoteEntry 这一层

1. 解耦"有什么"和"是什么"

  • remoteEntry.js 只告诉你"我有哪些模块、每个模块的 chunk 在哪"——这是"目录"。
  • 具体模块的代码（比如 Button 的实现）在单独的 chunk 文件里——这是"内容"。

先拿到目录，再按需加载内容。这样做的好处是：

  • 目录很小，加载快。
  • 不需要一次性加载所有模块，用哪个加载哪个。
  • Host 甚至可以先知道 remote 暴露了什么，再决定用不用。

2. shared 依赖协商必须先完成

在加载具体业务模块之前，必须先完成 shared 依赖的版本协商。

比如 Remote 用了 React，它需要知道 Host 那边有没有共享的 React、版本是多少。如果 Host 已经有了，Remote 就不用自己再加载一份了。

这个协商过程发生在 init() 调用时，必须在加载任何业务模块之前完成。而 init() 方法就在 remoteEntry.js 里。

3. 动态性与灵活性

有了 remoteEntry 这层间接：

  • Remote 可以随时新增模块，不用通知 Host 改配置。
  • 模块的 chunk 文件名（带 hash）可以变，remoteEntry 里记录最新的就行。
  • 可以做 A/B 测试：remoteEntry 里返回不同的模块列表。
  • 可以做权限控制：不同用户的 remoteEntry 内容不一样。

4. 模块名到文件的映射

Host 代码里写的是 import('app2/Button')，但它根本不知道 Button 对应哪个文件、在哪个 CDN。这个映射关系存在 remoteEntry 里。

如果没有 remoteEntry，Host 就必须在构建时知道所有远程模块的 URL，那就不叫"运行时"了。

能不能不用 remoteEntry，直接加载模块？

技术上可以，但会失去很多好处：

  1. 失去按需加载：Host 需要知道所有模块的 URL，或者一次性加载所有模块。
  2. 失去 shared 协商：无法提前知道依赖版本，可能加载多份。
  3. 失去动态性：Remote 加了新模块，Host 要重新构建才能知道。
  4. 失去统一入口：每个模块单独加载，管理混乱。

所以 remoteEntry 这层是必要的，它是 MF "运行时动态性"的基础。

加载流程的完整时序

更完整的加载时序是这样的：

Host 应用启动
  ↓
Host 自己的 shared 依赖先注册到共享作用域
  ↓
遇到 import('app2/Button')
  ↓
检查 app2 是否已初始化
  │
  ├─ 未初始化：
  │   1. 动态创建 <script> 加载 app2 的 remoteEntry.js
  │   2. remoteEntry 加载完成，window.app2 可用
  │   3. 调用 app2.init(共享作用域)
  │   4. shared 依赖版本协商，确定用哪个版本
  │   5. 标记为已初始化
  │
  └─ 已初始化：直接下一步
  ↓
调用 app2.get('./Button')
  ↓
get() 内部动态加载 Button 对应的 chunk
  ↓
Button 模块加载完成，返回模块的 exports
  ↓
Host 拿到模块，正常使用

一个容易踩的坑

remoteEntry.js 是不带 hash 的（文件名固定），因为 Host 配置里写死了 URL。这意味着：

  • remoteEntry 更新后，CDN 缓存要及时刷新。
  • 否则 Host 拿到的是旧版本的 remoteEntry，加载旧的模块。
  • 解决方式：remoteEntry.js 设成 no-cache，或者每次发版更新路径。

而具体的业务 chunk 文件是带 contenthash 的，所以可以放心强缓存。这是一个很典型的"入口文件协商缓存 + 资源文件强缓存"的架构设计。

追问 2：Module Federation 的 shared 依赖在多版本场景下具体是怎么协商的？singleton 为 true 和 false 有什么本质区别？ ​

答 ​

这是 MF 最核心也最容易踩坑的机制，必须深入理解。

共享作用域（Share Scope）

首先，所有 shared 依赖都注册在一个叫"共享作用域"的全局对象里。默认作用域名叫 'default'。

可以理解为页面上有一个全局的 map：

javascript
// 概念上的共享作用域
window.__webpack_share_scopes__.default = {
  react: {
    '18.2.0': { get: () => Promise.resolve(reactModule), loaded: true },
    '17.0.2': { get: () => Promise.resolve(reactModule17), loaded: false },
  },
  lodash: {
    '4.17.21': { get: () => ..., loaded: true },
    '4.17.20': { get: () => ..., loaded: false },
  },
  // ...
};

每个依赖名下，可能注册了多个版本。每个版本有自己的加载状态（loaded / from / 等）。

版本协商的过程

当一个应用（不管是 Host 还是 Remote）启动时，它会把自己的 shared 依赖注册到共享作用域里。然后当它需要用某个依赖时，就去共享作用域里"挑"一个版本来用。

挑版本的策略取决于 singleton 配置。

情况 1：singleton: false（默认值，允许多版本） ​

当 singleton: false 时，策略是：满足 requiredVersion 的最高版本。

举个例子：

共享作用域里的 react：
  - 18.2.0（Host 注册的）
  - 17.0.2（Remote A 注册的）
  - 16.14.0（Remote B 注册的）

应用 C 的 requiredVersion: '^17.0.0'
  → 挑 17.0.2（^17 范围内最高的）

应用 D 的 requiredVersion: '^18.0.0'
  → 挑 18.2.0

每个应用各挑各的，互不影响。结果就是：页面上可能加载了多份 React（16、17、18 各一份），各自用各自的。

适用场景：

  • 工具库（lodash、moment、dayjs），多份也不会出问题。
  • 各应用对版本要求差异大，无法统一。
  • 对体积不是特别敏感。

情况 2：singleton: true（单例模式） ​

当 singleton: true 时，策略是：全局只能有一份，所有人都用同一个版本。

具体规则：

  1. 所有声明了这个依赖且 singleton: true 的应用，大家共用同一份。
  2. 选哪个版本？所有注册版本中最高的那个。
  3. 不管你的 requiredVersion 是多少，强制用那个最高版本。
  4. 如果最高版本不满足你的 requiredVersion，会给一个 warning（开发环境），但还是用最高版本。

举个例子：

共享作用域里的 react（都是 singleton: true）：
  - 18.2.0（Host 注册的）
  - 17.0.2（Remote A 注册的）

选版本：
  → 18.2.0（最高版本）

Host 用 18.2.0 → 没问题（它自己就是 18.2.0）
Remote A 用 18.2.0 → 可能有问题（它是按 17 开发的，但被迫用 18）

为什么 React 必须 singleton

React 内部有一些"全局状态"，比如：

  • 当前的渲染上下文（Context Dispatcher）
  • Hooks 的链表（Fiber 节点上的 hook 链表）
  • 当前正在渲染的 Fiber

如果页面上有两份 React，它们各自有自己的全局状态：

  • A 应用的 React 创建的 Context，B 应用的 React 读不到。
  • 在 A 应用里调用 B 应用的组件，组件内部用了 useState，hooks 的顺序会乱（因为是两个不同的 React 实例在管理 hooks）。
  • 经典报错：Invalid hook call. Hooks can only be called inside of the body of a function component.

所以 React、ReactDOM、Vue 这类"框架核心"必须设 singleton，否则一定会出问题。

singleton 的风险

强制用最高版本可能导致兼容性问题：

  • Remote 是按 React 17 开发测试的，但线上被迫用 React 18，可能有 API 不兼容。
  • antd 3 和 antd 5 差异巨大，强制升版本会导致样式错乱。

怎么规避风险：

  1. 尽量统一版本：最好的方式是所有应用用同一个大版本。
  2. 严格测试：每次 Host 升级依赖版本，所有 Remote 都要回归测试。
  3. requiredVersion 声明：在 shared 里声明 requiredVersion: '^18.0.0'，开发环境下版本不匹配会有 warning，提前发现问题。
  4. strictVersion：设置 strictVersion: true，版本不满足直接报错（而不是警告），更严格但更安全。

eager 的作用 ​

eager: true 表示"这个共享依赖不要做成异步 chunk，直接打包进主 bundle"。

默认行为（eager: false）：

  • shared 依赖被做成异步的。
  • 只有当某个模块真正 import 了这个依赖时，才去加载它。
  • 好处是首屏可能不用加载（如果首屏没用到）。
  • 坏处是第一次用到时有加载延迟（虽然很短，但能感觉到）。

eager: true 的行为：

  • shared 依赖直接打进主 bundle，一开始就有。
  • 首屏体积大一点，但没有异步加载的延迟。
  • React、antd 这种首屏必须的依赖，建议设 eager。

shared 配置完整示例 ​

javascript
shared: {
  // 框架核心：单例 + 严格版本 + 预加载
  react: {
    singleton: true,
    requiredVersion: '^18.0.0',
    eager: true,
    strictVersion: true,
  },
  'react-dom': {
    singleton: true,
    requiredVersion: '^18.0.0',
    eager: true,
    strictVersion: true,
  },
  
  // 组件库：单例 + 预加载，但版本要求松一点
  antd: {
    singleton: true,
    requiredVersion: '^5.0.0',
    eager: true,
  },
  
  // 工具库：不强制单例，各用各的
  lodash: {
    singleton: false,
    requiredVersion: '^4.17.0',
  },
  
  // 状态管理：单例（否则 store 不共享）
  'react-redux': {
    singleton: true,
    requiredVersion: '^8.0.0',
  },
}

一个常见误解

很多人以为"配置了 shared，就不会重复打包了"。其实不对：

  • 构建时：每个应用还是会把 shared 依赖打进去（因为构建时不知道别的应用有没有）。
  • 运行时：加载了第一个应用的 shared 依赖后，后面的应用发现共享作用域里已经有了，就不会再加载自己的那份了。
  • 也就是说，构建时每个应用都有，运行时只用一份。

这也是为什么 MF 不能减少"构建产物体积"，只能减少"运行时加载的体积"。

追问 3：Module Federation 和 npm 包共享代码，该怎么选？两者各适合什么场景？能混用吗？ ​

答 ​

这是一个非常实际的架构决策问题。MF 和 npm 包不是非此即彼的关系，而是各有适用场景，可以混用。

核心区别

维度npm 包Module Federation
共享时机构建时运行时
代码位置每个消费方各自打包一份远端加载，运行时共享
版本同步各用各的版本，手动升级共享作用域，运行时协商
更新方式发版 → 消费方升级 → 重新构建 → 发版Remote 发版，Host 线上立即生效
依赖共享各打各的，可能重复shared 机制，运行时共享一份
类型支持原生支持，开箱即用需要额外处理（API 抽取、类型文件共享）
稳定性高（版本锁死，不会突然变）低（Remote 挂了 / 改了，Host 直接受影响）
调试难度低（本地就一份代码）高（多个应用混在一起）
安全高（本地代码，可控）低（加载远程代码执行）
部署解耦高（各发各的，互不影响）中（Remote 挂了 Host 也遭殃）

怎么选：决策框架

可以从以下几个维度判断：

维度 1：更新频率

  • 更新很频繁（每周甚至每天都要改）→ 优先 MF
    • 比如运营活动组件、动态配置的页面
    • 用 npm 的话，改一次要发版、所有消费方升级、再发版，链路太长
  • 稳定不怎么变 → 优先 npm
    • 比如基础工具函数、底层组件库基础版
    • 稳定的东西用 npm 更安心

维度 2：影响范围

  • 改了需要所有消费方立即生效 → 优先 MF
    • 比如安全修复、紧急 bug fix
    • 用 npm 的话，等所有应用升级完可能好几天过去了
  • 改了不着急，消费方慢慢升 → 优先 npm
    • 比如新功能、优化项
    • 各团队按自己的节奏升级，风险低

维度 3：依赖关系

  • 有框架级依赖（React、Vue、antd）需要共享 → MF 的 shared 有优势
    • 微前端场景下，多应用共用同一份 React 是刚需
    • npm 做不到运行时共享，只能 externals + CDN
  • 纯工具函数、无外部依赖 → npm 足够

维度 4：团队协作模式

  • 组件提供方和消费方是同一个团队 → npm 简单直接
  • 跨团队、跨部门协作 → MF 解耦更好
    • A 团队不需要知道 B 团队的发布节奏
    • B 团队也不需要等 A 团队升级

维度 5：稳定性要求

  • 核心链路、不能出问题 → 优先 npm
    • 支付、登录这些核心模块，宁可贵一点、慢一点，也要稳
    • npm 版本锁死了，不会因为 Remote 挂了就崩
  • 非核心、可以降级 → MF 没问题
    • 比如推荐位、广告位、非核心功能
    • 就算挂了也不影响主流程

典型场景对照表

场景推荐方案原因
底层工具函数库（lodash 风格）npm稳定、不怎么变、类型友好
基础组件库（Button / Input 等）npm + MF 混用稳定版走 npm，动态化需求走 MF
业务组件（带业务逻辑的）MF经常改、跨团队共享、要立即生效

能不能混用？当然可以，而且推荐混用

实际项目中，最佳实践是"npm 做基础，MF 做动态"——两者配合，各取所长。

一个典型的分层架构：

┌─────────────────────────────────────────┐
│  Layer 4: 业务组件 / 页面模块           │
│  （用 MF，动态共享，快速迭代）          │
├─────────────────────────────────────────┤
│  Layer 3: 业务组件库                    │
│  （用 npm 发稳定版，用 MF 做实时更新）  │
├─────────────────────────────────────────┤
│  Layer 2: 基础组件库 / 工具库           │
│  （用 npm，稳定优先，版本锁死）         │
├─────────────────────────────────────────┤
│  Layer 1: 框架 / 运行时（React / antd） │
│  （用 MF shared 或 externals + CDN）    │
└─────────────────────────────────────────┘

具体来说：

  1. 底层基础库（utils、hooks、基础组件）：走 npm，稳定、可靠、类型好。
  2. 框架层（React、antd、状态管理）：走 MF shared 或 CDN externals，确保单例。
  3. 业务组件：
    • 稳定版本走 npm 包（给要求稳的应用用）。
    • 同时作为 MF Remote 暴露（给要最新版、要热更新的应用用）。
  4. 页面模块 / 大型功能：走 MF，独立部署、动态加载。

混用的好处：

  • 该稳的地方稳（底层用 npm）。
  • 该快的地方快（上层用 MF）。
  • 风险可控：核心链路用 npm 稳定版，非核心功能尝试 MF。
  • 渐进式迁移：老项目先不动，新功能用 MF，慢慢过渡。

一个真实案例

某电商平台的架构：

  • 基础组件库（Button、Input 等 30+ 基础组件）：npm 包，每季度发一个大版本。
  • 业务组件库（商品卡片、价格组件、地址选择器等）：既发 npm 包，又通过 MF 暴露。
    • 核心应用（商品详情页、购物车）用 npm 稳定版，求稳。
    • 运营活动页、边缘业务用 MF 实时版，求快。
  • 跨应用页面模块（用户中心、订单列表）：纯 MF，独立团队开发部署。
  • 框架层（React 18 + antd 5）：MF shared，全局单例。

这样既保证了核心链路的稳定性，又让迭代快的业务享受了 MF 的便利。

总结：

  • npm 是"构建时共享"，特点是稳、慢、安全。
  • MF 是"运行时共享"，特点是快、动态、有风险。
  • 不是二选一，而是根据组件的层级和特性分层使用。
  • 底层稳定的用 npm，上层动态的用 MF，中间层可以两者都提供。
