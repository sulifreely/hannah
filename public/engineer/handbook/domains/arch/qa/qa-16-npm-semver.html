问答 16（进阶）：npm 依赖管理与语义化版本 ​

题目 ​

请说明 SemVer（语义化版本）规范、版本范围符号的含义（^ ~ * = > < -）、lock 文件的作用（package-lock.json / yarn.lock / pnpm-lock.yaml），并解释 peerDependencies、optionalDependencies、bundledDependencies 的区别，以及幽灵依赖、依赖地狱等问题。

参考答案 ​

一、SemVer 语义化版本 ​

SemVer（Semantic Versioning）是一套版本号命名规范，格式为：

主版本号.次版本号.修订号
MAJOR.MINOR.PATCH
1
2

每个数字的含义：

位名称含义例子
MAJOR（主版本）不兼容的 API 修改breaking change，老用户升级可能会挂1.x.x → 2.x.x
MINOR（次版本）向下兼容的功能性新增加了新功能，但老 API 还能用1.1.x → 1.2.x
PATCH（修订号）向下兼容的问题修正修 bug，不改功能1.0.0 → 1.0.1

为什么需要 SemVer：

没有规范之前，版本号很随意：有的项目用年份当版本（2024.1），有的用自增数字（v123），你根本不知道两个版本之间有没有 breaking change。

有了 SemVer，看到版本号就能大致判断升级风险：

  • 升级 PATCH：放心升，只是修 bug。
  • 升级 MINOR：可以升，加了新功能，不会破坏老的。
  • 升级 MAJOR：谨慎，可能有不兼容变化，看完 changelog 再升。

先行版本（Pre-release）

正式发布前，可以加先行版本号后缀：

1.0.0-alpha.1    // alpha 版，功能可能不全
1.0.0-beta.2     // beta 版，基本功能齐了，但可能有 bug
1.0.0-rc.1       // 候选版，基本没问题，再测测就正式发
1
2
3

先行版本的优先级低于正式版：1.0.0-alpha < 1.0.0。

构建元数据

还可以加构建号等元数据（不影响版本优先级）：

1.0.0+build.123
1.0.0+sha.abc123
1
2

二、版本范围符号 ​

package.json 里的依赖版本不是只能写死，还可以写范围。npm 安装时会安装"满足这个范围的最新版本"。

符号含义例子匹配范围
无符号 / =精确版本1.2.3 或 =1.2.3只能是 1.2.3
^兼容更新（不改变最左边非零数字）^1.2.3>=1.2.3 <2.0.0
~修订号范围~1.2.3>=1.2.3 <1.3.0
* / x / X通配符1.2.x 或 *1.2 任意修订号 / 任意版本
>大于>1.2.3比 1.2.3 大的
>=大于等于>=1.2.31.2.3 及以上
<小于<2.0.0小于 2.0.0
<=小于等于<=1.2.31.2.3 及以下
-范围（包含两端）1.2.3 - 1.4.5>=1.2.3 <=1.4.5
||或^1.0.0 || ^2.0.01.x 或 2.x

^ 和 ~ 是最常用的，重点理解：

^（Caret 范围） ​

规则：最左边的非零数字不变，右边的可以升。

^1.2.3   → >=1.2.3 <2.0.0    （主版本不变，次版本和修订号可以升）
^0.2.3   → >=0.2.3 <0.3.0    （0.x.x 的次版本不变，因为 0 开头是不稳定版）
^0.0.3   → =0.0.3            （0.0.x 是精确匹配）
1
2
3

为什么 0.x.x 特殊？因为 SemVer 规定：主版本号为 0 时（开发阶段），一切都可能变，次版本号也可能是 breaking change。所以 ^0.x.x 不会跨次版本。

^ 是 npm 默认的范围符号（npm install xxx 保存的版本就是 ^x.y.z）。

~（Tilde 范围） ​

规则：如果写了次版本号，修订号可以变；如果只写了主版本号，次版本号可以变。

~1.2.3   → >=1.2.3 <1.3.0    （次版本不变，修订号可以升）
~1.2     → >=1.2.0 <1.3.0    （同上）
~1       → >=1.0.0 <2.0.0    （主版本不变）
1
2
3

~ 比 ^ 更保守：^ 允许升次版本，~ 只允许升修订号。

怎么选：

  • 信任 SemVer，想自动拿最新功能 → ^（默认）
  • 比较保守，只想修 bug → ~
  • 完全锁死，不想自动升级 → 精确版本 + lockfile

三、lock 文件的作用 ​

package-lock.json（npm）、yarn.lock（Yarn）、pnpm-lock.yaml（pnpm）统称 lock 文件。

为什么需要 lock 文件 ​

package.json 里写的是版本范围，不是精确版本。比如 "react": "^18.0.0"，今天装可能是 18.2.0，明天 React 发了 18.3.0，再装就是 18.3.0 了。

这会导致一个经典问题："我本地好好的，你那里怎么挂了？"

  • 你昨天装的是 18.2.0，没问题。
  • 同事今天拉代码，npm install 装了 18.3.0，有 bug。
  • 你们用的是同一份 package.json，但实际安装的版本不一样。

lock 文件就是用来解决这个问题的——把每个依赖的精确版本、来源、哈希都记下来，保证任何人、任何时候、任何环境 npm install 出来的依赖树完全一致。

lock 文件里有什么 ​

以 package-lock.json 为例，它记录了：

  1. 每个依赖的精确版本号（不是范围）
  2. 每个依赖的下载地址（resolved）
  3. 每个依赖的完整性校验哈希（integrity，sha512）
  4. 完整的依赖树结构（每个依赖的子依赖）
  5. 依赖之间的依赖关系

有了它，npm install 时会严格按照 lock 文件里的版本来装，而不是去算最新版本。

lock 文件要不要提交到 Git ​

一定要提交。原因：

  • 保证团队所有成员的依赖版本一致。
  • 保证 CI/CD 环境和本地一致。
  • 出问题了可以回退 lock 文件版本，排查是哪个依赖升级导致的。

什么时候 lock 文件会变：

  • npm install 加新依赖。
  • npm update 更新依赖。
  • package.json 里的版本范围改了。
  • 手动改了 lock 文件（不推荐）。

四、dependencies 的几种类型 ​

package.json 里不是只有 dependencies，还有好几种，各有用途。

1. dependencies（生产依赖） ​

项目运行时必须的依赖。

json
"dependencies": {
  "react": "^18.0.0",
  "axios": "^1.0.0"
}
1
2
3
4
  • 执行 npm install（不带参数）会安装。
  • 会打到最终产物里。

2. devDependencies（开发依赖） ​

开发时才需要的依赖，运行时不需要。

json
"devDependencies": {
  "webpack": "^5.0.0",
  "eslint": "^8.0.0",
  "jest": "^29.0.0"
}
1
2
3
4
5
  • 构建工具、lint 工具、测试框架、类型定义（@types/*）都放这里。
  • npm install --production 或 NODE_ENV=production 时不会安装。
  • 注意：如果库项目里用了，但库的消费方不需要，就放 devDependencies。

3. peerDependencies（对等依赖） ​

你的包要求宿主环境必须安装的依赖，但你不自己打包进去，而是用宿主的那份。

json
// 比如一个 React 组件库
"peerDependencies": {
  "react": ">=16.8.0",
  "react-dom": ">=16.8.0"
}
1
2
3
4
5

为什么需要 peerDependencies：

以 React 组件库为例：

  • 如果组件库把 React 打进自己的包里，使用组件库的应用已经有自己的 React 了。
  • 结果页面上有两份 React：应用的一份 + 组件库的一份。
  • 这会导致 hooks 报错（因为 React 内部有全局状态，多份会乱）。

peerDependencies 的意思是：

  • "我是一个 React 组件库，我需要 React，但我不自己带 React。"
  • "用我的人，你自己必须有 React，而且版本要满足我的要求。"
  • "我用你提供的那份 React。"

npm 7 之前：peerDependencies 不满足时只会警告，不会自动安装。 npm 7+：默认会自动安装 peerDependencies（--legacy-peer-deps 可以关掉，回到老行为）。

4. optionalDependencies（可选依赖） ​

可选的依赖，装不上也没关系，不会导致安装失败。

json
"optionalDependencies": {
  "fsevents": "^2.0.0" // 只有 macOS 能用，Linux 装不上也没关系
}
1
2
3

适用场景：

  • 平台相关的依赖（比如 fsevents 只有 macOS 有）。
  • 不是必须的功能，装不上就降级（比如某个加速库，装不上就用普通实现）。

代码里要用 try/catch 包裹：

javascript
let fsevents;
try {
  fsevents = require('fsevents');
} catch (e) {
  // 装不上就不用
}
1
2
3
4
5
6

5. bundledDependencies / bundleDependencies（打包依赖） ​

发布 npm 包时，把这些依赖一起打进包里。

json
"bundledDependencies": ["lodash", "axios"]
1

适用场景：

  • 发布的包需要修改第三方依赖的源码。
  • 要确保用户拿到的依赖版本和你测试时完全一致。
  • 内部私有包，不想让用户自己去装。

不常用，了解即可。

五、幽灵依赖（Phantom Dependencies） ​

什么是幽灵依赖：你代码里 import 了一个包，但 package.json 里没声明它——你能 import 成功是因为它是某个依赖的依赖（被提升到了根目录的 node_modules）。

你的项目依赖了 A
A 依赖了 B

npm install 后，node_modules 里有：
  A
  B （因为 npm/yarn 的扁平化提升，B 被提到了根目录）

你的代码里写 import B from 'b' → 居然能运行！
但 package.json 里根本没声明 B
1
2
3
4
5
6
7
8
9

这就是幽灵依赖：你用了 B，但 B 不在你的 package.json 里，它是"飘"过来的。

幽灵依赖的危害：

  1. 版本不可控：A 升级了，B 的版本可能跟着变，你的代码可能突然就挂了，但你什么都没改。
  2. 别人装不上：同事拉代码，或者换个包管理器（比如 pnpm），可能装出来的依赖树不一样，B 没被提升，import 就报错了。
  3. 语义不清：看 package.json 根本不知道你用了 B，维护的人一脸懵。
  4. 依赖地狱的温床：各种隐式依赖纠缠在一起，理不清。

为什么会有幽灵依赖：

npm / yarn 经典模式的扁平化（hoisting）机制导致的。为了减少重复安装，npm 会把依赖的依赖尽量提升到根目录的 node_modules 里。提升了之后，你就能直接 import 到，但它们不应该是你能直接用的。

pnpm 怎么解决：

pnpm 用符号链接 + 非扁平化的结构：

  • 根目录的 node_modules 里只有你显式声明的依赖。
  • 子依赖在 .pnpm/ 目录里，通过符号链接组织。
  • 你 import 一个没声明的包 → 直接报错，因为根目录根本没有。
  • 从机制上杜绝了幽灵依赖。

这是 pnpm 的一大优势。

六、依赖地狱（Dependency Hell） ​

什么是依赖地狱：项目依赖多了之后，各种版本冲突、循环依赖、重复安装的问题，让你痛不欲生。

常见场景：

场景 1：版本冲突

你的项目依赖 A@^1.0.0 和 B@^1.0.0
A 依赖 C@^1.0.0
B 依赖 C@^2.0.0

C 有两个不兼容的版本，怎么办？
1
2
3
4
5

npm/yarn 的策略：

  • 把满足大多数需求的版本提升到根目录。
  • 另一个版本装在各自依赖的 node_modules 里（嵌套）。
  • 结果：C 被安装了两份，体积大，还可能有运行时问题（比如两个实例不共享状态）。

场景 2：嵌套地狱

早期 npm（v2 之前）是纯嵌套的：

npm 3+ 改成了扁平化，缓解了这个问题，但又带来了幽灵依赖。

场景 3：循环依赖

A 依赖 B

    你的项目

      A     B
      \     /
       \   /
         C

A 和 B 都依赖 C，C 的版本可能不同。如果版本兼容还好，不兼容就会装两份。

怎么缓解依赖地狱：

  1. 用 pnpm：更好的依赖管理，杜绝幽灵依赖，磁盘利用率高。
  2. 减少依赖：能自己写的就不用库，依赖越少问题越少。
  3. 锁定版本：lock 文件 + 精确版本，不要太宽松。
  4. 定期升级：用 npm outdated 检查过时依赖，定期维护。
  5. peerDependencies 用对：框架类依赖用 peer，避免多份实例。
  6. 依赖审计：npm audit 检查安全漏洞。
  7. 用 npm dedupe：去重，尽量减少重复安装。

追问 1：npm、yarn、pnpm 三种包管理器有什么核心区别？pnpm 的硬链接 + 符号链接方案是怎么工作的？为什么又快又省空间？ ​

答 ​

这三个是前端最常用的包管理器，理解它们的区别和原理对日常开发很有帮助。

演进历史

  • npm：Node.js 自带的包管理器，最古老，用户最多。npm 1/2 是嵌套结构，npm 3+ 改成扁平化。
  • yarn：2016 年 Facebook 推出，解决了早期 npm 的很多问题（没有 lockfile、安装慢、不确定）。引入了 yarn.lock、并行安装、缓存等特性。
  • pnpm：2017 年左右出现，用硬链接 + 符号链接的方式管理依赖，主打省空间、速度快、严格。

核心对比

维度npmyarnpnpm
诞生时间2010（随 Node.js）20162017
依赖结构扁平化（npm 3+）扁平化非扁平化 + 符号链接
幽灵依赖有有（默认）无（严格模式）
安装速度中等快（并行安装）最快（硬链接 + 缓存复用）
磁盘空间占空间多（每个项目一份）占空间多极省（全局一份，硬链接共享）

pnpm 的核心原理：硬链接 + 符号链接

这是 pnpm 又快又省空间的关键。先复习一下文件系统的概念：

  • 硬链接（Hard Link）：多个文件名指向同一个磁盘上的 inode（文件内容）。删除一个文件名，文件还在，只要还有硬链接指向它。硬链接不能跨文件系统，不能指向目录。
  • 符号链接（Symbolic Link / Soft Link）：一个文件指向另一个文件的路径，类似快捷方式。可以跨文件系统，可以指向目录。

pnpm 的存储结构

pnpm 有一个全局内容可寻址仓库（content-addressable store），所有下载的包都存在这里（通常在 ~/.pnpm-store）。

每个包的每个版本在全局仓库里只有一份文件。

项目里的 node_modules 结构：

node_modules/
  react/                 → 你显式声明的依赖，直接在根目录
    index.js             → 符号链接，指向 .pnpm/react@18.2.0/node_modules/react/index.js
    package.json
  .pnpm/                 → 所有包的真实位置（平铺存储）
    react@18.2.0/
      node_modules/
        react/           → 硬链接到全局仓库里的文件
        loose-envify/    → react 的依赖，也是符号链接
    loose-envify@1.4.0/
      node_modules/
        loose-envify/    → 硬链接到全局仓库
        js-tokens/       → 依赖的依赖
    ...

关键点：

  1. 全局仓库只有一份：不管你有多少个项目，react@18.2.0 在磁盘上只有一份文件。
  2. 硬链接到项目：每个项目的 node_modules 里的文件，是全局仓库文件的硬链接。不占额外磁盘空间。
  3. 符号链接组织依赖关系：通过符号链接构造依赖树，实现"每个包只能看到自己的依赖"（严格模式）。
  4. 根目录只有显式声明的依赖：你没在 package.json 里写的包，根目录 node_modules 里没有，杜绝幽灵依赖。

为什么快：

  1. 省了复制文件的时间：用硬链接，不用把文件从缓存复制到项目 node_modules，只是建个链接，瞬间完成。
  2. 缓存命中率高：全局仓库里有的包，不用再下载。A 项目装过的，B 项目直接用。
  3. 并行安装：pnpm 也是并行安装依赖的。
  4. 更高效的依赖解析：严格的依赖结构，不需要像 npm/yarn 那样做复杂的 hoisting 计算。

为什么省空间：

  1. 全局一份：100 个项目都用 react@18.2.0，npm/yarn 会存 100 份，pnpm 只存 1 份 + 100 个硬链接（几乎不占空间）。
  2. Content-addressable：相同内容的文件只存一份。就算是不同包，只要文件内容一样，也只存一份。
  3. 节省巨量空间：对于同时维护多个项目的开发者，pnpm 能省几个 GB 甚至几十 GB 的磁盘空间。

pnpm 的严格性

默认情况下，pnpm 是严格的：

  • 你只能 import 你在 package.json 里显式声明的依赖。
  • 依赖的依赖（子依赖）不会被提升到根目录，你 import 不到。
  • 从机制上杜绝了幽灵依赖。

这对新项目是好事，但对老项目可能不兼容——因为老项目可能到处都是幽灵依赖。pnpm 提供了 shamefully-hoist 选项，可以模拟 npm/yarn 的扁平化结构，方便迁移。

怎么选包管理器：

场景推荐原因
老项目，一直用 npm继续用 npm迁移有成本
团队用 yarn继续用 yarn保持一致
新项目pnpm快、省空间、严格，体验好
monorepo 项目pnpmworkspace 支持好，速度快
开源库都可以，说明兼容的包管理器让用户自己选

个人建议：新项目无脑选 pnpm，用过的都说好。老项目如果想迁移，先开 shamefully-hoist 过渡，再慢慢清理幽灵依赖。

追问 2：peerDependencies 具体是怎么工作的？npm 7 前后行为有什么变化？为什么会有 peer dependency 冲突？如何解决？ ​

答 ​

peerDependencies 是一个很多人搞不清但又非常重要的概念，尤其是做库开发的同学必须理解。

peerDependencies 的核心思想

普通依赖（dependencies）是"我把我的依赖带上，我用我自己的那份"。

对等依赖（peerDependencies）是"我需要这个依赖，但我不带，我用宿主的那份"。

为什么需要 peerDependencies

以 React 组件库为例：

应用 App
  └── 依赖 react@18.2.0
  └── 依赖 my-ui-lib
        └── （如果 my-ui-lib 把 react 放在 dependencies 里）
            └── react@18.2.0（自己带了一份）
1
2
3
4
5

结果：页面上有两份 React。这会导致：

  1. 体积大：React 被打包两次。
  2. hooks 报错：React 内部有全局状态（比如当前 dispatcher），两份 React 各管各的，在组件库里调用 hooks 会用组件库的 React，而应用的 React 上下文不一样，经典报错 Invalid hook call。
  3. Context 不共享：应用里的 React.createContext，组件库里消费不到，因为是两个不同的 React 实例。

peerDependencies 解决这个问题：

应用 App
  └── 依赖 react@18.2.0 （宿主提供）
  └── 依赖 my-ui-lib
        └── peerDependencies: { react: ">=16.8.0" } （我要求宿主有 react，但我不带）
1
2
3
4

组件库不打包 React，直接用宿主的那份。全页面只有一份 React，问题解决。

peerDependencies 的工作机制

npm 6 及之前：

  • 安装时，如果 peerDependencies 不满足，只会给一个 warning（警告），不会自动安装。
  • 需要用户自己手动安装 peer 依赖。
  • 经常看到"UNMET PEER DEPENDENCY"的警告，大家也习惯了忽略。

npm 7+（v7, v8, v9）：

  • 默认会自动安装 peerDependencies。
  • 安装时，npm 会自动把 peer 依赖装到合适的位置。
  • 如果有版本冲突（多个 peer 依赖要求的版本不兼容），安装会失败并报错。
  • 可以用 --legacy-peer-deps 关掉自动安装，回到 npm 6 的行为（只警告不安装）。

Yarn Classic（v1）和 npm 6 行为类似，只警告不自动安装。 Yarn Berry（v2+）默认是 PnP 模式，peer 处理更严格。 pnpm 的行为和 npm 7 类似，也会自动安装 peer 依赖，但严格性更高。

peer dependency 冲突是什么

当多个包的 peerDependencies 要求的同一个依赖的版本不兼容时，就会冲突。

举个例子：

你的项目有两个库：
  - antd@4.x → peerDependencies: { react: "^16.8.0 || ^17.0.0" }
  - my-lib@2.x → peerDependencies: { react: "^18.0.0" }

antd 4 要求 React 16 或 17
my-lib 2 要求 React 18
这两个要求没有交集 → 冲突！
1
2
3
4
5
6
7

npm 7+ 会直接报错，不让你装，因为它不知道该装 React 17 还是 18。

为什么会有冲突：

  1. 库的版本不同步：有的库已经升级到支持 React 18，有的还没。
  2. 版本范围太窄：库的 peerDependencies 写得太死（比如只支持 17.x），不给新版本留空间。
  3. 大型项目依赖多：依赖越多，遇到冲突的概率越大。

如何解决 peer dependency 冲突

方案 1：升级库（推荐）

看看有没有新版本的库支持更高版本的 peer 依赖。

bash
# 比如 antd 4 不支持 React 18，但 antd 5 支持
# 升级 antd 到 5
npm install antd@5
1
2
3

这是最推荐的方式——保持依赖更新，从根源解决。

方案 2：降级某些依赖

如果某个库还不支持最新版本，你可以先降级另一个。

# my-lib 要 React 18，但 antd 4 只支持到 17
# 那就先用 React 17
npm install react@17 react-dom@17
1
2
3

短期可行，长期还是要等库升级。

方案 3：--legacy-peer-deps / --force（临时方案）

如果确定没问题（比如虽然 peer 写的是 17，但 18 其实也能用），可以强制安装：

bash
npm install --legacy-peer-deps   # 回到 npm 6 行为，只警告不报错
npm install --force              # 强制安装，忽略冲突
1
2
bash
# pnpm 也有类似选项
pnpm install --legacy-peer-deps
# 或者在 .npmrc 里配置
strict-peer-dependencies=false

4

注意：这是临时 workaround，不是真正解决了问题。可能会有运行时 bug，用了就要自己承担风险。

强行指定某个依赖的版本，覆盖 peer 的要求。

    "react": "18.2.0"  // 强制所有依赖都用这个版本的 react
  }
}

方案 5：fork 库，自己改 peerDependencies

如果库已经不维护了，你又必须用，可以 fork 一份，自己改 peerDependencies 的版本范围，然后用自己的 fork。

不推荐，但实在没办法了可以用。

作为库开发者，怎么写好 peerDependencies

  1. 范围写宽松点：比如 "react": ">=16.8.0" 比 "react": "17.x" 好，给用户更多空间。
  2. 及时更新：新版本的 peer 依赖出来后，测试兼容性，及时放宽 peerDependencies 的范围。
  3. 写清楚为什么需要 peer：在 README 里说明哪些是 peer 依赖、为什么。
  4. 提供多个主版本：如果 breaking change 太大，就发新的主版本，分别支持不同的 peer 版本范围。

总结：peerDependencies 是"库用宿主的依赖"，避免多份实例。npm 7+ 会自动安装 peer 依赖，冲突会报错。解决冲突的优先顺序是：升级库 → 降级依赖 → overrides 强制指定 → --legacy-peer-deps 绕过。

追问 3：什么是依赖扁平化（hoisting）？npm 为什么要做扁平化？扁平化有什么副作用？pnpm 是怎么在不扁平化的情况下解决重复依赖问题的？ ​

答 ​

依赖扁平化是理解 npm/yarn/pnpm 差异的核心概念之一。

什么是依赖扁平化（Hoisting）

先看 npm 2 时代的"嵌套结构"：

你的项目
  └── A@1.0.0
  │    └── B@1.0.0
  │         └── C@1.0.0
  └── D@1.0.0
       └── B@1.0.0
            └── C@1.0.0
1
2
3
4
5
6
7

每个包都把自己的依赖装在自己的 node_modules 里，嵌套很深。

问题：

  1. 重复安装：B 和 C 各被装了两次，浪费空间。
  2. 路径太长：Windows 下路径长度有限制，嵌套深了会报错。
  3. 加载慢：Node.js 找模块要一层层往上找。

npm 3 以后改成了扁平化结构：尽量把所有依赖提升到根目录的 node_modules 里。

你的项目
  └── node_modules
       ├── A@1.0.0
       ├── B@1.0.0  （A 和 D 都依赖 B，提升到根目录，只装一份）
       ├── C@1.0.0  （同理）
       └── D@1.0.0
1
2
3
4
5
6

A 和 D 都依赖 B@1.0.0，B 被提升到根目录，只装一份，A 和 D 都能用。

这就是hoisting（提升/扁平化）。

npm 是怎么决定提升谁的

扁平化的算法大概是这样的：

  1. 遍历所有依赖，构建完整的依赖树。
  2. 从根开始，尽量把依赖往上层提升。
  3. 如果两个依赖版本相同 → 提升一份，共享。
  4. 如果版本不同 → 谁先遇到就提升谁，另一个保留在自己的 node_modules 里（不提升）。

举个例子：

项目依赖 A 和 D
A 依赖 B@1.0.0
D 依赖 B@2.0.0
1
2
3

扁平化后：

node_modules/
  A@1.0.0
  B@1.0.0   （先遇到 A 的 B，提升 1.0.0）
  D@1.0.0
    node_modules/
      B@2.0.0  （D 的 B 版本不同，不能提升，留在 D 里面）
1
2
3
4
5
6

B@1.0.0 被提升到根目录，B@2.0.0 留在 D 的 node_modules 里。这叫半扁平化——能提升的提升，不能提升的嵌套。

为什么要有这个算法：尽量减少重复，但又不能完全消除重复（版本不同的没办法）。

扁平化的副作用

扁平化虽然解决了空间和路径问题，但带来了新的问题：

副作用 1：幽灵依赖（Phantom Dependencies）——最严重的问题

前面讲过，你没声明的依赖，因为被提升到了根目录，你也能 import 到。

你只声明了 A
A 依赖 B
B 被提升到根目录

你代码里写 import B → 能运行
但 package.json 里根本没 B
1
2
3
4
5
6

危害：版本不可控、别人装不上、语义不清。

副作用 2：依赖分身（Doppelgangers）

当版本冲突时，同一个包会被装多份，分别在不同的位置。

node_modules/
  B@1.0.0
  D/
    node_modules/
      B@2.0.0
1
2
3
4
5

如果某个库判断"是不是同一个实例"（比如 instanceof 判断），可能出问题。

副作用 3：不确定性

安装顺序可能影响提升结果。比如你先装 A 再装 D，和先装 D 再装 A，提升的 B 版本可能不一样。

npm 5 之后有了 lockfile，这个问题基本解决了（lockfile 记录了确定的提升结果）。但早期没有 lockfile 的时候是个大问题。

副作用 4：扁平化计算开销

要算清楚哪些能提升、哪些不能，算法复杂度不低。大型项目依赖多了，安装时计算依赖树很慢。

pnpm 的方案：不扁平化，用符号链接

pnpm 的思路是：既然扁平化有这么多问题，那我干脆不扁平化了。但我也不嵌套，我用符号链接来组织依赖关系。

pnpm 的结构：

node_modules/
  A@1.0.0/                    → 你声明的依赖（直接在根目录）
    node_modules/
      B → ../../.pnpm/B@1.0.0/node_modules/B  （符号链接，指向 A 的依赖 B）
  D@1.0.0/
    node_modules/
      B → ../../.pnpm/B@2.0.0/node_modules/B  （D 的 B 是 2.0.0，另一个符号链接）
  .pnpm/
    B@1.0.0/
      node_modules/
        B/                    → 硬链接到全局仓库（真实文件在这里）
          node_modules/
            C → ...           （B 的依赖，继续符号链接）
    B@2.0.0/
      node_modules/
        B/                    → 硬链接到全局仓库（另一个版本）
    C@1.0.0/
      node_modules/
        C/                    → 硬链接到全局仓库
    ...

关键点：

  1. 根目录只有你显式声明的依赖：A 和 D 在根目录，B 不在根目录。所以你不能直接 import B——从机制上杜绝了幽灵依赖。
  2. 每个包有自己独立的 node_modules：A 有自己的 node_modules（里面是它的依赖 B@1.0.0），D 也有自己的 node_modules（B@2.0.0）。互不干扰，版本严格隔离。
  3. 通过符号链接组织依赖关系：A 的 node_modules 里的 B 是符号链接，指向 .pnpm/ 里的真实位置。不占额外空间。
  4. .pnpm 目录是平铺的：所有版本的所有包都在 .pnpm/ 里平铺存储，没有嵌套。
  5. 文件内容用硬链接共享：每个包的真实文件在全局仓库里，.pnpm 里的是硬链接，不重复占用磁盘。

pnpm 方案的优势：

  1. 没有幽灵依赖：根目录只有你声明的，import 没声明的直接报错。严格但安全。
  2. 版本严格隔离：A 用 B@1.0.0，D 用 B@2.0.0，互不影响，不会混乱。
  3. 极省空间：全局一份 + 硬链接，多少项目都不重复。
  4. 更快：不需要计算复杂的 hoisting 策略，硬链接比复制文件快得多。
  5. 确定性更高：符号链接的结构是确定的，不会因为安装顺序不同而不同。

pnpm 方案的代价：

  1. 不兼容幽灵依赖：老项目如果有大量幽灵依赖，迁移到 pnpm 会报错，需要一个个修。pnpm 提供了 shamefully-hoist=true 选项来模拟扁平化，方便迁移。
  2. 工具兼容性：有些工具假设 node_modules 是扁平化的，在 pnpm 下可能有问题（不过现在大部分主流工具都兼容了）。
  3. 学习成本：符号链接 + .pnpm 目录，理解起来比扁平化稍复杂。

为什么叫"shamefully-hoist"

"shameful"（可耻的）这个词很有意思——pnpm 觉得扁平化导致幽灵依赖是不好的 practice，如果你非要用，那就是"shameful"的。所以那个选项叫 shamefully-hoist，带有一丝"你这样做我都替你害臊"的意味。

当然这是个玩笑话，实际项目中为了迁移方便，用 shamefully-hoist 也很正常。

总结：

  • npm 2：纯嵌套，重复多，路径长。
  • npm 3+ / yarn 经典：扁平化，减少重复，但引入幽灵依赖。
  • pnpm：符号链接 + 全局硬链接，严格、省空间、快，无幽灵依赖。

演进方向就是：从"空间换简单"到"简单换空间"再到"用链接两者兼得"。
