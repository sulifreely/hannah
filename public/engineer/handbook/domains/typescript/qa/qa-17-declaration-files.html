问答 17（进阶）：声明文件（.d.ts）编写与发布 ​

题目 ​

请解释 TypeScript 声明文件（.d.ts）的作用和编写方法：declare 关键字怎么用？什么是 declare global？模块扩充（Module Augmentation）是什么？三斜线指令有什么用？DefinitelyTyped 是什么？如何在 package.json 中配置 types 和 typesVersions？

参考答案 ​

声明文件是干什么的

JS 库本身没有类型信息，TS 怎么知道 jQuery 的 $ 有哪些方法？靠的就是声明文件（.d.ts）。

.d.ts 文件只包含类型声明，不包含实现（没有运行时代码）。它描述了一个模块/库的"形状"——有哪些函数、类、对象、接口，它们的类型是什么。

可以把 .d.ts 理解成"类型层面的头文件"——告诉编译器"有这些东西存在，它们长这样"。

declare 关键字

declare 用来"声明"某个东西的存在，但不提供实现。它可以修饰：

变量 / 常量 ​

ts

全局声明 vs 模块声明

.d.ts 文件有两种模式：

模式特征作用域
全局声明没有 import / export全局可见，所有文件都能用
模块声明有 import / export模块作用域，需要 import 才能用
ts
// global.d.ts —— 全局模式（没有 import/export）
declare var __APP_VERSION__: string;

// utils.d.ts —— 模块模式（有 export）
export interface User { id: string; name: string }

注意：只要文件里出现了 import 或 export，就自动变成模块模式，里面的 declare 不再是全局的。想在模块模式下声明全局的东西？用 declare global。

declare global：模块里加全局声明

如果你在一个模块文件（有 import/export）里，想声明全局的类型或值，用 declare global：

ts
// 这是一个模块文件（有 import）
import { SomeType } from './some-module';

declare global {
  interface Window {
    myApp: {
      version: string;
      start(): void;
    };
  }

  function myGlobalFunction(): void;
}

// 不导出点东西，这个文件就不是模块了
export {};

模块扩充（Module Augmentation）

模块扩充可以给已有模块添加新的类型声明。比如你用了一个第三方库，想给它加点自定义方法，就要扩展它的类型。

ts
import { StringDecoder } from 'string_decoder';

declare module 'string_decoder' {
  interface StringDecoder {
    // 给 StringDecoder 加一个新方法
    decodeAll(buffer: Buffer[]): string;
  }
}

和 declare global 的区别：

  • declare global：扩展全局作用域
  • declare module 'xxx'：扩展某个具体模块

模块扩充的常见场景：

  • 给第三方库打类型补丁（加方法、加属性）
  • Vue 插件给 Vue 实例加属性
  • Express 中间件给 Request / Response 加字段

三斜线指令（Triple-Slash Directives）

三斜线指令是写在 .d.ts 文件顶部的特殊注释，用来告诉编译器"这个文件依赖什么"。

ts
/// <reference path="./other.d.ts" />
/// <reference types="node" />
/// <reference lib="es2020" />
1
2
3
指令作用
/// <reference path="..." />引入另一个声明文件（路径引用）
/// <reference types="..." />引入某个包的类型（包引用）
/// <reference lib="..." />引入 TS 内置库（比如 es2020、dom）

在现代 TS 项目中，三斜线指令用得越来越少了——大部分场景可以用 tsconfig.json 的 types、lib、include 来代替。但在 .d.ts 文件的头部，/// <reference types="node" /> 这种写法仍然常见。

DefinitelyTyped 是什么

DefinitelyTyped 是社区维护的最大的 TS 类型声明仓库，地址是 github.com/DefinitelyTyped/DefinitelyTyped。

里面有几千个 JS 库的类型声明，对应 npm 上 @types/xxx 的包——比如：

  • @types/react → React 的类型
  • @types/node → Node.js 的类型
  • @types/lodash → Lodash 的类型

为什么叫 DefinitelyTyped？ 因为它的目标是"确定性的类型"——让那些没有类型的 JS 库，也能有确定的 TS 类型定义。

package.json 中的类型配置

  "main": "dist/index.js",
  "types": "dist/index.d.ts"

字段作用
types / typings指定入口类型声明文件的路径（两者等价，types 更常用）
main指定 JS 入口文件（运行时用的）

没有 types 字段会怎样？ TS 会自动找 main 同目录下同名的 .d.ts 文件。比如 main: "index.js"，就会找 index.d.ts。但显式写 types 更清晰可靠。

typesVersions：按 TS 版本分发不同类型

TS 4.1+ 支持 typesVersions，可以根据用户的 TS 版本提供不同的类型声明。当你用了新的 TS 特性（比如模板字面量类型），又想兼容旧版本 TS 的用户时，就会用到。

json
{
  "name": "my-lib",
  "version": "1.0.0",
  "types": "./dist/index.d.ts",
  "typesVersions": {
    "<4.1": { "*": ["dist/ts4.0/*"] }
  }
}

  • TS 4.1 以下：用 dist/ts4.0/ 里的类型（降级版本）

这是库作者的高级玩法，业务开发一般用不到。

写声明文件的最佳实践

  1. 优先用 TS 写库，自动生成 .d.ts：源码就是 TS，编译时开 declaration: true 自动生成声明文件，最不容易出错。
  2. 手写声明要写测试：写一个 .test-d.ts 文件，实际用一下你的类型，确保声明是对的。可以用 tsd 或 expect-type 做类型单元测试。
  3. 导出的类型要精简：不要导出内部实现细节，保持公共 API 简洁。
  4. 遵循 DefinitelyTyped 规范：如果要给 DefinitelyTyped 贡献类型，要遵循它们的规范（dtlint 检查）。

追问 1：interface 的"声明合并"和"模块扩充"有什么关系？什么是声明合并？ ​

答 ​

声明合并（Declaration Merging）是 TS 的一个特性：如果你多次声明同一个名字的 interface（或 namespace），TS 会把它们合并成一个。

ts
// 第一个声明
interface User {
  id: string;
}

// 第二个声明——会和上面的合并
interface User {
  name: string;
}

// 最终效果：interface User { id: string; name: string }
const u: User = { id: '1', name: 'Alice' }; // ✅

这就是为什么"模块扩充"能工作——你 declare module 'xxx' 里声明的 interface，会和原模块里同名的 interface 合并在一起。

ts
declare module 'express' {
  interface Request {
    userId?: string;  // 给 Express 的 Request 加字段
  }
}

2
3
4
5

这里 express 模块里本来就有 Request 接口，你在模块扩充里再写一个，TS 会把它们合并。

声明合并的规则：

  • interface 和 interface 可以合并
  • namespace 和 namespace 可以合并
  • class 不能合并（不能声明两个同名 class）
  • type 不能合并（不能声明两个同名 type alias）
  • interface 和 namespace 也可以合并（但比较少见）

声明合并 vs 模块扩充：

  • 声明合并是"机制"
  • 模块扩充是"用这个机制去扩展第三方模块的类型"

追问 2：什么时候该把类型声明放在 .d.ts 里，什么时候放在普通 .ts 文件里？ ​

答 ​

这是个很实际的工程问题。简单的判断标准：

场景放在哪里
描述 JS 代码的类型（第三方库、老代码）.d.ts
全局类型 / 全局变量声明global.d.ts
你的业务代码本身就是 TS普通 .ts 文件里，直接 export interface
类型和实现绑定在一起就在实现的 .ts 文件里定义
类型需要跨项目共享单独的 types/ 目录，或发布独立的 types 包

几个常见误区：

❌ 误区 1："所有类型都要放 .d.ts 里才专业" 不对。TS 项目里类型就写在 .ts 文件里，和代码在一起。.d.ts 是给"没有 TS 源码"的场景准备的。

❌ 误区 2：".d.ts 里可以写逻辑代码" 不行。.d.ts 里只能有类型声明，不能有实现（不能有函数体、不能有 const x = 1 这种值的定义）。

❌ 误区 3："全局类型越多越好" 全局类型会污染命名空间。能用模块 import 的就别放全局。只有真正的"全局变量"（比如 window 上的属性）才适合放全局。

最佳实践：

  • 业务代码：类型和实现放一起，用 export 导出
  • 通用类型：放在 src/types/ 目录下，按需 import
  • 全局声明：集中在 src/types/global.d.ts，保持精简
  • 给 JS 写类型：写对应的 .d.ts 文件

追问 3：手写 .d.ts 和自动生成的 .d.ts 各有什么优缺点？什么情况下需要手写？ ​

答 ​

自动生成（tsc --declaration）：

优点缺点
不会和实现不一致（同源）可能导出过多内部类型，不够干净
维护成本低（改代码自动同步）复杂的泛型/重载可能推断得不够精确
有实现支撑，可信度高对 JS 库来说用不了

手写 .d.ts：

优点缺点
可以精确控制对外暴露的类型 API容易和实现不一致，维护成本高
可以给纯 JS 库加类型手写容易有 bug，需要测试
可以做"比实现更精确"的类型（比如函数重载）复杂类型写起来费时间

什么时候需要手写？

  1. 给纯 JS 库写类型：没有 TS 源码，只能手写。
  2. 类型比实现更"精确"：比如实现里是 any，但你想对外暴露精确的重载签名。
  3. 需要隐藏内部类型：自动生成会把内部类型都导出来，手写可以只暴露公共 API。
  4. DefinitelyTyped 贡献：给第三方库提交类型，必须手写。

一个实用建议： 如果你的库是用 TS 写的，优先自动生成，然后用 @internal 注释标记内部类型，用 stripInternal: true 在生成时去掉。这样既保证了一致性，又能控制对外 API。

如果发现自动生成的类型不够好（比如推断太宽、重载不够精确），再考虑手写一部分，或者用 export type 手动重新导出精简后的类型。
