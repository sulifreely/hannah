问答 14（进阶）：类型守卫与类型断言 ​

题目 ​

请系统梳理 TypeScript 中"收窄类型"和"断言类型"的各种手段：typeof / instanceof / in / 自定义类型守卫（is 关键字）/ 断言函数（asserts）/ as const / 非空断言（!）/ 双重断言，分别说明它们的用法、适用场景和注意事项。

参考答案 ​

TS 的类型系统是"静态的"，但 JS 的值是"动态的"。当我们有一个"宽类型"的值，需要把它当"窄类型"来用时，有两类手段：

  • 类型守卫（Type Guard）：通过运行时检查，让 TS 自动把类型收窄——安全，推荐优先用。
  • 类型断言（Type Assertion）：你告诉 TS"相信我，它就是这个类型"——跳过检查，慎用。

一、类型守卫（Type Guard）

类型守卫是"运行时检查 + 编译期收窄"的组合，是最安全的收窄方式。

1. typeof：原始类型守卫 ​

ts
function foo(x: string | number) {
  if (typeof x === 'string') {
    x.toUpperCase(); // x 收窄为 string
  } else {
    x.toFixed(2);    // x 收窄为 number
  }
}

2. instanceof：类实例守卫 ​

ts
function print(x: Date | Error) {
  if (x instanceof Date) {
    x.toISOString();    // x 收窄为 Date
  } else {
    x.message;          // x 收窄为 Error
  }
}

3. in：属性存在性守卫 ​

ts
type Cat = { meow(): void };
type Dog = { bark(): void };

function speak(animal: Cat | Dog) {
  if ('meow' in animal) {
    animal.meow();       // 收窄为 Cat
  } else {
    animal.bark();       // 收窄为 Dog
  }
}

4. 可辨识联合（Discriminated Union） ​

其实不是新的守卫，而是一种设计模式——给联合的每个成员加一个共同的"标签字段"，配合 switch 做穷尽收窄：

ts
type Shape =
  | { kind: 'circle'; r: number }
  | { kind: 'rect'; w: number; h: number };

function area(s: Shape) {
  switch (s.kind) {
    case 'circle': return Math.PI * s.r ** 2;
    case 'rect':   return s.w * s.h;
  }
}

5. 自定义类型守卫：x is T ​

当内置守卫不够用时，自己封装判断逻辑，返回值类型写成 x is T：

ts
interface Fish { swim(): void }
interface Bird { fly(): void }

function isFish(pet: Fish | Bird): pet is Fish {
  return (pet as Fish).swim !== undefined;
}

function move(pet: Fish | Bird) {
  if (isFish(pet)) {
    pet.swim();    // 收窄为 Fish
  } else {
    pet.fly();     // 收窄为 Bird
  }
}

6. 断言函数：asserts x is T ​

TS 3.7 引入。如果一个函数"不抛错就意味着条件成立"，可以用 asserts：

ts
function assertString(x: unknown): asserts x is string {
  if (typeof x !== 'string') throw new TypeError('not a string');
}

function foo(x: unknown) {
  assertString(x);
  x.toUpperCase();  // x 收窄为 string
}

二、类型断言（Type Assertion）

类型断言是"我比编译器更清楚"的手动覆盖，不会做运行时检查，直接跳过类型校验。

1. 基础断言：as T ​

ts
const input = document.getElementById('email') as HTMLInputElement;
input.value; // 直接用，不检查
1
2

2. 非空断言：! ​

告诉 TS"这个值肯定不是 null / undefined"：

ts
function getLength(s: string | null) {
  return s!.length;  // 我保证 s 不是 null
}
1
2
3

! 就是 . 前面的断言，等价于 (s as string).length。

3. as const：字面量收窄 + 只读 ​

as const 作用于值表达式，效果有三个：

  • 对象/数组属性递归变 readonly
  • 字面量类型收窄到最精确（1 而不是 number）
  • 数组变成 readonly 元组
ts
const obj = { name: 'Alice', age: 30 } as const;
// obj: { readonly name: 'Alice'; readonly age: 30 }

const arr = [1, 2, 3] as const;
// arr: readonly [1, 2, 3]

和 Readonly<T> 的区别：Readonly<T> 是类型层面的，只加顶层 readonly，不收窄字面量；as const 是值层面的，递归+收窄+变元组。

4. 双重断言：as unknown as T ​

当两个类型之间没有继承关系，直接 as 会报错，可以先转成中间类型（通常是 unknown 或 any），再转成目标类型：

ts
const str = 'hello';
const num = str as unknown as number; // 先 unknown，再 number
1
2

警告：双重断言几乎等于"关掉类型检查"，非常危险。不到万不得已不要用。

三、使用原则

  1. 优先用类型守卫：能 typeof/instanceof/in 的，不用断言。
  2. 优先用 schema 校验：复杂数据用 zod/valibot/io-ts 等库，运行时校验 + 类型收窄一步到位。
  3. 断言是最后手段：只有你 100% 确定类型、且编译器无法推断时才用。
  4. as const 是好东西：配置、常量、枚举定义多用 as const，既能保留字面量精度，又不用重复写类型。

追问 1：is 和 asserts 除了"分支收窄 vs 作用域收窄"，还有什么深层区别？ ​

答 ​

深层区别在于函数的"语义"不同：

维度x is Tasserts x is T
返回值返回 boolean不返回有意义的值（返回 void）
语义"判断一下，是就 true，不是就 false""断言一下，不是就抛错，正常返回就说明是"
收窄位置if 分支内调用点之后的整个作用域
失败处理走 else 分支，程序继续抛错/终止，程序不往下走
典型场景条件判断、过滤、分支逻辑前置校验、参数检查、不变量维护

还有一个容易忽略的点：asserts 有一种简写形式 asserts value（不带 is），表示"如果函数正常返回，value 就是真值"：

ts
function assertNonNull<T>(x: T | null | undefined): asserts x is T {
  if (x == null) throw new Error('null');
}

// 简写版（只断言真值）：
function assertTruthy(x: any): asserts x {
  if (!x) throw new Error('falsy');
}

追问 2：as const 和 "枚举（enum）" 比，各自什么情况下用？ ​

答 ​

这是个很好的工程问题。TS 中表示"一组有限的可选值"有多种方式：

方案性质运行时产物优点缺点
as const 数组 + [number]值 + 类型真实数组有真实值、可遍历、字面量精确不能反向映射
字符串字面量联合 type T = 'a' | 'b'纯类型无轻量、不产生 JS 代码运行时拿不到所有值
字符串枚举 enum E { A = 'a' }类型 + 对象枚举对象有命名空间感、IDE 支持好有运行时代价、不能 tree-shake
数字枚举 enum E { A, B }类型 + 对象枚举对象 + 反向映射有自动递增、反向映射容易出意外的数字值、不安全

选择建议：

  1. 只是类型约束，不需要运行时遍历 → 用字符串字面量联合。最轻量。
  2. 需要在运行时拿到所有可选值（比如遍历生成下拉选项） → 用 as const 数组，再 typeof arr[number] 提取联合。一石二鸟。
  3. 有命名空间需求、或者团队习惯枚举风格 → 字符串枚举也可以，但要知道它有运行时体积。
  4. 数字枚举尽量别用：容易出现"魔数"问题，可读性差，也不安全。

as const + 数组/对象的模式在现代 TS 项目中越来越流行，因为它"一份定义，值和类型两用"，而且零额外成本。

追问 3：什么情况下你会用双重断言？有没有比双重断言更好的替代方案？ ​

答 ​

双重断言（as unknown as T）是"类型系统的逃生舱"——理论上能把任何类型转成任何类型，但几乎等于放弃类型安全。

什么时候真的需要？

  1. 和动态类型的 JS 库交互：旧库没有类型定义，返回值类型混乱，你又比 TS 更清楚实际返回什么。
  2. 测试 / mock 场景：写测试用例时构造假数据，不想满足完整类型。
  3. 类型系统的"已知盲区"：比如某些复杂的类型体操结果，TS 推断不出来，但你逻辑上能证明是对的。

有没有更好的替代？

替代方案适用场景为什么更好
写类型守卫 is能写运行时判断的有运行时检查，安全
schema 校验（zod 等）外部数据（API、表单）运行时校验 + 类型推导，最安全
泛型 + 重载函数返回值不确定在类型系统内解决，保留类型安全
改进数据结构设计类型设计本身有问题从根源消除"需要断言"的需求
satisfies需要约束但不丢失字面量见 qa-15，比断言更安全

工程原则：代码 review 时遇到 as unknown as 应该多问一句"有没有更安全的写法？"——双重断言是代码的"坏味道"，偶尔用一下可以，但到处都是说明类型设计有问题。
