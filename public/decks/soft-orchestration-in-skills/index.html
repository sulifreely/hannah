Conference Talk · Agent Skill 设计

Skill 中的
软编排

当执行者是一个概率模型时，我们如何给它一条
有检查点、有护栏、但依然轻量的路线。

一个天天在用，却很少被追问的问题

我们每天都在用 Skill，甚至自己在写 Skill —— 但有没有真正拆开看过，它里面到底藏着哪些结构、哪些模式？

今天想和大家一起，把这层"结构"挖出来看看。

01

两种编排，两种重量

硬编排
  • 状态机 / DAG / workflow engine
  • 可视化节点，拖拽调度
  • 把每一步焐死成代码节点
软编排
  • 写进 Skill 的操作手册
  • 顺序、分叉、检查点、退出条件
  • 把行动边界写清楚，而非焐死
一个复杂任务，应该按什么顺序、以什么边界、由谁来完成？

编排要回答的，就是这一句话。任务足够简单时不需要它；任务一复杂，就会长出岔路。

02

今天想聊三件事

Skill = 指令 + 编排模式 + 工具协议 + 验证门禁
  • 常见的六类编排模式：链式、并行、路由、循环、编排、层级
  • 这些模式在 Skill 里怎么落地表达
  • 执行者是概率模型时，如何尽量保障结果确定
03

六种常见编排模式

01Chain 链式上一步输出是下一步输入
02Parallel 并行子任务互不依赖，同时收集
03Route 路由不同输入走不同路径
04Loop 循环执行、验证、修正、收敛
05Orchestrate 编排中心协调者拆任务、做判断
06Hierarchy 层级父任务拆子任务，逐层收敛
04 · Chain

链式 — 上一步的输出，是下一步的输入

理解需求
读取上下文
制定方案
执行修改
验证结果

关键不是编号本身，而是每一步的输入输出要清楚 —— 没看清楚火从哪冒出来，就急着往墙上浇水，只会把屋子弄得更湿。

参考 Skill → executing-plans↗
05 · Parallel

并行 — 没有强依赖，就同时做

主任务：理解模块
查入口
查模型
查测试
查改动
统一格式，汇总上下文

并行的收益是快，风险是散。Skill 要把"散"收回来：各查各的边界，但按同一种格式把证据交回来。

参考 Skill → dispatching-parallel-agents↗ code-review↗
06 · Route

路由 — 不同输入，走不同路径

用户请求
实现
调试
Review
写作
  • 好的路由规则有三层：何时用、何时不用、进入后怎么细分
  • 条件一模糊，模型就会凭语感选路 —— 路牌不能只写"差不多往那边"
参考 Skill → prototype↗ triage↗
07 · Loop

循环 — 执行、验证、修正，直到收敛

执行
验证
通过？
结束
↺ 不通过：分析失败原因、更新假设，再回到"执行"
  • 一定要写清楚退出条件，否则会烧干上下文和耐心
  • 最多重复 N 次，每轮都要留下判断记录，不是瞎撞
参考 Skill → loop-on-ci↗ tdd↗ diagnosing-bugs↗
08 · Orchestrate

编排 — 谁来协调，谁做最终判断

编排者：定义边界 · 保留最终判断
读代码
调搜索
子 Agent
跑测试

和并行的区别：Parallel 关注"能不能同时做"，Orchestrate 关注"谁来决定下一步"。工具和子 Agent 能搬砖，但房子歪没歪，得有人抬头看一眼。

参考 Skill → subagent-driven-development↗
09 · Hierarchy

层级 — 父任务拆子任务，子任务还能再拆

父任务：迁移组件体系
梳理现状
设计目标 API
实现脚本
补测试文档
  • 子任务必须带明确的输入、输出、边界
  • 父任务必须在关键节点重新汇总，不能一路放飞
参考 Skill → orchestrate↗ to-issues↗
10

在 Skill 里，编排可以拆成五层

L1触发条件何时进入
L2阶段定义拆出可见步骤
L3模式嵌入注入编排规则
L4工具协议用证据代替猜测
L5交付格式约束最终产物

从上到下，逐渐从"要不要做、做哪些步骤"收敛到"怎么做、交付什么"。

11

一个软编排 Skill 的最小模型

## 什么时候使用 当……时使用；当……时不要使用 ## 工作流 1. 判断任务类型 # 路由 2. 收集上下文 # 并行 3. 规划 → 执行 → 验证 # 链式 / 循环 4. 带着证据总结 ## 输出 改了什么 · 验证了什么 · 还有什么不确定
12

工作流什么时候该停下来问用户？

自己查，别打扰
  • 能用读取 / 搜索 / 命令自行确认
  • 只是措辞、格式这类低风险选择
  • 用户已经明确给出方向
该问，就给选择题
  • 存在多个合理路径，明显影响产物
  • 即将执行高影响副作用动作
  • 给 2-4 个选项，推荐项放第一位
13

概率模型，也能有确定的落点

语言表达
允许自由变化
决策过程
尽量稳定
外部动作
必须可控
交付结果
必须可验证

越往右，越不能靠模型"自由发挥"。我们不是把风变成石头，而是给帆、舵和锚各自安排好位置。

14

六条压住"确定性光谱"右侧的做法

1显式状态别让判断只藏在模型心里
2受限选择把开放生成变成选择题
3工具校验用证据替代模型的自信
4循环预算设上限，强制更新假设
5副作用门禁高风险动作集中管理
6交付检查清单简单、便宜、有效的最后一道门
15

软编排不是万能药

适合软编排
  • 日常 AI Coding 操作规范
  • 写作、Review、调研、排障
  • 半结构化：目标明确，输入多变
更适合重型系统
  • 多小时运行的数据管道
  • 涉及资金、权限、生产发布
  • 高风险自动化闭环
16

如果只记住三件事

  • 编排的本质是给复杂任务建立结构 —— Chain 处理顺序，Parallel 处理并发，Route 处理分支，Loop 处理迭代，Orchestrate 处理协调，Hierarchy 处理拆解
  • 在 Skill 中实现编排，不一定要写成 DAG —— 把触发条件、阶段、分支、循环退出、工具协议、交付格式写清楚就够了
  • 概率模型不可能天然确定，但可以让关键过程尽量确定：显式状态、受限选择、工具校验、循环预算、副作用门禁、检查清单
Skill 不是魔法咒语，而是一份认真写过的团队协作规范 —— 只不过团队里多了一个很会写、很会猜，偶尔也会一本正经胡说八道的 Agent。
