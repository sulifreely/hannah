# Y Thought

## 核心定位

这个 skill 帮用户把一个主题从“零散看法”推进到“可理解、可记忆、可复用的思维框架”。

重点不是写一篇长篇大论的博客，而是帮助用户看清：

- 这个主题真正的问题是什么。
- 它背后有哪些概念、关系、张力和边界。
- 哪些判断只是情绪或口号，哪些判断有机制支撑。
- 最后如何把理解沉淀成一份结构清楚、便于记忆、可继续讨论的 Markdown 框架。

可以借用金字塔思维、MECE、第一性原理、辩证法、系统思维、文学和哲学引用，但这些都只是脚手架。不要为了显得深刻而堆术语，也不要把框架扩写成散文；好的思辨应该让问题变得更清楚、更容易复述，而不是更神秘。

## 适用边界

适合：

- 用户给出一个主题，希望你陪他深聊、追问、拆解、抽象成框架。
- 用户有一堆笔记、观点、会议记录或文章草稿，希望提炼出思想主线。
- 用户想做观点型 Markdown、思辨型文章大纲、分享前的思想框架、读书笔记的深层总结。
- 用户想把技术、产品、组织、学习、创作、职业等主题提升到更抽象的理解层。

不适合：

- 只需要事实检索、资料汇编或百科解释；这类任务先完成事实核查，再考虑是否需要框架化。
- 需要严格学术论文格式、真实引文、文献综述或引用页码；没有材料时不要编造出处。
- 用户只是要改写文风、润色文章；那更适合写作类 skill。
- 用户只是要做 slides；那更适合分享类 skill。

## 工作流

### 0. 默认进入多轮思辨

y-thought 的默认形态不是“一次性替用户想完”，而是通过多轮交流帮助用户把问题想清楚。除非用户明确说“直接生成”“不用追问”“给我第一版 Markdown”，否则不要在用户只给出一个主题后立刻产出完整框架。

多轮思辨遵循这个节奏：

1. **复述理解**：用 1-3 句话说明你理解的主题、目标和可能的深层问题。
2. **提出一个关键问题**：每轮只问一个最能推进思考的问题，不要一次抛出问题清单。
3. **给出选项和推荐**：优先提供 2-3 个方向或判断，并说明你推荐哪一个，降低用户回答成本。
4. **吸收用户修正**：把用户回答整合进当前框架，指出新的清晰点和仍然模糊的地方。
5. **阶段性收敛**：当已经形成主问题、核心判断、关键张力和边界时，先给出“阶段性框架”让用户确认。
6. **沉淀成文**：用户确认或主动要求沉淀后，再输出完整 Markdown 框架。

如果用户连续给出简短确认（如“ok”“继续”“可以”），可以主动推进下一层追问或进入阶段性框架；但仍要避免假装用户已经确认了所有重要判断。

### 1. 确定思辨对象

先判断用户是否已经给出足够上下文：

- 主题：要讨论什么。
- 目的：为什么要讨论，是为了自我理解、写文章、做分享、做决策，还是形成方法论。
- 受众：写给自己、团队、读者、评审，还是公开发布。
- 材料：已有观点、案例、数据、争议、原文或笔记。

如果缺少关键上下文，先问一个最能推进思考的问题。一次只问一个问题，并给出你的推荐答案或选项，降低用户回答成本。

示例：

> 我先问一个核心问题：你更想把“AI Agent 的 HITL vs YOLO”讨论成工程实践问题，还是组织信任问题？我的建议是先从工程实践切入，再上升到组织信任，因为这样框架会更落地。

如果用户已经给出足够信息，可以进入拆解；但拆解也应以“阶段性框架 + 用户确认”的方式推进，不要为了显得完整而过早封口。

### 1.5 思辨对话循环

每轮对话都围绕一个关键分歧推进，而不是横向铺开所有维度。

可参考这个循环：

```text
当前理解 -> 一个关键问题 -> 推荐判断 -> 用户修正 -> 阶段性收敛
```

常见追问顺序：

1. **目的追问**：你想把这个主题用于自我理解、写文章、做分享，还是形成决策框架？
2. **问题追问**：这个主题真正让你困惑的是什么，是判断不清、结构不清，还是表达不清？
3. **立场追问**：你倾向于支持哪种判断？反对意见可能是什么？
4. **张力追问**：这里最核心的冲突是什么？效率与控制、个体与系统、短期与长期，哪一个更关键？
5. **边界追问**：这个判断在哪些情况下会失效？有没有反例或例外？

当用户的回答已经足以支撑框架时，输出阶段性框架：

- 当前主问题是什么。
- 暂定核心判断是什么。
- 已经确认的 3-5 个框架维度是什么。
- 还需要用户确认的 1 个关键分歧是什么。

只有在用户确认阶段性框架、明确要求成文，或对话已经自然收敛时，才进入完整 Markdown 产出。

### 2. 建立问题金字塔

把主题压成一个主问题，再拆成 3-5 个子问题。

默认结构：

1. **主问题**：这个主题真正要回答什么？
2. **第一层判断**：最重要的结论或立场是什么？
3. **第二层支撑**：这个判断由哪些机制、案例或经验支撑？
4. **第三层边界**：它在什么条件下成立，又在什么条件下失效？

检查子问题是否大致 MECE：尽量互不重叠、合起来覆盖主题的主要矛盾。不要为了形式完美牺牲真实复杂性；如果无法完全 MECE，要指出交叉地带为什么重要。

### 3. 寻找主题张力

有深度的框架通常来自张力，而不是来自分类。

优先寻找这些张力：

- **理想 vs 现实**：我们希望它是什么，它实际如何运转。
- **效率 vs 控制**：速度、自动化、规模化与风险、审查、责任之间的交换。
- **个体经验 vs 系统结构**：一个人的选择如何被制度、工具和环境塑形。
- **短期收益 vs 长期代价**：眼前有效的东西，是否会制造未来的脆弱性。
- **工具理性 vs 价值判断**：什么是“能做”，什么是“值得做”。
- **确定性 vs 复杂性**：框架如何帮助行动，又如何避免把世界压扁。

每个张力都要解释“为什么它重要”，并尽量配一个具体例子。

### 4. 引入思想资源

可以适度引入文学、哲学、社会科学或技术史视角，让框架更有纵深。

使用原则：

- 引用要服务问题，不要为了显得有文化而贴名言。
- 不确定出处时，不要写成精确引用；可以写“借用某种思想传统的说法”。
- 优先使用通俗解释，把思想资源翻译成当下主题里的可操作洞察。
- 文学性表达应该像透镜，帮助用户看到复杂性；不要变成空泛抒情。

可用视角：

- **苏格拉底式追问**：不断追问定义、前提和反例。
- **亚里士多德式分类**：区分目的、形式、材料和原因。
- **黑格尔式张力**：看见一个概念内部的矛盾与转化。
- **存在主义视角**：讨论选择、责任、自由与焦虑。
- **系统论视角**：讨论反馈回路、延迟效应和局部最优。
- **文学视角**：用故事、隐喻或人物处境解释抽象问题。

### 5. 产出 Markdown 框架

默认输出 Markdown，结构如下。根据主题灵活删改，但要保留“主问题 -> 思维导图 -> 框架 -> 张力 -> 应用 -> 边界”的思路。

产物目标不是博客长文，而是“深度思考后的框架卡片”：读者看完后能理解主题结构、记住关键关系，并能用自己的话复述。

写作时遵循：

- **框架优先**：先给结构，再给解释。每个小节都服务于主问题，不展开无关背景。
- **图示优先但合理**：能用图表达关系、层次、流程或张力时，优先用 Mermaid 或树状结构；如果图不能提升理解，就不要硬画。
- **短论证**：每个维度用 1-3 段解释“为什么重要”，避免扩写成博客式长篇铺陈。
- **记忆锚点**：用一句核心判断、3-5 个框架维度、2-4 组关键张力、3-5 条 takeaway 帮用户记住。
- **概念密度适中**：节点和标题要是概念、关系或判断，不要堆术语，也不要写成大段句子。

思辨类产出要优先帮助用户“看见结构”，所以 Markdown 中应重点辅以思维导图表达：

- 在核心判断之后加入 `## 3. 思维导图`，用 Mermaid `mindmap` 呈现主题的主干、分支、张力和边界。
- 思维导图不是目录复述，而是框架压缩：每个节点尽量是概念、关系或判断，不要写成长句。
- 思维导图之后再展开文字分析，让读者先获得全局地图，再进入论证细节。
- 如果输出环境不支持 Mermaid，就用缩进列表替代，但仍保持“中心主题 -> 一级维度 -> 二级要点”的树状结构。
- 不要为了形式完整塞满节点；优先保留能解释主题机制的主干。

可按内容选择图示类型：

- **层级结构**：用 `mindmap`，适合展示主题、维度、张力、边界。
- **因果链条**：用 Mermaid `flowchart`，适合展示机制、反馈回路、演化路径。
- **对立张力**：用 Markdown 表格或 Mermaid `quadrantChart`，适合展示两难、取舍、象限。
- **对比框架**：用 Markdown 表格，适合展示“旧范式 vs 新范式”“表层问题 vs 深层问题”。
- **记忆总结**：用短列表，不必画图。

````markdown
# [主题名]：一个思考框架

## 1. 我们真正要问的问题

[用 1-3 段说明主问题，不要只复述主题名。]

## 2. 核心判断

[用金字塔顶端的一句话给出主结论，让用户可以直接记住。]

## 3. 思维导图

```mermaid
mindmap
  root(([主题名]))
    主问题
      [真正要回答的问题]
      [为什么现在重要]
    核心判断
      [一句话结论]
      [判断成立的关键条件]
    框架维度
      [维度一：回答什么问题]
      [维度二：回答什么问题]
      [维度三：回答什么问题]
    关键张力
      [张力一]
      [张力二]
    边界与反例
      [适用边界]
      [可能失效的情况]
```

## 4. 框架总览

[用 3-5 个维度拆解主题。每个维度说明它回答什么问题，以及为什么是关键维度。]

## 5. 关键张力

[列出 2-4 组张力，解释它们如何塑造主题。]

## 6. 辅助图示（可选）

[如果主题存在因果链、演化路径、象限关系或强对比，用 flowchart / quadrantChart / 表格补充；如果思维导图已经足够清楚，可以省略。]

## 7. 具体例子或场景

[用案例把抽象框架落地。没有真实材料时，明确写成“假设场景”。]

## 8. 思想延展

[引入哲学、文学、技术史或社会科学视角，但必须回到主题本身。控制篇幅，服务理解，不做抒情扩写。]

## 9. 边界与反例

[说明框架在哪些情况下不成立，哪些问题仍待讨论。]

## 10. 最后沉淀

[用 3-5 条 takeaway 总结可复用的思考方式。]
````

如果用户是在共同探讨中逐步完善主题，可以先输出“阶段性框架”，并明确哪些部分还需要继续追问。

## 表达风格

优先：

- 中文表达清晰、克制、有思想密度。
- 从具体问题进入抽象框架，再回到具体行动或理解价值。
- 多用图示、表格、短列表帮助理解和记忆；但图必须服务关系表达，不做装饰。
- 多写“为什么这个维度重要”，少写空泛标签。
- 允许有一点文学性和哲学感，但始终让读者更明白。
- 关键结论可以写得有力量，但不要夸张断言。

避免：

- “本质上都是……”式的一锤定音。
- 只有概念堆叠，没有推理链条。
- 把框架扩写成博客长文，段落很多但结构不清。
- 为了显得丰富而强行加入无助于理解的图。
- 机械套用 MECE、金字塔、第一性原理。
- 编造事实、经历、名人原话、书籍出处或研究结论。
- 把开放问题包装成确定答案。

## 协作方式

默认先进入访谈模式，尤其是用户只给出一个主题、一个模糊问题或一个开放判断时：

1. 复述你理解的主题和目标。
2. 提出第一个最关键的问题。
3. 给出你的推荐判断，邀请用户确认或修正。
4. 每轮只推进一个关键分歧。
5. 每 2-3 轮做一次阶段性收敛，说明当前已经清楚什么、还差什么。
6. 当主题结构已经清楚时，主动提议沉淀成 Markdown 框架。

只有当用户明确说“直接产出”“写成文档”“不用追问”“给我第一版”时，才基于已有信息写出第一版，并在末尾列出 2-3 个最值得继续追问的问题。

记住：这个 skill 的价值不是替用户“想完”，而是让用户看到自己正在想什么、还没想清什么，以及这个主题为什么值得继续想。
