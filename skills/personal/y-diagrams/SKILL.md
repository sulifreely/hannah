# Y Diagrams

## 核心定位

y-diagrams 负责把技术文章里的复杂概念变成清晰、可读、可维护的图示。

它不负责写完整文章，也不负责生成装饰插画。它只回答一个问题：这张图要解决哪个理解问题，以及用哪种图最省力地解决它。

默认原则：图 > 表 > 文，但图必须服务理解。不能让读者更快看懂的图，就不要画。

## 输入判断

先判断图示任务属于哪一类：

- **直觉模型图**：解释一个抽象概念的心智模型。
- **流程图**：解释步骤、调用链、数据流、状态流转。
- **架构图**：解释模块边界、依赖关系、分层和职责。
- **对比图**：解释旧方案 vs 新方案、错误模型 vs 正确模型。
- **风险图**：解释风险点、兜底链路、观测点和恢复路径。

如果用户只说“帮我画个图”，先根据上下文给出推荐图型和理由；只有缺少主题、读者或目标时，才问一个简短问题。

## 绘图流程

1. **定问题**：用一句话写清楚这张图要让读者理解什么。
2. **定图型**：在模型图、流程图、架构图、对比图、风险图里选一种主图型。
3. **定载体**：默认白板感用 Excalidraw；精确时序、复杂分支、源码级可审阅图用 Mermaid；小模型可用 ASCII。
4. **控复杂度**：一张图只讲一个主问题。节点超过 8 个、连线超过 12 条，优先拆成多张图或改成表格。
5. **写图前后文**：图前说明为什么需要这张图，图后解释从图里能看到什么。
6. **验图**：检查节点是否短、关系是否清楚、箭头是否不乱穿、图是否真的比文字更清楚。

完成标准：读者不看正文长段，也能从图名、节点和图后解释里复述主链路。

## 载体选择

优先级不是固定审美，而是表达匹配：

- **Excalidraw**：直觉模型、概念关系、旧流程 vs 新流程、8 个节点以内的主流程、白板讲解气质。
- **Mermaid**：精确时序、复杂分支、跨模块数据流、需要源码级可审阅和可复制的图。
- **ASCII 简图**：只需要快速表达一个小模型，不值得单独落图。
- **表格**：方案对比、指标解释、风险清单、角色分工。不要把表格更清楚的内容硬画成图。

不要手搓 SVG/HTML 图。它们维护成本高，也容易出现箭头乱穿、文本错位和主题适配问题。

## Excalidraw 流程

只有确定需要产出 Excalidraw 文件时，读取并执行 [excalidraw-diagrams.md](references/excalidraw-diagrams.md)。

如果 Excalidraw 环境不可用，先说明原因并给用户两个选择：安装 Excalidraw 相关 skill，或退回 Mermaid。用户明确选择 Mermaid 后，再退回 Mermaid。

## Mermaid 规则

退回 Mermaid 时：

- 直接在 Markdown 正文里使用 `mermaid` 代码块，不落 `.mmd` 文件。
- 图前说明为什么用 Mermaid，例如“这里分支和时序比较精确，用 Mermaid 比手绘草图更不容易产生歧义”。
- 节点和边的标签保持短，超过一行的解释放到图后正文。
- 中文、标点、`<br/>` 换行或 emoji 节点用双引号包起来，避免解析错误。

## 资产落盘

当任务会写入本地文章、博客或文档目录时：

- Excalidraw 产物放到文章同级或约定的 `assets/` / 图片目录中，保留 `.excalidraw` 和渲染后的 `.png`。
- 文件名使用语义化短横线，例如 `old-vs-new-flow.png`、`agent-state-model.excalidraw`。
- Markdown 正文只保留图片引用、图名前后的解释和必要图注，不内联 Excalidraw JSON、SVG 或 HTML。
- 临时草图不要落进最终文章目录；先用图片占位说明。

## 自检

交付前检查：

- 这张图解决了一个明确理解问题吗？
- 图型是否匹配内容，而不是为了好看？
- 节点是否足够短，长解释是否放回正文？
- 是否有图前引导和图后解释？
- Excalidraw 是否按 reference 完成环境检查、生成和视觉校验？
- Mermaid 是否可读、可复制，并避免中文解析问题？
