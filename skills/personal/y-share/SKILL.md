# Y Share

## 核心原则

把已有内容改造成适合演讲的 HTML slides，而不是把 Markdown 段落机械拆成页面。

必须自包含完成：不要要求再读取其它 slides/design skill。生成结果默认是一个可直接打开的单文件 HTML，CSS/JS 内联、零构建依赖、可键盘翻页、可投屏、可导出 PDF。

四条硬约束：

- **讲述优先**：先提炼主问题、听众、结论和节奏，再拆页。
- **一屏一意**：每页只讲一个观点；内容超限就拆页，不滚动、不硬塞。
- **设计有立场**：每套 slides 选择一个明确审美方向，避免通用 AI 感。
- **视口不溢出**：每张 `.slide` 必须刚好适配 100vh/100dvh，任何尺寸都不能需要页内滚动。

## 适用边界

适合：

- 用户给出一个 Markdown 文件、文章、技术笔记或方案文档，希望做成技术分享、读书分享、方案汇报、内部培训或可投屏演示。
- 用户已经有文章内容，但需要改成更适合口头讲述的节奏。
- 用户想要 HTML/web slides，而不是传统 `.pptx`。
- 用户需要一个可本地打开、可导出 PDF、可部署分享的演示页面。

不适合：

- 从零写文章内容；这类需要先向用户确认主题、听众、目标和素材，不要凭空生成事实。
- 创建复杂白板图或重绘架构图；需要时先询问用户是否要额外生成图源文件。
- 把 PowerPoint 转 HTML；除非用户明确给出 `.pptx`，否则不要走 PPT 转换流程。
- 读取、导出、创建、更新或发布外部文档平台内容；这些是文档平台操作，不属于 slides 创作本身。若输入来源是在线文档链接，先由对应文档 skill 取得 Markdown 或正文素材，再回到 y-share 制作 slides。

## 工作流

1. **归一化输入**：本地 Markdown、文章或用户提供的正文素材直接读取和整理。保留事实、代码、链接、图片引用、标题层级和附件信息；不要在本 skill 内执行在线文档读取、导出或发布操作。
2. **建立演讲线**：提炼主问题、目标听众、核心结论、3-5 个 section、每节 takeaway。不要逐段搬运原文。
3. **生成 slide outline**：默认按“封面 -> 为什么要听 -> 直觉模型 -> 关键机制 -> 案例/demo -> 取舍风险 -> 总结”组织；根据内容增删。
4. **处理素材**：图片用相对路径放入 `assets/`；复杂表格转卡片/对比页/分组列表；长代码只保留 8-10 行关键片段。
5. **规划图片丰富度**：判断哪些页面需要视觉资产来提升理解或记忆，例如封面 hero、章节过渡、机制解释图、场景插画、缩略图或社交分享图。
6. **选择视觉方向**：先回答“给谁看、解决什么问题、希望留下什么记忆点、有哪些技术约束”，再确定一个具体气质，并设计字体、色彩、背景、空间、动效和视觉隐喻。
7. **生成 HTML**：创建独立输出目录和 `index.html`，所有 CSS/JS 内联；需要图片时复制到 `assets/`，不覆盖原输入。
8. **验证交付**：确认本地可打开、键盘翻页可用、图片路径正确、投屏字号足够、1280x720 不溢出、中文排版自然、页面间距舒适、打印/PDF 预览可用。

## Slide 设计要求

- 一页只讲一个观点，标题必须像口头表达的句子，不要全是名词短语。
- 每 5-7 页设置一次节奏变化：问题页、对比页、图解页、代码页、引用页、总结页交替出现。
- 标题页最多 1 个标题 + 1 个副标题 + 1 个标签；内容页最多 4-6 个 bullet 或 2 段短文；网格最多 6 张卡片；代码页最多 8-10 行。
- 优先用视觉层级表达关系：大标题、卡片、时间线、左右对比、流程节点。
- 动效只做轻量渐入和当前页强调，不要做影响理解的炫技动画。
- 中文技术分享要亲切、清晰、有工程判断，但不要绑定某个写作 skill，也不要编造经历或数据。

## 视觉设计规则

不要生成默认模板感、通用 AI 感或“紫色渐变白底卡片”式页面。每次制作 slides 前，先根据主题选一个具体风格方向，例如工程白板、终端控制台、杂志社论、工业系统、复古实验室、极简黑板、柔和教学卡片、赛博网格、纸张索引卡等；不要只写“现代、简洁、科技感”。

先做 4 个设计判断：

- **Purpose**：这场分享解决什么问题？听众为什么要在此刻听？
- **Tone**：选择一个有边界的审美方向，可以极简、工业、复古未来、杂志社论、柔和教学、奢华克制、玩具感、原始粗粝，但不能停在“好看、科技感”。
- **Constraints**：投屏、性能、可访问性、离线打开、字体加载、图片路径和 PDF 导出会限制设计；视觉复杂度不能破坏这些底线。
- **Differentiation**：观众结束后记住的一个画面是什么？例如多工作台、分叉轨道、黑板推导、控制台雷达、纸质索引卡。

落地时重点检查：

- **字体**：选择有性格的标题字体与可靠正文字体组合，避免默认 Arial、Roboto、Inter 或纯系统字体堆叠。标题字体可以更有态度，正文字体必须清晰耐读。可用组合包括 Archivo Black + Space Grotesk、Cormorant + IBM Plex Sans、Fraunces + Work Sans、JetBrains Mono、Bodoni Moda + DM Sans。
- **色彩**：有主色、背景色和强调色，不平均撒颜色；主色承担气质，强调色只服务重点。强对比、克制双色、纸张暖色、终端绿、工业橙都可以，但必须和主题有关系。
- **空间**：允许非对称、留白、重叠、对角线动势、破格网格、受控密度；极简设计靠留白和对齐取胜，信息密集设计靠层级和分区取胜。
- **背景和细节**：使用纹理、渐变网格、几何线条、光效、边框、噪声、扫描线、纸张颗粒或分层透明营造氛围，避免纯色平铺的模板感。
- **动效**：用 CSS 渐入、上移、缩放、模糊、轻量视差或卡片 hover；一个编排良好的入场动画胜过到处乱动。尊重 `prefers-reduced-motion`，不因炫技导致滚动或溢出。
- **精修程度**：视觉复杂时，代码可以更 elaborate；视觉克制时，要把间距、对齐、字重、行高、边框透明度和阴影调细。美感来自执行精度，不来自元素堆叠。
- **记忆点**：每套 slides 至少有一个和主题强绑定的视觉隐喻，并在封面、章节页和总结页形成呼应。

可选风格库：

- **Bold Signal**：深色背景 + 高饱和强调卡片 + 巨大章节编号，适合自信结论型分享。
- **Electric Studio**：黑白/蓝分屏 + 精准排版，适合专业方案汇报。
- **Notebook Tabs**：纸张卡片 + 彩色索引标签，适合学习笔记和层级知识。
- **Terminal Green**：GitHub dark + 终端绿 + 等宽字体，适合工程工具、CLI、调试主题。
- **Swiss Modern**：黑白红 + 可见网格 + 非对称布局，适合抽象概念和系统性观点。
- **Paper & Ink**：暖纸色 + 衬线标题 + pull quote，适合读书笔记、复盘和观点型分享。

禁止模式：Inter/Roboto/Arial 做 display、`#6366f1` 泛紫色渐变、全居中 hero、同质卡片网格、无语义玻璃拟态、无目的阴影、每次都使用同一字体组合或同一配色。

## 中文排版增强

这个 skill 主要面向中文分享，HTML slides 应该内置中文排版最佳实践：优化中西文混排、照顾全角标点、控制标题断行、保证暗色主题下的中文可读性。不要为了排版引入额外构建流程；默认保持分享页可直接打开。

建议给主要内容容器加上 `heti` 或同等语义类名，方便集中管理中文排版规则，但不要强制依赖外部排版库：

```html
<main class="slides heti">
  <section class="slide">
    <div class="slide-content">
      <h1>为什么要把 Markdown 变成 Web Slides</h1>
      <p>让技术内容从“适合阅读”变成“适合讲述”。</p>
    </div>
  </section>
</main>
```

可以内联轻量中文排版 CSS：

```css
.heti {
  font-variant-east-asian: proportional-width;
  line-break: strict;
  overflow-wrap: break-word;
}

.heti h1,
.heti h2,
.heti h3 {
  text-wrap: balance;
  word-break: keep-all;
}
```

如果需要自动中西文混排美化，可以补充轻量 `autoSpacing()` 逻辑；遍历正文文本节点，在中文与英文/数字之间插入窄空格或普通空格。但不要处理 `code`、`pre`、`script`、`style`、URL 和文件路径，避免破坏代码与链接。

生成 CSS 时关注：

- **字体栈**：中文正文优先使用可靠中文字体栈，例如 `"Noto Serif SC"`, `"Source Han Serif SC"`, `"Noto Sans SC"`, `"PingFang SC"`, `"Microsoft YaHei"`，并和英文字体明确分工。
- **行高与字距**：中文正文行高通常比英文更需要呼吸感；投屏正文建议 `line-height: 1.45-1.7`，标题可更紧但不能压迫。
- **中西文混排**：中文与英文、数字、缩写、代码标识之间要有自然间隔；不要手工到处塞空格，优先用 CSS/轻量 JS 后处理统一规则。
- **全角标点**：注意中文引号、括号、顿号、冒号、破折号等标点的视觉挤压；标题末尾通常不需要句号。
- **代码与术语**：行内代码、API 名、文件名要用等宽字体和轻微背景区分，但不要破坏中文行高。
- **避头尾问题**：不要让中文标点孤立在行首；短标题尽量人工调整断句，避免一行只有一两个字。
- **深浅色适配**：暗色主题下中文细字重容易发虚，适当提高字重或对比度。

生成后必须检查：中英文和数字之间是否自然留白，标题是否没有奇怪断行，中文标点是否没有孤立在行首，暗色主题下细字重是否仍然清晰。

## 图片丰富度

图片不是装饰，而是让 slides 更容易被理解和记住。优先复用输入文档中的真实图片、截图、架构图和素材；当现有素材不足、且图片会明显改善表达时，再生成或补充 raster image。

适合补图的场景：

- **Hero art**：封面或章节页需要一个强记忆点。
- **Editorial illustration**：抽象观点需要具象隐喻。
- **Explainer image**：机制、流程、对比关系需要一眼看懂。
- **Scene/product image**：业务场景、工具使用、用户动作需要代入感。
- **Poster/thumbnail/share visual**：用户需要对外分享、发群或做预告。

图片工作流：

- 只生成 slides 实际会使用的图片；不要为了“更丰富”批量生成无处安放的图。
- 写图片提示词时包含：主题、画面主体、构图、风格、色彩、比例、避免事项、是否禁止文字。
- 不要在生成图片里放事实性结论、小字号文字、代码或复杂表格；这些内容应该由 HTML 文本承载。
- 检查生成结果，必要时迭代 1-2 次；只保留可用终稿。
- 把最终图片复制到 `share-slides/assets/` 或同等输出目录，用相对路径引用。
- 在最终回复中记录使用了哪些新增图片、保存路径，以及关键提示词摘要，方便用户复用或追溯。

## Viewport 约束

每个 HTML deck 必须内置这些规则：

- `html, body` 高度 100%，纵向 scroll snap；禁止横向溢出。
- `.slide { width: 100vw; height: 100vh; height: 100dvh; overflow: hidden; scroll-snap-align: start; }`
- `.slide-content` 使用 flex 布局居中，`max-height: 100%`，`overflow: hidden`。
- 字体和间距全部用 `clamp(min, preferred, max)`；不要固定 px/rem。
- 图片 `max-width: 100%`，`max-height: min(50vh, 400px)`，`object-fit: contain`。
- 必须包含 `max-height: 700px / 600px / 500px` 的高度断点，短屏时减少 padding、字号并隐藏装饰元素。
- 不要写 `-clamp()`、`-min()`、`-max()`；负值必须用 `calc(-1 * clamp(...))`。

## 生成后间距审查

生成 HTML 后必须做一次视觉间距检查。很多 slides 的问题不是内容错，而是元素贴得太近、分页圆点撞到卡片、标题与正文距离失衡，导致页面显得廉价或拥挤。

逐页检查这些关系：

- **框与文本**：卡片、代码框、callout、图片容器内边距要足够，文本不能贴边；普通卡片至少保留 `clamp(1rem, 2vw, 2rem)` 级别 padding。
- **文本与文本**：标题、副标题、正文、列表项之间要有清楚层级；标题和正文不要黏连，列表项不要像段落墙。
- **组件与组件**：多张卡片、左右分栏、时间线节点之间保持一致 gap；相邻元素不要因为 `absolute`、负 margin 或装饰图层互相压迫。
- **分页圆点/页码与内容**：导航点、页码、进度条不能贴住卡片、图片或正文；必要时加安全区，或把导航放到固定角落并降低存在感。
- **装饰与主体**：光效、纹理、几何形状、生成图片不能遮挡文字，也不能和标题形成视觉切线。
- **边界留白**：内容不能贴近视口边缘；短屏断点下仍要保留最小呼吸感，而不是只追求塞下。

如果发现间距异常，不要只调单个元素。先判断是内容过多、层级不清、布局容器过窄，还是装饰/导航占位错误；优先通过拆页、统一 spacing token、调整 grid/flex gap、移动导航安全区来修复。

## HTML 输出约定

默认生成结构。不要覆盖原输入内容，也不要把 slides 的 `index.html` 写到已有博客目录根部，除非用户明确要求：

```text
<input-dir-or-workspace>/share-slides/
  index.html
  assets/        # 仅当需要复制或整理图片时创建
```

`index.html` 至少包含：

- `slides` 数据或语义化 `<section class="slide">`。
- 中文排版增强规则，例如 `.heti` 主内容容器、中文字体栈、标题断行控制和可选轻量 `autoSpacing()`。
- 左右方向键、空格、PageUp/PageDown 翻页，`Home`/`End` 跳转。
- 触控/滚轮导航、进度条或导航点。
- 当前页码和总页数。
- `@media print`，方便浏览器导出 PDF。
- `IntersectionObserver` 给当前页添加 `.visible`，触发 `.reveal` 动效。
- `prefers-reduced-motion` 支持。
- 图片失败时不让整页布局崩掉。
- speaker notes 使用每页内的 `<aside class="notes">` 或等价数据字段保存，默认不显示在投屏视图。

如果用户明确要求浏览器内编辑，再加入编辑模式；否则不要生成编辑相关按钮、localStorage、导出逻辑。

## 交付与分享

验证时优先用浏览器或静态文件服务打开 HTML；如果没有自动化浏览器环境，至少检查文件存在、关键资源路径存在，并在最终回复说明未做真实浏览器预览。

交付后告诉用户：文件位置、slide 数量、选用视觉方向、键盘导航方式、如何用浏览器打印为 PDF。只有用户明确要求部署或导出 PDF 时，再执行分享/导出流程。

PDF 导出注意：动画不会保留；图片必须是相对路径；`.slide` 类名不能变；大 deck 生成的 PDF 可能较大，必要时降低导出分辨率。

## 示例决策

用户说：“基于 `blogs/xxx/index.md` 做个分享。”

应该先读 `index.md` 和附近图片，然后输出一个可打开的 HTML slides。不要直接套 Markdown 渲染器；先把长文章压缩成演讲主线，再把重点内容分配到页面。若文章里已有 Mermaid、Excalidraw 或图片，优先复用；如果图太复杂，先用占位说明并询问是否需要重绘。

## 常见错误

- **逐段搬运 Markdown**：文章适合阅读，slides 适合讲述，必须重排叙事。
- **页面信息过密**：一页塞多个段落、完整代码或大表格，会导致投屏不可读。
- **忽略中文排版**：中英文和数字黏在一起、标点挤压难看、行高过紧、代码破坏行距，会让中文分享显得粗糙。
- **忽略中文排版增强**：只靠默认浏览器样式会让中文标题断行、中英文混排和暗色正文显得粗糙。
- **间距没有复检**：框贴文字、标题正文黏连、分页圆点撞内容、装饰压主体，都会让 slides 显得粗糙。
- **忽略资源路径**：图片相对路径、空格文件名、暗色背景透明图都要检查。
- **为了丰富而堆图**：图片必须服务观点、节奏或记忆点；无用插图会稀释重点。
- **只做视觉不做导航**：HTML slides 必须能键盘翻页，并能导出 PDF。
- **默认引入复杂依赖**：除非用户明确要求，否则优先单 HTML，避免让分享交付变成构建工程。
- **混入文档平台操作**：不要在 y-share 内处理在线文档读取、导出、发布、权限或同步；先让对应文档 skill 准备好素材，再专注 slides 创作。
- **引用外部 skill**：本 skill 已包含必要设计和 slides 规则，不要要求用户再安装或读取其它 skill。
